# AI 阅读助手输出渲染与流式输出设计决策

## 背景

AI 阅读助手已经进入 P1：

- 普通对话返回结构化 JSON。
- 文本回答使用 `answer`。
- 推荐新书使用 `recommendedBooks` 结构化数组。
- 前端将推荐书渲染为可操作列表，并支持加入本地候选书架。

在这个阶段，需要判断是否引入：

- Markdown 渲染。
- 流式输出。

两者都能提升聊天体验，但也会显著增加边界复杂度，尤其当前助手依赖严格 JSON schema 和结构化字段。

## 决策结论

P1 不引入完整 Markdown 渲染，也不引入流式输出。

当前采用：

- `answer` 作为普通文本展示。
- `white-space: pre-wrap` 保留换行和编号。
- `recommendedBooks` 等结构化字段用专门 UI 组件渲染。

P2 已小步落地 Markdown-lite。

流式输出放到更后阶段，至少晚于微信读书搜索确认、推荐卡片体验稳定之后。

## 为什么暂不做完整 Markdown 渲染

### 1. 当前输出已经结构化

阅读助手不是开放式文档生成器，而是阅读场景助手。

目前核心输出形态：

```json
{
  "answer": "文本回答",
  "recommendedBooks": [],
  "suggestions": [],
  "basisNotice": "依据说明"
}
```

推荐书、后续追问、依据说明都已经有专门字段，直接 Markdown 渲染会和结构化 UI 重叠。

### 2. 完整 Markdown 会扩大安全边界

完整 Markdown 通常涉及：

- HTML 过滤。
- 外链处理。
- 图片渲染。
- 表格布局。
- 代码块。
- 自定义 heading。

这些在右侧小面板中容易造成排版失控，也需要额外 XSS 防护。

### 3. 右侧面板不适合长文档排版

AI 阅读助手面板是随手追问工具，不是文章编辑器。

如果允许大标题、复杂表格和长代码块，会让面板从“对话工具”变成“文档阅读器”，与当前产品定位不一致。

## 当前 P1 渲染策略

### 文本回答

继续使用纯文本：

```tsx
<p>{message.content}</p>
```

配合 CSS：

```css
white-space: pre-wrap;
```

这样可以支持：

- 多行回答。
- 编号问题。
- 简单列表。

### 结构化内容

结构化内容不用 Markdown 表达，而是使用专门组件。

例如：

- `recommendedBooks` -> 推荐书列表。
- `usedContext` -> 依据标签。
- `suggestions` -> 回答后的快捷追问按钮。

这种方式更可控，也更适合后续加入操作按钮。

### 快捷追问

`suggestions` 是模型返回的下一步追问建议。前端在对应助手回答气泡内展示最多 3 条，点击后填入输入框，由用户再次确认发送。

约束：

- 不自动发送。
- 点击后聚焦输入框，光标放到文本末尾，方便用户二次编辑。
- 不写入正式 AI 资产。
- 不从历史消息反推追问建议。
- 空值和重复项在前端丢弃。

## Markdown-lite 方案

P2 只做 Markdown-lite，而不是完整 Markdown。

### 允许语法

只允许：

- 有序列表。
- 无序列表。
- 加粗。
- 行内代码。
- 普通段落。

### 禁止语法

禁止：

- HTML。
- 图片。
- iframe。
- 表格。
- 外链自动预览。
- 标题级排版。
- 代码块高亮。

### 实现建议

实现采用本地轻量解析器和 React 文本节点渲染，不使用 `dangerouslySetInnerHTML`，不引入完整 Markdown 依赖。

渲染边界：

- 输入文本最多解析 6000 字符，超出后追加省略提示。
- 最多渲染 80 个块。
- 单个列表最多渲染 120 个条目。
- 单段内联格式节点最多解析 240 个。
- 链接不解析、不渲染。
- 图片不解析、不渲染。
- HTML 只作为普通文本展示。

### 触发条件

满足以下条件后，进入 Markdown-lite：

- 用户频繁要求“整理清单/复盘框架/分层总结”。
- 纯文本列表可读性明显不足。
- 面板内答案长度稳定超过 8-10 行。

## 为什么暂不做流式输出

### 1. 当前后端依赖完整 JSON

阅读助手请求使用 JSON schema。

完整响应类似：

```json
{
  "answer": "...",
  "recommendedBooks": [...],
  "suggestions": [...],
  "basisNotice": "..."
}
```

流式输出时，JSON 在完成前不可安全解析。

这会导致：

- `recommendedBooks` 无法边生成边渲染。
- `suggestions` 无法提前可靠显示。
- `basisNotice` 可能到最后才出现。
- 中途失败会留下半截 JSON。

### 2. 流式会引入新的状态机

需要新增：

- Tauri 事件流。
- 请求取消。
- partial message 状态。
- 失败恢复。
- 历史落库时机。
- UI 光标/加载状态。

这些都超出 P1 的最小闭环。

### 3. 结构化推荐比逐字输出更重要

P1 的主要价值是：

```text
推荐新书 -> 用户确认 -> 加入候选书架
```

流式只改善等待体验，不改善核心闭环。

因此优先级低于：

- 搜索确认微信读书可用。
- 推荐卡片持久化。
- 候选书架联动增强。

## 后续流式方案草案

如果未来要做流式，建议分两种模式。

### 模式 A：文本流式，结构化最终返回

流程：

```text
stream answer 文本
  -> 完成后返回完整 JSON
  -> 渲染 recommendedBooks/suggestions/basisNotice
```

优点：

- 用户能先看到文字。
- 最终仍能拿到结构化字段。

缺点：

- 需要 provider 支持或双请求。
- answer 可能与最终 JSON 不一致。

### 模式 B：非结构化对话才流式

只有普通问答走流式。

以下场景不流式：

- 新书推荐。
- 候选书架决策。
- 需要结构化字段的输出。

优点：

- 实现更稳。
- 不破坏结构化推荐。

缺点：

- 用户体验不一致。

### 推荐路线

如果后续实现流式，优先模式 B。

原因：

- P1/P2 的重点是结构化推荐和候选操作。
- 普通解释型回答才最需要流式体验。
- 可以减少 JSON 半截解析问题。

## 产品分期

### P1

不做 Markdown。

不做流式。

继续使用：

- 普通文本。
- 结构化推荐列表。
- 快捷追问按钮。

### P2

已落地 Markdown-lite。

范围：

- 仅助手正常回答启用。
- 用户消息和错误消息保持纯文本。
- 推荐书仍通过结构化组件展示。
- 不解析 HTML、图片、链接、表格、标题和代码块。

### P3

可评估流式输出。

前提：

- AI 调用耗时明显影响体验。
- 已明确哪些场景需要结构化完整输出。
- 已设计取消、失败、历史落库策略。

## 验收标准

### 当前 P1

- 编号问题可通过换行正确显示。
- 推荐书通过结构化组件展示，而不是 Markdown 文本。
- answer 中即使出现简单列表，也不会破坏面板布局。
- 不出现 HTML、图片或外链自动渲染。

### Markdown-lite 进入标准

- 助手回答支持普通段落、简单列表、加粗和行内代码。
- HTML、链接和图片只作为文本显示。
- 不使用 `dangerouslySetInnerHTML`。
- 推荐书、依据和快捷追问仍使用结构化 UI。

### 流式进入标准

- 已明确流式只覆盖普通文本问答，或已解决结构化 JSON 流式解析。
- 有取消请求设计。
- 有失败回滚设计。
- 有历史落库时机设计。

## 结论

AI 阅读助手当前应坚持“结构化优先、文本克制”的输出策略。

P1 不做完整 Markdown 和流式输出。这样可以保持实现简单、面板稳定，并优先完成阅读助手最关键的闭环：基于本地阅读理解用户，并把推荐结果沉淀为可操作的本地候选。
