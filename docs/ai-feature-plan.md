# AI Feature Plan

## 目标

AI 功能当前阶段服务于个人阅读复盘、阅读规划和候选取舍。已覆盖单本书笔记总结、阅读统计复盘、AI 阅读指南 / 跨书路线图和选书决策；不做通用聊天、不做全书内容问答、不做自动推荐，也不在后台自动上传用户数据。

当前阶段的用户价值是：用户可以主动生成基于单本笔记的结构化复盘，也可以基于阅读统计、当前书和本地候选生成可执行的阅读建议、跨书路线和选书取舍。

AI 输出不能停留在“看起来聪明的分析”。每个 AI 能力必须至少满足以下一类价值：

- 行动：给出用户下一步能执行的阅读、复盘、整理动作。
- 决策：帮助用户在书、笔记、复盘对象或阅读顺序之间做取舍。
- 交付物：生成可保存、可导出、可分享或可继续写作的复盘、索引、报告、卡片或行动清单。

统计、主题标签、画像和趋势只能作为依据；如果一个 AI 功能只增加解释文本、不能触发行动或形成交付物，默认不进入开发范围。

“AI 阅读指南 / 跨书路线图”不进入自动化范围。预研结论是：它更适合作为书籍详情页的手动动作，默认基于当前书、已生成复盘和本地阅读状态生成“本书阅读指南”；只有用户显式选择候选书后，才扩展为跨书阅读路线图。不应出现在发现页主流程，也不应在进入详情页时自动生成。

“选书决策助手”只解决候选书取舍，不做泛推荐。它应该回答“下一本为什么现在读它、为什么暂缓另外几本、下一步做什么”，而不是生成更长书单。入口归属“候选书架”，在候选书架内用三步弹窗确认输入；生成成功后进入内部“选书决策”结果页，不出现在全局侧边栏。发现页只负责发现、搜索、相似探索和保存候选，不承载完整决策结果。输入只包含用户已经保存到本机的候选书、用户手动选择的参考因子、可选当前书/最近本地统计信号和已生成摘要的结构化结论，不读取全量书架、不同步远端笔记、不发送原始笔记正文。

## 当前范围

### 功能入口

- 入口放在单本笔记页 `BookNotesPage` 的操作区。
- 用户点击 `AI 总结` 后进入独立页面 `BookAiSummaryPage`。
- AI 总结不会自动生成；用户必须点击 `读取缓存/生成` 才会读取缓存或调用 AI。
- 已生成结果优先读取本地持久化缓存。
- 用户可以点击 `重新生成`，显式重新调用 AI 并覆盖同 key 缓存。
- 复盘中心的“书籍复盘”页展示已生成复盘和建议生成复盘的候选书；候选只基于本地笔记统计，不自动读取或发送笔记正文。
- 书籍复盘页承接复盘导出能力：导出只使用已生成复盘缓存，支持后续扩展搜索、筛选和批量选择；导出不会自动生成缺失复盘。

### 输入数据

只允许发送当前书籍相关数据：

- 书名、作者、阅读进度等基础元数据。
- 当前书的划线内容。
- 当前书的想法/点评；如果微信读书返回了依附原文，则一并发送该想法的 `abstractText`、章节和位置，用于区分“原文摘录”和“我的想法”。
- 章节标题和笔记分组。
- 书签数量可以作为统计信息发送，但不能伪造成书签正文。

不发送：

- 全量书架。
- 其他书籍笔记。
- 微信读书 API Key 或 AI API Key。
- 本地数据库路径。
- 原始 WeRead 响应中与总结无关的字段。

当前单本复盘输入结构补充：

- 划线条目使用 `type=highlight`、`chapterTitle`、`text`、`createdAt`。
- 想法条目使用 `type=thought`、`chapterTitle`、`text`、`abstractText`、`range`、`star`、`createdAt`。
- `abstractText` 只来自微信读书返回的依附原文，不从全文或其他接口推断；缺失时保持为空。

统计页 AI 复盘只允许发送当前周期结构化统计：

- `mode` 和 `baseTime`。
- 阅读天数、总阅读时长、自然日均、环比。
- 趋势分桶。
- 最长阅读内容的标题、作者、类型、阅读时长和标签。
- 分类偏好的标题、父分类、时长、数量或权重。

统计页 AI 复盘不发送：

- 笔记正文。
- 书籍全文。
- 全量书架。
- 原始 WeRead API 响应 `raw`。

### 输出结构

AI 总结使用稳定结构，方便 UI 展示和后续导出：

- `overview`: 3-5 句核心概览。
- `keyIdeas`: 3-8 条关键观点。
- `myFocus`: 用户笔记反映出的关注点。
- `actionItems`: 可执行行动。
- `themeTags`: 5-10 个短主题标签。
- `representativeQuotes`: 3-6 条代表性摘录，每条包含 `quote`、`reason`、`chapter`、`noteType`。
- `reflectionQuestions`: 3-6 个复盘问题。
- `sourceStats`: 使用了多少条划线、想法和章节。
- `generatedAt`: 生成时间。
- `promptVersion`: 提示词版本。
- `basisNotice`: 固定说明，标注“基于本地笔记生成，不代表整本书全文内容”。

`actionItems` 和 `reflectionQuestions` 是单本复盘从“阅读报告”进入“行动产出”的关键字段。当前已支持行动项四态反馈、复盘问题轻量回答状态和清单复制，状态保存在前端本地存储中，不扩展成复杂任务管理系统。

`representativeQuotes.quote` 必须来自提供的划线或想法原文，可以截短但不可改写。
当代表性摘录来自带依附原文的想法时，应优先引用 `abstractText` 或明确标注其来自用户想法，避免把用户解读误当成书中原文。

## 技术边界

### 调用边界

AI 调用必须放在 Tauri/Rust trusted layer。前端只调用本地命令，不直接保存或读取 AI Key。

当前命令：

- `get_ai_settings_state`
- `validate_ai_credential`
- `save_ai_credential`
- `save_ai_settings`
- `test_ai_connection`
- `remove_ai_credential`
- `get_ai_cached_output`
- `summarize_book_notes`
- `export_book_notes_summary_markdown`
- `export_book_notes_summaries_markdown`
- `list_book_notes_summaries`
- `summarize_reading_stats`
- `summarize_reading_route`
- `export_reading_route_markdown`
- `summarize_book_decision`
- `get_latest_book_decision`

### 配置

第一阶段支持 OpenAI-compatible HTTP 接口配置：

- `base_url`
- `api_key`
- `model`

`api_key` 存入本机安全存储。`base_url` 和 `model` 可以作为普通设置保存。前端可以输入新 Key，但不能读取或显示已保存 Key 明文。

设置页支持 AI 连通性测试。未输入新 Key 时，测试会复用本机已保存 Key；保存 Provider 设置也不要求必须同时修改 Key。

### 缓存与持久化

AI 输出持久化到 SQLite 表 `ai_outputs`：

```sql
CREATE TABLE IF NOT EXISTS ai_outputs (
  feature TEXT NOT NULL,
  scope_id TEXT NOT NULL,
  prompt_version TEXT NOT NULL,
  input_hash TEXT NOT NULL,
  output_json TEXT NOT NULL,
  source_count INTEGER,
  provider_model TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  PRIMARY KEY(feature, scope_id, prompt_version, input_hash)
);
```

缓存 key 规则：

- `feature`: `book-notes-summary`
- `scope_id`: `bookId`
- `prompt_version`: 当前固定为 `book-notes-summary-v3`
- `input_hash`: 当前书笔记输入 JSON 的稳定 hash

当笔记内容变化时，`input_hash` 改变，旧总结不会被误用。用户重新生成时覆盖同 key 输出，保持实现简单。

当前阶段不新增复杂分析表，不把主题、摘录、问题拆成多表；结构化结果统一保存到 `ai_outputs.output_json`。只有当后续需要跨书搜索、聚合统计或引用级筛选时，再评估独立表或向量索引。

AI 复盘反馈持久化到 SQLite 表 `ai_feedback_records`，用于保存用户对行动项和复盘问题的本地执行状态：

```sql
CREATE TABLE IF NOT EXISTS ai_feedback_records (
  feature TEXT NOT NULL,
  scope_id TEXT NOT NULL,
  input_hash TEXT NOT NULL,
  item_kind TEXT NOT NULL,
  item_id TEXT NOT NULL,
  status TEXT NOT NULL,
  note TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  PRIMARY KEY(feature, scope_id, input_hash, item_kind, item_id)
);
```

反馈记录边界：

- `feature` 使用页面资产语义，首版为 `book-review`，预留 `reading-route`，不复用内部缓存名 `book-notes-summary`。
- `scope_id + input_hash` 绑定到某一版 AI 资产；笔记变化导致新 `input_hash` 时，不把旧反馈静默套到新复盘。
- `item_kind` 区分 `actionItem` 和 `reflectionQuestion`，但前端仍展示为同一套轻量反馈能力，避免拆出两个任务系统。
- 反馈记录是用户本地资产，不默认发送给 AI；清理 AI 输出缓存只删除 `ai_outputs`，不删除反馈记录。
- 旧 `localStorage` 的 `aiReviewFeedback / aiActionItems / aiReflectionQuestions` 只作为迁移兼容读取；迁移成功后写入以本地数据库为准，暂不删除旧数据。

阅读统计复盘缓存：

- `feature`: `reading-stats-review`
- `scope_id`: `mode:baseTime`
- `prompt_version`: 当前固定为 `reading-stats-review-v1`
- `input_hash`: 当前周期结构化统计输入 JSON 的稳定 hash

## Prompt 策略

提示词必须版本化，放在 Rust 侧，不从前端拼接核心规则。

当前版本 `book-notes-summary-v3` 固定要求：

- 只基于提供的笔记回答。
- 不补写未提供的书籍内容。
- 如果笔记数量太少，应明确说明总结依据有限。
- 输出 JSON，字段稳定。
- 使用简体中文。
- 引用来源时只引用书名、章节、笔记类型，不输出内部 ID。
- 代表性摘录必须可回溯到用户提供的划线或想法。

当前版本 `reading-stats-review-v1` 固定要求：

- 只基于结构化统计回答。
- 不编造未提供的阅读记录、书籍内容或评分。
- 输出 `overview`、`rhythmInsights`、`preferenceInsights`、`focusItems`、`nextActions`。
- 使用简体中文。
- 明确基于统计生成，不包含笔记正文。

## UI 策略

单本笔记页只保留 `AI 总结` 入口，详细内容在独立 `BookAiSummaryPage` 展示：

- 未配置 AI Key：显示设置入口。
- 未生成：显示缓存/生成按钮和隐私提示。
- 生成中：显示进度状态。
- 已生成：展示概览、主题标签、关键观点、关注点、行动项、代表性摘录、复盘问题、来源统计和缓存元信息，并支持分段复制和完整复盘复制。
- 生成失败：如果存在同 prompt version 的旧缓存，则返回旧缓存并显示错误。
- 复制成功或失败使用全局 toast，不挤占阅读内容区。
- 待生成：展示数据边界提示，明确没有本地缓存时只有点击“生成复盘”才会读取并发送当前书笔记。

独立页面比弹窗更适合承载长文本、摘录、复盘问题和后续导出能力；弹窗只适合短反馈，不适合阅读型分析内容。

书籍复盘导出归属“书籍复盘”页：

- 单本复盘页继续保留当前复盘的 Markdown 导出。
- 书籍复盘页提供独立导出入口，只导出已生成复盘缓存，并生成复盘导出索引。
- 笔记页批量导出可以兼容附带已有复盘缓存，但不承载复盘专属筛选、搜索和批量选择。
- 批量生成复盘不是导出功能；如果后续要做，必须作为显式生成队列单独设计。

统计页内增加 `AI 阅读复盘` 卡片：

- 不作为一级导航。
- 不自动生成。
- 点击后优先读取本地缓存；没有缓存或重新生成时才调用 AI。
- 只展示节奏洞察、偏好洞察、关注项和下一步行动，避免泛化成聊天入口。

AI 阅读指南 / 跨书路线图的 UI 边界：

- 不放在发现页首屏，避免让发现页从搜索/推荐工具变成 AI 规划页。
- 适合放在书籍详情页的“本书管理区”，作为 `本书阅读指南` 手动按钮。
- 详情页只承载入口和短状态，不直接铺开长指南内容。
- 不选择候选书时，结果页必须表达为“本书阅读指南”，展示当前书的阅读、复盘和沉淀步骤。
- 选择候选书后，结果页才表达为“跨书路线图”，展示多本书的先后顺序、依赖和复盘节点。
- 生成仍必须由用户显式点击触发，不做后台自动调用。
- 后续管理入口归属 `复盘 > 阅读指南`，但首版作为“按书聚合的阅读资产库”，不做纯历史列表，也不把所有历史版本平铺出来。
- 阅读指南库主页面使用轻量书籍资产卡片：书名、当前进度、本地追踪状态、已有指南/路线/复盘数量和是否建议刷新；具体指南、路线、复盘内容放入书籍资产详情页。
- 书籍资产详情页使用页面内标签承载 `阅读指南 / 跨书路线 / 书籍复盘`；完整内容在页面内展示，版本、来源边界和历史记录后续折叠到详情内，不使用弹窗承载长内容。
- 弹窗或抽屉只用于短确认、少量候选选择、发送长文本确认和归档确认，不用于展示完整阅读指南、跨书路线或历史版本。

阅读追踪与章节数据边界：

- 阅读追踪第一版采用“进度阶段驱动”，不做实时章节任务系统。追踪的价值是判断当前最该阅读、整理、刷新指南还是生成复盘，而不是精确派发每一章任务。
- 固定阶段为 `0-15% 起步`、`15-40% 建立主线`、`40-70% 深入推进`、`70-95% 收束整理`、`100% 完成归档`；第一版不提供用户自定义阈值。
- 进度跨阶段、新增笔记、长时间停滞和阅读完成只触发“建议刷新 / 建议复盘 / 建议归档”的可见提示，不自动调用 AI，不自动同步远端笔记。
- 章节数据只作为增强依据，用于章节分组、章节回跳、引用来源、最近笔记集中章节和已缓存目录中的当前章节提示。
- 不承诺实时追踪用户读到的具体章节，不按完整章节目录逐章派任务，不做跨设备即时章节同步。
- 缺章节或目录未缓存时，必须回退到阅读进度百分比、最近笔记、本地状态和已有复盘摘要；页面和导出不得表现为生成失败。

选书决策助手的 UI 边界：

- 全局侧边栏只保留 `书架 > 候选书架`，不增加“选书决策”常驻菜单。
- 候选书架承接候选池管理和 `推荐下一本` 入口；点击后在候选书架内打开三步输入弹窗，确认后才生成。
- 内部 `选书决策` 页只承载结果消费态：标题区提供状态、`导出 Markdown`、`重新生成` 和返回候选书架；不再保留默认引导页或输入摘要卡。
- “调整输入范围”向导必须单步单职责：第 1 步只选本次选书目标，第 2 步只选候选书，第 3 步只确认参考信号和最终摘要；不使用左右双列配置面板。
- 发现页只在本地候选轻量区域提供“去候选书架决策”跳转，不展开完整决策结果。
- 没有本地候选书时不展示生成入口，只提示先保存候选。
- 点击后展示输入弹窗和隐私说明，明确“只读取本地候选，不同步远端笔记”。
- 未配置 AI Key 时展示设置入口；已配置时用户在输入弹窗点击“生成决策”才调用 AI。
- 输出最多 3 本下一本候选，每本必须包含“为什么现在读 / 取舍理由 / 预计投入 / 前置动作 / 复盘触发点”。
- 输入范围包含用户显式选择的 `本次选书目标`，首版固定为 `轻松读 / 延续当前主题 / 推进长期书 / 只有 30 分钟 / 读完能复盘`，目标只影响取舍侧重点和行动表述，不扩大数据来源；目标选择不和候选书列表挤在同一步。
- 参考因子必须由用户显式勾选：近期阅读书籍、历史已读完成书、阅读习惯和已生成复盘摘要都不能默认隐式扩大输入。
- 候选书选择步骤使用紧凑单列 checklist，默认不显示封面和每行详情按钮；详情查看属于候选书架或书籍详情页职责，不在输入向导内承载。
- 输出必须包含“暂缓项”和“下一步动作”，避免只给推荐理由。
- 结果页首屏优先展示结构化消费视图：主推荐、取舍对比、行动清单和依据说明；长依据默认折叠，不直接把 AI 返回字段顺序贴到页面。
- 结果页或卡片不做复杂年度计划，不新增任务系统，不写回微信读书。

### AI 阅读指南 / 跨书路线图预研结论

定位：

- 默认用于回答“围绕当前书，接下来怎么读完、怎么复盘、沉淀什么输出”。
- 只有用户显式选择候选书后，才用于回答“下一步读哪些候选书、按什么顺序读、每一步复盘什么”。
- 不替代发现页搜索、相似探索和本地候选书管理。
- 不做自动推荐流、不做后台预生成、不在详情页直接铺开长路线内容。
- 首版只做单本书出发的阅读指南和可选跨书路线图，不做全局年度阅读计划。

输入边界：

- 当前书基础元数据：`bookId`、书名、作者、分类、阅读进度、阅读状态和本地整理状态。
- 当前书计算阅读阶段：由阅读进度百分比映射为 `起步`、`建立主线`、`深入推进`、`收束整理` 或 `完成归档`，用于约束下一步行动的粒度。
- 当前书章节线索：只使用已缓存目录、笔记章节标题、`chapterUid`、range 和最近笔记所在章节作为辅助依据；缺失时必须降级到进度、最近笔记和本地状态。
- 用户显式选择的候选书：来自发现页候选、相似探索结果或书架中用户选择的书；首版建议限制为 3-8 本。
- 已生成的单本 AI 复盘摘要：只使用 `overview`、`keyIdeas`、`myFocus`、`actionItems`、`themeTags` 和 `sourceStats`，默认不再次发送代表性摘录原文。
- 已生成的统计 AI 复盘或本地统计标签：只使用节奏、偏好、分类和重点内容的结构化结论。
- 本地阅读状态：候选、待复盘、已整理等本地状态；仅用于排序和提醒，不写回微信读书。

不允许发送：

- 全量书架。
- 未经用户选择的其他书籍。
- 其他书的原始划线、想法或章节笔记。
- 当前书的原始笔记正文；如果没有单本复盘缓存，阅读指南首版只基于元数据和候选书信息生成，不隐式触发单本复盘。
- 微信读书 API Key、AI API Key、本地数据库路径和原始 API 响应。
- 实时阅读位置、逐章完成状态或任何需要后台监听微信读书客户端才能得到的数据。

输出结构：

- `routeOverview`: 总览；无候选时说明本书阅读指南解决什么阅读/复盘问题，有候选时说明跨书路线图解决什么阅读顺序问题。
- `books`: 推进步骤列表；无候选时至少包含当前书节点，有候选时表达路线中的书籍顺序。每项包含 `bookId`、`title`、`author`、`order`、`role`、`readingPurpose`、`estimatedEffort`、`localStatus` 和 `basis`。`basis` 优先说明进度阶段、最近笔记、已生成复盘和可用章节线索，不把章节作为强制任务。
- `dependencies`: 跨书依赖关系，每项包含前置书、后续书和依赖原因；无候选书或没有依赖时返回空数组。
- `reviewCheckpoints`: 建议复盘点，每项包含触发时机、复盘问题和建议输出物。
- `nextActions`: 用户下一步可以执行的动作，例如先读哪本、先生成哪本复盘、先补充哪本候选。动作必须匹配当前阅读阶段，并包含时间、范围或完成标准中的至少两项。
- `sourceStats`: 使用的当前书、候选书数量、已生成复盘数量、统计信号数量和本地状态数量。
- `generatedAt`: 生成时间。
- `promptVersion`: 固定提示词版本。
- `basisNotice`: 固定说明，标注“基于本地缓存、已生成复盘和用户选择的候选书生成，不代表微信读书远端计划，也不会写回微信读书”。

提示词收紧规则：

- 当前阅读指南提示词小版本收紧为 `reading-route-v2.1`，不升级为 prompt v3，不拆成全新的单书和跨书两套提示词。
- 单书指南必须按阅读阶段生成建议：起步阶段强调阅读目的和验证问题，中段强调主线判断和笔记沉淀，后段强调收束整理和复盘产物，完成阶段强调生成复盘和归档。
- 章节线索只能作为 `basis` 的辅助证据；不得生成逐章任务清单，不得承诺实时章节追踪，不得输出“每天自动读第 X 章”这类自动安排。
- 缺少章节目录、章节标题或当前章节时，必须继续基于进度、最近笔记、本地状态和已有复盘生成建议，不允许以章节缺失为理由拒绝生成。
- 跨书路线仍以当前书为主书归档。候选书只在用户显式选择后进入路线，不因为阶段变化自动扩展输入范围。

缓存与导出：

- `feature`: `reading-route`
- `scope_id`: `book:{bookId}`；如果包含候选书，则使用 `book:{bookId}:candidates:{candidate_hash_12}`。
- `prompt_version`: 当前阅读指南为 `reading-route-v2.1`，在 `reading-route-v2` 基础上收紧为“进度阶段驱动、章节辅助、缺章节可回退”的单书阅读处方和跨书路线。
- `input_hash`: 对规范化后的路线输入 JSON 做稳定 hash；候选书变化、复盘缓存变化或本地状态变化都会得到新缓存。
- 导出 Markdown 必须包含指南或路线正文、候选书范围、`promptVersion`、`sourceStats`、生成时间和 `basisNotice`。
- 单书阅读指南导出必须和页面详情区保持一致，使用“推进任务 / 复盘点 / 下一步行动 / 来源统计”结构，复盘点保留“输出 / 验收”，下一步行动保留“完成标准”。

AI 资产库第一版规划：

- 长期目标是把 AI 输出从一次性页面结果升级为“按书管理的阅读资产”，但第一版只做可落地的聚合入口，不实现完整版本系统。
- `复盘` 侧边栏下增加 `阅读指南` 二级入口，和 `书籍复盘 / 阅读报告` 并列；该入口展示阅读指南库，不进入发现页或候选书架。
- 阅读指南库按书聚合展示，不按 AI 输出版本平铺。每张卡片只展示书名、作者、当前阅读进度、本地追踪状态、当前是否有本书指南、跨书路线数量、是否已有书籍复盘和最近更新时间。
- 点击书籍资产卡进入书籍资产详情，详情页使用 `阅读指南 / 跨书路线 / 书籍复盘` 标签。第一版只展示当前有效内容和跳转入口，历史版本不在主列表展示。
- 旧的 `reading-route` 缓存、跨书路线缓存和已生成书籍复盘缓存先映射为“当前版本”。第一版不修改 `ai_outputs` 表结构，新增聚合服务层从 `ai_outputs`、书架缓存和本地阅读状态组装视图模型。
- 页面不得直接耦合 `ai_outputs.output_json` 原始结构。第一版增加薄领域层，向页面提供 `AIAssetSummary`、`AIAssetDetail` 和 `AssetVersionRef` 这类视图模型。
- 本地追踪状态后续应从 `候选 / 在读 / 待复盘 / 已整理` 收敛为 `候选 / 在读 / 待整理 / 已整理`，避免 `待复盘` 和 `书籍复盘` 页面语义冲突。第一版不强制迁移历史状态，只在新增文案中优先使用“待整理”。
- 微信读书正在读但尚未进入本地追踪的书，后续应提示 `加入追踪 / 忽略`。`加入追踪` 不等于加入候选池，也不自动进入选书决策候选。

AI 资产库第二阶段规划：

- 第二阶段目标不是继续增加新的 AI 入口，而是把“本书阅读指南 / 跨书路线 / 书籍复盘”从单次生成结果推进为“按书管理、可回看、可更新、可触发行动的本地阅读资产”。
- 第二阶段继续保持“按书聚合”而不是“按时间平铺”。历史版本只能在书籍上下文中查看，不做全局 AI 生成历史时间线，避免形成输出垃圾堆。
- 页面层级固定为三层：`阅读指南资产库 -> 书籍资产详情 -> 单资产版本详情`。不再把所有元信息、当前结果、历史版本和完整长内容塞在同一层页面。
- 第一层“阅读指南资产库”中的书籍卡是导航入口，整卡可点击；职责是帮助用户判断“哪本书有资产、是否建议更新、值不值得进入查看”。
- 第二层“书籍资产详情”中的 `阅读指南 / 跨书路线 / 书籍复盘` 标签继续保留，但标签内默认只展示“当前有效版本摘要卡 + 明确动作按钮”，不把摘要卡本身设计成跳转型整卡按钮。
- 第三层“单资产版本详情”承载完整长内容、来源统计、行动清单、导出、版本变化说明和后续历史版本入口；只有明确点击 `查看完整内容` 或历史版本条目时才进入。
- 第二层的“当前有效版本摘要卡”职责是说明状态，不是继续导航。摘要卡内只展示标题、生成时间、prompt version、scope、provider model、刷新原因和阶段信息；交互通过明确按钮承载，例如 `查看完整内容 / 导出 Markdown / 重新生成 / 查看历史版本`。
- 列表卡负责进入，摘要卡负责说明当前状态，长内容页负责阅读与处理。这条规则适用于阅读指南、跨书路线和书籍复盘，避免同一类卡片在不同页面承担相互冲突的职责。
- 第二阶段的历史版本只放在“书籍资产详情”的当前标签内，以默认折叠的轻量列表承载；每行只展示生成时间、当时阶段、当时进度、触发原因、prompt version 和状态，不直接平铺完整内容。
- 历史版本条目可点击进入“单资产版本详情”；但资产库首页和书籍资产详情默认态都不直接展开所有历史正文。
- 第二阶段的更新动作必须是手动触发。用户点击重新生成前，页面应展示本次刷新会参考的依据：当前阅读阶段、距上一版的新增笔记/划线数量、停滞情况、已完成行动数和上一版摘要；同时明确不会自动读取未缓存的远端内容。
- 新版本生成时不只读取当前输入，还应优先参考上一版结构化摘要、当前阅读阶段、本地状态变化、已完成行动和新增笔记量。这样更新才能体现“继续保留什么 / 新增什么 / 为什么变化”，而不是每次重写一篇无上下文的新结果。
- 第二阶段继续不做复杂任务系统。行动反馈收敛为 `2.1` 最小闭环：只支持 `待处理 / 已完成 / 暂不做 / 不适合` 四态，并允许可选极短补充说明；不引入长表单、提醒系统、日历或任务排期。
- 数据结构上继续保留 `ai_outputs` 作为原始缓存，不在第二阶段一开始重做底层表结构；优先在 `ai-assets` 聚合层上补充“当前版本、历史版本列表、版本详情和行动状态”的读取模型。
- 版本详情页可以按资产类型分别展示结构化内容：阅读指南展示推进任务、复盘点、下一步行动和验收标准；跨书路线展示阅读顺序、切换理由、依赖和行动；书籍复盘展示关键观点、行动项、复盘问题和摘录。
- 第二阶段仍不做：全局 AI 历史中心、后台自动刷新、未缓存数据的静默远端读取、复杂任务系统、prompt v3 重写或跨书路线自动接续。

第二阶段信息架构与交互规则：

- `阅读指南资产库`：
  - 对象是“书”。
  - 书籍卡整卡可点击。
  - 只显示书名、作者、进度、阅读阶段、本地状态、资产数量、刷新建议和最近更新时间。
  - 不显示具体 prompt、scope、完整版本列表或长摘要。
- `书籍资产详情`：
  - 对象是“这本书当前有效的 AI 资产”。
  - 使用 `阅读指南 / 跨书路线 / 书籍复盘` 标签。
  - 每个标签统一顺序为：当前状态条 -> 当前有效版本摘要卡 -> 动作区 -> 历史版本折叠区。
  - 当前有效版本摘要卡不整卡可点击；通过按钮进入后续操作。
- `单资产版本详情`：
  - 对象是“某一个具体版本”。
  - 承载完整内容、变化说明、来源统计、导出、重新生成前对比和行动状态。
  - 如果存在上一版，可展示“相较上一版的变化”，但第二阶段先做只读差异摘要，不做复杂 diff UI。

第二阶段建议实现顺序：

1. 先补 `阅读阶段计算 / 刷新建议原因 / 章节缺失回退`，让资产库和详情页的“为什么建议更新”有稳定依据。
2. 再补“单资产版本详情页”，把当前有效摘要卡的去向明确下来。
3. 再补“历史版本折叠区”，只做按书、按资产类型查看，不做全局历史中心。
 4. 再补行动状态持久化和轻量继承，让新版本不至于把用户已完成动作全部重置。
 5. 最后评估版本变化说明和与选书决策的联动，不提前扩散到更多页面。
 
第二阶段 2.1 最小闭环：

- 行动反馈和复盘问题反馈只做本地持久化，不新增远端命令和复杂任务系统。
- 阅读指南和书籍复盘的行动项都使用同一套状态：`待处理 / 已完成 / 暂不做 / 不适合`。
- 单本书籍复盘反馈统一存储为一份本地状态，内部按 `actionItems` 和 `reflectionQuestions` 区分；复盘问题显示 `待思考 / 已回答 / 暂不答 / 不适合`，但不再拆成独立存储 key。
- 默认只保存状态变更和可选反馈记录；反馈记录保存在本机，不默认发送给 AI。
- 资产版本详情页顶部展示当前版本行动反馈摘要，并展示重新生成前的核对边界说明。
- 第二阶段其余内容仍按本节既有规划执行，不扩展到完整版本时间线或 prompt v3。

### 选书决策助手首版设计

定位：

- 用于回答“候选书里下一本读哪本，为什么现在读它”。
- 服务于已经沉淀到本机的候选书架，不替代搜索、推荐、相似探索和阅读路线图。
- 阅读路线图解决“当前书和候选书按什么顺序读”；选书决策助手解决“候选书之间如何取舍，下一本先读谁”。
- 不做全局年度阅读计划，不做无限推荐，不做评分排行榜。

输入边界：

- 用户保存到本机的候选书：`bookId`、书名、作者、分类、本地状态、保存备注；首版最多纳入 8 本。
- 本次选书目标：用户显式选择的固定枚举值，用于约束“为什么现在读”和下一步行动的侧重点。
- 可用的结构化统计信号：只使用最近本地统计的分类、最长阅读内容和已生成统计复盘行动建议。
- 可用的已生成单本复盘摘要：只使用 `overview`、`keyIdeas`、`myFocus`、`actionItems`、`themeTags` 和 `sourceStats`。
- 当前书可选；如果从发现页直接生成，则当前书为空，决策只围绕候选池。

不允许发送：

- 全量书架。
- 未保存为本地候选的推荐书或搜索结果。
- 任意书籍的原始划线、想法、章节笔记或书签正文。
- 微信读书 API Key、AI API Key、本地数据库路径和原始 API 响应。
- 未缓存数据的远端同步请求；生成决策不能触发微信读书接口。

输出结构：

- `decisionOverview`: 一段取舍总览，说明本次决策依据和局限。
- `topCandidates`: 最多 3 本下一本候选，每项包含 `bookId`、`title`、`author`、`rank`、`whyNow`、`tradeoff`、`estimatedEffort`、`prerequisiteAction`、`reviewTrigger` 和 `basis`。
- `deferredCandidates`: 暂缓项，每项包含 `bookId`、`title` 和 `reason`。
- `nextActions`: 2-5 条可执行动作，例如先打开哪本详情、先读多少分钟、读完后产出什么。
- `sourceStats`: 使用的候选书数量、已生成复盘数量、统计信号数量和本地状态数量。
- `generatedAt`: 生成时间。
- `promptVersion`: 固定提示词版本。
- `basisNotice`: 固定说明，标注“基于本地候选、已生成复盘和结构化统计信号生成，不代表微信读书远端推荐，也不会写回微信读书”。

缓存：

- `feature`: `book-decision`
- `scope_id`: `candidates:{candidate_hash_12}`，其中 hash 包含候选书 ID 和本次选书目标，避免不同目标复用同一份决策缓存。
- `prompt_version`: 首版固定为 `book-decision-v1`
- `input_hash`: 对规范化后的候选池、本次选书目标和本地结构化信号做稳定 hash；候选书变化、目标变化、复盘缓存变化或统计信号变化都会得到新缓存。

首版实现顺序：

1. [x] 后端增加选书决策输入构建、prompt、输出规范化和缓存读取。
2. [x] 前端首版曾放在发现页本地候选区；后续调整为 `书架 > 候选书架` 只承载入口，内部 `选书决策` 任务页承载完整生成流程和参考因子选择。
3. [x] E2E 覆盖入口不自动调用 AI、点击生成后展示推荐下一本、取舍理由和下一步动作。
4. [x] 验证 prompt 输入不包含全量书架、原始笔记正文、API Key、数据库路径或原始接口响应。

## 安全与隐私

- 默认不自动发送任何笔记内容。
- 每次 AI 调用必须由用户动作触发。
- 前端不接触已保存 AI Key 明文。
- 日志不得打印 prompt 全文、API Key、原始响应全文。
- 清除本地缓存时应包含 `ai_outputs`。
- 设置页提供 AI 输出缓存独立清理入口，只删除 `ai_outputs`，不删除 API Key、微信读书缓存、本地阅读状态或 AI 反馈记录。
- 移除 AI Key 时不删除历史 AI 输出；只有全量清除本地缓存或独立清理 AI 输出缓存才删除历史输出。
