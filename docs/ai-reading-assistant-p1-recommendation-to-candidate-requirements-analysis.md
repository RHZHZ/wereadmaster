# AI 阅读助手 P1：推荐新书加入候选书架需求分析

## 背景

P0 阶段已经完成 AI 阅读助手的核心对话能力：

- 基于阅读统计、阅读画像、当前书、复盘摘要和阅读记忆回答问题。
- 支持“推荐 3 本可加入候选书架的新书”。
- 明确区分“AI 推荐新书”和“候选书架内选书决策”。
- 候选书架在 P0 中只作为排除清单，不作为推荐池。
- AI 不会声称已加入候选书架，也不会静默写入本地状态。

P1 的自然下一步是打通“推荐结果 -> 用户确认 -> 加入候选书架”的闭环。

这不是让 AI 自动替用户管理书架，而是让 AI 给出建议后，由用户确认其中一本或多本，再写入本地候选书架。

## 问题定义

当前 P0 推荐链路存在断点：

```text
用户：推荐 3 本可加入候选书架的新书
  -> AI 给出书名、作者、理由、风险
  -> 用户需要自己去搜索或手动保存候选
```

问题：

- 推荐结果是纯文本，不能直接操作。
- 用户需要复制书名再去发现页搜索，链路偏长。
- 系统无法记录“这本书来自 AI 推荐”的来源信息。
- 重复加入、标题相似、微信读书不可用等边界没有统一处理。

P1 要解决的是“用户确认后加入候选”的最小闭环，而不是完整的智能书目管理系统。

## 目标

### 产品目标

- 用户可以从 AI 推荐的新书中选择一本或多本加入候选书架。
- 加入前必须由用户明确确认。
- 加入后候选书架、仪表盘候选区、阅读路线等已有功能可以复用这些候选书。
- 推荐来源和 AI 理由可以保留为候选书备注，方便后续回看。

### 技术目标

- 复用现有 `reading_item_states` 本地候选书架能力。
- 尽量不新增数据库表。
- 推荐结果先结构化，前端再渲染为可操作卡片。
- 写入候选书架走现有阅读状态 upsert 能力或薄封装命令。
- 保持本地优先，不写回微信读书。

## 非目标

P1 不做：

- 不自动把所有 AI 推荐写入候选书架。
- 不承诺推荐书一定存在于微信读书。
- 不做远端微信读书书架写回。
- 不做复杂向量知识库。
- 不做跨平台书目数据库。
- 不做自动购买、下载或打开外部商城。
- 不把候选书架变成推荐池。

## 用户故事

### 用户故事 1：从推荐结果加入候选

作为用户，我希望 AI 推荐新书后，可以直接点击“加入候选”，这样我不用复制书名再去搜索。

验收：

- AI 推荐结果中每本书都有明确的加入入口。
- 点击后需要用户确认。
- 确认后本地候选书架出现该书。

### 用户故事 2：避免重复加入

作为用户，我希望系统能识别已经在候选书架或已读书架里的书，避免重复添加。

验收：

- 已在候选书架的书不再显示“加入候选”，或显示“已在候选”。
- 已读/已在书架的书显示提醒，避免误以为是新书。

### 用户故事 3：保留推荐理由

作为用户，我希望加入候选后还能看到 AI 当时推荐它的理由，方便之后取舍。

验收：

- 候选书备注中保留 AI 推荐理由摘要。
- 备注不写入敏感原始上下文。

### 用户故事 4：处理无法确认的书

作为用户，我希望如果系统无法确认书籍是否存在，也能先保存为本地候选，后续再搜索核对。

验收：

- P1 可以保存“标题 + 作者”的本地候选。
- UI 明确提示“未确认微信读书可用”。

## P1 范围

### P1 必做

1. 推荐结果结构化
   - AI 新书推荐结果需要提供结构化书目数组。
   - 每项至少包含：
     - `title`
     - `author`
     - `reason`
     - `fit`
     - `risk`

2. 推荐卡片展示
   - 前端将结构化推荐渲染为卡片。
   - 卡片包含书名、作者、推荐理由、适合点、风险。
   - 卡片提供“加入候选”动作。

3. 用户确认
   - 点击“加入候选”后显示确认。
   - 确认文案必须说明：
     - 只加入本地候选书架。
     - 不代表已加入微信读书。
     - 未确认微信读书一定可用。

4. 本地写入
   - 写入 `reading_item_states`：
     - `itemType = "candidate"`
     - `status = "toRead"`
     - `title = 推荐书名`
     - `author = 推荐作者`
     - `note = AI 推荐理由摘要`
   - `itemId` 需要稳定生成，避免同书重复。

5. 去重与状态
   - 加入前检查本地候选书架。
   - 标题 + 作者命中时视为已存在。
   - 已存在时不重复写入。

### P1 可选

- 加入后展示“查看候选书架”入口。
- 支持一次加入多本。
- 保存推荐来源标签，例如 `来自 AI 推荐`。
- 在候选书架中展示 AI 推荐理由来源。

## P1 不纳入

- 自动搜索微信读书并绑定真实 `bookId`。
- 多来源书目合并。
- 作者名模糊匹配。
- ISBN 级别去重。
- 复杂推荐解释评分。
- 用户长期不感兴趣反馈模型。

这些可以进入 P2 或单独设计。

## 业务规则

### 1. 必须用户确认

AI 推荐不能直接写入候选书架。

正确流程：

```text
AI 推荐书籍
  -> 用户点击加入候选
  -> 确认弹窗
  -> 写入本地候选
```

错误流程：

```text
AI 推荐书籍
  -> 自动写入候选书架
```

### 2. 候选书架仍然不是推荐池

P1 只是在推荐结果之后写入用户确认的书。

推荐生成时：

- 候选书架继续作为排除清单。
- 不从候选书架里抽书假装“新书推荐”。

候选决策时：

- 只对已经确认加入候选书架的书排序和取舍。

### 3. 本地候选优先

P1 写入的是本地候选书架，不写微信读书远端。

这意味着：

- 不需要微信读书写权限。
- 不承诺远端书架同步。
- 后续如果接搜索，可以再把本地候选与微信读书搜索结果关联。

### 4. 推荐理由可保存，但要压缩

候选备注可以保存 AI 推荐理由，但不能保存完整对话上下文。

建议备注格式：

```text
来自 AI 阅读助手推荐：{简短理由}
适合点：{fit}
风险：{risk}
```

备注长度需要限制，避免候选列表展示膨胀。

## 信息架构

### AI 回复结构

P1 后，新书推荐回复建议包含：

```json
{
  "answer": "基于你的阅读画像，我建议先看这 3 本。",
  "recommendedBooks": [
    {
      "title": "书名",
      "author": "作者",
      "reason": "推荐理由",
      "fit": "为什么适合你",
      "risk": "可能的取舍或风险"
    }
  ],
  "suggestions": [],
  "basisNotice": "基于本地阅读统计、阅读画像和阅读记忆生成，未确认微信读书可用。"
}
```

P1 可以保持 `answer` 字段兼容，同时新增 `recommendedBooks`。

### 前端展示

```text
AI 回复：
  基于你的阅读画像，我建议先看这 3 本。

推荐卡片 1
  书名 / 作者
  推荐理由
  适合点
  风险
  [加入候选]

推荐卡片 2
...
```

### 加入确认

确认弹窗：

```text
加入本地候选书架？

《xxx》会保存到本地候选书架，用于后续选书决策和阅读路线。
这不会写入微信读书，也不代表已确认微信读书可用。

[取消] [加入候选]
```

## 数据需求

### 推荐书结构

前端类型建议：

```ts
type ReadingAssistantRecommendedBook = {
  title: string;
  author?: string;
  reason: string;
  fit: string;
  risk: string;
  candidateState?: "available" | "adding" | "added" | "exists" | "failed";
};
```

### 写入候选状态

复用现有阅读状态输入：

```ts
type ReadingItemStateInput = {
  itemId: string;
  itemType: "candidate";
  status: "toRead";
  title: string;
  author?: string;
  note?: string;
};
```

### itemId 生成

P1 可以使用稳定本地 ID：

```text
ai-rec:{normalizedTitle}:{normalizedAuthor}
```

要求：

- 去除首尾空格。
- 标题和作者归一化。
- 限制长度。
- 避免暴露完整 prompt 或用户隐私。

## 交互流程

### 新书推荐到候选

```text
用户点击“推荐 3 本可加入候选书架的新书”
  -> ask_reading_assistant
  -> AI 返回 answer + recommendedBooks
  -> 前端渲染推荐卡片
  -> 用户点击“加入候选”
  -> 检查是否已存在
  -> 用户确认
  -> upsert reading_item_states
  -> 卡片状态变为“已加入”
```

### 已存在候选

```text
用户点击加入候选
  -> 本地匹配到同标题同作者
  -> 不重复写入
  -> 卡片显示“已在候选”
```

### 写入失败

```text
用户确认加入
  -> 写入失败
  -> 卡片显示失败状态
  -> 提供重试
```

## 权限与隐私

- 只写入本地数据库。
- 不写入微信读书远端。
- 不上传候选写入结果给模型。
- 备注中不保存完整 prompt、原始笔记或隐私路径。
- 若以后接入微信读书搜索，搜索动作需要单独说明数据流。

## 验收标准

### 功能验收

- 新书推荐结果能展示为可操作卡片。
- 用户确认后能加入本地候选书架。
- 已存在候选不会重复加入。
- 加入后候选书架页面能看到该书。
- 候选备注包含简短 AI 推荐理由。

### 边界验收

- AI 推荐没有结构化书目时，仍展示普通文本回答。
- 推荐书缺少作者时，允许保存标题候选，但提示信息要稳妥。
- 写入失败时不影响当前对话。
- 取消确认后不写入。

### 回归验收

- P0 对话能力不受影响。
- 候选书架内选书决策仍只使用本地已确认候选。
- `npx tsc --noEmit --pretty false` 通过。
- 相关前端测试通过。
- Rust 阅读状态或 AI 阅读助手相关测试通过。
- `npm run build` 通过。

## 风险分析

### 风险 1：AI 推荐书名不真实

缓解：

- UI 明确标注“未确认微信读书可用”。
- P1 只保存为本地候选，不声明可读。
- P2 再接搜索确认。

### 风险 2：重复书籍识别不准

缓解：

- P1 只做标题 + 作者的保守去重。
- 命中不确定时允许用户手动处理。
- 不做复杂模糊匹配。

### 风险 3：候选书架质量下降

缓解：

- 必须用户确认。
- 卡片展示风险/取舍，不只展示推荐理由。
- 加入后仍可在候选书架删除。

### 风险 4：AI 输出结构不稳定

缓解：

- 后端 schema 增加 `recommendedBooks` 可选字段。
- 解析失败时降级为普通文本。
- 前端不依赖纯文本解析。

## 分阶段建议

### P1a：结构化推荐展示

- 后端输出支持 `recommendedBooks`。
- 前端展示推荐卡片。
- 不提供写入按钮或按钮先 disabled。

### P1b：确认加入本地候选

- 接入本地 reading state upsert。
- 加入确认弹窗。
- 卡片状态更新。

### P1c：候选书架联动

- 加入后提示查看候选书架。
- 候选书架展示 AI 推荐备注。
- 仪表盘候选区同步出现。

## 结论

P1 可以进入，但应优先做“AI 推荐新书 -> 用户确认 -> 加入本地候选书架”的最小闭环。

本阶段不做自动搜索、不做远端写回、不做向量知识库。这样既能显著提升 AI 推荐的可操作性，又能保持本地优先和用户确认边界。
