# 第三阶段规划

## 背景

第一版已经把阅读指南、跨书路线、书籍复盘、选书决策和导出边界收口到可用状态；第二版和 2.1 行动反馈闭环也已经完成。第三阶段不再追求新增入口，而是解决一个更核心的问题：让现有 AI 资产真正“连续更新”，而不是每次重新生成一份彼此割裂的结果。

## 阶段目标

第三阶段只收四件事：

1. 跨书路线真正接续到上一版资产链。
2. 更新时能明确说明“基于什么变更”。
3. 书籍复盘并入同一套版本体系，但保留独立资产语义。
4. 正式发布接入 GitHub Releases 一键更新，形成可验证的桌面端发布闭环。

目标不是做更多页面，也不是把历史结果铺成时间线。

## 当前状态

- AI 资产库已经按书聚合阅读指南、跨书路线和书籍复盘。
- 版本详情页已经能展示 `previousVersion`、更新依据和更新前确认弹窗。
- 书籍复盘已经复用版本详情、历史版本和反馈状态，不再是悬而未决的独立体系。
- GitHub Releases 一键更新链路已有独立文档和基础实现，详见 `docs/github-release-updates.md`。

## 剩余问题

- 跨书路线已有同 scope 历史链，但候选书集合变化导致 scope 变化时，仍缺少显式父子关系。
- 更新依据已经能展示阶段、进度、刷新原因、结构化约束和行动反馈摘要，但“上一版保留什么、修改什么、为什么改”还可以继续强化。
- 历史版本列表必须保证 `previousVersion` 指向真正更早的上一版，而不是倒序列表中的更新版本。
- 发布链路仍需要首次正式发布前的 Secrets、签名产物和 release 验证。

## 规划边界

### 要做

- 为跨书路线补真正的父子版本关系。
- 更新输入时纳入上一版结构化摘要、当前阅读阶段、本地状态变化和已完成行动。
- 让版本详情页更清楚地表达“差异来源”和“更新依据”。
- 让书籍复盘与阅读指南共享版本链、历史和反馈口径，但不合并入口。
- 让桌面端正式发布走 GitHub Releases，并支持应用内检查更新和安装更新。
- 让更新包、签名和版本描述文件在同一条发布链路里生成和分发。

### 不做

- 不新增全局 AI 历史中心。
- 不把历史版本平铺到主列表。
- 不做后台自动刷新。
- 不做未缓存远端内容的静默读取。
- 不把第三阶段扩展成新的推荐系统或知识库问答。
- 不把一键更新开放给开发调试构建。

## 优先级

1. 跨书路线版本链。
2. 更新依据与变化说明。
3. 书籍复盘是否并入版本体系。

## 预期产出

- 资产链路更完整，跨书路线不再是孤立结果。
- 用户重新生成时能看见更强的上下文连续性。
- 后续如果要做版本归档、差异对比或导出策略统一，会有稳定基础。

## 实施拆分

### 3.1 跨书路线版本链

目标：让跨书路线和上一版资产形成可追踪的父子关系。

当前状态：

- 同一 `scopeId` 下的阅读指南、跨书路线和书籍复盘已经能读取历史版本。
- 版本详情会读取并展示上一版引用。
- 历史版本列表已修正为指向更早的上一版，避免把更新版本误标为“上一版”。

- 新增或补齐跨书路线的父版本引用字段。
- 在版本详情中展示上一版与当前版的接续关系。
- 生成新跨书路线时优先沿用上一版可复用上下文。

验收标准：

- 用户能从当前跨书路线看回上一版。
- 重新生成后，路线不会像一份全新结果一样断链。

### 3.2 更新依据与变化说明

目标：让每次更新都能回答“为什么变了”。

当前状态：

- 版本详情页已展示更新依据。
- 更新确认弹窗已集中展示本次重新生成前的核对项、行动反馈摘要和与上一版相比的结构化变化。
- 反馈作为阅读成果的独立设计已收口到 `docs/feedback-outcome-review-plan.md`，后续在本阶段按“成果回顾”而非“用户评价”落地。

- 更新前展示本次会参考的输入变化。
- 更新后展示相对上一版的变化摘要。
- 输入中固定纳入上一版结构化摘要、当前阅读阶段、本地状态变化和已完成行动。

验收标准：

- 用户能明确看到这次更新的依据。
- 用户能区分“内容变化”和“上下文延续”。

### 3.3 书籍复盘并入结论

目标：确认书籍复盘已采用与阅读指南一致的版本体系。

当前状态：

- 复盘版本已复用同一套版本详情、历史和反馈表达。
- 复盘仍保留独立资产类型，不额外新增入口层级。
- 导出、反馈和更新上下文沿用统一口径。

验收标准：

- 书籍复盘与阅读指南的版本阅读方式一致。
- 不再保留“是否并入”的悬而未决状态。

## 现阶段判断

第三阶段已经从“评估是否并入”推进到“已并入但不扩入口”。当前重点剩下两类：跨 scope 的版本链父子关系，以及首次正式发布前的一键更新验证。

## 实施顺序

1. 先把同 scope 历史版本链保持正确，避免上一版引用方向出错。
2. 再评估跨 scope 的跨书路线父子关系是否需要新增持久化字段。
3. 继续强化更新依据和变化摘要，但不把它扩展成全局历史中心。
4. 同步完成 GitHub Releases 首次正式发布前的签名、Secrets 和 release 验证。

这个顺序的原则很简单：先解决结构连续性，再解决表达连续性，最后才决定是否统一复盘体系。
