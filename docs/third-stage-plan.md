# 第三阶段规划

## 背景

第一版已经把阅读指南、跨书路线、书籍复盘、选书决策和导出边界收口到可用状态；第二版和 2.1 行动反馈闭环也已经完成。第三阶段不再追求新增入口，而是解决一个更核心的问题：让现有 AI 资产真正“连续更新”，而不是每次重新生成一份彼此割裂的结果。

## 阶段目标

第三阶段只收三件事：

1. 跨书路线真正接续到上一版资产链。
2. 更新时能明确说明“基于什么变更”。
3. 书籍复盘并入同一套版本体系，但保留独立资产语义。
4. 正式发布接入 GitHub Releases 一键更新，形成可验证的桌面端发布闭环。

目标不是做更多页面，也不是把历史结果铺成时间线。

## 当前问题

- 跨书路线目前仍更像“单次生成结果”，缺少清晰的 `parentAssetId / parentRouteId` 链接。
- 版本更新已经有手动触发和刷新原因，但对“上一版保留什么、修改什么、为什么改”还不够强。
- 书籍复盘和阅读指南已经共用一部分反馈状态，但版本语义还没有彻底统一。

## 规划边界

### 要做

- 为跨书路线补真正的父子版本关系。
- 更新输入时纳入上一版结构化摘要、当前阅读阶段、本地状态变化和已完成行动。
- 让版本详情页更清楚地表达“差异来源”和“更新依据”。
- 让书籍复盘与阅读指南共享版本链、历史和反馈口径，但不合并入口。
- 让桌面端正式发布走 GitHub Releases，并支持应用内检查更新和安装更新。
- 让更新包、签名和版本描述文件在同一条发布链路里生成和分发。

### 不做

- 不新增全局 AI 历史中心。
- 不把历史版本平铺到主列表。
- 不做后台自动刷新。
- 不做未缓存远端内容的静默读取。
- 不把第三阶段扩展成新的推荐系统或知识库问答。
- 不把一键更新开放给开发调试构建。

## 优先级

1. 跨书路线版本链。
2. 更新依据与变化说明。
3. 书籍复盘是否并入版本体系。

## 预期产出

- 资产链路更完整，跨书路线不再是孤立结果。
- 用户重新生成时能看见更强的上下文连续性。
- 后续如果要做版本归档、差异对比或导出策略统一，会有稳定基础。

## 实施拆分

### 3.1 跨书路线版本链

目标：让跨书路线和上一版资产形成可追踪的父子关系。

- 新增或补齐跨书路线的父版本引用字段。
- 在版本详情中展示上一版与当前版的接续关系。
- 生成新跨书路线时优先沿用上一版可复用上下文。

验收标准：

- 用户能从当前跨书路线看回上一版。
- 重新生成后，路线不会像一份全新结果一样断链。

### 3.2 更新依据与变化说明

目标：让每次更新都能回答“为什么变了”。

- 更新前展示本次会参考的输入变化。
- 更新后展示相对上一版的变化摘要。
- 输入中固定纳入上一版结构化摘要、当前阅读阶段、本地状态变化和已完成行动。

验收标准：

- 用户能明确看到这次更新的依据。
- 用户能区分“内容变化”和“上下文延续”。

### 3.3 书籍复盘并入结论

目标：确认书籍复盘已采用与阅读指南一致的版本体系。

- 复盘版本复用同一套版本详情、历史和反馈表达。
- 复盘仍保留独立资产类型，不额外新增入口层级。
- 导出、反馈和更新上下文沿用统一口径。

验收标准：

- 书籍复盘与阅读指南的版本阅读方式一致。
- 不再保留“是否并入”的悬而未决状态。

## 现阶段判断

第三阶段已经从“评估是否并入”推进到“已并入但不扩入口”。当前重点只剩版本链与更新依据的收口。

## 实施顺序

1. 先做 3.1 跨书路线版本链，补父子关系和接续提示。
2. 再做 3.2 更新依据与变化说明，确保每次更新都能解释差异来源。
3. 同步补齐 GitHub Releases 发布配置、签名产物和应用内更新入口。
4. 最后确认 3.3 书籍复盘并入结论，避免后续再分叉出另一套版本体系。

这个顺序的原则很简单：先解决结构连续性，再解决表达连续性，最后才决定是否统一复盘体系。
