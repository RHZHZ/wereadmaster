# 本地书库与阅读器边界规划

## 结论

当前阶段已经从“只做本地书库资产管理”调整为“桌面端轻量本地阅读器”。原因是用户明确希望在桌面应用内阅读本地图书，并围绕正文完成划线、想法和后续 AI 提问记录。新的边界不是做完整阅读器大工程，而是先做可验证的 TXT/EPUB 文本阅读闭环。

新的判断很直接：

- 当前产品的核心价值仍是微信读书资产同步、本地整理、复盘、统计、候选决策和导出。
- 桌面端本地阅读器能补上“用户真正拥有的本地图书文件”的阅读和思考闭环。
- 完整阅读器仍会立刻引入精排版、EPUB CFI、PDF 标注、分页引擎、文件版本和跨来源去重等复杂度。
- 微信读书里也可能存在“导入书”，如果本地阅读器直接复用 `bookId` 或笔记表，会造成来源混乱。

本规划的目标是保留未来扩展空间，同时把当前已经落地的轻量阅读能力写清楚：本地来源和微信读书来源隔离，AI 不自动读取整本书，不把本地批注写回微信读书。

## 产品边界

### 近期应该做

- 明确微信书架、候选书架、本地书库的边界。
- 为后续本地书籍导入预留来源命名空间。
- 支持桌面端导入用户本机 TXT/EPUB，并进入轻量正文阅读器。
- 支持本地阅读进度、划线、想法、书内搜索、轻量目录和本地 Markdown 导出。
- 允许未来把本地文件作为阅读资产手动关联到某本微信读书条目。
- 聚合展示时区分来源，不合并原始进度和原始笔记。

### 近期不做

- 不做移动端阅读器；手机上继续建议使用微信读书 App 或系统阅读器。
- 不做 PDF 阅读器、PDF 坐标标注和复杂分页排版。
- 不做完整 EPUB 精排版、EPUB CFI 锚点或跨设备同步。
- 不做自动全文问答。
- 不把微信读书导入书和本地文件自动合并。
- 不自动去重或覆盖任何微信读书笔记。

### 必须坚持

- 只管理用户合法持有、无 DRM 的本地图书文件。
- 不破解 DRM，不导入微信读书正文，不代理或缓存第三方受版权保护阅读页。
- AI 只处理用户主动选择的片段、笔记或结构化上下文；配置 AI 后也只处理用户主动选区和问题，未配置时保存草稿态记录，不调用模型。
- 本地文件路径、API Key、数据库路径不进入 AI 输入。

## 当前已落地能力

桌面端本地阅读器第一阶段已经形成轻量闭环：

- 本地书库入口独立于微信书架和候选书架。
- 本地书库卡片和阅读器标题会标识“本地版本”，并提示与微信书架隔离，降低同名书误操作风险。
- 本地书库会基于已同步微信书架电子书做只读疑似匹配，展示“可能有微信版本”；微信书籍详情页也会基于本地书库做反向提示，二者都不自动关联、不合并笔记、不改变打开行为。
- 本地图书 API 返回 `source: "local"`，导出的 Markdown 也保留 `source_kind: local`；前端已有 `SourceItemKey` 工具区分 `local` 和 `weread` 来源，后续统一资产页不依赖裸 `bookId` 判断来源。
- TXT/EPUB 本地图书可以进入正文阅读器；EPUB 当前走保守文本抽取，不承诺精排版。
- 本地书库当前支持选择文件或粘贴路径导入，不展示未实现的拖拽导入承诺；`Ctrl + K` 可聚焦书库搜索；重复导入同一文件时保留现有记录并提示“已在本地书库”。
- 阅读器支持滚动阅读、阅读进度、轻量目录、书内搜索和主题/字号/行距偏好。
- 用户选中文本后，在正文附近浮层执行划线、写想法、问 AI、标记、复制等动作。
- 选区浮层只保留真实且互不重复的动作；黄色划线不再额外展示“重点”等同义按钮。
- 写想法在正文附近浮层完成，保存后自动形成本地划线。
- 点击正文划线后显示浮层；侧边栏划线卡片只展示摘要，点击后进入划线详情弹窗查看完整内容。
- 想法和划线的长文本都采用“摘要卡 + 详情弹窗”，避免撑大侧栏卡片。
- 想法详情弹窗支持复制原文/想法、编辑、定位原文和删除确认；划线详情弹窗支持复制、定位原文、写想法、问 AI、切换划线类型和删除确认；长 URL 或长 token 不撑开卡片。
- 本地 Markdown 导出只导出本地阅读器划线、想法和 AI 提问记录；即使只有 AI 提问记录，也保留空划线和空想法占位，不读取微信读书笔记，不触发新的 AI 请求。
- AI 提问记录已收纳到 AI tab 摘要列表，完整选区、问题、回答和追问通过详情弹窗查看；未配置 AI 时只保存草稿态记录，不展示赞同/不赞同等假反馈按钮。
- 未接入的书库批量操作、排序切换等能力不展示为空按钮；当前只保留可执行操作和静态状态说明。

这些能力仍然属于“轻量文本阅读器”，不代表已经进入完整 EPUB/PDF 阅读器阶段。

## 关键风险：微信导入书与本地导入书

微信读书上的导入书仍然属于微信读书来源：

```text
source = "weread"
sourceId = 微信 bookId
```

本地书库导入的 TXT/EPUB 属于本地来源；未来如果支持 PDF，也必须继续属于本地来源：

```text
source = "local"
sourceId = 本地生成 id
```

即使两边标题、作者、文件内容高度相似，也不要自动视为同一条记录。它们只能被手动关联为同一本阅读资产下的两个来源版本。

## 笔记冲突原则

微信读书笔记和本地笔记必须分来源存储：

```ts
type NoteSource = "weread" | "local";

type NoteRecord = {
  id: string;
  source: NoteSource;
  sourceBookId: string;
  sourceNoteId: string;
  text: string;
  anchor: unknown;
};
```

原因：

- 微信读书笔记使用微信自己的 `bookId`、章节 UID、划线位置和想法结构。
- 本地阅读器未来可能使用文件 hash、EPUB CFI、页码、字符 offset 或 PDF 坐标。
- 两套锚点不能天然互换，强行合并容易导致错章、错页、重复显示或覆盖。

允许做：

- 在统一书籍详情中按来源聚合展示。
- 提示“疑似同一本书”或“疑似重复笔记”。
- 用户手动确认后建立关联关系。

不允许做：

- 自动删除疑似重复笔记。
- 自动把微信划线转换成本地划线。
- 自动把本地批注写入微信读书条目。
- 用标题和作者相同作为强合并依据。

## 推荐信息架构

短期保持：

```text
书架
├─ 微信书架
└─ 候选书架
```

当本地书库进入实施时再增加：

```text
书架
├─ 微信书架
├─ 本地书库
└─ 候选书架
```

统一资产视图后置：

```text
阅读资产
├─ 微信读书来源
├─ 本地文件来源
└─ 用户手动关联关系
```

## 数据模型草案

先设计来源命名空间和轻量阅读字段，不设计完整阅读器：

```ts
type ReadingAssetSource = "weread" | "local";

type SourceItemKey = {
  source: ReadingAssetSource;
  sourceId: string;
};

type ReadingAsset = {
  id: string;
  title: string;
  author?: string;
  category?: string;
  createdAt: number;
  updatedAt: number;
};

type ReadingSourceItem = {
  assetId?: string;
  source: ReadingAssetSource;
  sourceId: string;
  title: string;
  author?: string;
  cover?: string;
  fileHash?: string;
  format?: "epub" | "txt" | "pdf";
  linkedBy: "system" | "user";
  createdAt: number;
  updatedAt: number;
};
```

说明：

- `ReadingAsset` 表示“这本书”的概念层。
- `ReadingSourceItem` 表示它来自哪里。
- 微信书架条目可以先不落入 `ReadingAsset`，等用户手动关联或统一资产页上线后再生成。
- 本地导入文件必须使用 `fileHash` 去重，但不以 `fileHash` 覆盖微信来源。

## 本地书库 MVP

第一版已从“只做资产管理”扩展为“资产管理 + 轻量阅读器”：

- 导入本地图书元数据。
- 计算文件 hash。
- 保存格式、大小、标题、作者、封面候选。
- 支持 TXT/EPUB 正文文本阅读。
- 支持本地阅读进度、划线、想法、AI 提问记录和 Markdown 导出。
- 支持正文附近浮层操作，避免把阅读页变成复杂工作台。
- 支持手动关联到某个微信读书条目，但关联不等于合并数据。
- 支持备注、候选、待复盘等本地状态后续接入。

第一版仍不做：

- 移动端阅读器。
- PDF 阅读、PDF 坐标标注和图片批注。
- EPUB CFI 精确锚点、复杂分页和跨设备同步。
- 本地阅读进度同步到微信读书。
- 本地划线或想法写回微信读书。
- 自动合并微信读书导入书和本地文件。
- 全文 AI 问答或后台自动 AI 调用。

## 重型阅读器触发条件

只有满足以下条件时，才单独立项重型阅读器：

- 轻量 TXT/EPUB 阅读器已有稳定使用数据。
- 用户明确需要精排版、分页、复杂目录、跨章节锚点或 PDF 标注。
- 用户需要本地笔记和微信读书笔记在统一资产页中并排管理。
- 能接受阅读器开发带来的长期维护成本。

立项后再评估：

- EPUB 是否引入成熟渲染库或 CFI 锚点。
- PDF 是否独立为只读预览、批注工具或继续外部打开。
- 本地笔记是否进入现有笔记页，还是保持在本地阅读器侧栏。
- AI 是否仍只处理用户选区和本地笔记摘要。

继续不做：

- PDF 坐标标注。
- 微信读书正文导入。
- 自动同步、自动去重或自动改写用户笔记。

## 与当前书架筛选的关系

当前书架筛选仍只处理微信书架：

```ts
type ShelfEntryType = "book" | "album" | "mp";
```

不要把本地书库硬塞进 `book / album / mp` 类型里。未来如果要做统一视图，应新增来源筛选：

```text
来源：全部 / 微信读书 / 本地导入
类型：电子书 / 有声书 / 文章收藏
分类：文学 / 计算机 / ...
```

默认不把 `来源` 常驻放进微信书架页，避免当前页面变复杂。

## 分期计划

### P0：来源边界（已完成）

- 明确 `source: "weread" | "local"` 作为长期命名空间。
- 文档和后续模型设计都不再用裸 `bookId` 表示跨来源书籍。
- 笔记、进度、AI 输出都保留来源字段。

### P1：本地书库资产管理（已完成第一版）

- 新增本地书库入口。
- 支持导入文件元数据和 hash。
- 支持 TXT/EPUB 本地图书进入轻量阅读器。
- 支持本地阅读进度。

### P2：轻量本地阅读器（已完成第一版）

- 支持正文阅读、目录识别、书内搜索、主题/字号/行距偏好。
- 支持正文附近选区工具条、划线、想法、标记和复制。
- 支持划线和想法详情弹窗处理长文本边界。
- 支持本地 Markdown 导出，导出可包含当前 AI 提问记录，但不触发新的模型请求。
- 支持 AI 提问记录；配置 AI 后只基于用户选区请求模型，未配置时保存草稿态记录，不展示假回答反馈。

下一阶段实施文档见 `docs/local-reader-second-stage-plan.md`。第二阶段重点不是扩展重型阅读器，而是把当前第一版推进为真实 TXT/EPUB 导入、稳定阅读、来源隔离、导出验收和可选选区 AI 的桌面端闭环。

### P3：来源聚合展示

- 在统一资产详情中展示多个来源版本。
- 笔记来源可筛选：全部 / 微信读书 / 本地。
- 疑似重复只提示，不自动合并。

### P4：评估重型阅读器

- 只有在轻量本地阅读器真实使用成立后，再评估重型阅读器。
- EPUB/TXT/PDF 分开评估，不一次性承诺完整阅读体验。

## 工程原则

- KISS：本地阅读器先保持文本阅读和正文附近轻操作，不做复杂工作台。
- YAGNI：不提前实现 PDF、EPUB CFI、跨设备同步和全文问答。
- DRY：复用现有本地状态、候选、复盘和导出能力。
- SOLID：来源条目、统一资产、笔记记录分层建模，避免互相覆盖。
