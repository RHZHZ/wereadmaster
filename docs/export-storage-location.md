# 导出保存位置设置

## 背景

设置页原有“自定义存储位置”实际控制本地 SQLite 数据库目录迁移，但用户容易理解为导出文件保存位置。导出笔记、批量导出、书籍复盘、统计复盘、阅读指南和诊断信息此前都默认写入本地数据目录下的 `exports` 子目录，路径不够直观，也不适合直接保存到文档目录、同步盘或 Obsidian Vault。

## 设计决策

- “本地数据库位置”和“导出保存位置”是两个独立概念。
- 本地数据库位置只负责 SQLite、WAL/SHM、缓存和状态文件；迁移需要确认，成功后需要重启。
- 导出保存位置只负责后续导出的 Markdown、批量导出目录、书籍复盘导出和诊断信息；修改后立即影响新导出，不移动历史导出文件。
- 首版只提供一个统一导出目录，不拆分笔记目录和复盘目录，避免增加不必要的设置复杂度。
- 默认导出目录仍为当前数据目录下的 `exports`，兼容旧行为。

## 实现范围

- 设置状态新增 `exportData`，包含当前导出目录、默认导出目录和是否自定义。
- 后端新增统一导出根目录解析，所有导出服务使用同一个目录来源。
- 设置页新增“导出保存位置”卡片，原“自定义存储位置”改为“本地数据库位置”。
- 用户可以通过系统目录选择器选择导出目录，也可以使用手动输入作为兜底；支持恢复默认目录。
- 当前版本不迁移历史导出文件。

## 约束

仓库新增项目级 Cargo 配置，覆盖全局失效的 USTC crates.io git index，改用当前可访问的 TUNA sparse registry。这样可以稳定拉取 `tauri-plugin-dialog`，并把导出目录设置接入系统目录选择器。

本次构建失败不是单一“镜像不可用”：

- Cargo 依赖索引层：全局 USTC crates.io 配置不可用，项目级配置已切到 TUNA sparse registry。
- 原生归档层：`libsodium-sys-stable` 的构建脚本会额外访问 `download.libsodium.org` 下载 `libsodium` 归档，这不走 crates.io 镜像；若网络不稳定，可在本机临时设置 `SODIUM_DIST_DIR` 指向已有缓存目录。

如果后续换机器构建，需要重新确认本机是否具备 `libsodium` 归档缓存；不建议把 `SODIUM_DIST_DIR` 这类本机绝对路径写进仓库配置。

## 验收标准

- [x] 设置页明确区分“导出保存位置”和“本地数据库位置”。
- [x] 默认导出目录仍保持 `数据目录/exports`。
- [x] 自定义导出目录只影响后续导出，不移动历史文件。
- [x] 单本笔记、批量笔记、书籍复盘、统计复盘、阅读指南和诊断导出都走统一导出目录。
- [x] 前端类型映射和 Tauri 命令权限有测试覆盖。
