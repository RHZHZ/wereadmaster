# AI 阅读助手 P2：推荐书微信读书搜索确认需求分析

## 背景

P1 已完成“AI 推荐新书 -> 用户确认 -> 加入本地候选书架”的闭环。

但 P1 仍然保留一个重要边界：AI 推荐的新书不确认微信读书是否可用。加入候选时，系统会提示“未确认微信读书可用”。

P2 要解决的是：当 AI 推荐一本书后，用户可以在加入候选前或加入后，通过微信读书搜索确认该书是否存在，并把本地候选与真实微信读书搜索结果关联。

## 当前状态

已有能力：

- `searchBooks` 可调用微信读书搜索。
- 发现页已经支持搜索并保存候选。
- 本地候选书架使用 `reading_item_states`。
- P1 推荐卡片已经包含：
  - `title`
  - `author`
  - `reason`
  - `fit`
  - `risk`
  - 加入候选动作。

缺口：

- 推荐卡片不能直接搜索确认。
- 加入本地候选后，`itemId` 是本地生成的 `ai-rec-*`，不一定是真实微信读书 `bookId`。
- 候选书架无法区分“已确认微信读书书籍”和“AI 推荐本地候选”。
- 用户仍然可能加入一本微信读书里搜不到的书。

## 问题定义

当前链路：

```text
AI 推荐《某书》
  -> 用户加入本地候选
  -> 系统提示未确认微信读书可用
  -> 用户需要手动去发现页搜索
```

P2 目标链路：

```text
AI 推荐《某书》
  -> 用户点击“搜索确认”
  -> 展示微信读书搜索结果
  -> 用户选择匹配项
  -> 加入候选书架
```

## 目标

### 产品目标

- 用户可以从推荐卡片直接搜索微信读书。
- 搜索结果能帮助用户确认书名、作者、封面和类别。
- 用户选择确认结果后，候选书架保存真实微信读书 `bookId`。
- 无搜索结果时，仍允许保存为本地候选，但明确标注未确认。

### 技术目标

- 复用现有 `searchBooks` API。
- 复用现有 `upsertReadingItemState` 候选写入。
- 不新增远端写回。
- 尽量不新增数据库表。
- 在 UI 层区分“已确认”和“未确认”的候选来源。

## 非目标

P2 不做：

- 不自动替用户选择搜索结果。
- 不自动写入微信读书远端书架。
- 不做多平台书目搜索。
- 不做 ISBN 级去重。
- 不做复杂模糊匹配模型。
- 不做后台自动补全所有历史 AI 推荐。

## 用户故事

### 用户故事 1：推荐卡片直接搜索

作为用户，我希望 AI 推荐一本书后，可以直接点“搜索确认”，不用复制书名去发现页。

验收：

- 推荐卡片有“搜索确认”入口。
- 点击后显示搜索状态。
- 搜索使用书名和作者作为关键词。

### 用户故事 2：选择微信读书匹配项

作为用户，我希望看到微信读书搜索结果，然后自己选择最匹配的书。

验收：

- 搜索结果显示书名、作者、封面、类别。
- 用户点击某个结果后确认加入候选。
- 候选书架保存真实 `bookId`。

### 用户故事 3：无结果时保留本地候选

作为用户，我希望如果微信读书搜不到，也可以先保存为本地候选。

验收：

- 无结果时显示“未找到可确认书籍”。
- 用户可以选择“仍保存为本地候选”。
- 候选备注保留 AI 推荐理由和未确认状态。

### 用户故事 4：避免重复候选

作为用户，我希望如果搜索结果已经在候选书架或我的书架里，系统不要重复保存。

验收：

- 命中已有候选时显示“已在候选”。
- 命中书架已有书时提示“已在书架”。
- 不重复写入。

## P2 范围

### P2 必做

1. 推荐卡片新增搜索确认入口
   - 按钮：`搜索确认`
   - 状态：`搜索中`、`已确认`、`未找到`、`失败`

2. 搜索关键词生成
   - 优先：`书名 作者`
   - 作者为空：`书名`
   - 去掉多余空格。

3. 搜索结果展示
   - 在推荐卡片下方展示最多 3-5 个搜索结果。
   - 每项展示书名、作者、封面、类别。
   - 用户点击结果后可确认加入候选。

4. 使用真实微信读书 `bookId` 写入候选
   - `itemId = searchResult.bookId`
   - `itemType = "candidate"`
   - `status = "toRead"`
   - `title/author/cover/category` 来自搜索结果
   - `note` 保留 AI 推荐理由摘要。

5. 无结果降级
   - 保留 P1 的“加入本地候选”能力。
   - 明确提示“未确认微信读书可用”。

### P2 可选

- 搜索结果支持“查看更多”，跳转发现页。
- 自动把最相似结果放在第一位，但仍需用户确认。
- 候选书架中展示“已确认微信读书”标识。
- 对已保存 `ai-rec-*` 候选提供“重新搜索确认”入口。

## P2 不纳入

- 自动选择搜索结果。
- 远端书架写入。
- 根据 ISBN 或出版信息去重。
- 搜索结果长期缓存。
- AI 参与搜索结果判定。

## 业务规则

### 1. 搜索确认必须由用户选择

搜索结果即使只有一条，也不能自动写入。

原因：

- 书名可能重复。
- 作者可能译名不同。
- 微信读书搜索结果可能包含同名书、听书或合集。

### 2. 已确认候选优先使用真实 bookId

用户选择微信读书搜索结果后，候选写入应使用真实 `bookId`。

这样后续：

- 可以打开书籍详情。
- 可以参与阅读路线。
- 可以更准确去重。

### 3. 未确认候选仍然允许存在

AI 推荐不一定都能在微信读书找到。

P2 不应阻止用户保存未确认候选，但必须显式标注：

- 未确认微信读书可用。
- 后续可能需要手动搜索或替换。

### 4. 不覆盖用户已有候选备注

如果真实 `bookId` 已经存在本地候选，搜索确认不应覆盖用户原备注。

只更新卡片状态为“已在候选”。

## 信息架构

### 推荐卡片新增区域

```text
推荐书
  书名 / 作者
  推荐理由
  适合你
  取舍

  [搜索确认] [加入本地候选]

  搜索结果：
    结果 1 [确认加入]
    结果 2 [确认加入]
```

### 状态文案

- 未搜索：`搜索确认`
- 搜索中：`搜索中`
- 有结果：`选择匹配项`
- 已确认加入：`已加入候选`
- 无结果：`未找到匹配书籍`
- 失败：`搜索失败，重试`

## 数据需求

### 前端推荐卡片状态

```ts
type RecommendedBookSearchState =
  | "idle"
  | "searching"
  | "found"
  | "notFound"
  | "failed";
```

### 搜索结果缓存

当前会话内即可：

```ts
type RecommendedBookSearchResultState = {
  status: RecommendedBookSearchState;
  results: SearchResult[];
  errorMessage?: string;
};
```

P2 不要求持久化搜索结果。

### 候选备注

确认结果写入候选时：

```text
来自 AI 阅读助手推荐：{reason}
适合点：{fit}
风险：{risk}
已通过微信读书搜索确认。
```

未确认本地候选：

```text
来自 AI 阅读助手推荐：{reason}
适合点：{fit}
风险：{risk}
未确认微信读书可用。
```

## 交互流程

### 搜索并确认加入

```text
用户点击推荐卡片“搜索确认”
  -> searchBooks({ keyword: "书名 作者" })
  -> 展示搜索结果
  -> 用户选择结果
  -> 确认加入候选
  -> upsertReadingItemState(realBookId)
  -> 刷新候选书架
```

### 搜索无结果

```text
用户点击“搜索确认”
  -> 搜索返回空
  -> 显示未找到
  -> 用户可点击“加入本地候选”
  -> upsertReadingItemState(ai-rec-*)
```

### 已存在候选

```text
用户选择搜索结果
  -> 检查 reading_item_states
  -> 已存在 candidate/toRead
  -> 显示已在候选
```

### 已在书架

```text
用户选择搜索结果
  -> 检查 bookshelf entries
  -> 已存在
  -> 提示已在书架
  -> 不写入候选
```

## 错误处理

### 无微信读书凭据

如果搜索需要凭据但当前未登录：

- 搜索按钮显示可用，但点击后提示去设置。
- 不阻止加入本地候选。

### 搜索失败

- 展示错误文案。
- 允许重试。
- 保留“加入本地候选”按钮。

### 搜索结果为空

- 显示空状态。
- 不显示确认加入。
- 保留本地候选降级入口。

## 隐私与安全

- 搜索关键词只包含书名和作者，不包含用户阅读画像。
- AI 推荐理由不发送给微信读书搜索。
- 搜索结果写入本地候选，不写远端书架。
- 不把搜索结果回传给 AI。

## 验收标准

### 功能验收

- 推荐卡片可发起搜索确认。
- 搜索结果能在卡片内展示。
- 用户选择搜索结果后能加入候选。
- 写入候选使用真实 `bookId`。
- 无结果时仍可保存为本地候选。
- 已存在候选不重复写入。

### 回归验收

- P1 的直接加入本地候选仍可用。
- 普通 AI 对话不显示搜索确认。
- 候选书架决策仍只基于已确认加入的候选。
- `npx tsc --noEmit --pretty false` 通过。
- 相关前端测试通过。
- `npm run build` 通过。

## 风险分析

### 风险 1：搜索结果不准确

缓解：

- 必须用户手动选择。
- 结果展示作者、封面、分类。

### 风险 2：UI 变复杂

缓解：

- 默认折叠搜索结果。
- 只展示前 3-5 个。
- 保留“去发现页查看更多”作为可选增强。

### 风险 3：重复候选

缓解：

- 真实 `bookId` 优先去重。
- 本地推荐候选继续用标题 + 作者保守去重。

### 风险 4：搜索依赖网络和凭据

缓解：

- 搜索失败不阻断本地候选保存。
- 保留 P1 降级路径。

## 分阶段建议

### P2a：搜索结果展示

- 推荐卡片新增“搜索确认”。
- 调用 `searchBooks`。
- 展示前 3-5 个搜索结果。
- 不写入候选。

### P2b：选择结果加入候选

- 点击搜索结果后确认加入。
- 使用真实 `bookId` 写入候选。
- 刷新候选书架。

### P2c：候选确认状态增强

- 候选书架展示“已确认微信读书”。
- 对 `ai-rec-*` 候选提供“搜索确认”。

## 结论

P2 应优先解决“AI 推荐书是否能在微信读书找到”的确认链路。

推荐路线是：推荐卡片内搜索、用户选择搜索结果、用真实 `bookId` 加入本地候选。这样可以提升推荐结果的可信度，也为后续阅读路线、书籍详情和候选书架决策提供更稳定的数据基础。
