# AI 阅读助手新书推荐设计文档

## 背景

AI 阅读助手已经支持基于本地阅读画像、阅读统计、AI 资产和候选书架进行对话。但“推荐新书”和“从候选书架里选择下一本”容易混在一起。

当前风险：

- 如果严格限制“只能基于本地上下文回答”，AI 不敢给出具体新书，只会推荐类别或引导用户去使用 AI 选书。
- 如果完全放开推荐，AI 可能编造书籍、推荐用户已读/已候选的书，或让用户误以为系统已经写入候选书架。
- 如果聊天里直接写入候选书架，会过早引入搜索、去重、写入确认和失败处理复杂度。

因此需要把新书推荐设计为一个受控能力：AI 可以基于用户阅读画像和通用书目知识提出新书候选，但不能静默写入候选书架。

## 目标

P0 目标：

- 用户点击“推荐可加入候选书架的新书”后，AI 直接给出具体书名建议。
- 推荐基于本地阅读画像、阅读统计和已有 AI 资产摘要。
- 已读书、已在候选书架的书只作为排除清单，不作为推荐池。
- AI 必须说明推荐依据、不确定性和下一步动作。
- 用户确认前，不写入候选书架。

## 非目标

P0 不做：

- 不在聊天里一键加入候选书架。
- 不调用微信读书远端搜索。
- 不保证推荐书一定存在于微信读书。
- 不把候选书架作为新书推荐来源。
- 不生成完整选书决策资产。
- 不建立向量知识库或长期推荐模型。

## 术语边界

### 新书推荐

新书推荐指 AI 根据用户阅读画像和统计信号，推荐用户可能感兴趣、且不在本地已读/候选范围内的书。

输出应是“可加入候选书架的建议”，不是已经加入候选书架的结果。

### 候选书架决策

候选书架决策只处理用户已经加入候选书架的书。

目标是回答：

- 下一本先读哪本。
- 哪些暂缓。
- 为什么现在读。
- 读前需要确认什么。

### 排除清单

排除清单包含：

- 已读书标题。
- 当前候选书标题。
- 已归档/暂缓但仍在本地记录中的标题。

排除清单只用于避免重复推荐，不用于让 AI 在其中排序。

## 产品流程

```text
用户点击快捷问题
  -> AI 阅读助手识别为新书推荐意图
  -> 构建阅读画像 + 统计 + AI 资产摘要 + 排除清单
  -> AI 输出 3-5 本新书建议
  -> 用户确认方向
  -> 用户前往发现/搜索/候选书架流程加入候选
  -> 后续在候选书架里做选书决策
```

## 交互设计

### 快捷问题

全局助手默认快捷问题建议：

- `推荐 3 本可加入候选书架的新书`
- `总结我的阅读偏好`
- `帮我制定本周阅读计划`

候选书架场景快捷问题建议：

- `从候选书架里先读哪本？`
- `这些候选书怎么取舍？`
- `帮我缩小到 3 本`

### 推荐结果

AI 回答建议结构：

```text
基于你的阅读画像和近期阅读记录，我建议先看这 3 本：

1. 《书名》 作者
   推荐理由：...
   适合你的原因：...
   可能风险：...

2. ...

这些是 AI 基于阅读画像提出的新书候选，不代表已经加入候选书架。下一步可以先搜索其中 1-2 本加入候选，再用候选书架决策功能排序。
```

### 不确定性说明

必须说明：

- 推荐基于阅读画像、统计和已生成 AI 资产。
- 书籍可用性需要用户搜索确认。
- 未经用户确认，不会加入候选书架。

## 上下文设计

### 新书推荐使用的上下文

P0 使用：

- `readingStats`
- `readingPersona`
- `aiAssetSummary`
- `conversationHistory`
- `bookExclusionList`，P0 可先复用受限的候选/书架标题摘要实现

不使用：

- 原始笔记正文。
- 全量书架明细。
- 全量候选书详情作为推荐池。
- API Key、数据库路径、原始 WeRead 响应。

### 候选书架决策使用的上下文

候选书架场景使用：

- `candidateBooks`
- `readingStats`
- `readingPersona`
- `aiAssetSummary`
- `conversationHistory`

这里的 `candidateBooks` 是决策输入，不是新书推荐输入。

## 后端设计

### 意图识别

P0 不新增复杂意图分类器，使用轻量关键词判断即可。

推荐新书意图触发条件：

- 用户消息包含 `推荐新书`
- 用户消息包含 `加入候选`
- 用户消息包含 `推荐` 且包含 `书`
- 用户消息包含 `下一本` 且当前 scope 不是 `candidateShelf`

候选书架决策意图触发条件：

- 当前 scope 是 `candidateShelf`
- 或用户明确说 `从候选书架`
- 或用户明确说 `候选书`

如两者冲突，优先尊重明确的候选书架表达。

### Prompt 约束

`reading_assistant_system_prompt()` 需要补充新书推荐例外：

- 默认仍只基于输入中的本地阅读上下文回答。
- 当用户明确要求推荐新书时，可以结合阅读画像和通用书目知识提出新书候选。
- 新书推荐必须输出具体书名和作者。
- 不得声称推荐书来自用户本地书架。
- 不得声称书籍已经加入候选书架。
- 不得声称已经确认微信读书可用。
- 必须说明推荐只是一组待确认候选。

### Payload 约束

`build_reading_assistant_payload()` 可增加推荐意图字段：

```json
{
  "intent": "newBookRecommendation",
  "recommendationPolicy": {
    "allowGeneralBookKnowledge": true,
    "candidateBooksAsExclusionOnly": true,
    "mustNotPersistCandidates": true,
    "recommendedItemCount": "3-5"
  }
}
```

非推荐新书场景不需要该字段，避免扩大普通对话能力边界。

### 输出格式

P0 不扩展 JSON schema，仍保持：

```json
{
  "answer": "...",
  "suggestions": [],
  "basisNotice": "..."
}
```

原因：

- KISS：前端暂时只渲染文本回答。
- YAGNI：一键加入候选之前，不需要结构化推荐项。
- 兼容已有模型 JSON schema。

P1 如果要做“加入候选”按钮，再扩展：

```json
{
  "recommendedBooks": [
    {
      "title": "...",
      "author": "...",
      "reason": "...",
      "fit": "...",
      "risk": "..."
    }
  ]
}
```

## 前端设计

### P0

修改：

- 全局快捷问题改为 `推荐 3 本可加入候选书架的新书`。
- 候选书架场景保留 `从候选书架里先读哪本？`。
- 回答仍作为普通对话消息展示。

不做：

- 不新增“加入候选”按钮。
- 不新增推荐书卡片。
- 不新增候选写入 API。

### P1

当后端返回 `recommendedBooks` 后，可以升级为：

- 每本书一张轻量推荐卡。
- 卡片提供“搜索这本书”“加入候选”。
- 加入候选前必须二次确认。
- 写入后刷新候选书架。

## 数据与隐私

P0 不新增表。

对话历史可以保存用户问题和 AI 回答，但不保存额外的外部搜索结果。

排除清单应只包含最小必要字段：

- title
- author，可选
- status，可选

不发送：

- 原始笔记正文。
- 本地数据库路径。
- 微信读书原始响应。
- API Key。

## 验收标准

### 新书推荐

- 点击全局快捷问题后，AI 应输出 3-5 本具体新书建议。
- 回答不得只输出“人文科技”“哲学小说”等类别。
- 回答不得说“请使用 AI 选书功能”作为主要结果。
- 回答必须说明这些书需要用户确认后再加入候选书架。
- 回答必须说明依据来自阅读画像/统计/AI 资产，而不是本地已有候选书。

### 候选书架决策

- 在候选书架 scope 下，AI 应只围绕已有候选书排序和取舍。
- 不推荐候选书架之外的新书。
- 不把候选书架决策说成新书推荐。

### 安全边界

- AI 不声称已写入候选书架。
- AI 不声称已写回微信读书。
- AI 不声称已确认远端书库可用。

## 实施步骤

### 阶段 1：文案和 Prompt

- 修改全局快捷问题。
- 修改 `reading_assistant_system_prompt()`。
- 在 payload 中增加轻量 `intent` 和 `recommendationPolicy`。
- 保持现有 JSON schema 不变。

### 阶段 2：排除清单

- 构建最小书名排除清单。
- 候选书和已读书仅作为去重来源。
- 控制数量，避免发送全量书架。

### 阶段 3：验证

- 用阅读画像场景提问新书推荐。
- 验证输出具体书名。
- 验证候选书架场景仍只基于已有候选。
- 跑 Rust 测试、TypeScript 检查和前端构建。

## 风险

### 推荐不准确

风险：模型可能推荐不存在、不适合或微信读书不可用的书。

控制：

- 明确“待确认候选”。
- 不自动加入候选。
- 后续通过搜索确认。

### 推荐重复

风险：推荐已读或已候选的书。

控制：

- 增加排除清单。
- P0 控制为标题级去重，不做复杂匹配。

### 过度复杂

风险：过早做卡片、搜索、写入候选。

控制：

- P0 只改 Prompt、上下文和文案。
- P1 再做结构化推荐项和加入候选按钮。

## 结论

新书推荐应作为 AI 阅读助手的受控例外能力：默认仍本地上下文优先，但当用户明确要求推荐新书时，允许 AI 基于阅读画像和通用书目知识提出具体候选。候选书架不作为推荐池，只作为去重排除和后续决策入口。
