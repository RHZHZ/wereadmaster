# 官网桌面端布局设计文档

## 文档状态

- 状态：草案，用于后续桌面端 UI 修复与验收。
- 范围：官网单页 `website/` 的桌面端布局与区块节奏。
- 目标页面：`website/App.tsx`、`website/styles.css`、`website/index.html`。
- 核心目标：解决桌面端首屏、产品截图、新增 FAQ 和开源区块的宽度与视觉节奏问题。

## 背景

官网已经完成下载转化方向的内容增强，新增了：

- “为什么是桌面端？”
- “下载前常见问题”
- FAQ 结构化数据
- 下载与开源信任说明

但桌面端视觉仍存在明显不协调：首屏标题断行、产品截图裁切、中段区块宽度不统一、FAQ 像输入框列表、开源区块和上方 FAQ 衔接弱。当前问题不适合继续零散 patch，应先统一布局规则，再按区块修复。

## 设计原则

### KISS

只引入两套页面宽度规则，不新增复杂布局系统：

```text
普通内容区：用于截图、说明卡片、开源区块。
阅读内容区：用于 FAQ、长段解释和问答。
```

### YAGNI

本阶段不做：

- 不新增路由。
- 不新增 CMS。
- 不新增动画系统。
- 不替换产品截图。
- 不重做完整视觉品牌。

### DRY

所有中段区块复用同一组宽度 token，避免每个区块单独写 `980px`、`1120px`、`1480px`。

### 单一职责

每个区块只承担一个清晰任务：

- Hero：说明产品是什么，并引导下载。
- 产品截图：让用户看清真实界面。
- 为什么是桌面端：解释下载必要性。
- FAQ：解除安装前疑虑。
- 开源区：建立源码、分发和数据边界信任。
- 下载区：完成二次转化。

## 当前问题分析

### 1. Hero 标题断行

现象：

```text
微信读书本地管理
工具
```

这个断行不是有意设计，而是左侧文本列宽不足导致。用户会感觉标题排版粗糙。

根因：

- `hero-copy` 最大宽度偏小。
- H1 字号偏大。
- Hero 网格左右列比例没有给中文标题留足自然宽度。

设计要求：

- 在 1440px 桌面视口下，H1 应优先单行展示。
- 如果必须换行，应按语义换成：

```text
微信读书本地管理工具
```

或明确设计为：

```text
微信读书本地管理
本地阅读工作台
```

不能让“工具”单独掉到下一行。

### 2. Hero 产品截图被裁切

现象：

- 右侧产品截图只显示局部。
- 界面右侧和底部内容被 `object-fit: cover` 裁掉。
- 截图边缘还出现半截 UI，像没有处理完的裁切。

根因：

- 预览容器比例不适合真实截图。
- 截图使用 `cover`，优先填满容器而不是完整展示。

设计要求：

- Hero 产品预览必须优先展示真实界面完整性。
- 桌面端首屏截图建议使用 `object-fit: contain`。
- 预览容器采用稳定比例：

```text
16 / 10 或 16 / 9
```

- 允许用浅色背景、边框和阴影承接留白，但不允许裁掉核心界面。

### 3. Hero 左右比例失衡

现象：

- 左侧文案区域很空，标题却被挤换行。
- 右侧截图较大但不可读。
- 首屏像“文案 + 被裁切的截图”，不是一个完成度高的桌面端 landing section。

根因：

- 左列宽度和 H1 字号不匹配。
- 右列截图容器没有按截图原始比例设计。
- 首屏缺少明确的内容边界。

设计要求：

桌面端 Hero 使用双栏：

```text
左侧：产品名、副文案、CTA、信任点
右侧：完整产品截图
```

推荐布局：

```text
Hero 容器宽度：min(1320px, calc(100vw - 96px))
左列：minmax(560px, 0.9fr)
右列：minmax(620px, 1.1fr)
列间距：48px - 72px
```

在 1280px 以下可以降级为上下结构，避免双栏互相挤压。

### 4. 中段区块宽度不统一

现象：

- 产品截图区较宽。
- “为什么是桌面端”区块较窄。
- FAQ 更窄。
- 开源区块又变宽。

这会让页面像多个独立模块拼接，而不是同一个设计系统。

根因：

- 每个区块单独定义宽度。
- 缺少统一内容宽度 token。
- FAQ 和开源区块没有明确的容器层级。

设计要求：

定义两套宽度：

```css
--site-content-width: min(1120px, calc(100vw - 40px));
--site-reading-width: min(880px, calc(100vw - 40px));
```

使用规则：

| 区块 | 宽度 |
| --- | --- |
| Hero | 独立 hero 宽度，最大约 1320px |
| 功能价值区 | `--site-content-width` 或现有四列宽度 |
| 流程区 | `--site-content-width` |
| 产品截图区 | `--site-content-width` |
| 为什么是桌面端 | `--site-content-width` |
| FAQ | `--site-reading-width` |
| 开源透明 | `--site-content-width` |
| 下载区 | `--site-content-width` |

### 5. FAQ 像输入框列表

现象：

- 每个 FAQ 项高度较扁。
- 圆角、阴影和白底让它像输入框。
- 没有清晰的展开状态提示。

根因：

- `summary` 缺少右侧 chevron。
- 问题文本和容器之间的内边距偏小。
- 展开态没有视觉变化。

设计要求：

FAQ 项应呈现为可展开问答：

- 每项最小高度：`64px`。
- 左侧保留小圆点或状态点。
- 右侧增加 chevron。
- 展开后 chevron 旋转。
- 展开后答案区域和问题之间有清晰层级。
- FAQ 列表整体宽度使用 `--site-reading-width`。

### 6. 开源区块和 FAQ 衔接弱

现象：

- FAQ 是居中窄列表。
- 开源透明突然变为宽双栏。
- 右侧信任点较碎，信息权重不清晰。

根因：

- 开源区块仍按较宽的营销卡片布局。
- 右侧信任点没有和左侧 repo 卡片形成紧凑关系。

设计要求：

- 开源区块宽度使用 `--site-content-width`。
- 左侧 repo 卡片占主要视觉权重。
- 右侧信任点保持竖向列表，但宽度控制在 `300px - 360px`。
- 与 FAQ 之间保持足够顶部间距，避免像一个列表突然接另一个模块。

## 推荐信息架构

桌面端页面顺序保持不变，但调整视觉任务：

```text
1. Hero：产品名 + 完整真实截图 + 主下载 CTA
2. 功能区：同步书架 / 整理笔记 / AI 复盘 / Markdown 导出
3. 流程区：同步 -> 整理 -> 复盘 -> 导出
4. 产品截图：真实界面导读
5. 为什么是桌面端：3 个原因
6. FAQ：下载前疑虑
7. 开源透明：GitHub、分发、数据边界
8. 下载区：最终 CTA
```

## 详细设计

### Hero

目标：

- 桌面端第一眼要像真实桌面应用官网，而不是模板化 SaaS 首屏。
- 用户能同时看到产品名、下载按钮和完整界面截图。

布局：

```text
左侧宽度：560px - 620px
右侧宽度：620px - 720px
Hero 高度：640px - 720px
```

H1：

```text
微信读书本地管理工具
```

H1 规则：

- 1440px 视口下单行优先。
- 字号建议：`clamp(3rem, 4vw, 4.35rem)`。
- 最大行宽应让完整标题自然放下。

截图：

- 使用真实 `shelf.png`。
- `object-fit: contain`。
- 背景使用浅色面板承接空白。
- 不允许裁切主要导航、统计卡片、筛选区和书籍列表。

CTA：

- 主按钮：下载 Windows 版。
- 次按钮：查看 GitHub 项目。
- 信任点保持三项：

```text
GitHub Releases 分发
开源可查
Key 保存在本机
```

### 产品截图区

目标：

- 让用户看清应用真实界面。
- 不使用大面积裁切截图作为背景。

要求：

- 截图容器宽度使用 `--site-content-width`。
- 大图使用稳定比例。
- 截图导读不压缩到过窄列。
- tab 或按钮需要保持明显可点击状态。

### 为什么是桌面端

目标：

- 解释“为什么不能直接网页使用”。
- 承接下载疑虑，而不是重复功能卖点。

布局：

```text
容器：--site-content-width
卡片：3 列
卡片间距：16px - 20px
```

卡片内容保持短句：

- 本地缓存
- 安全存储
- 长期整理

### FAQ

目标：

- 用较短问答解决下载前阻力。
- 视觉上像问答，不像表单。

布局：

```text
容器：--site-reading-width
列表：单列
问题项高度：64px - 72px
项间距：14px - 16px
```

交互：

- 使用原生 `details / summary`。
- summary 右侧增加 chevron。
- `details[open]` 时 chevron 旋转。
- 展开答案保持左侧缩进，但不能过度缩小字号。

### 开源透明

目标：

- 建立信任，不抢下载主流程。
- 让 GitHub 仓库、发布来源和数据边界更可信。

布局：

```text
容器：--site-content-width
左列：repo card
右列：trust points
```

推荐比例：

```text
左列：minmax(0, 1fr)
右列：minmax(300px, 0.52fr)
gap：28px - 32px
```

视觉要求：

- repo card 不超过内容容器。
- trust point 使用竖向列表。
- 避免多个小卡片堆叠造成信息碎片化。

### 下载区

目标：

- 作为页面底部二次转化，不再承担过多解释。
- 保持 Windows / Android 双平台入口清晰。

要求：

- 宽度使用 `--site-content-width`。
- 截图或设置图不应压过 CTA。
- 移动端优先展示 Android 文案，桌面端优先展示 Windows 文案。

## 响应式策略

### 大桌面：`>= 1280px`

- Hero 使用双栏。
- H1 单行优先。
- 产品截图完整展示。
- 内容区宽度最大 1120px。
- FAQ 最大 880px。

### 中等桌面和平板：`861px - 1279px`

- Hero 可改为上下结构。
- 产品截图居中，最大宽度约 760px。
- “为什么是桌面端”卡片从 3 列降为 1 列或 2 列，避免挤压。
- 开源区块改为单列。

### 移动端：`<= 860px`

- 所有内容宽度使用 `--site-width`。
- Hero 上下结构。
- FAQ 保持单列。
- 下载 CTA 高度不低于 44px。
- 移动菜单保持清晰不穿透。

## CSS Token 方案

建议集中定义：

```css
:root {
  --site-width: min(1480px, calc(100vw - 40px));
  --site-content-width: min(1120px, calc(100vw - 40px));
  --site-reading-width: min(880px, calc(100vw - 40px));
  --site-hero-width: min(1320px, calc(100vw - 72px));
}

@media (max-width: 860px) {
  :root {
    --site-width: min(100vw - 32px, 1240px);
    --site-content-width: var(--site-width);
    --site-reading-width: var(--site-width);
    --site-hero-width: var(--site-width);
  }
}
```

使用边界：

- `--site-width`：导航、全局页脚、需要接近宽屏边界的结构。
- `--site-content-width`：主要内容区。
- `--site-reading-width`：FAQ 和长文本阅读区。
- `--site-hero-width`：Hero 双栏区。

## 验收标准

### 桌面端视觉验收

在 `1440 x 900` 下：

- Hero H1 不出现“工具”单字掉行。
- Hero 真实截图完整展示，不裁切主要界面。
- 主 CTA 在首屏可见。
- “为什么是桌面端”宽度不超过 1120px。
- FAQ 宽度不超过 880px。
- 开源区宽度不超过 1120px。
- 页面无横向滚动。

在 `1280 x 720` 下：

- Hero 左右两列不互相挤压。
- H1 换行必须语义自然。
- 右侧截图仍可识别主要界面。
- FAQ 锚点跳转后标题不被 sticky header 遮挡。

### 移动端回归验收

在 `390 x 844` 下：

- 页面无横向滚动。
- Hero 截图不溢出。
- FAQ 每项可点击，展开内容不溢出。
- 移动菜单可打开和关闭。
- 主下载按钮不低于 44px。

### 技术验收

必须通过：

```text
npx tsc --noEmit --pretty false
npm run build:website
```

浏览器检查：

```text
http://127.0.0.1:5173/website/
```

检查项：

- 页面非空。
- 无 Vite / React 错误覆盖层。
- 控制台无新增 error/warn。
- 下载按钮仍指向 GitHub Releases。
- GitHub 按钮仍指向项目仓库。

## 风险与取舍

### 风险 1：Hero 改为 contain 后右侧留白变多

处理：

- 使用浅色截图面板承接留白。
- 保持边框和阴影，让截图像产品窗口，而不是孤立图片。
- 不为了填满容器牺牲界面完整性。

### 风险 2：FAQ 收窄后页面显得更长

处理：

- FAQ 控制在 6 条。
- 每条回答保持短句。
- 不把教程内容塞进首页 FAQ。

### 风险 3：开源区收窄后信息拥挤

处理：

- repo card 保持左侧主视觉。
- 右侧 trust point 控制为 3 条。
- 删除重复的技术标签或过细说明。

### 风险 4：桌面修复影响移动端

处理：

- 所有桌面专用布局放在默认或大屏规则。
- 移动端在 `max-width: 860px` 中重置宽度 token。
- 每次改动都同时跑 `390 x 844` 回归。

## 结论

当前官网桌面端不需要推倒重做，问题集中在：

```text
Hero 双栏比例
截图完整性
中段容器宽度
FAQ 交互表达
开源区块节奏
```

最小有效方案是先统一宽度 token，再修 Hero 和 FAQ。这样能保留现有内容改造成果，同时把页面从“拼接模块”收敛成一个一致的桌面端官网。
