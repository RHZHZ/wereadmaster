# AI 阅读助手 P3 阶段规划

## 背景

P0-P2 已经完成 AI 阅读助手的可用闭环：

- 右侧非模态对话面板。
- 历史、设置、上下文 chips。
- 新书推荐到候选书架。
- 微信读书搜索确认。
- 候选书源确认状态。
- Markdown-lite 渲染。
- 回答内快捷追问。
- Codex 风格一体化输入区。

P3 不应继续在面板上堆功能，而应解决“长期可用”和“等待体验”的问题。

## P3 核心目标

P3 的目标是让 AI 阅读助手从“可用聊天入口”升级为“稳定可回看、可持续追问、等待体验更好”的阅读助手。

优先目标：

1. 历史回放能恢复结构化 AI 输出。
2. 普通解释型问答支持更好的等待体验。
3. 阅读记忆检索能力有清晰边界，不直接跳到完整向量知识库。

## P3 阶段原则

P3 需要控制范围，避免把“阅读助手体验增强”扩大成“完整 AI 知识库平台”。

执行原则：

- 先验收现有 P0-P2，再进入结构性改造。
- 先持久化已有稳定输出，再新增新的 AI 交互形态。
- 涉及数据库结构变更时单独确认，不在普通 UI 优化中夹带。
- 阅读记忆必须可解释、可关闭、可追溯来源。
- 不继续压缩 AI 面板主对话区，不把历史列表、设置、推荐卡片重新挤回同一视图。
- 不为了流式牺牲结构化输出的稳定性。

## 非目标

P3 暂不做：

- 不做全量向量知识库。
- 不做整书全文问答。
- 不把聊天结果自动写成正式 AI 资产。
- 不把所有结构化输出都改成流式。
- 不引入远端同步对话历史。
- 不大拆 `reading-api.ts` 或重构整个 AI 服务层。

## 候选方向评估

### 方向 A：历史结构化输出恢复

现状：

- `askReadingAssistant` 当前返回 `recommendedBooks` 和 `suggestions`。
- 当前会话内可以展示推荐卡片和快捷追问。
- 但历史消息读取结构只包含文本、上下文、模型信息和错误信息。
- 重新打开历史会话后，推荐卡片和快捷追问无法恢复。

价值：

- 用户回看历史推荐时仍能搜索确认、加入候选。
- 对话历史不再只是纯文本快照。
- 为后续“对话转资产”提供更稳定的数据基础。

代价：

- 需要为 `ai_assistant_messages` 增加结构化输出字段，例如 `output_json`。
- 属于数据库结构变更，实施前需要明确确认。

建议优先级：P3.1。

### 方向 B：普通问答流式输出

现状：

- 当前 AI 助手依赖完整 JSON schema。
- `recommendedBooks`、`suggestions`、`basisNotice` 都需要完整响应后才能稳定解析。

推荐策略：

- 只让普通解释型回答流式。
- 新书推荐、候选书决策、结构化推荐卡片仍保持完整 JSON 返回。
- 流式文本只是等待体验优化，最终历史落库仍以完整响应为准。

价值：

- 降低等待焦虑。
- 更像现代 AI 对话产品。

代价：

- 需要 Tauri 事件流或等价的前端订阅机制。
- 需要取消请求、失败回滚和落库时机设计。

建议优先级：P3.2。

### 方向 C：阅读记忆检索-lite

现状：

- 当前已有阅读统计、阅读画像、AI 资产摘要、候选书上下文。
- 还没有完整向量检索。

推荐策略：

- 先做“阅读记忆检索-lite”，不做向量库。
- 只从现有结构化资产中抽取少量高置信摘要：
  - 最近复盘摘要。
  - 最近阅读路线。
  - 候选书决策。
  - 高频主题标签。
- 每次回答展示明确来源，避免“我记得你……”但无法追溯。

价值：

- 提升跨页面追问的连续性。
- 不引入 embedding、索引任务和隐私边界复杂度。

代价：

- 需要定义记忆片段结构和最大注入预算。
- 需要避免重复上下文和过期资产。

建议优先级：P3.3。

## 推荐执行顺序

### P3.0：整体验收与风险清单

目标：

- 在进入结构性改造前，确认当前 P0-P2 没有明显体验回归。

验收：

- AI 面板打开、关闭、Esc 关闭正常。
- 初始态只展示通用快捷问题。
- 有回答后只展示回答内追问。
- 新书推荐卡片能搜索确认和加入候选。
- 候选书架能区分已确认、待确认和轻管理候选。
- Markdown-lite 不渲染 HTML、链接和图片。

产出：

- P0-P2 体验验收结果。
- 当前已知缺陷和延后事项清单。
- 是否进入 P3.1 的实施判断。

### P3.1：历史结构化输出持久化

目标：

- 历史会话恢复推荐卡片和快捷追问。

建议设计：

```text
ai_assistant_messages
  + output_json TEXT NULL
```

`output_json` 只保存助手结构化输出的稳定子集：

```json
{
  "suggestions": [],
  "recommendedBooks": [],
  "basisNotice": ""
}
```

边界：

- 不迁移历史旧消息。
- 旧消息 `output_json` 为空时正常显示纯文本。
- 不从自然语言回答中反推书名或追问。
- 不保存搜索结果列表，搜索结果仍按需实时获取。

注意：

- 这是数据库结构变更，实施前需要单独确认。

产出：

- 数据结构设计与兼容策略。
- 后端读写实现。
- 前端历史回放恢复推荐卡片和快捷追问。
- 覆盖旧消息兼容、新消息持久化、历史恢复的测试。

实施状态：

- 已确认进入 P3.1。
- 已为 `ai_assistant_messages` 增加可空 `output_json`。
- 已保存并恢复助手回答的 `suggestions`、`recommendedBooks`、`basisNotice`。
- 旧历史消息 `output_json` 为空时继续按纯文本显示。
- 未持久化微信读书搜索结果列表。
- 验证已通过：TypeScript 类型检查、相关前端测试、完整 Rust 测试、生产构建。

### P3.1.1：数据范围选择与总计阅读检索

目标：

- 让 AI 助手在回答前明确本次使用的数据范围，避免把年度统计当作总计阅读。

触发背景：

- 用户询问“我读过的理财类书籍”时，AI 只基于 2026 年度统计回答，遗漏总计阅读和本地结构化书籍匹配。

建议设计：

- 详见 `docs/ai-reading-assistant-p3-1-1-context-range-design.md`。

推荐范围：

- `自动`
- `总计阅读`
- `本年阅读`
- `最近阅读`
- `当前页面`
- `候选书架`

边界：

- 不做完整向量知识库。
- 不读取整本书全文。
- 不把最近 5 本阅读当作全部已读书籍。
- 不拿最新年度统计冒充总计阅读。

验收：

- “我读过的理财类书籍有哪些？”默认走总计阅读或本地结构化匹配。
- “我今年读过哪些理财类书？”限定为年度范围。
- 没有总计统计缓存时，回答必须说明限制。
- chips 展示 `总计阅读统计`、`2026 年度阅读统计` 等明确标签。

建议优先级：P3.1.1，应先于 P3.2 流式输出。

### P3.2：普通问答流式输出

目标：

- 普通解释型回答能边生成边显示。

流式适用场景：

- 总结阅读偏好。
- 解释某个阅读问题。
- 制定阅读计划。
- 基于已有上下文回答开放追问。

不流式场景：

- 新书推荐。
- 推荐书卡片。
- 候选书架决策。
- 任何必须依赖完整结构化 JSON 的输出。

验收：

- 普通回答显示增量文本。
- 生成中可以取消。
- 失败时保留可理解错误状态。
- 最终仍以完整响应落库。
- 历史回放不会出现半截流式文本。

产出：

- 流式适用场景判断。
- 前端流式渲染状态。
- 取消、失败、重试策略。
- 普通问答流式验证，不影响结构化推荐链路。

实施状态：

- 已新增 `reading-assistant-stream` 事件协议。
- 已新增 `ask_reading_assistant_stream` 命令。
- 已新增 `cancel_reading_assistant_stream` 命令。
- 普通问答 intent 走流式 JSON 的 `answer` 增量展示。
- 新书推荐、候选书决策等结构化 intent 仍走完整 JSON 返回。
- 最终仍以完整 JSON 解析、归一化、落库和恢复历史。
- 生成中可取消，取消后前端移除 pending 消息，后端不落库半截文本。
- 已补充前端 API 测试与 Rust SSE/stream payload 测试。
- 验证已通过：TypeScript 类型检查、相关前端测试、完整 Rust 测试、生产构建。

### P3.3：阅读记忆检索-lite

目标：

- 在不做向量库的前提下，提高跨资产追问质量。

建议输入：

- 最近 3-5 个正式 AI 资产摘要。
- 最近候选书决策。
- 最近统计复盘摘要。
- 画像标签和主题标签。

边界：

- 不读取全量原始笔记。
- 不读取整本书全文。
- 不做后台自动索引。
- 不生成不可追溯的“长期记忆”。

验收：

- 回答能说明来自哪些阅读资产。
- 上下文预算可控。
- 用户关闭阅读记忆后不注入这些片段。

产出：

- 记忆片段结构。
- 来源展示规则。
- 注入预算和去重策略。
- 设置项或开关策略。

实施状态：

- 已新增 P3.3 设计文档。
- 已新增 `readingMemory` 轻量上下文，来源包括最近 AI 资产、统计复盘、选书决策和少量候选状态。
- 每个记忆片段带 `sourceRef`，通过 used context 记录“阅读记忆”来源数量。
- `useReadingMemory=false` 时不注入 `readingMemory`。
- 默认全局助手和候选书架改用 `readingMemory` 承载跨资产记忆，不再通过宽泛 `aiAssetSummary` 默认注入最近资产。
- 不读取全文、原始笔记、原始 WeRead 响应或本地文件路径。
- 验证已通过：TypeScript 类型检查、相关前端测试、完整 Rust 测试、生产构建。

### P3 收尾：页面级 QA

目标：

- 验证 P3 能力在真实构建产物中的可见体验。
- 确认 AI 面板桌面和移动端没有明显重叠、裁切或横向溢出。
- 确认流式输出、取消生成状态、阅读记忆上下文和助手设置可见。

实施状态：

- 已新增最终 QA 记录：`docs/ai-reading-assistant-p3-final-qa.md`。
- 已使用 `vite preview` 对 `dist` 构建产物进行 Playwright 页面级验收。
- Browser skill 可见，但运行时所需 `node_repl js` 工具未暴露，已记录为 Browser invocation failed，并回退到 Playwright。
- 桌面视口 `1360 x 820` 通过，移动视口 `390 x 844` 通过。
- Console error、page error、框架 overlay 均未发现。
- 普通问答流式 mock 通过：`reading-assistant-stream` 事件、生成中“取消生成”、最终 Markdown-lite 回答、used context 和快捷追问均可见。
- 剩余风险仅限真实 Tauri 桌面端 AI Provider 网络流式联调和长数据压力场景。

## P3 决策点

进入具体实施前需要明确以下决策：

1. 是否允许 P3.1 修改本地数据库结构。
2. 流式输出是否只覆盖普通解释型问答。
3. 阅读记忆检索-lite 是否作为用户可开关能力。
4. 历史旧消息是否保持只读兼容，不做批量迁移。
5. P3 是否仍以桌面本地数据为主，不引入远端同步。

## 不建议进入 P3 的事项

- 直接做完整向量知识库。
- 所有 AI 输出统一流式。
- 对话自动沉淀为正式复盘。
- 历史消息批量迁移或自动解析旧文本。
- 点击外部区域自动关闭 AI 面板。
- 对主内容做高斯模糊。

## P3 启动建议

建议先做 P3.0 体验验收，然后选择 P3.1。

原因：

- P3.1 解决的是已有能力的回放一致性，比流式更基础。
- 推荐卡片和快捷追问已经稳定，值得持久化。
- 但它涉及数据库结构变更，必须在实施前确认范围。

如果不希望触碰数据库，则先做 P3.2 的流式预研文档，不直接实现。
