# 阅读洞察追问交互改造文档

## 背景

单本 AI 复盘和复盘中心版本详情中已经展示“阅读洞察”区块。当前洞察卡片由现有 `BookAiSummary` 字段组合生成：

- 洞察标题：主要来自 `myFocus`。
- 洞察说明：主要来自 `keyIdeas` 或 `overview`。
- 来源摘录：来自 `representativeQuotes`。
- 可继续追问：来自 `reflectionQuestions`。

当前实现中，洞察卡片会展示“可继续追问”问题列表，但卡片底部只有一个“追问”按钮。点击该按钮时，实际提交给 AI 阅读助手的是：

```text
围绕这条阅读洞察继续追问：「{insight.title}」。
洞察说明：{insight.description}
请结合当前复盘和来源摘录，给出 3 个后续问题，并指出最值得先处理的一步。
```

也就是说，用户看到的“可继续追问”问题和点击后进入助手的问题不是同一个对象。

## 问题定义

### 现象

用户在洞察卡片中看到：

```text
可继续追问
- 如果你身处那个战乱与疫情交织的时代，你会如何平衡个人安危与救死扶伤的使命？
- 姚英子、方三响、孙希三人的命运在相遇之前就已交织，你认为是什么力量在推动这种宿命般的连接？
```

点击“追问”后，助手输入框却预填：

```text
围绕这条阅读洞察继续追问：「历史的真实自带凝重质感，超越一切艺术创作。」。
洞察说明：公共卫生的成效取决于金字塔底，即政府能否从最广泛的底层汲取力量。
请结合当前复盘和来源摘录，给出 3 个后续问题，并指出最值得先处理的一步。
```

### 本质问题

当前 UI 发生了“显示 A、提交 B”的语义错位：

- 页面展示的是 `reflectionQuestions`。
- 点击按钮提交的是 `insight.title + insight.description`。
- 用户会自然认为“追问”对应可见问题，而不是整张洞察卡片。

### 影响

1. 用户预期不稳定：看到人物伦理问题，实际追问公共卫生洞察。
2. AI 助手草稿可解释，但不符合点击对象。
3. 洞察标题、说明和追问问题来自不同字段，语义可能不同轴。
4. 用户难以判断“可继续追问”只是展示信息，还是可操作入口。

### 截图复核结论

本次截图中的洞察卡片仍呈现旧状态：

- “可继续追问”下面展示了两个具体问题。
- 卡片底部只有一个通用“追问”按钮。
- 点击后进入 AI 阅读助手的草稿围绕的是洞察标题和洞察说明，不是用户看到的某一个问题。

这说明问题不在 AI 输出本身，而在前端交互映射：`reflectionQuestions` 已经被展示为“可继续追问”，但没有被设计成可点击对象。用户看到问题后，合理预期是“点追问 = 问这些问题之一”；系统实际执行的是“围绕洞察整体继续生成问题”。因此必须把可见问题升级为独立操作入口。

## 改造目标

1. 让用户看到的可追问内容与提交给 AI 助手的草稿一致。
2. 保留“围绕整条洞察追问”的能力。
3. 让 `reflectionQuestions` 成为真实可点击问题，而不是只读列表。
4. 不新增后端字段，不改缓存结构，不扩大 AI 调用链路。
5. 单本复盘页和复盘中心版本详情复用同一组件行为。

## 非目标

本次不做：

- 不新增独立洞察表。
- 不改 `BookAiSummary` 类型。
- 不重新设计 AI 复盘 prompt。
- 不把追问结果自动写回复盘。
- 不把所有复盘问题迁移到洞察卡片。
- 不做复杂的洞察与问题语义匹配算法。
- 不调用 `gpt-image2` 或其他图像生成模型补资源；本问题是文案、状态和交互对象不一致，不需要新增视觉资产。

## 设计方案

### 方案概览

把洞察卡片中的追问能力拆成两层：

```text
洞察卡片
  ├─ 围绕洞察追问
  └─ 可继续追问
       ├─ 问这个问题
       ├─ 问这个问题
       └─ 问这个问题
```

### 1. 卡片级按钮改名

当前：

```text
追问
```

建议改为：

```text
围绕洞察追问
```

作用：

- 明确按钮作用对象是整条洞察。
- 避免用户误以为它会提交下方某个问题。

草稿保持当前结构，但文案更收敛：

```text
围绕这条阅读洞察继续追问：「{insight.title}」。
洞察说明：{insight.description}
请结合当前复盘和来源摘录，说明这条洞察最值得继续展开的方向，并给出 3 个后续问题。
```

### 2. 每条问题增加操作入口

在“可继续追问”列表中，每条问题旁增加一个轻量按钮：

```text
问这个问题
```

草稿模板：

```text
围绕这个复盘问题继续追问：
「{question}」

关联洞察：「{insight.title}」
洞察说明：{insight.description}

请结合当前复盘、阅读洞察和来源摘录回答，并给出 1 个最值得继续展开的方向。
```

首版带上关联洞察，但把用户点击的问题放在最前面，避免助手回答偏离点击对象。

### 3. 视觉层级

建议结构：

```text
阅读洞察

洞察 1
历史的真实自带凝重质感，超越一切艺术创作。
公共卫生的成效取决于金字塔底...

来源摘录
...

可继续追问
1. 如果你身处那个战乱与疫情交织的时代...
   [问这个问题]
2. 姚英子、方三响、孙希三人的命运...
   [问这个问题]

[围绕洞察追问]
```

按钮层级：

- `问这个问题`：靠近具体问题，使用小号 text button。
- `围绕洞察追问`：放在卡片底部，使用当前洞察追问按钮样式。

### 4. 空状态

如果 `followUpQuestions.length === 0`：

- 不展示“可继续追问”区块。
- 只保留“围绕洞察追问”按钮。

如果没有 `onAskInsight`：

- 继续只读展示，不显示任何追问按钮。

### 5. AI 输出内容是否合理

截图中的洞察输出内容本身基本合理，但交互表达不合理。

合理部分：

- 洞察标题、说明、来源摘录和后续问题都有阅读复盘语义。
- “可继续追问”的问题适合作为 AI 阅读助手的下一轮输入。
- 来源摘录能支撑用户继续追问，不需要额外生成素材。

不合理部分：

- “可继续追问”问题与卡片底部“追问”按钮没有一一对应关系。
- 按钮文案过短，隐藏了“围绕洞察整体追问”的真实意图。
- 当洞察标题、说明和问题来自不同字段时，点击结果可能出现语义跳轴。

处理结论：

- 不需要让 AI 重新生成洞察。
- 不需要调用图像生成模型补缺失资源。
- 需要改造为“问题级追问”和“洞察级追问”两个入口。

## 文案规范

### 推荐文案

| 场景 | 文案 |
| --- | --- |
| 卡片级按钮 | 围绕洞察追问 |
| 问题级按钮 | 问这个问题 |
| 问题区标题 | 可继续追问 |
| 草稿首句 | 围绕这个复盘问题继续追问 |

### 不建议文案

| 文案 | 原因 |
| --- | --- |
| 追问 | 对象不明确 |
| 继续 | 动作含糊 |
| 生成问题 | 会误解为再生成一批问题 |
| 深入分析 | 和洞察/问题对象绑定不清 |

## 技术改造点

### 组件

主要修改：

```text
src/components/BookInsightSection.tsx
```

建议新增两个 draft builder：

```ts
function buildInsightDraft(title: string, description: string): string

function buildInsightQuestionDraft(question: string, insightTitle: string, insightDescription: string): string
```

也可以后续拆到：

```text
src/lib/book-insight-drafts.ts
```

首版为了保持 KISS，可以先留在组件文件内；如果 `BookAiSummaryPage`、`ReadingHubPage` 或测试也需要复用，再抽纯函数。

### 数据模型

不变：

```ts
type BookInsightViewModel = {
  id: string;
  title: string;
  description: string;
  sourceQuotes: BookAiRepresentativeQuote[];
  followUpQuestions: string[];
};
```

不新增字段。

### 后端路由

无需新增后端 action。

但必须继续保证以下草稿不会触发正式 `BookReview` 动作卡：

```text
围绕这个复盘问题继续追问：
「...」

请结合当前复盘、阅读洞察和来源摘录回答...
```

当前后端已经把 `当前复盘`、`阅读洞察`、`来源摘录`、`后续问题` 等追问场景排除出正式复盘路由。实现后仍需加测试覆盖。

### 实现状态

当前改造应落在以下范围：

- `BookInsightSection` 统一负责单本 AI 复盘页和复盘中心版本详情的洞察卡片。
- `buildInsightDraft` 只生成“围绕洞察追问”的草稿。
- `buildInsightQuestionDraft` 只生成“围绕这个复盘问题继续追问”的草稿。
- Rust 路由识别必须把问题级草稿视为普通追问，不能替换为“应进入本应用的单本 AI 复盘生成，不在聊天里临时替代完成。”
- E2E 必须覆盖问题级追问打开 AI 阅读助手后，输入框包含用户点击的问题原文。

## 验收标准

1. 用户点击“可继续追问”下某个问题时，助手输入框必须包含该问题原文。
2. 用户点击卡片级按钮时，助手输入框必须围绕洞察标题和说明。
3. 阅读洞察卡片内不再出现对象不明确的单独“追问”按钮。
4. 三类入口语义清晰：
   - 阅读洞察：围绕洞察追问。
   - 可继续追问：问这个问题。
   - 下一步行动：拆解。
5. 问题级追问不会触发“生成 AI 复盘”动作卡。
6. 单本复盘页和复盘中心版本详情行为一致。
7. 移动宽度下按钮不挤压问题文本，不产生横向滚动。

## 风险与取舍

### 风险 1：按钮增多导致卡片变重

控制方式：

- 问题级按钮使用轻量 text button。
- 每条洞察默认最多展示现有数量的追问问题，不增加问题数量。
- 不新增额外说明文案。

### 风险 2：洞察和问题本身语义仍可能不同轴

控制方式：

- 问题级追问以用户点击的问题为主。
- 洞察标题和说明只作为关联上下文。
- 不尝试自动判断问题是否属于该洞察。

### 风险 3：正式复盘路由误判

控制方式：

- 草稿中明确包含“当前复盘、阅读洞察、来源摘录、继续追问”等上下文。
- 后端测试覆盖该草稿不触发 `BookReview` action。

## 当前阶段判断

适合进入实现验证阶段，但不适合直接进入下一阶段的大范围能力扩展。

原因：

- 当前缺口已经明确：展示问题、点击对象和助手草稿不一致。
- 不需要新增后端模型字段，也不需要重做 AI prompt。
- 已有组件可以复用，变更范围可控。
- 仍需通过 E2E 验证真实页面不再出现旧按钮“追问”和问题不可点击的状态。

下一阶段只有在上述验收通过后再考虑：

- 是否把洞察追问结果沉淀到历史复盘。
- 是否把高质量追问沉淀到成果库。
- 是否对洞察与复盘问题做语义归类。

## 结论

当前问题不是“洞察不能追问”，而是追问对象不明确。改造后应形成明确对应关系：

```text
点击洞察按钮 -> 围绕洞察追问
点击问题按钮 -> 围绕这个问题追问
点击行动按钮 -> 拆解下一步行动
```

这样能保持现有架构不变，同时消除用户看到的问题和助手收到的问题不一致的体验断点。
