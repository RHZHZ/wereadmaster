# 设置页作者支持入口设计文档

## 结论

适合加入，但应作为低频、克制、透明的设置入口。

当前应用是本地优先、开源发布的阅读资产工作台，主流程集中在书架、笔记、统计、复盘和导出。作者赞助和联系方式属于“项目信任、反馈、持续维护”信息，不应进入主工作流，也不应在同步、导出、生成报告等关键操作后弹出。

推荐首版在设置页新增一个独立分类：`关于与支持`。该分类下展示两张用户提供的图片：

- 第一张：赞赏码，用于 `赞赏作者`。
- 第二张：联系方式，用于 `联系作者`。

不建议把两张图合并成一个入口，因为它们的用户意图不同：赞赏是自愿支持，联系方式是反馈和交流。

## 设计目标

1. 不打扰主流程

- 不在首页、报告生成、导出完成、同步完成后强提示。
- 不把赞助做成任务、弹窗或红点提醒。
- 只在用户主动进入设置页时可发现。

2. 明确自愿边界

- 赞助不解锁功能。
- 不影响同步、AI、导出和本地缓存。
- 不记录用户是否赞赏。
- 不收集支付或联系方式数据。

3. 保持信任感

- 使用真实二维码图片，但不内嵌第三方支付 WebView。
- 用户自行扫码完成赞赏或添加联系方式。
- 对外链、二维码和个人信息展示保持清晰说明。

4. 实现简单

- 不新增支付 SDK。
- 不新增后端接口。
- 不新增账号系统。
- 不新增赞助状态记录。

## 信息架构

### 设置分类

新增设置分类：

```ts
export type SettingsCategoryId =
  | "account"
  | "ai"
  | "appearance"
  | "export"
  | "updates"
  | "support"
  | "advanced";
```

分类建议：

```ts
{
  id: "support",
  label: "关于与支持",
  description: "作者、反馈、赞赏",
  heroDescription: "查看项目来源、联系作者或自愿赞赏维护工作。",
  icon: HeartHandshake
}
```

位置建议：

- 放在 `应用更新` 之后。
- 放在 `高级维护` 之前。

理由：

- `应用更新` 和 `关于与支持` 都属于低频项目信息。
- `高级维护` 涉及缓存、备份、数据库和诊断，风险更高，仍应留在最后。

## 页面结构

### 推荐布局

```text
关于与支持

[ 项目说明卡 ]
  微信读书个人阅读管理是本地优先的开源阅读工具。
  赞赏完全自愿，不解锁额外功能，也不会影响你的本地数据。

[ 赞赏作者 ]       [ 联系作者 ]
  赞赏码图片          联系方式二维码
  自愿支持维护        反馈问题 / 交流建议
```

桌面端：

- 项目说明卡占满一行。
- `赞赏作者` 和 `联系作者` 两张卡片并排。

移动端：

- 三张卡片上下排列。
- 二维码图片保持足够可扫尺寸。
- 不横向溢出。

## 内容设计

### 项目说明卡

标题：

```text
开源项目，感谢支持
```

正文：

```text
这个工具会继续围绕本地阅读管理、复盘和导出体验迭代。赞赏完全自愿，不会解锁额外功能，也不会影响本地数据。
```

辅助说明：

```text
项目代码和安装包仍以 GitHub Releases 为准；问题反馈建议优先通过 GitHub 相关入口反馈，私下交流可扫码联系作者。
```

操作：

- `访问 GitHub`
- `查看更新`

### 赞赏作者卡

标题：

```text
赞赏作者
```

正文：

```text
如果这个工具节省了你的整理时间，可以扫码自愿支持维护。
```

边界说明：

```text
赞赏不会解锁功能，应用也不会记录或校验赞赏状态。
```

图片：

- 使用第一张赞赏码。
- 图片应完整显示，不裁切。
- 可设置最大宽度，避免在桌面端过大。

### 联系作者卡

标题：

```text
联系作者
```

正文：

```text
扫码添加作者，适合反馈使用问题、交流需求或提供复现信息。
```

边界说明：

```text
添加好友是你主动在微信中完成的操作，应用不会读取或上传联系人信息。
```

图片：

- 使用第二张联系方式二维码。
- 图片应完整显示，不裁切。

建议补充：

- 如果有可公开展示的文字联系方式，建议提供 `复制微信号` 或 `复制邮箱`。
- 如果不希望公开文字联系方式，首版只展示二维码即可。

## 视觉规范

### 总体风格

- 延续设置页现有 `settings-card`、`settings-panel`、`settings-icon` 体系。
- 不使用营销式大横幅。
- 不使用强对比广告样式。
- 二维码区域保持干净、可扫、无遮挡。

### 卡片建议

项目说明卡：

- 使用浅色设置卡样式。
- 重点放在透明边界和开源链接。

赞赏作者卡：

- 可以使用轻微暖色强调，但不要做强商业化样式。
- 图片背景保持白色或中性浅色，保证扫码识别。

联系作者卡：

- 使用中性浅色样式。
- 文案聚焦反馈和交流，不暗示客服承诺。

### 图片尺寸

桌面端：

```css
.settings-support-qr {
  width: min(100%, 320px);
  aspect-ratio: 1 / 1;
  object-fit: contain;
}
```

联系方式图片不是严格 1:1，可以单独处理：

```css
.settings-support-qr.is-contact {
  aspect-ratio: 880 / 1131;
}
```

移动端：

- 宽度使用 `min(100%, 300px)`。
- 居中展示。
- 卡片之间保持 12px 到 16px 间距。

## 交互设计

### 基础交互

- 点击设置导航 `关于与支持` 打开该分区。
- 二维码图片默认直接展示。
- 可选提供 `放大查看`，但首版不强制。

### 推荐按钮

项目说明卡：

- `访问 GitHub`
- `查看更新`

赞赏作者卡：

- `查看赞赏码` 或无按钮直接展示。

联系作者卡：

- `查看联系方式` 或无按钮直接展示。
- 如提供文字联系方式，可加 `复制联系方式`。

### 不做的交互

- 不做赞助弹窗。
- 不做赞助记录。
- 不做支付跳转。
- 不做支付成功回调。
- 不做“已赞赏用户”身份。

## 风险与控制

### 风险 1：个人二维码随开源仓库公开

控制：

- 落地前确认作者接受公开发布。
- 文档中明确这是个人收款码和联系方式。
- 如果后续需要撤回，集中放在 `src/assets/support` 方便替换。

### 风险 2：赞赏入口被理解为付费功能

控制：

- 文案明确 `赞赏不会解锁额外功能`。
- 不放在主流程。
- 不做红点、弹窗、强 CTA。

### 风险 3：二维码不可扫

控制：

- 图片 `object-fit: contain`。
- 不裁切、不加模糊、不叠加遮罩。
- 保持白底或中性底。
- 移动端保证宽度不小于 240px。

### 风险 4：联系方式被误解为客服承诺

控制：

- 文案使用 `反馈问题、交流建议`。
- 不使用 `官方客服`、`实时支持` 等承诺性表达。

## 是否遗漏或过当

### 不遗漏

- 已覆盖赞赏码和联系方式两类不同意图。
- 已覆盖设置页入口、桌面端、移动端和资产路径。
- 已覆盖开源分发、隐私和用户误解风险。
- 已包含单元测试、e2e 和构建验证建议。

### 不过当

- 不引入支付 SDK。
- 不新增后端。
- 不记录赞赏状态。
- 不做弹窗和强引导。
- 不新增复杂页面系统，首版只作为设置页低频分区。

## 推荐首版范围

首版建议只做：

- 新增 `关于与支持` 设置分类。
- 展示项目说明、赞赏码、联系方式二维码。
- 提供 GitHub / 更新页入口。
- 补单元测试和一条移动端布局 e2e。

暂不做：

- 全屏二维码查看器。
- 复制微信号。
- 支付跳转。
- 赞助名单。
- 远程配置二维码。

这样能满足当前需求，同时保持 SOLID/KISS/DRY/YAGNI：功能边界清楚、实现简单、低耦合、没有提前引入商业化复杂度。
