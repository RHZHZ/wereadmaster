# 阅读指南模块去冗余与行动边界改造方案

## 背景

当前阅读指南页已经形成几类内容模块：

- `reading-guide-map`：路线图节点，展示单书指南或跨书路线的推进结构。
- `GuideNodeDetailDialog`：节点详情弹窗，展示单个节点的完整字段。
- `reading-route-focus-card`：本书重点或路线主线，提炼先读哪里、为什么读、如何收束。
- `完整阅读指南`：展示推进任务、复盘点、依赖关系、来源统计等完整结构。
- `下一步行动`：用户可以标记状态、记录反馈、复制清单的执行项。

这些模块的出发点不同，但在单书指南场景中，部分内容已经出现重叠：

- 路线图节点的“先读哪里”与本书重点的“先读哪里”重复。
- 路线图节点的“带问题读”与完整指南中的“复盘点”重复。
- 路线图节点的“交付物”与下一步行动、复盘点存在表达交叉。
- 完整指南中的“推进任务”再次展示了阅读任务、投入和依据。

本次改造目标不是把所有内容压缩到一个模块，而是明确每个模块的职责边界，减少重复表达，让用户能快速分辨：

- 哪些是路径结构。
- 哪些是解释依据。
- 哪些是真正需要执行并反馈的行动。

## 核心结论

### 1. 节点内容不应整体并入下一步行动清单

路线图节点是导航信息，下一步行动清单是执行信息。二者语义不同。

如果把所有节点都并入行动清单，会产生三个问题：

- 行动清单变重：`当前书`、`延伸判断`、`依据`、`预计投入` 等说明性信息会被误认为任务。
- 反馈记录失真：用户对说明性节点的反馈不是阅读行动成果，混入 `ai_feedback_records` 后会污染后续成果回顾。
- 信息层级变平：路线结构、复盘问题、行动标准被放到同一列表里，用户难以判断现在该做什么。

因此，节点不能作为行动清单的直接来源。

### 2. 行动清单只接收“可执行、可反馈”的内容

能进入下一步行动清单的条目必须同时满足：

- 用户可以在现实中完成它。
- 它有明确完成标准。
- 它值得记录状态或反馈。
- 它会影响下一版阅读指南或复盘生成。

例如：

- `今天安排 45 分钟读完第 2 章`
- `标出 3 条可以直接实践的专注规则`
- `输出 1 页本书复盘`

不应进入行动清单的内容：

- 当前书标题、作者、本地状态。
- 阅读依据、来源统计、依赖关系。
- `是否加入候选书` 这类判断说明，除非被明确改写成可执行动作。
- 节点完整说明中的解释性文本。

### 3. 节点可以关联行动，但不承载反馈

路线图节点可以在详情弹窗中展示“关联行动”，帮助用户知道这个节点对应哪些可执行项。

推荐表达：

- 节点卡片：只展示路径摘要。
- 节点弹窗：展示完整目的、依据、投入和关联行动。
- 行动清单：仍然是唯一的反馈入口。

这样可以保留节点与行动的关系，同时避免出现两个地方都能记录反馈的状态分裂。

## 用户视角

改造后的理想体验：

1. 用户进入阅读指南，先看路线图，理解这份指南的推进结构。
2. 用户点击某个节点，查看为什么这样安排、需要投入多少、依据是什么。
3. 用户看到节点详情中的关联行动，知道真正要执行的是哪几项。
4. 用户到下一步行动清单中勾选状态、记录反馈。
5. 用户的反馈只沉淀在行动项上，后续新版生成时能准确复用。

用户不需要理解内部模块差异，但界面应该自然传达：

- 路线图回答“怎么走”。
- 节点详情回答“为什么这样走”。
- 行动清单回答“现在做什么”。
- 完整指南回答“需要深读时看全部细节”。

## 模块职责边界

| 模块 | 保留职责 | 删除或弱化职责 |
|---|---|---|
| 路线图节点 | 展示路径结构、节点顺序、阅读目的摘要 | 不记录反馈，不展示长篇行动标准 |
| 节点详情弹窗 | 展示单节点完整目的、依据、投入、关联行动 | 不展示整份指南，不提供反馈入口 |
| 本书重点 / 路线主线 | 提供首屏 3 条摘要 | 避免与路线图逐字重复 |
| 完整阅读指南 | 展示推进任务、复盘点、依赖关系、来源统计 | 不重复承载行动反馈 |
| 下一步行动清单 | 承载可执行动作、完成标准、反馈记录 | 不收纳说明性节点和依据字段 |

## 信息架构建议

### 推荐页面顺序

首版建议保持现有页面结构，但调整表达密度：

1. 路线图
2. 下一步行动清单
3. 本书重点 / 路线主线
4. 完整阅读指南
5. 来源统计与生成信息

原因：

- 路线图先建立结构。
- 行动清单紧跟路线图，让用户立即看到“现在可做什么”。
- 本书重点如果保留，应作为轻摘要，不与路线图抢信息。
- 完整指南适合放在后面，服务深读和核对。

如果担心改动过大，可以先不调整布局，只先完成字段去冗余和关联行动。

### 单书指南

单书指南中的路线图节点建议保留 4 类：

- 当前书：说明本指南围绕哪一本书。
- 先读哪里：说明下一段阅读推进范围。
- 带问题读：说明阅读时需要验证的问题。
- 读完产出：说明本轮阅读结束后的输出。

`是否加入候选书` 节点建议降级处理：

- 如果只是说明，不进入行动清单。
- 如果用户已经选择候选书，可以作为跨书路线入口提示。
- 如果需要行动化，应改写为明确动作，例如 `完成本书复盘后，再决定是否加入候选书`。

### 跨书路线

跨书路线节点本质上是书籍顺序，不应并入行动清单。

行动清单只保留：

- 当前书完成后的接续动作。
- 切换到下一本前的判断动作。
- 路线复盘输出。

节点详情可以展示：

- 本书角色。
- 阅读目的。
- 预计投入。
- 前置依赖。
- 后续依赖。
- 关联行动。

## 去冗余策略

### 1. 路线图节点瘦身

节点默认只保留：

- `eyebrow`
- `label`
- 一句短说明
- 关键 meta，例如投入或状态

完整依据、验收标准、依赖关系放到节点详情弹窗。

### 2. 本书重点去重复

`reading-route-focus-card` 如果继续保留，应避免直接复刻路线图节点。

建议改成更高层摘要：

- `当前优先级`
- `阅读时要验证的判断`
- `本轮收束方式`

而不是再次使用：

- `先读哪里`
- `带什么问题读`
- `读完产出什么`

### 3. 完整指南保留完整，但减少卡片标题重复

完整指南中的推进任务、复盘点、行动清单仍可保留，但每个区块要有明确角色：

- 推进任务：展示阅读任务和依据。
- 复盘点：展示问题和输出要求。
- 下一步行动：展示可勾选的执行项。

避免在推进任务和行动清单中用完全相同句子作为标题。

### 4. 关联行动只做视图模型，不改 AI 输出

首版不修改 prompt、schema 或后端数据结构。

在前端根据现有 `route.nextActions` 派生节点关联行动：

```ts
type GuideMapNode = {
  id: string;
  eyebrow: string;
  label: string;
  detail: string;
  meta?: string;
  fullDetail?: string;
  fullMeta?: string;
  fields?: Array<{
    label: string;
    value: string;
  }>;
  associatedActions?: Array<{
    title: string;
    done: string;
  }>;
};
```

节点详情弹窗只展示关联行动文本，不提供反馈控件。

反馈仍然只写入行动清单中已有的 `AiActionFeedbackChecklist`。

## 技术方案

### 1. 明确行动判定函数

新增或复用一个纯函数，把行动清单入口限制在 `guideDetails.actions`：

```ts
function isFeedbackAction(item: { title: string; done: string }) {
  return Boolean(item.title.trim() && item.done.trim());
}
```

此函数只用于行动清单，不用于路线图节点。

### 2. 节点增加关联行动字段

在 `buildSingleBookGuideNodes` 中建立最小映射：

- `continue-reading` 关联第一个阅读推进行动。
- `book-review` 关联复盘相关行动。
- `single-book-output` 关联输出相关行动。

如果无法可靠匹配，只展示空关联，不做猜测。

### 3. 节点详情弹窗展示关联行动

在 `GuideNodeDetailDialog` 中新增只读区块：

- 标题：`关联行动`
- 内容：行动标题 + 完成标准
- 文案提示：不出现说明性教程，只展示内容本身。

弹窗仍只保留：

- 复制节点内容
- 关闭

不新增：

- 行动反馈控件
- 节点完成状态
- 节点编辑入口

### 4. 调整本书重点文案来源

把 `buildGuidePrescriptionItems` 从“重复路线节点”调整为“摘要层”。

可选方向：

- 保留函数名，改变 label 文案。
- 或新增 `buildGuideFocusItems`，让路线图和重点卡不共用同一份视图模型。

建议采用第二种：

```ts
buildGuideMapNodes(...)
buildGuideFocusItems(...)
buildGuideDetailSections(...)
```

理由：

- 路线图、重点卡、完整指南是不同展示层。
- 共用同一个 `GuidePrescriptionItem` 会自然导致重复。
- 拆成纯函数后职责更清晰，测试也更直接。

### 5. 行动清单位置提权

如果要进一步改善最终用户视角，可以把下一步行动清单从完整指南右侧移到路线图下方。

首版可作为第二阶段处理，避免一次性调整太多布局。

## 数据与存储

首版不新增数据库字段，不修改后端 schema。

继续保持：

- 用户反馈只绑定 `route.nextActions` 派生出的行动项。
- 反馈记录继续写入 `ai_feedback_records`。
- 节点关联行动只作为前端展示关系，不持久化。
- `feedbackOutcomeSummary` 继续只读取真实行动反馈，不读取说明性节点。

如果后续发现关联行动需要跨版本稳定追踪，再评估是否增加节点到行动的稳定映射字段。

## 补充注意事项

### 1. 行动项文案要保持语义稳定

阅读指南反馈继承依赖行动项文本和匹配键。后续如果为了去冗余调整 `nextActions` 的展示文案，可能导致历史反馈无法继承。

因此：

- 不要为了 UI 精简随意改写行动清单的核心动词、对象和完成标准。
- 可以调整路线图节点、重点卡和完整指南的展示文案，但行动清单文案应尽量保持稳定。
- 如果必须改写行动文案，应先确认 `buildAiActionItemId`、反馈匹配键和跨版本继承策略不会失效。

这是本次改造里最重要的数据连续性风险。

### 2. 关联行动不要过度智能匹配

节点关联行动只能做确定映射，不做模糊猜测。

推荐：

- `continue-reading` 只关联明确的阅读推进动作。
- `book-review` 只关联明确的复盘或问题回答动作。
- `single-book-output` 只关联明确的输出动作。

不推荐：

- 用关键词模糊搜索所有 `nextActions`。
- 因为标题相似就强行挂载行动。
- 在没有明确对应关系时生成“伪关联行动”。

如果无法可靠匹配，节点详情中不展示关联行动，比展示错误关联更好。

### 3. `是否加入候选书` 节点要单独处理

`是否加入候选书` 是最容易混淆的节点。

默认情况下，它是判断说明，不是行动项，不进入下一步行动清单。

只有当它被明确改写为可执行动作时，才可以进入行动清单，例如：

- `完成本书复盘后，决定是否加入候选书`
- `从候选书中选择 1 本接续阅读`

判断标准仍然是：用户能否执行、能否完成、是否值得反馈。

### 4. 复制与导出要保持结构一致

节点详情复制可以包含关联行动，但不要让导出的内容变成另一份行动清单。

建议保持：

- 节点复制：节点标题、目的、依据、投入、关联行动。
- 行动清单复制：只包含可执行行动、完成标准和用户反馈。
- Markdown 导出：路线结构、完整指南、行动清单分区展示，不把三者混成一个列表。

这样可以避免用户在外部文档里看到重复任务。

### 5. 反馈入口必须唯一

后续任何 UI 改造都要守住这一点：

- 只有下一步行动清单能记录反馈。
- 节点详情弹窗最多展示关联行动，不展示完成状态控件。
- 完整指南中的推进任务和复盘点不单独出现反馈入口。

这能避免一个行动在多个地方有不同状态，导致版本更新时无法判断哪条反馈才是真实阅读成果。

### 6. 空状态不能强行补内容

以下场景必须允许自然为空：

- 没有 `nextActions`。
- 没有 `reviewCheckpoints`。
- 跨书路线没有依赖关系。
- 节点没有可确定的关联行动。
- 当前书没有作者、本地状态或预计投入。

这些情况不应为了填满 UI 强行生成默认行动或伪关联。空状态文案应短，并避免暗示用户缺少操作。

### 7. 布局调整分阶段推进

把下一步行动清单提到路线图下方是合理方向，但不建议和字段去重、关联行动同时一次性完成。

推荐节奏：

- 先完成语义边界和节点关联行动。
- 再观察页面信息密度。
- 最后再调整布局顺序。

这样能降低回归风险，也方便判断每一步改造的真实效果。

## 实施步骤

### 第一阶段：明确边界，减少误读

- 新增本文档并作为后续改造依据。
- 确认节点不进入行动清单。
- 确认节点详情不提供反馈入口。
- 在代码中保持 `ActionChecklistCardList` 只读取 `guideDetails.actions`。

### 第二阶段：节点关联行动

- 扩展 `GuideMapNode.associatedActions`。
- 在单书节点中做保守映射。
- 在跨书节点中只展示当前书相关的接续动作，不做强行匹配。
- 弹窗展示关联行动只读区块。
- 更新复制节点内容格式，包含关联行动。
- 确保没有关联行动时不展示空标题或占位列表。

### 第三阶段：去重展示层

- 拆分 `buildGuidePrescriptionItems` 的职责。
- 新增 `buildGuideFocusItems`。
- 让路线图节点、重点卡、完整指南使用不同视图模型。
- 评估是否把下一步行动清单提到路线图下方。

### 第四阶段：验证反馈成果闭环

- 确认记录进展只出现在行动清单。
- 确认重新生成时只参考行动反馈，不参考节点说明。
- 确认版本详情中的“上次沉淀”不把路线节点当成已完成行动。
- 确认行动项文案调整不会破坏已有反馈继承。

## 测试计划

### 单元测试

覆盖：

- `buildSingleBookGuideNodes` 不把所有节点转为行动项。
- 节点关联行动只在有明确匹配时出现。
- `formatGuideNodeDetail` 包含关联行动但不包含反馈状态。
- `formatReadingRouteActionChecklist` 仍只输出 `guideDetails.actions`。
- 行动清单文案调整后，已有反馈匹配策略仍可继承有效反馈。
- `是否加入候选书` 默认不会进入行动清单。

### 组件 / SSR 测试

覆盖：

- 路线图节点仍可点击查看详情。
- 节点详情弹窗展示关联行动。
- 下一步行动清单只展示可反馈行动。
- 页面不出现两个相同语义的反馈入口。
- 没有关联行动时，节点弹窗不展示空的关联行动区块。
- 节点弹窗中不出现 `记录反馈`、`已完成`、`暂不做` 等反馈控件。

### E2E 测试

覆盖：

- 用户点击节点可以看到完整依据和关联行动。
- 用户只能在下一步行动清单中记录反馈。
- 记录反馈后重新生成时，后端只接收到行动反馈上下文。
- 用户关闭节点弹窗后，焦点回到触发节点。
- Markdown 导出中路线结构、完整指南和行动清单仍然分区清晰。

### 负例测试

需要显式覆盖：

- 说明性节点不会进入行动清单。
- 依赖关系不会被当成行动项。
- 来源统计不会被当成行动项。
- 没有 `nextActions` 时，不会生成伪行动。
- 模糊文本相似时，不会错误关联行动。

## 验收标准

- 路线图仍能快速说明阅读路径。
- 下一步行动清单更聚焦，只包含真实执行项。
- 节点详情能解释节点，但不会诱导用户在节点上记录反馈。
- 单书指南中“先读哪里 / 带问题读 / 读完产出”不再在多个模块中逐字重复。
- 用户反馈记录不被说明性节点污染。
- 行动清单文案调整不会破坏已有反馈继承。
- 空状态下不出现伪行动、空标题或误导性占位。
- 导出和复制内容仍能区分路线结构、节点详情和行动清单。
- 不修改 AI 输出结构、不新增数据库字段、不引入新依赖。

## 不做范围

- 不把阅读指南改造成任务管理系统。
- 不新增节点级完成状态。
- 不新增节点级反馈记录。
- 不把节点内容批量写入 `ai_feedback_records`。
- 不把完整指南搬进节点弹窗。
- 不新增全局历史行动页。

## 最终结论

节点内容不应并入下一步行动清单。

正确的改造方向是：

- 路线图节点负责结构。
- 节点详情负责解释。
- 下一步行动清单负责执行和反馈。
- 完整指南负责深读和核对。

为了减少冗余，应建立“节点关联行动”的只读关系，并逐步拆分路线图、重点卡和完整指南的视图模型，避免不同模块共用同一组摘要文本。
