# 手机端图片成果保存到相册设计文档

## 结论

手机端适合设计“保存到相册”，但只应覆盖图片成果，不应替代当前 Markdown、复盘文档、阅读指南和诊断信息等文件导出。

首版建议：

- 手机端图片成果主操作改为 `保存到相册`。
- 手机端图片成果补充 `分享`，调用系统分享面板或 Web Share 兜底。
- 桌面端继续使用 `下载 PNG` / `导出图片`，不出现 `保存到相册`。
- 文档成果继续走应用导出目录或用户选择的导出目录，不进入相册。

这不是一项 AI 能力，也不需要调用 `gpt-image2` 补资源。当前问题是图片成果在移动端的落点不符合用户预期，而不是缺少视觉资产。

当前代码库已经落地图片成果导出基础层，并接入 Android 原生保存到相册和系统分享，同时补上了 iOS 原生保存到相册与分享的源码骨架；桌面端与浏览器仍保留下载兜底。现在更像是“实现收口 + 真机验收”阶段，不再是纯设计阶段。

## 当前实现状态

- 已完成：图片成果导出基础层、前端 capability 探针、Android 原生保存到相册、系统分享、桌面/浏览器兜底。
- 已完成：`src-tauri/plugins/tauri-plugin-image-artifact-mobile` 自研插件接入，前端通过 `image-artifact-mobile` 命令调用原生能力。
- 已完成：保存到相册失败或误入不支持平台时，导出层会回退到分享；分享不可用时继续回退到浏览器下载。
- 已完成：Android 分享临时文件仅通过 `cacheDir/image-artifacts/` 授权给 `FileProvider`，不开放整个外部存储路径。
- 已确认：当前 Tauri Android 生成工程没有把自研插件的 Android manifest 合并进 app，因此 `FileProvider` 的有效声明仍落在 app 壳层 manifest；长期源头改为插件 `build.rs` 自动注入，插件运行时代码只负责写入临时文件和发起分享。
- 仍需补齐：Android 真机回归、iOS 工具链 / 真机验证、批量保存全部页的完整体验。
- 说明：`src-tauri/gen/android/tauri.settings.gradle` 是生成产物，不是源配置；插件 include、`FileProvider` 注入和 `file_paths.xml` 收窄的真实来源是 `src-tauri/plugins/tauri-plugin-image-artifact-mobile/build.rs`。

## Android 原生配置落点

Android 首版保持两层职责：

- 插件 `build.rs` 把 `FileProvider` 注入到 app 壳层 manifest，authority 使用 `${applicationId}.fileprovider`。
- 自研插件只负责接收 PNG、写入相册、写入临时分享文件、调起系统分享。

当前落点：

| 文件 | 职责 | 维护约束 |
| --- | --- | --- |
| `src-tauri/plugins/tauri-plugin-image-artifact-mobile/build.rs` | Android app 级配置源头 | 负责注入 `FileProvider`、生成受限 `file_paths.xml`，并监听这两个输出以便重建后回补 |
| `src-tauri/gen/android/app/src/main/AndroidManifest.xml` | `FileProvider` 的最终生效位置 | 由插件 build script 维护，避免依赖插件 manifest 合并 |
| `src-tauri/gen/android/app/src/main/res/xml/file_paths.xml` | 限定分享文件可授权路径 | 由插件 build script 维护，只允许 `cache-path image-artifacts/` |
| `src-tauri/plugins/tauri-plugin-image-artifact-mobile/android/src/main/java/.../ImageArtifactMobilePlugin.kt` | 保存到相册与系统分享实现 | 临时文件必须写入 `activity.cacheDir/image-artifacts/` |

后续如果重新生成 Android 工程，必须复核插件 build script 是否已重新写入以上两个 app 壳层文件。只要 `file_paths.xml` 被重置为 `external-path path="."` 或 `cache-path path="."`，就应视为安全回退，需要由 `build.rs` 恢复到 `image-artifacts/` 子目录。

## 背景

当前产品已经有两类图片成果：

- 笔记卡片：单条划线 / 想法图片、组合笔记图片。
- 阅读报告：月报海报、横版报告、多页报告图、长期复盘图。

现有桌面语义基本成立：

- 笔记卡片入口文案偏 `导出图片`。
- 阅读报告预览入口使用 `下载 PNG`、`下载横版 PNG`、`下载当前页`、`下载全部页`。
- 阅读报告 PNG 在 Tauri 运行时走应用导出目录，普通浏览器环境使用下载兜底。
- 笔记卡片当前直接通过浏览器下载 Canvas PNG。

但在手机端，用户对图片成果的自然预期不是“下载到某个目录”，而是：

- 保存到系统相册，方便在微信、朋友圈、小红书、相册里继续使用。
- 直接调起系统分享面板，选择微信、备忘录、文件、图片编辑器等目标。

因此，手机端需要把图片成果从“文件下载心智”调整为“相册 / 分享心智”。

## 目标

1. 让手机端图片成果具备明确的 `保存到相册` 操作。
2. 保留 `分享` 作为高频社交分发路径。
3. 桌面端导出行为不退化，继续保持文件下载和导出目录语义。
4. 不把 Markdown、复盘正文、阅读指南等文档成果混入相册。
5. 不自动写入相册，不在应用启动时请求相册权限。
6. 不改变图片生成内容，只改变图片成果的出口能力。

## 非目标

本次不做：

- 不把 Markdown 渲染成长图保存到相册。
- 不把完整书籍复盘正文做成图片。
- 不自动保存每次生成的报告图。
- 不新增 AI 图片生成流程。
- 不改变当前阅读报告、笔记卡片的 Canvas 绘制内容。
- 不把所有导出统一抽成一个复杂的大流程。
- 不承诺 iOS、Android、Web 三端首版同时完整支持相册写入；能力不可用时必须有兜底。

## 成果类型边界

### 适合保存到相册

| 成果 | 首版处理 | 说明 |
| --- | --- | --- |
| 单条笔记卡片图片 | 支持 | 天然是分享图 |
| 组合笔记卡片图片 | 支持 | 适合沉淀为相册图片 |
| 月报海报 PNG | 支持 | 手机端最强相册场景 |
| 阅读报告横版 PNG | 支持 | 可保存，但移动端主推海报或多页卡片 |
| 多页阅读报告图 | 支持 | 可保存当前页，也可批量保存 |
| 长期复盘分享图 | 支持 | 与阅读报告同一类图片成果 |

### 不适合保存到相册

| 成果 | 处理方式 | 原因 |
| --- | --- | --- |
| 单本笔记 Markdown | 导出目录 / 分享文件 | 结构化文本，不适合进相册 |
| 批量笔记导出 | 导出目录 | 多文件成果，不适合相册 |
| 单本 AI 复盘 Markdown | 导出目录 / 分享文件 | 正文较长，图片化会牺牲可读性 |
| 书籍复盘批量导出 | 导出目录 | 多文档和索引成果 |
| 阅读指南 / 选书决策文档 | 导出目录 / 复制 / 分享文本 | 更接近文档成果 |
| 诊断信息 | 导出目录 | 不应进入个人相册 |

## 交互设计

### 术语统一

为了避免文案和实现概念打架，后续统一使用以下术语：

| 术语 | 含义 |
| --- | --- |
| 图片成果 | 所有可视化 PNG 导出结果 |
| 文档成果 | Markdown、复盘正文、指南、诊断等文本导出结果 |
| 保存到相册 | 写入系统相册 / 照片库 |
| 分享 | 调起系统分享面板 |
| 导出文件 | 写入应用导出目录或浏览器下载 |
| capability | 当前运行时可用能力探针 |
| intent | 用户显式点击的导出意图 |
| source | 最终实际落点 |

术语约束：

- 不把“保存到相册”简称成“下载”。
- 不把“导出文件”描述成“保存到相册”。
- 不把“分享”与“复制图片”混写。
- UI 上尽量只展示一个主要动作词，避免一个按钮塞三种出口。

### 平台文案

| 平台 | 图片成果主操作 | 次操作 | 兜底 |
| --- | --- | --- | --- |
| 手机原生壳 | 保存到相册 | 分享 | 导出文件 |
| 手机浏览器 / WebView | 分享 | 下载图片 | 长按图片保存，作为说明兜底 |
| 桌面 Tauri | 下载 PNG / 导出图片 | 打开导出位置 | 不展示保存到相册 |
| 桌面浏览器 | 下载 PNG / 导出图片 | 无 | 浏览器下载 |

用户可见文案建议：

- 手机端报告图：`保存到相册`
- 手机端笔记卡片：`保存图片`
- 手机端多页报告：`保存当前页`、`保存全部页`
- 分享入口：`分享`
- 桌面端：继续使用 `下载 PNG`、`下载横版 PNG`、`导出图片`

### 文案矩阵

| 场景 | 手机端文案 | 桌面端文案 | 备注 |
| --- | --- | --- | --- |
| 单张报告海报 | 保存到相册 | 下载 PNG | 手机端强调相册落点 |

| 系统分享 | 分享 | 分享 | 仅在能力可用时展示 |

不建议使用的文案：

| 文案 | 原因 |
| --- | --- |
| 一键保存 | 隐藏权限和平台差异 |

| 同步到相册 | 容易误解为持续同步 |
| 导入相册 | 用户语义不自然 |
| 生成分享图 | 当前动作是保存 / 分享，不是重新生成 |


多页报告卡组：

```text
[保存当前页]  [保存全部页]  [分享]
```

桌面端保持现状：

```text
[下载 PNG]

[下载当前页] [下载全部页]
```

### 笔记卡片

当前按钮文案是 `导出图片`。建议按平台切换：

- 手机端：`保存图片`
- 桌面端：`导出图片`

组合笔记按钮同理：

- 手机端：`保存组合图`
- 桌面端：`导出组合图` 或继续沿用当前按钮语义

首版不需要新增复杂的分享预览页。现有 Canvas 生成后直接进入保存 / 分享流程即可。

### 触屏与无障碍

手机端图片成果操作需要满足已有触屏基线：

- 所有主操作按钮最小触控热区为 44 × 44。
- `保存到相册`、`分享`、`导出文件` 不应只依赖图标表达。
- 批量保存时按钮进入 loading 状态后不可重复点击。
- 结果摘要不能只依赖 toast，尤其是多页报告。
- 权限错误和平台不支持错误需要可被屏幕阅读器读到。
- 横屏短视口下，报告弹窗操作区不能被底部导航或键盘遮挡。

## 权限与状态

### 权限触发时机

只在用户点击 `保存到相册` 时请求相册写入权限。

不在以下时机请求：

- 应用启动。
- 进入统计页。
- 打开报告弹窗。
- 生成预览图。
- 同步笔记或生成 AI 复盘。

### 成功状态

保存成功后展示 toast：

```text
已保存到相册：{fileName}
```

如果平台只能返回成功但不能返回最终路径，不展示路径。

### 权限拒绝

如果用户拒绝相册权限：

```text
没有相册保存权限。你可以改用分享，或导出为文件。
```

可提供：

- `分享`
- `导出文件`
- `去设置开启权限`，仅在原生平台能稳定打开权限设置时展示

### 权限文案草案

权限文案应解释“为什么需要写入相册”，不解释业务功能细节。

建议：

| 场景 | 文案 |
| --- | --- |
| 系统权限说明 | 用于将你生成的阅读报告或笔记图片保存到相册。 |
| 权限前置说明 | 保存到相册需要照片写入权限，仅会保存你当前生成的图片。 |
| 权限拒绝后 | 没有相册保存权限。你可以改用分享，或导出为文件。 |
| 平台不支持 | 当前环境不支持直接保存到相册。 |
| 去设置引导 | 你可以在系统设置中开启照片写入权限后重试。 |

文案约束：

- 不承诺读取相册内容。
- 不使用“同步”“自动保存”等持续性表达。
- 不把权限说明写成营销文案。
- 不在权限弹出前展示长段解释，避免打断主流程。

### 设置页入口

首版不建议在设置页新增相册权限管理项。

原因：

- 相册权限只和图片成果导出有关，不是全局基础配置。
- 在设置页常驻展示会让用户误以为应用会持续访问相册。
- 系统权限状态读取在不同平台上不稳定，首版容易增加复杂度。

如果后续真机反馈显示用户频繁遇到权限拒绝，可以再考虑在导出失败结果区提供 `去设置开启权限`，而不是新增全局设置项。

### 对外文案全集

建议把手机端和桌面端的对外文案固定下来，后续实现只做能力切换，不再临时发明新说法。

| 场景 | 文案 |
| --- | --- |
| 手机端主按钮 | 保存到相册 |
| 手机端分享按钮 | 分享 |
| 手机端文件兜底 | 导出文件 |
| 桌面端主按钮 | 下载 PNG / 导出图片 |
| 单张成功 | 已保存到相册：{fileName} |
| 分享面板打开 | 已打开分享：{fileName} |
| 导出成功 | 已导出：{fileName} |
| 权限前置说明 | 保存到相册需要照片写入权限，仅会保存你当前生成的图片。 |
| 权限拒绝 | 没有相册保存权限。你可以改用分享，或导出为文件。 |
| 平台不支持 | 当前环境不支持直接保存到相册。 |
| Canvas 失败 | 生成图片失败，请稍后重试。 |
| 保存失败 | 保存到相册失败。 |
| 分享取消 | 已取消分享。 |
| 批量部分成功 | 已保存 3 张，1 张失败。 |

文案约束：

- 不在成功文案里重复解释技术细节。
- 不把错误文案写成产品教程。
- 不出现“自动”“同步”“一键”这类容易误导的词。
- 笔记卡片和阅读报告的成功文案可以不同，但失败文案尽量统一。

### 部分成功

多页报告批量保存时可能出现部分成功。结果应展示：

```text
已保存 3 张，1 张失败。
```

失败时保留当前预览状态，允许重试失败项。首版可以先重试整批，但错误文案必须说明已保存数量，避免用户重复保存全部图片后产生混乱。

### 错误分类

| 类型 | 用户文案 | 后续动作 |
| --- | --- | --- |
| 平台不支持相册写入 | 当前环境不支持直接保存到相册。 | 展示 `分享` 或 `导出文件` |
| 用户拒绝权限 | 没有相册保存权限。 | 展示 `分享`、`导出文件`，可选 `去设置开启权限` |
| Canvas 生成失败 | 生成图片失败，请稍后重试。 | 保留当前页面状态 |
| 原生保存失败 | 保存到相册失败。 | 展示原始错误摘要和兜底操作 |
| 系统分享取消 | 已取消分享。 | 不作为错误 toast 强提示 |
| 浏览器下载失败 | 下载图片失败。 | 提示换用桌面端或系统分享 |

错误处理原则：

- 用户主动取消分享不计入失败。
- 权限拒绝不自动再次弹权限申请，避免打扰。
- 批量保存遇到单张失败时继续处理后续图片，再汇总结果。
- 不把底层技术错误直接暴露为主文案，可以在详情或日志里保留原始错误。

### 错误码草案

建议内部错误码保持短小稳定，便于日志、埋点和测试断言。

| 错误码 | 含义 | 是否可重试 |
| --- | --- | --- |
| `IMAGE_ARTIFACT_UNSUPPORTED` | 当前运行时不支持目标出口 | 是 |
| `IMAGE_ARTIFACT_PERMISSION_DENIED` | 相册或分享所需权限被拒绝 | 是 |
| `IMAGE_ARTIFACT_CANVAS_FAILED` | Canvas 转 PNG 失败 | 是 |
| `IMAGE_ARTIFACT_SHARE_CANCELLED` | 用户主动取消分享 | 否 |
| `IMAGE_ARTIFACT_SAVE_FAILED` | 原生保存到相册失败 | 是 |
| `IMAGE_ARTIFACT_DOWNLOAD_FAILED` | 浏览器下载失败 | 是 |

错误码约束：

- 错误码只用于内部识别，不直接暴露给普通用户。
- 同一错误码应尽量对应同一类恢复动作。
- 批量场景可以记录每张图片的错误码，但结果展示仍应汇总。

### 运行时分流规则

首版不建议仅按屏幕宽度切换，而应按“平台能力 + 用户意图”双条件判断：

1. 用户显式点击 `保存到相册`，才走相册写入。
2. 用户显式点击 `分享`，才走系统分享面板。
3. 如果当前运行时不支持相册写入，就隐藏该入口或降级为 `导出文件`。
4. 如果当前运行时不支持分享，就只保留文件导出或下载。
5. 浏览器环境始终保留下载兜底，不把失败直接暴露给用户。

这样可以避免“手机窄屏但其实是桌面浏览器”“平板但没有相册权限”“WebView 但不支持分享”的误判。

### 组件拆分建议

首版建议把页面内的图片导出相关逻辑拆成三层：

#### 1. 入口层

负责展示按钮和收集用户意图。

典型组件：

- `ReportGenerationWizardDialog`
- `BookNotesPage` 的卡片操作区

职责：

- 根据 capability 决定显示哪个按钮。
- 把 `保存到相册`、`分享`、`导出文件` 变成明确意图。
- 不直接处理 Canvas、权限或下载细节。

#### 2. 导出层

负责把 Canvas 变成 PNG，并路由到正确出口。

典型模块：

- `src/lib/image-artifact-export.ts`
- `src/features/reading-stats/report-image-export.ts`

职责：

- 接收 `canvas`、`fileName`、`intent`。
- 执行 `toBlob`、Data URL 转换、浏览器下载或原生命令。
- 统一返回成功、失败或部分成功结果。

#### 3. 结果层

负责显示导出结果和兜底操作。

典型区域：

- 报告弹窗结果区
- 笔记卡片 toast

职责：

- 展示保存成功、权限拒绝、分享取消、部分成功。
- 给出下一步动作，比如 `分享`、`导出文件`、`重试`。
- 不再重复导出判断逻辑。

拆分约束：

- 入口层不要知道原生命令细节。
- 导出层不要知道页面布局。
- 结果层不要重新触发导出流程，除非用户显式点重试。

### 入口组件 Props 草案

如果后续抽共享操作条，可以先保持很薄：

```ts
type ImageArtifactActionBarProps = {
  capabilities: ImageArtifactCapabilities;
  isBusy: boolean;
  primaryLabel: string;
  onSaveToAlbum?: () => void;
  onShare?: () => void;
  onExportFile?: () => void;
};
```

约束：

- `ImageArtifactActionBar` 不接收 `canvas`。
- `ImageArtifactActionBar` 不调用原生命令。
- `primaryLabel` 由页面结合运行时和成果类型决定。
- 按钮显隐由 capability 和 handler 是否存在共同决定。

首版也可以不抽组件，只按这个 Props 边界约束页面内实现。等阅读报告和笔记卡片出现足够重复后，再抽共享组件。

### 建议抽象

新增轻量图片成果导出层，不和 Markdown 导出混用：

```ts
type ImageArtifactExportIntent =
  | "saveToAlbum"
  | "share"
  | "exportFile";

type ImageArtifactExportSource =
  | "photoAlbum"
  | "shareSheet"
  | "exportDir"
  | "browserDownload";
```

建议补充能力探针：

```ts
type ImageArtifactCapabilities = {
  canSaveToAlbum: boolean;
  canShareImage: boolean;
  canExportFile: boolean;
  canBrowserDownload: boolean;
};
```

建议文件：

```text
src/lib/image-artifact-export.ts
```

职责：

- 接收 `HTMLCanvasElement`、文件名、成果类型和用户意图。
- 统一把 Canvas 转成 PNG Blob / Data URL。
- 根据平台能力分发到相册、分享面板、应用导出目录或浏览器下载。
- 返回统一结果，供 toast 和错误提示使用。

不建议把它命名为通用 `exportArtifact`。首版只服务图片成果，保持边界简单。

### 前端接口草案

```ts
export async function getImageArtifactCapabilities(): Promise<ImageArtifactCapabilities>;

export async function exportImageArtifact(options: {
  canvas: HTMLCanvasElement;
  fileName: string;
  intent: ImageArtifactExportIntent;
  fallbackIntent?: ImageArtifactExportIntent;
}): Promise<{
  fileName: string;
  source: ImageArtifactExportSource;
  path?: string;
  exportedAt?: string;
}>;
```

接口约束：

- `intent` 必须来自用户点击，不做自动推断保存到相册。
- `fallbackIntent` 只用于权限拒绝或平台不支持后的明确兜底。
- 返回 `source` 由底层真实结果决定，不能只按用户点击的按钮推断。
- Canvas 转 PNG 失败时直接返回错误，不进入分享或文件导出。

### 状态模型草案

建议前端和原生层共享同一组结果语义：

```ts
type ImageArtifactExportStatus =
  | "idle"
  | "preparing"
  | "saving"
  | "sharing"
  | "exporting"
  | "success"
  | "error"
  | "partialSuccess";

type ImageArtifactExportResult = {
  fileName: string;
  source: ImageArtifactExportSource;
  status: ImageArtifactExportStatus;
  path?: string;
  exportedAt?: string;
  errorCode?: string;
  retryable?: boolean;
};
```

语义说明：

- `status` 由导出层统一产出，不由页面自己猜。
- `errorCode` 对应内部错误码草案。
- `retryable` 只描述当前结果是否还能再试一次，不等于是否值得重试。
- `partialSuccess` 只用于批量场景，不用于单张图片。

如果未来要把结果区做成更完整的面板，可以直接复用这组状态，不需要再造另一套模型。

### 后端 / 原生命令草案

如果 Tauri 移动端需要自定义命令，建议先控制在两个窄接口：

```rust
save_image_to_album(file_name: String, data_url: String) -> ImageArtifactNativeResult

share_image_file(file_name: String, data_url: String) -> ImageArtifactNativeResult
```

结果结构建议：

```rust
struct ImageArtifactNativeResult {
    file_name: String,
    exported_at: String,
    path: Option<String>,
}
```

设计约束：

- 命令只接收 PNG Data URL，不接收任意本地路径，减少路径访问风险。
- 命令只保存当前用户显式生成的图片，不读取数据库或远端数据。
- Android / iOS 权限差异封装在原生层，前端只消费成功、拒绝、不可用、失败四类结果。
- 桌面端不需要实现相册写入命令。

### 平台能力分层

建议按能力判断，而不是只按屏幕宽度判断：

1. 原生移动端支持相册写入：显示 `保存到相册`。
2. 支持系统分享文件：显示 `分享`。
3. 支持应用导出目录：显示 `导出文件` 或桌面 `下载 PNG`。
4. 只支持浏览器下载：使用现有 `<a download>` 兜底。

UI 文案可以结合视口和运行时：

- 粗指针 / 窄屏优先手机文案。
- Tauri 桌面优先文件文案。
- Web 环境只展示实际可执行的能力，不展示无法完成的相册保存。

### 与导出目录的关系

`导出保存位置` 仍然只管理文件导出目标。

`保存到相册` 是一次用户显式触发的系统相册写入，不应改变导出保存位置，也不应把相册路径写入导出设置。

如果用户选择 `导出文件`，才继续走当前应用导出目录或浏览器下载兜底。

### 文件命名规则

图片成果保存时，文件名应继续沿用当前“书名 + 成果类型”的可读风格，不因为出口从文件变成相册就改成纯随机串。

建议：

- 笔记卡片：`{书名}-划线.png`、`{书名}-想法.png`
- 组合笔记：`{书名}-笔记组合.png`
- 月报海报：沿用现有月报文件名规则
- 横版报告：在现有文件名后附加 `-16-9` 或等价标识
- 多页报告：每页文件名保持可区分，方便相册中排序

约束：

- 文件名必须稳定、可读、可排序。
- 同一本书多次导出时可以重复，但要保证每次生成的结果在本地路径或相册展示中不容易混淆。
- 中文书名需要保留可读性，不要过早转成纯拼音或哈希。

### 导出状态机

建议前端把图片成果导出收敛成一个小状态机，而不是散在每个页面里自己维护：

```text
idle
  -> preparing
  -> saving | sharing | exporting
  -> success
  -> error
```

说明：

- `preparing`：Canvas 转 PNG、组装文件名、检查能力。
- `saving`：写入相册。
- `sharing`：调起系统分享。
- `exporting`：下载到文件或导出目录。
- `success`：展示 toast，必要时附带 `fileName`。
- `error`：展示可恢复错误和兜底操作。

批量场景额外增加：

- `partialSuccess`
- `retryableError`

这样页面只负责“触发什么意图”，导出层负责“最终落到哪里”。

### 数据流

单张图片成果的数据流建议固定为：

```text
用户点击
  -> 页面传入 Canvas 与文件名
  -> image-artifact-export 转 PNG Blob
  -> 根据 intent 与 capability 选择出口
  -> 相册 / 分享 / 导出目录 / 浏览器下载
  -> 返回统一结果
  -> 页面展示 toast 或弹窗结果
```

批量图片成果的数据流建议固定为：

```text
用户点击保存全部
  -> 生成待处理页列表
  -> 按顺序逐张导出
  -> 每张记录 success / error
  -> 汇总成功数和失败数
  -> 允许重试失败项
```

设计约束：

- 页面层不直接关心图片最终落到哪个系统 API。
- 导出层不负责重新绘制图片，只消费已经生成的 Canvas。
- 原生层不读取业务数据，只接收最终 PNG 数据。
- 批量导出不并发写入相册，首版按顺序执行，降低权限和内存风险。

### 性能与资源约束

图片导出应避免把大对象长期留在 React 状态或本地存储中。

约束：

- Canvas 转 Blob 后尽快释放临时 Object URL。
- 不把 Base64 Data URL 写入 localStorage、IndexedDB 或 SQLite。
- 不把大图 Base64 放进 React state，只在调用命令时短暂传递。
- 批量保存按顺序处理，首版不做并发。
- 多页报告保存全部时需要展示进度，避免用户以为页面卡死。
- Canvas 生成失败要尽早中断当前图片，不继续进入权限或分享流程。

大图场景建议：

- 横版报告和多页报告优先复用现有 Canvas 尺寸，不在保存阶段再次放大。
- 如果未来支持超高清导出，应作为独立设置，不默认启用。
- 分享临时文件保存到应用缓存目录，分享完成或下次启动时清理。

### 风险与控制

| 风险 | 表现 | 控制方式 |
| --- | --- | --- |
| 权限误触发 | 打开页面就弹相册权限 | 只在点击 `保存到相册` 时申请 |
| 文案误导 | Web 环境显示 `保存到相册` 但无法执行 | capability 控制入口显隐 |
| 出口混乱 | 同一按钮有时下载、有时保存相册 | 用户意图和实际 source 都要明确 |
| 批量卡顿 | 保存全部页时页面无反馈 | 顺序处理并展示进度 |
| 内存膨胀 | Base64 大图长期驻留 | 不持久化 Base64，及时释放临时 URL |
| 隐私越界 | 应用读取用户相册 | 只写入最终 PNG，不读取相册内容 |
| 重复保存 | 用户连续点击导致多份图片 | loading 状态禁用重复提交 |
| 桌面回归 | 桌面下载入口被相册文案替换 | 桌面端测试固定检查下载文案 |

## 数据与隐私

- 图片在本地 Canvas 生成，不需要新增网络请求。
- 不上传图片到 AI 服务。
- 不把 API Key、数据库路径、Cookie、原始接口响应写入图片。
- 批量保存图片时不自动同步微信读书远端数据。
- 保存到相册只应写入用户显式生成的最终 PNG，不写入中间缓存。

## 配置与数据库边界

首版不需要新增数据库表、字段或迁移脚本。

不新增：

- 相册保存历史表。
- 图片成果索引表。
- 相册权限状态缓存。
- 图片 Base64 缓存字段。
- 用户相册路径字段。

配置侧只允许引入能力开关，不引入用户可配置项：

- `enableImageArtifactRefactor`
- `enableImageShare`
- `enableSaveToAlbum`

约束：

- 能力开关用于发布控制，不作为用户偏好设置。
- 不在设置页新增“默认保存到相册”。
- 不记录用户每次选择了相册、分享还是导出文件作为长期偏好。
- 不因为一次权限拒绝就永久隐藏保存入口，权限状态以系统为准。

如果后续要做用户偏好，例如“总是优先分享”或“默认导出文件”，需要另起设计，并证明它不会和平台 capability 冲突。

## 日志与诊断

导出相关日志只记录最小必要信息。

建议记录：

- `intent`
- `source`
- `fileName`
- `platform`
- `errorCategory`
- `retryable`

不建议记录：

- PNG Blob 原文
- Base64 Data URL
- 用户相册路径
- 原始系统分享内容
- API Key 或同步数据

原则：

- 用户可见错误和开发日志分开。
- 开发日志可以保留分类和简短栈信息，但不要包含图片本体。
- 批量导出日志只记录统计，不逐帧持久化失败图片内容。

## 成果记录边界

`保存到相册` 是外部落点，不等同于进入应用内成果库。

首版建议：

- 保存成功后只展示 toast 或弹窗内结果状态。
- 阅读报告弹窗可以保留本次导出结果，例如文件名、保存方式、完成时间。
- 不新增“相册图片成果库”。
- 不把保存到相册的图片路径写入本地数据库。
- 不把用户相册路径作为可追踪历史记录保存。

原因：

- 相册是系统级个人媒体库，应用不应假设后续还能稳定访问该图片。
- 用户可能在相册中删除、编辑、移动或转发图片。
- 如果应用内再维护一份图片成果库，会带来同步、删除、权限和隐私复杂度。

如果后续要做应用内图片成果历史，应另起设计，不和本次相册保存混在一起。更合理的对象不是“相册记录”，而是“已生成的报告配置 / 预览缓存 / 导出记录”，并且需要明确生命周期。

## 结果展示位置

保存动作完成后，结果反馈应尽量贴近原入口。

### 单张图片

- 笔记卡片导出成功后，在当前卡片操作区附近展示 toast。
- 文案只给出结果，不展开导出链路。
- 失败时保留当前按钮所在状态，允许再次点击。

### 阅读报告

- 报告弹窗内显示当前步骤的导出状态。
- 如果是多页报告，结果摘要放在弹窗结果区，不要只依赖全局 toast。
- 已保存页数、失败页数和可重试动作应在同一处展示。

### 统一原则

- toast 只负责短反馈。
- 步骤型弹窗负责结果详情。
- 页面级历史不新增“相册成果记录”。
- 如果后续真的需要导出历史，应复用已有导出目录概念，而不是读取相册。

## 平台权限清单

实现前需要按目标平台补齐权限配置和能力探针。

### Android

需要确认：

- 当前 Tauri Android 壳层的目标 SDK。
- 保存图片到系统相册是否需要媒体写入权限，或是否可通过系统 MediaStore / 分享 Intent 完成。
- 分享图片是否需要先写入应用临时缓存目录，再通过 content URI 授权给目标应用。
- 临时分享文件是否在分享完成后清理。

设计约束：

- 不在启动时申请相册或媒体权限。
- 权限拒绝后不阻塞文件导出兜底。
- 分享用临时文件不进入长期导出目录。

### iOS

当前决策：

- `build.rs` 注入 `NSPhotoLibraryAddUsageDescription`；旧版 iOS 兜底同时注入 `NSPhotoLibraryUsageDescription`。
- iOS 14 及以上优先申请 add-only 照片权限，不读取用户照片库。
- iOS 13 走系统兼容权限路径，仅用于写入当前 PNG。
- 系统分享使用缓存目录中的临时 PNG 文件，不进入长期导出目录。
- 保存失败映射为 `image_artifact_save_failed`，权限拒绝映射为 `image_artifact_album_permission_denied`，用户取消分享返回 `cancelled: true`。

设计约束：

- 首选“只写入 / 添加照片”的最小权限。
- 不读取用户相册内容。
- 不在应用内保存相册 asset identifier，除非后续有明确的成果管理需求。

### Web / 浏览器

需要确认：

- 当前浏览器是否支持 `navigator.share`。
- 当前浏览器是否支持文件类型分享。
- 不支持 Web Share 时，是否仍可通过 `<a download>` 下载 PNG。

设计约束：

- Web 环境不展示不可执行的 `保存到相册`。
- Web Share 失败或不可用时回退浏览器下载。
- 不要求用户长按保存作为唯一主路径。

## 运行时兼容矩阵

| 运行时 | 保存到相册 | 分享 | 导出文件 | 浏览器下载 | 备注 |
| --- | --- | --- | --- | --- | --- |
| Android 原生壳 | 目标支持 | 目标支持 | 目标支持 | 否 | 首先补原生命令 |
| iOS 原生壳 | 目标支持 | 目标支持 | 目标支持 | 否 | 优先最小权限 |
| Android / iOS WebView | 目标支持，若能力已接入 | 目标支持，若能力已接入 | 可作为兜底 | 作为兜底 | 取决于壳层能力 |
| 桌面 Tauri | 不展示 | 可选 | 支持 | 支持 | 保持现有下载语义 |
| 移动浏览器 | 不展示 | 仅在 Web Share 可用时展示 | 不展示 | 支持 | 不做相册假入口 |
| 桌面浏览器 | 不展示 | 可选 | 不展示 | 支持 | 继续浏览器下载 |

- 没有能力就不展示，不用灰掉的假按钮占位。
- 有明确兜底时可以展示第二入口，但不要把兜底伪装成主路径。
- 桌面端不因为窄屏就切成手机语义。

## 按钮状态规范

### 主按钮

主按钮只承载一个最强动作：

- 手机端图片成果：`保存到相册`，或能力未接入时 `保存图片` / `保存当前页`。
- 桌面端图片成果：`下载 PNG` / `导出图片`。

主按钮状态：

- `default`：可点击。
- `loading`：显示生成中，禁止重复点击。
- `success`：恢复默认状态，展示结果。
- `error`：恢复默认状态，展示错误和兜底动作。

### 次按钮

次按钮用于同级兜底，不应与主按钮语义冲突：

- `分享`
- `导出文件`

次按钮规则：

- 仅在能力可用时展示。
- 不可用时优先隐藏，不做长驻禁用态。
- 当主操作失败时，可以保留为可立即切换的备选项。

### 批量按钮

批量保存按钮需要额外状态：

- `preparing`：计算待处理项。
- `saving`：逐项执行中。
- `partialSuccess`：部分成功，显示成功和失败数量。
- `retryableError`：允许只重试失败项，或重试整批。

批量按钮在执行中必须禁用，防止重复写入相册或重复分享同一张图。

## 典型流程

### 手机端保存月报海报

1. 用户打开阅读报告弹窗。
2. 页面展示 `保存到相册` 和 `分享`。
3. 用户点击 `保存到相册`。
4. 系统仅在此时申请相册权限。
5. 成功后显示 `已保存到相册`。
6. 失败后保留当前弹窗，并提供 `分享` 或 `导出文件` 兜底。

### 手机端保存笔记卡片

1. 用户在笔记页点击单条卡片导出。
2. 页面根据能力展示 `保存图片` 或 `导出图片`。
3. 用户完成操作后获得 PNG 结果。
4. 若原生能力不可用，浏览器下载兜底仍可工作。

### 桌面端导出横版报告

1. 用户在桌面打开报告生成入口。
2. 页面展示 `下载 PNG` 或 `下载横版 PNG`。
3. 用户点击后直接下载到当前导出路径。
4. 不出现相册、权限或分享面板。

### 多页报告批量保存

1. 用户选择保存全部页。
2. 页面进入 `preparing` 和 `saving`。
3. 每页按顺序执行。
4. 完成后展示成功数和失败数。
5. 用户可对失败页重试。

## 开发任务拆分

> 说明：下面的“MVP 范围锁定 / 任务拆分 / 里程碑”保留为当时的实施路线记录，实际状态以“当前实现状态”为准。

### MVP 范围锁定

MVP 只解决“图片成果在移动端如何被用户拿走”这个问题。

MVP 包含：

- 阅读报告 PNG 的统一导出基础层。
- 笔记卡片 PNG 的统一导出基础层。
- 能力探针。
- 手机端可执行入口的文案切换。
- 浏览器下载兜底。

MVP 可选：

- Web Share。
- 原生系统分享。

MVP 不包含：

- iOS 原生相册保存的完整实现，除非平台适配已经准备好。
- 多页报告批量保存全部页。
- 应用内图片成果历史。
- Markdown 转长图。
- 自动生成或自动保存任何图片。

MVP 判定标准：

- 现有桌面 PNG 下载不退化。
- 图片成果导出逻辑完成 DRY 收敛。
- 未接入能力的平台不会展示不可执行的相册入口。
- 后续接入相册保存时不需要重写页面绘图逻辑。

### 优先级

| 优先级 | 范围 | 理由 |
| --- | --- | --- |
| P0 | 抽出图片成果导出基础层 | 不改变用户行为，先降低重复和后续接入成本 |
| P0 | 阅读报告 PNG 复用基础层 | 当前已有导出目录能力，是最稳定的迁移对象 |
| P1 | 笔记卡片复用基础层 | 当前仍在页面内直接下载，适合收敛 |
| P1 | 能力探针与平台化文案 | 为移动端相册和分享做入口控制 |
| P1 | Web Share / 系统分享兜底 | 分享是手机端高频路径，但必须受能力控制 |
| P2 | 原生保存到相册 | 涉及平台权限和原生命令，需单独验证 |
| P2 | 多页报告批量保存汇总 | 复杂度高，等单张链路稳定后接入 |

排序原则：

- 先做无用户行为变化的 DRY 收敛。
- 再做能力探针和文案切换。
- 最后接入平台权限相关能力。
- 批量保存放在单张保存稳定之后。

### 任务 1：抽出图片成果导出基础层

目标：

- 从笔记卡片和阅读报告中抽出 Canvas 转 PNG、文件名规范化、浏览器下载兜底。
- 保持现有用户行为不变。
- 不引入相册权限和系统分享能力。

建议改动：

- 新增 `src/lib/image-artifact-export.ts`。
- 为 `canvasToPngBlob`、`ensurePngFileName`、`downloadBlob` 增加单测。
- 让 `src/features/reading-stats/report-image-export.ts` 复用新基础层。
- 暂不移动笔记卡片绘制函数，只替换末尾 `downloadCanvas` 的导出实现。

验收：

- 桌面端阅读报告仍能下载 PNG。
- 浏览器环境仍能下载 PNG。
- 笔记卡片导出文件名和现有行为保持一致。

### 任务 2：加入能力探针和平台化文案

目标：

- UI 根据能力展示正确文案。
- 在没有原生相册能力前，不展示不可执行的 `保存到相册`。
- 手机端可以先展示 `保存图片` / `分享` 的可用入口，但相册入口必须由能力探针控制。

建议改动：

- 新增 `getImageArtifactCapabilities`。
- 阅读报告弹窗根据 capability 渲染按钮文案。
- 笔记卡片按钮根据 capability 和粗指针环境渲染 `保存图片` 或 `导出图片`。
- 补组件测试，确保桌面端不出现相册文案。

验收：

- 当前桌面 Tauri 行为不变。
- 普通浏览器行为不变。
- 不支持相册能力时不会展示 `保存到相册`。

### 任务 3：接入分享能力

目标：

- 优先支持系统分享或 Web Share。
- 分享不可用时回退下载。
- 用户取消分享不作为错误处理。

建议改动：

- 前端 `exportImageArtifact` 支持 `intent: "share"`。
- Web 环境检测 `navigator.canShare` 和 `navigator.share`。
- 原生端如果暂未实现分享命令，则 capability 返回不可用。

验收：

- 支持分享的环境可以调起分享。
- 不支持分享的环境不会展示无效入口。
- 分享取消不出现失败 toast。

### 任务 4：接入移动端保存到相册

目标：

- 只在原生移动端支持 `保存到相册`。
- 权限只在用户点击时申请。
- 权限拒绝时回退 `分享` 或 `导出文件`。

建议改动：

- 后端 / 原生层新增 `save_image_to_album` 命令。
- 前端 capability 增加相册能力检测。
- 阅读报告、笔记卡片接入 `intent: "saveToAlbum"`。
- 补权限拒绝和保存失败的错误归一化。

验收：

- 第一次点击保存才触发权限。
- 保存成功有明确 toast。
- 权限拒绝后不重复弹权限申请。

### 任务 5：批量保存和结果汇总

目标：

- 多页报告支持保存当前页和保存全部页。
- 批量保存有进度、部分成功和失败汇总。
- 不阻塞单页报告和笔记卡片主流程。

建议改动：

- 为多页报告增加批量执行状态。
- 单张失败后继续处理后续图片。
- 结果 toast 展示成功张数和失败张数。

验收：

- 全部成功、部分失败、全部失败都有明确状态。
- 用户可以在失败后重试。
- 已生成的预览状态不被清空。

## 首个实现切片（路线回放）

按原实施路线，首个 PR 建议只做任务 1，不做任务 2-5。

首个 PR 范围：

- 新增 `src/lib/image-artifact-export.ts`。
- 复用现有阅读报告 PNG 导出逻辑。
- 让笔记卡片导出复用基础层。
- 增加基础单测。

首个 PR 不做：

- 不改按钮文案。
- 不展示 `保存到相册`。
- 不新增原生命令。
- 不改 Tauri 权限。
- 不增加系统分享。
- 不改 Markdown 导出。

这样第一步只完成 DRY 收敛，不改变用户可见能力，风险最小。

### 首个 PR 详细步骤

建议按以下顺序开发：

1. 新增 `src/lib/image-artifact-export.ts`。
2. 迁入通用 `canvasToPngBlob`、`ensurePngFileName`、`downloadBlob`。
3. 保留 `exportCanvasAsReportImage` 的外部函数签名不变。
4. 让 `report-image-export.ts` 调用新基础层，继续在 Tauri 环境走 `exportReportImage`。
5. 将 `BookNotesPage.tsx` 的 `downloadCanvas` 改为调用新基础层。
6. 保持 `exportNoteCardImage` 和 `exportNoteGroupImage` 的返回值仍为 `fileName`。
7. 增加 `src/lib/image-artifact-export.test.ts`。
8. 跑现有阅读报告导出测试和新增测试。

拆分原则：

- 第一轮只移动导出基础逻辑，不移动 Canvas 绘制逻辑。
- 不引入 `intent` / `capability` 的完整状态机，避免第一 PR 过大。
- 新工具函数只暴露当前需要的最小 API。

### 首个 PR 推荐 API

首个 PR 可以先提供两个函数：

```ts
export async function canvasToPngBlob(
  canvas: HTMLCanvasElement,
  errorMessage: string
): Promise<Blob>;

export async function downloadCanvasAsPng(
  canvas: HTMLCanvasElement,
  fileName: string,
  errorMessage: string
): Promise<string>;
```

如果阅读报告仍需要 Tauri 导出目录能力，可以再补一个内部工具：

```ts
export function ensurePngFileName(fileName: string, fallback?: string): string;
```

暂不建议首个 PR 一次性暴露 `exportImageArtifact`。该接口依赖 `intent`、`capability`、分享和相册能力，放到任务 2 更合适。

### 代码命名约束

建议命名保持业务边界清晰：

- `image-artifact-export`：图片成果导出基础层。
- `canvasToPngBlob`：只负责 Canvas 到 Blob。
- `downloadCanvasAsPng`：只负责浏览器下载兜底。
- `exportCanvasAsReportImage`：保留阅读报告语义。
- `exportNoteCardImage` / `exportNoteGroupImage`：保留笔记卡片语义。

不建议命名：

- `saveImage`：无法区分保存到相册还是文件下载。
- `shareImage`：当前首个 PR 不做分享。
- `exportArtifact`：范围过宽，会把文档成果也卷进来。
- `mobileExport`：首个 PR 不应绑定移动端。

### 回滚方案

如果首个 PR 出现导出回归，应能快速回滚到旧逻辑：

- `report-image-export.ts` 可以恢复本地 `canvasToPngBlob`、`downloadBlob`。
- `BookNotesPage.tsx` 可以恢复页面内 `downloadCanvas`。
- 新增 `image-artifact-export.ts` 可以暂时保留未使用，也可以整体删除。

首个 PR 不改 UI 文案、不改原生命令、不改权限配置，因此回滚风险应限制在图片下载路径，不影响 Markdown 导出、AI 复盘或同步流程。

### 首个 PR 完成定义

- 阅读报告 PNG 导出测试通过。
- 新增图片导出工具函数单测通过。
- 笔记卡片仍能生成 PNG 文件名。
- 桌面端按钮文案不变。
- 浏览器下载兜底不变。
- 没有新增相册、分享或权限相关依赖。

### 旧行为保护

首个 PR 必须保持以下旧行为：

| 旧行为 | 保护要求 |
| --- | --- |
| 阅读报告 Tauri 环境导出到应用导出目录 | `source` 仍为 `exportDir`，保留 `path` 和 `exportedAt` |
| 阅读报告浏览器环境下载 PNG | 继续触发浏览器下载兜底 |
| 笔记卡片导出返回文件名 | `exportNoteCardImage` 仍返回 `fileName` |
| 组合笔记导出返回文件名 | `exportNoteGroupImage` 仍返回 `fileName` |
| 导出失败文案 | 继续使用原有错误文案或等价中文提示 |
| 按钮 loading 状态 | 不因工具函数抽取而改变 |

如果旧行为保护与新抽象发生冲突，首个 PR 应优先保护旧行为，推迟抽象完整度。

## 责任边界

### 前端

- 负责按钮文案、状态机、能力探针调用和结果展示。
- 不直接处理相册权限细节。
- 不持久化图片二进制内容。

### 原生层

- 负责相册写入、系统分享和平台权限。
- 不理解业务语义，只处理最终 PNG 数据。

### 产品层

- 负责区分图片成果和文档成果。
- 负责决定哪些场景进入 MVP，哪些后置。
- 不把相册保存和成果库概念混成一体。

## 发布策略

建议按“基础层先行，能力逐步放开”的方式发布。

### 阶段 1：纯重构发布

- 只合入图片成果导出基础层。
- 不改变按钮文案和运行时入口。
- 目标是先验证没有行为回归。

### 阶段 2：文案切换灰度

- 根据 capability 控制手机端文案。
- 先对内测或少量设备开放。
- 桌面端保持不变。

### 阶段 3：分享能力开放

- 先在支持分享的原生壳和 Web Share 环境打开。
- 不支持的环境继续保持下载兜底。

### 阶段 4：相册能力开放

- 只在原生移动端开启。
- 先小流量验证权限、失败和重试。
- 出现系统性失败时可以快速回退到分享或文件导出。

### 回退开关

建议保留以下可快速关闭的能力开关：

- `enableImageArtifactRefactor`
- `enableImageShare`
- `enableSaveToAlbum`

回退时优先关闭高风险能力：

1. 先关相册写入。
2. 再关系统分享。
3. 最后保留基础层重构。

### 开关配置建议

建议开关粒度保持三档，不要过细：

- `enableImageArtifactRefactor`：控制基础层是否启用。
- `enableImageShare`：控制系统分享入口。
- `enableSaveToAlbum`：控制相册保存入口。

开关原则：

- 默认先关高风险能力，先放基础层。
- 发布阶段只改一个开关的默认值，避免同时改变多条用户路径。
- 关闭开关时，页面必须自动回退到次优但可用的路径。
- 不要在 UI 上展示“正在灰度”之类的元信息。

## 指标与观测

发布时建议重点观察以下指标：

| 指标 | 期望 | 异常含义 |
| --- | --- | --- |
| `save_click_rate` | 手机端图片成果主按钮点击率稳定 | 文案不清或入口位置不对 |
| `save_success_rate` | 相册保存成功率高 | 原生命令、权限或平台能力异常 |
| `share_invocation_rate` | 分享入口被合理使用 | 主入口失效或用户更偏向分享 |
| `permission_deny_rate` | 不应异常升高 | 权限文案或触发时机有问题 |
| `fallback_download_rate` | 只在不可用环境明显出现 | 平台能力判断不准确 |
| `share_cancel_rate` | 可存在但不应成为失败噪音主因 | 用户主动取消或分享流程过长 |
| `batch_partial_success_rate` | 批量场景可接受少量部分成功 | 顺序执行或资源消耗需要优化 |

### 埋点事件草案

建议事件只记录导出过程，不记录图片内容：

| 事件 | 触发时机 | 关键字段 |
| --- | --- | --- |
| `image_artifact_export_click` | 用户点击主操作 | `intent`, `platform`, `page` |
| `image_artifact_export_prepare` | 开始生成 PNG | `intent`, `fileName`, `page` |
| `image_artifact_export_success` | 导出成功 | `intent`, `source`, `fileName`, `platform` |
| `image_artifact_export_error` | 导出失败 | `intent`, `errorCode`, `retryable`, `platform` |
| `image_artifact_export_fallback` | 触发兜底路径 | `fromIntent`, `toIntent`, `reason` |
| `image_artifact_permission_denied` | 权限被拒绝 | `platform`, `intent` |
| `image_artifact_share_cancelled` | 用户取消分享 | `platform`, `intent` |

埋点原则：

- 不上报 PNG 原文、Base64、相册路径、系统分享载荷。
- 不把同一张图片的每个中间步骤都打成独立事件，避免噪音过大。
- 批量场景只需要记录汇总和失败分类，不逐页上报过多细节。

### 观测判读

- 如果 `save_success_rate` 低于预期，优先查原生命令和权限，而不是先改文案。
- 如果 `fallback_download_rate` 偏高，说明 capability 判断过于保守或平台能力接入不完整。
- 如果 `permission_deny_rate` 偏高但 `save_success_rate` 也正常，说明权限申请时机可能过早。
- 如果 `share_cancel_rate` 高，但 `share_success_rate` 正常，通常不是缺陷。

## 已定决策与剩余事项

以下问题已经定案并进入实现约束：

1. 原生移动端的相册写入与系统分享由同一个自研插件承接，但前端保持 `保存到相册` 与 `分享` 两个用户意图。
2. `保存到相册` 只对图片成果开放，不覆盖 Markdown、AI 复盘正文、阅读指南、诊断信息等文档成果。
3. 多页报告“保存全部页”按顺序逐张处理，后续补部分成功与重试体验。
4. 分享失败后的兜底统一回落到文件导出或浏览器下载，目标是让用户仍能拿到 PNG。
5. 手机端笔记卡片在 capability 支持时使用 `保存图片`，桌面端继续保留 `导出图片`。

仍需补齐：

1. Android 真机验收记录。
2. iOS 原生相册保存与分享的 Mac / Xcode 构建验证、真机权限验证。
3. 多页批量保存的结果汇总、部分失败重试和用户取消语义。

## 默认决策建议

如果实现阶段没有新的平台约束，建议按以下默认决策推进：

| 问题 | 默认决策 | 理由 |
| --- | --- | --- |
| 相册与分享是否共用原生模块 | 前端保持两个意图，原生层可共用内部实现 | 保持 UI 和业务语义清晰 |
| `保存到相册` 覆盖范围 | 只覆盖图片成果 | 避免文档成果被错误图片化 |
| 批量保存全部页 | 按顺序逐张处理 | 降低内存和权限风险 |
| 分享失败兜底 | 回落到 `导出文件` | 给用户稳定拿到 PNG 的路径 |
| 笔记卡片手机文案 | 能力可用时用 `保存图片` | 更符合手机用户预期 |
| Web 环境相册入口 | 不展示 | Web 无法稳定承诺相册写入 |
| 应用内历史记录 | 不新增 | 避免把相册当成果库 |

这些默认决策可以覆盖 MVP 和首个实现切片。后续如果原生平台能力发生变化，再单独调整对应项。

## 回退策略

如果原生能力暂时不可用，首版回退顺序应固定为：

1. 相册写入不可用时，回退到分享。
2. 分享不可用时，回退到导出文件。
3. 导出文件不可用时，回退到浏览器下载。
4. 所有回退都必须保持当前页面状态，不打断用户再次尝试。

回退策略的目标不是“尽量完成”，而是“让用户始终能拿到最终 PNG”。

## 实施顺序

### 第一阶段：设计与文案落地

- 为图片成果按钮增加平台化文案。
- 明确手机端 `保存到相册`、`分享`、`导出文件` 的展示条件。
- 保持桌面端现有下载文案。

### 第二阶段：抽出图片成果导出层

- 从 `BookNotesPage.tsx` 中抽出 `downloadCanvas` 的通用能力。
- 让阅读报告和笔记卡片共用 Canvas 到 PNG 的基础流程。
- 不改图片绘制逻辑。

### 第三阶段：接入移动端相册能力

- 在 Tauri 移动端或平台适配层中封装相册写入命令。
- 权限只在点击保存时请求。
- 权限失败时回退到分享或导出文件。

### 第四阶段：分享能力

- 原生移动端优先使用系统分享面板。
- Web 环境在支持文件分享时使用 Web Share。
- 不支持文件分享时回退浏览器下载或展示可复制 / 可导出的兜底。

### 第五阶段：回归验证

- 单张笔记卡片保存成功。
- 组合笔记卡片保存成功。
- 月报海报保存成功。
- 横版报告保存成功。
- 多页报告当前页和全部页保存成功。
- 权限拒绝、平台不支持、Canvas 生成失败都有明确错误提示。

### 验证清单

- 手机端首次点击 `保存到相册` 才触发权限。
- 手机端拒绝权限后能继续使用 `分享` 或 `导出文件`。
- 手机端保存成功后 toast 文案与文件导出文案不同。
- 桌面端仍保持下载语义，不出现相册入口。
- Markdown、复盘正文、阅读指南不进入相册逻辑。
- 浏览器环境仍可下载 PNG，不依赖原生能力。
- 批量保存失败时保留已完成结果，不让用户误以为全部失败。

## 测试策略

### 真机验收清单

真机验收优先覆盖“权限、相册、分享、回退”四类行为。

#### Android 真机

- 首次点击 `保存到相册` 才触发权限。
- 允许权限后，图片出现在系统相册。

- 拒绝权限后，结果区提供兜底操作。

## 阻断条件

出现以下任一情况，不建议进入下一阶段发布：

- 桌面端阅读报告 PNG 下载失败。
- 桌面端按钮文案被误改为相册语义。
- Markdown、复盘正文、阅读指南进入图片成果导出链路。
- 页面打开时触发相册或媒体权限请求。
- Web 环境展示不可执行的 `保存到相册`。
- Canvas 生成失败后仍继续请求权限或调起分享。
- Base64 图片内容被写入 localStorage、IndexedDB 或 SQLite。
- 批量保存失败被展示为全部成功。

这些阻断条件优先级高于“功能完整度”。只要命中，就应先回退或修复，再继续扩大范围。
