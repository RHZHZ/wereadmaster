# 单本 AI 复盘行动边界设计文档

## 背景

单本 AI 复盘当前包含三类容易混淆的输出：

- `复盘内容`：`overview`、`keyIdeas`、`myFocus`、`themeTags`、`representativeQuotes`、阅读洞察等。
- `AI 复盘行动`：`actionItems`。
- `复盘问题`：`reflectionQuestions`。

其中 `actionItems` 和 `reflectionQuestions` 已支持本地反馈状态，并会影响后续复盘更新。随着“阅读洞察”和 AI 助手追问入口加入，用户会更频繁地从一条洞察进入后续处理。如果 `actionItems` 仍然输出“继续思考”“深入理解”“进一步复盘”这类弱动作，它会和复盘内容、复盘问题重复，降低复盘页的信息密度。

本设计文档用于明确三者边界，并把 `AI 复盘行动` 收口为可执行、可反馈、可继承的“下一步行动”。

## 问题定义

用户看完单本 AI 复盘后，实际有三个连续问题：

1. 我从这本书和笔记里读出了什么？
2. 我还应该继续追问什么？
3. 我接下来具体做什么？

当前重复风险主要来自第 2 和第 3 点边界不清：

- 复盘问题是问句，用于打开思考。
- AI 复盘行动是任务，用于推进执行。
- 复盘内容是结论和依据，用于沉淀资产。

如果行动项只是把问题改写成陈述句，就会重复。例如：

```text
复盘问题：气节和个人选择之间有什么关系？
行动项：继续思考气节和个人选择之间的关系。
```

这不是行动，只是弱化后的问题。行动必须包含明确对象、处理范围或完成标准。

## 目标

1. 明确 `复盘内容`、`复盘问题`、`下一步行动` 的职责边界。
2. 把 `actionItems` 的产品语义从“AI 建议”调整为“下一步行动”。
3. 降低行动项与复盘问题、阅读洞察、关键观点的重复。
4. 保留现有反馈机制，不新增复杂任务管理系统。
5. 让后续 AI 助手追问可以围绕行动拆解，而不是触发正式复盘生成。

## 非目标

本阶段不做：

- 不新增一级导航“行动中心”。
- 不把单本复盘行动升级为完整任务系统。
- 不新增数据库表或复杂依赖关系模型。
- 不强制所有复盘都生成行动项。
- 不把 AI 助手聊天结果默认写入正式复盘资产。
- 不要求跨书自动合并相似行动。
- 不在首期新增独立“成果库”或全局反馈历史中心。

## 核心设计判断

### 1. 复盘是结论资产

复盘回答：

```text
这本书和我的笔记说明了什么？
```

它可以包含观点、洞察、主题、摘录和依据，但不要求用户立刻执行。

### 2. 复盘问题是思考入口

复盘问题回答：

```text
我还可以继续问什么？
```

它必须是问句，适合进入 AI 助手继续追问，也适合作为用户后续写作或自我反思的题目。

### 3. 下一步行动是执行出口

下一步行动回答：

```text
我接下来要完成什么？
```

它必须是任务，不是观点，不是问题，也不是泛化建议。用户应该能对它做四态反馈：

- 待处理
- 已完成
- 暂不做
- 不适合

## 字段边界

| 字段 / 模块 | 角色 | 输出形态 | 允许内容 | 禁止内容 |
| --- | --- | --- | --- | --- |
| `overview` | 复盘概览 | 3-5 句总结 | 本书总结、依据范围、核心判断 | 行动清单、问题清单 |
| `keyIdeas` | 关键观点 | 列表 | 观点、概念、判断 | 可勾选任务 |
| `myFocus` | 我的关注点 | 列表 | 用户反复标记的主题 | 执行动作 |
| `representativeQuotes` | 来源依据 | 摘录对象 | 划线、想法、章节 | AI 改写后的行动 |
| `reflectionQuestions` | 复盘问题 | 问句列表 | 可继续追问的问题 | 非问句任务、空泛建议 |
| `actionItems` | 下一步行动 | 任务列表 | 可执行、可反馈、可验收的动作 | 问句、观点复述、泛化建议 |
| 阅读洞察 | 组合展示 | 洞察卡片 | 关注点、观点、摘录、追问 | 新增无依据结论 |

## 行动项准入规则

`actionItems` 中每一条必须同时满足：

1. 以动词开头。
2. 包含明确对象，例如“这 2 条摘录”“气节相关笔记”“第 3 个复盘问题”。
3. 包含范围、数量、时间或完成标准中的至少一项。
4. 用户完成后能判断状态。
5. 不依赖后台自动任务。

推荐句式：

```text
整理 3 条“气节”相关摘录，写成一段不超过 150 字的个人判断。
选择 1 个复盘问题发起 AI 助手追问，并把回答压缩成 3 条可复用原则。
把本书中关于“成功”的 2 个观点加入年度复盘草稿，并标记适用场景。
```

不合格句式：

```text
继续思考气节和人生选择。
深入理解作者观点。
进一步复盘这本书。
这个问题值得长期关注。
```

## 复盘问题准入规则

`reflectionQuestions` 中每一条必须满足：

1. 必须是问句。
2. 只提出一个核心问题，避免多个问题串联。
3. 能基于当前复盘、洞察或摘录继续追问。
4. 不直接要求完成任务。

推荐句式：

```text
我为什么会被“气节”相关内容反复吸引？
这本书里关于成功的判断，和我已有经验哪里一致、哪里冲突？
这条洞察最适合迁移到写作、职业判断还是生活选择？
```

不合格句式：

```text
整理 3 条气节相关摘录。
下周写一页复盘。
继续深入理解这本书。
```

## 展示设计

### 单本 AI 复盘页

建议展示顺序：

```text
复盘概览
阅读洞察
代表性摘录
复盘问题
下一步行动
反馈记录 / 导出
```

设计理由：

- 用户先理解结论，再看到依据。
- 复盘问题作为追问入口，承接阅读洞察。
- 下一步行动放在后面，作为阅读成果处理出口。
- 行动不应抢在洞察前面，否则用户还没理解就进入执行。

如果页面希望更强调执行，也可以把“下一步行动”放在“复盘问题”前，但必须保持文案边界：

```text
复盘问题 = 继续想什么
下一步行动 = 具体做什么
```

### 复盘中心版本详情

版本详情页应复用同一套结构：

- 书籍复盘详情展示 `下一步行动` 和 `复盘问题`。
- 两者分别读取自己的反馈状态。
- 不把行动反馈和问题反馈混在一个列表里。
- 如果当前版本包含 `feedbackOutcomeSummary`，在版本详情中展示“反馈沉淀”。

## 反馈分析与历史复盘沉淀

行动反馈和复盘问题反馈需要 AI 分析，但首期结果应进入历史复盘版本详情，不进入独立成果库。

### 核心边界

| 类型 | 定义 | 展示位置 | 是否进入成果库 |
| --- | --- | --- | --- |
| 原始反馈 | 用户标记的状态、备注、复盘问题回答 | 当前复盘页、版本详情 | 否 |
| AI 分析 | AI 基于明确反馈生成的反馈沉淀 | 历史复盘版本详情、当前再生成版本 | 否 |
| 正式成果 | 用户确认值得长期保存的判断、原则、写作素材 | 后续成果库能力 | 首期不做 |

首期只做前两层。这样能保证用户看到“反馈被吸收了”，但不会把普通勾选、临时备注或未确认回答包装成长期成果。

### 分析触发

AI 反馈分析只在以下场景触发：

- 用户基于已有反馈重新生成单本 AI 复盘。
- 用户打开包含 `feedbackOutcomeSummary` 的历史复盘版本。
- 后续 P1 中，用户在 AI 助手中主动追问“这次反馈沉淀说明了什么”。

不做后台自动分析，不扫描全量历史，不跨书汇总。

### 输入范围

只允许使用同一书籍、同一复盘资产上下文内的明确反馈：

- `actionItems` 的状态和备注。
- `reflectionQuestions` 的状态、备注或回答。
- 上一版复盘中与这些反馈对应的行动和问题文本。
- 当前重新生成时传入的 `updateContext`。

不得使用：

- 没有被用户反馈的行动或问题。
- 其他书籍的反馈。
- AI 推测的完成情况。
- 用户未确认的聊天内容。

### 输出结构

首期复用现有可选字段：

```ts
type FeedbackOutcomeSummary = {
  summary: string;
  appliedChanges: string[];
};
```

字段语义：

- `summary`：1-2 句，说明上一版反馈沉淀出了什么。
- `appliedChanges`：1-3 条，说明本次复盘如何吸收反馈。

推荐展示：

```text
反馈沉淀

上一版反馈显示，你已完成了核心观点整理，并将 1 个复盘问题连接到现实场景。

本次吸收：
- 不再重复生成已完成的观点整理动作
- 避开已标记不适合的章节卡片方式
- 将已回答问题转为新的关注点
```

### 展示规则

历史复盘版本详情：

- 有 `feedbackOutcomeSummary` 时展示“反馈沉淀”。
- 展示在复盘概览之后、阅读洞察之前。
- 同时保留原始反馈状态，避免 AI 分析替代用户事实记录。

当前复盘页：

- 如果当前版本是基于反馈重新生成，并返回 `feedbackOutcomeSummary`，可以展示“反馈沉淀”。
- 没有真实反馈时不展示空区块。

Dashboard：

- 不展示反馈沉淀长文本。
- 只展示仍需处理的行动、复盘问题或洞察入口。

成果库：

- 首期不新增。
- 后续只有用户明确确认某条沉淀为成果时，才允许进入成果库。

### AI 分析约束

AI 必须遵守：

- 只能分析用户明确提交的反馈。
- 不能声称用户完成了未标记完成的行动。
- 不能把 `暂不做` 解释为失败。
- 不能把 `不适合` 解释为用户能力不足。
- 不能评价用户执行力、坚持程度或表现。
- 不能把复盘问题回答包装成事实结论，除非用户明确写了回答。
- 没有真实反馈时，不生成“上一版沉淀”。

### Dashboard 工作台

工作台只展示需要继续处理的最小行动入口：

```text
继续处理《明朝那些事儿》的 1 条下一步行动
```

展示规则：

- 优先展示未处理行动。
- 没有未处理行动但有未处理洞察时，展示洞察入口。
- 不在工作台展示长篇复盘问题列表。

## AI 助手联动

行动项旁可以提供“拆解”或“追问”入口，打开 AI 阅读助手。

助手职责：

- 拆解行动的执行步骤。
- 解释行动依据。
- 把行动转成写作提纲、复盘草稿或短清单。
- 根据用户反馈判断是否需要调整行动。

助手不应：

- 把行动追问误判成“生成正式 AI 复盘”。
- 默认保存聊天结果为正式复盘资产。
- 在缺少上下文时编造整本书内容。

推荐预填模板：

```text
围绕这条下一步行动继续拆解：「{actionItem}」。
请结合当前复盘、阅读洞察和来源摘录，给出 3 个执行步骤，并说明最小可完成版本。
```

## Prompt 约束

单本 AI 复盘 prompt 应加入以下规则：

```text
actionItems 必须是“下一步行动”，不是观点复述或复盘问题。
每条 actionItem 必须以动词开头，并包含对象、范围或完成标准。
如果当前笔记不足以支持具体行动，可以返回空数组，不要强行生成泛化行动。
reflectionQuestions 必须是问句，用于继续追问，不得写成执行任务。
不要在 actionItems 和 reflectionQuestions 中输出语义重复的条目。
如果输入包含 updateContext，只能基于用户明确反馈生成 feedbackOutcomeSummary。
feedbackOutcomeSummary 只总结反馈沉淀和本次吸收，不评价用户表现，不编造未完成事项。
```

AI 阅读助手 prompt 应加入以下规则：

```text
当用户追问已有复盘、阅读洞察、来源摘录或下一步行动时，必须直接回答。
只有用户明确要求生成、创建、保存正式复盘时，才引导使用单本 AI 复盘功能。
```

## 数据与兼容性

首版不改数据结构。

继续使用现有字段：

```ts
type BookAiSummary = {
  overview: string;
  keyIdeas: string[];
  myFocus: string[];
  actionItems: string[];
  themeTags: string[];
  representativeQuotes: BookAiRepresentativeQuote[];
  reflectionQuestions: string[];
  feedbackOutcomeSummary?: FeedbackOutcomeSummary;
};

type FeedbackOutcomeSummary = {
  summary: string;
  appliedChanges: string[];
};
```

兼容策略：

- 字段仍叫 `actionItems`，页面展示名改为“下一步行动”。
- 旧缓存中的弱行动不做迁移，只在新生成版本中通过 prompt 和校验改善。
- 导出 Markdown 统一使用“下一步行动”章节名，不再使用“行动与复盘”。
- 反馈存储继续按 `actionItems` 和 `reflectionQuestions` 分开。
- `feedbackOutcomeSummary` 是可选字段；旧缓存没有该字段时不展示“反馈沉淀”。
- 首期不新增成果库表，不新增全局反馈索引。

## 输出校验

可以在前端或后端增加轻量校验，不做复杂 NLP。

### 行动项校验

启发式规则：

- 不能以 `？` 或 `?` 结尾。
- 不能包含“是否”“为什么”“如何看待”等明显问句开头。
- 不应只包含“继续思考”“深入理解”“进一步复盘”“长期关注”等空泛短语。
- 建议包含数字、时间、对象名或产物词，例如“摘录”“草稿”“清单”“原则”“问题”“段落”。

### 复盘问题校验

启发式规则：

- 应以 `？` 或 `?` 结尾，或包含“为什么”“如何”“哪些”“是否”等问句结构。
- 不能以“整理”“完成”“写”“标记”“复制”“导出”等任务动词开头。

### 降级策略

如果 AI 返回不合格行动：

1. 前端仍展示，但不提升到 Dashboard。
2. 可在下一版生成时通过反馈上下文减少类似行动。
3. P1 可增加后端 sanitize，把明显问题句从 `actionItems` 移到 `reflectionQuestions`，但首版不做自动重写。

## 验收标准

1. 页面文案中 `AI 复盘行动` 统一展示为“下一步行动”。
2. `reflectionQuestions` 展示项必须是复盘问题，不与行动项混排。
3. 新生成复盘中，行动项不能只是“继续思考 / 深入理解 / 进一步复盘”。
4. 行动项能被标记为 `已完成 / 暂不做 / 不适合`，复盘问题能独立记录状态。
5. 用户点击洞察或行动追问后，结果展示在 AI 阅读助手对话面板。
6. 洞察追问和行动追问不会触发“生成 AI 复盘”动作卡。
7. 被用户标记为“不适合”的行动，下一版复盘不应原样再次出现。
8. Markdown 导出中行动和问题分为两个章节。
9. 有真实行动反馈或复盘问题反馈时，重新生成复盘可以包含 `feedbackOutcomeSummary`。
10. 没有真实反馈时，不生成“反馈沉淀”区块。
11. 历史复盘版本详情能展示 `feedbackOutcomeSummary`，但 Dashboard 不展示长文本。
12. `feedbackOutcomeSummary` 不进入独立成果库，也不替代原始反馈记录。

## 实施计划

### P0：文案与边界收口

- 将页面标题“行动与复盘”改为“下一步行动”。
- 保留字段名 `actionItems`，不改类型。
- 更新复制和导出章节名。
- 更新单本 AI 复盘 prompt，强化行动项和复盘问题边界。
- 复用 `feedbackOutcomeSummary`，在历史复盘版本详情展示“反馈沉淀”。
- 更新单本 AI 复盘 prompt，限制反馈分析只能基于明确反馈。
- 补充测试覆盖标题、导出、prompt 约束和无反馈不生成沉淀。

### P1：轻量质量校验

- 增加 `isWeakAiActionItem` 和 `isReflectionQuestionLike` 等本地规则。
- Dashboard 只提升合格的未处理行动。
- 弱行动仍留在复盘页，不进入工作台提醒。
- 对明显问句行动增加 UI 标识或降低优先级。
- 当前复盘页展示“反馈沉淀”。
- Markdown 导出包含“反馈沉淀”章节。
- AI 助手支持追问“这次反馈沉淀说明了什么？”。

### P2：助手联动增强

- 在行动项旁增加“拆解”入口。
- 打开 AI 助手并预填行动拆解模板。
- scope 优先使用 `bookNotes` 或 `aiAsset`。
- 助手直接回答行动拆解，不触发正式复盘动作卡。
- 用户手动确认某条反馈沉淀为正式成果，再评估成果库或知识卡片。

## 测试计划

### 单元测试

- `BookAiSummaryPage` 展示“下一步行动”和“复盘问题”。
- Markdown 导出包含“## 下一步行动”和“## 复盘问题”。
- 行动反馈和复盘问题反馈仍分别读写。
- 弱行动不会进入 Dashboard 工作台。
- 洞察追问、行动追问不会生成 `BookReview` action。
- 有真实反馈时，版本详情展示 `feedbackOutcomeSummary`。
- 无真实反馈时，不展示“反馈沉淀”。

### 集成测试

- 生成单本 AI 复盘后，页面展示顺序符合设计。
- 用户标记行动为“不适合”，重新生成时 updateContext 包含该反馈。
- 复盘中心版本详情页展示同样的行动 / 问题边界。
- 重新生成复盘时，AI 只吸收明确反馈，不复述已标记不适合的行动。

### 回归测试

- 旧缓存仍能正常展示。
- 旧反馈状态不会丢失。
- 导出 Markdown 不缺失行动项和复盘问题。
- AI 助手普通“生成正式复盘”请求仍能触发单本 AI 复盘动作卡。
- 旧版本缺少 `feedbackOutcomeSummary` 时不会渲染空区块。

## 风险与取舍

### 风险 1：过度校验导致有价值行动被过滤

首版不自动删除行动，只控制 Dashboard 提升和 prompt 输出。这样可以避免误伤用户已有内容。

### 风险 2：字段名与展示名不一致

`actionItems` 是代码和缓存字段，展示为“下一步行动”是产品语义。首版不改字段名，避免扩大迁移范围。

### 风险 3：行动项仍可能和洞察重复

通过 prompt 和验收规则降低重复，但不追求完全去重。行动项只要包含明确产物和完成标准，就可以和洞察共享主题。

## 结论

`AI 复盘行动` 不应删除，但应从“复盘建议”收口为“下一步行动”。它和复盘、复盘问题的边界是：

```text
复盘：沉淀结论
复盘问题：打开追问
下一步行动：推进执行
```

只要行动项保持可执行、可反馈、可验收，它就不会和复盘问题重复，而是成为阅读资产闭环的出口。

## 下一阶段准入

当前实现适合进入 P1 内测和灰度验证，但不直接进入正式发布。准入依据：

- 产品边界已收口：复盘沉淀结论，复盘问题打开追问，下一步行动推进执行。
- `feedbackOutcomeSummary` 已限制为只基于明确反馈生成，不进入独立成果库。
- 阅读洞察追问、反馈沉淀追问、下一步行动拆解都进入 AI 阅读助手草稿，不自动发送。
- 助手侧已排除这些追问触发正式 `BookReview` 动作卡的误判。
- 已通过完整前端构建、相关前端单测、Rust 全量测试和 Playwright 全量 e2e。

P1 不新增数据库表，不新增全局成果库，不做自动重写行动项。P1 只验证现有闭环在真实数据下是否稳定。

### P1 灰度目标

1. 验证新生成复盘中的 `actionItems` 是否足够具体。
2. 验证用户是否能理解“反馈沉淀”和“原始反馈记录”的区别。
3. 验证三类追问入口是否真的减少用户手动组织问题的成本。
4. 验证重新生成复盘时，已完成、暂不做、不适合的反馈不会被重复忽略。

### P1 样本范围

建议使用 5-8 本真实书籍样本：

- 2 本笔记密集、摘录较多的书。
- 2 本只有少量想法或划线的书。
- 1-2 本已经有行动反馈的书。
- 1 本行动项曾被标记为“不适合”的书。
- 1 本旧缓存复盘，用于验证兼容和空区块隐藏。

每本书至少完成：

1. 打开当前复盘，检查阅读洞察、反馈沉淀、下一步行动、复盘问题的顺序和可读性。
2. 点击一条阅读洞察追问，确认助手草稿可直接编辑。
3. 点击一条下一步行动拆解，确认助手不出现“生成 AI 复盘”动作卡。
4. 记录一条行动反馈或复盘问题反馈。
5. 重新生成复盘，检查 `feedbackOutcomeSummary` 是否只总结真实反馈。

### P1 不做事项

- 不把 `feedbackOutcomeSummary` 自动沉淀到成果库。
- 不把 AI 助手聊天结果自动写回正式复盘。
- 不把弱行动自动删除或自动迁移为复盘问题。
- 不在 Dashboard 展示反馈沉淀长文本。
- 不改变 `actionItems` 字段名和缓存结构。

### 发布前门槛

正式发布前必须满足：

- 真实样本中没有明显把复盘问题写成下一步行动的高频问题。
- “继续思考 / 深入理解 / 进一步复盘”这类弱行动不再作为主要输出出现。
- 用户能在历史复盘版本详情中找到反馈沉淀，但不会误认为它是独立成果库。
- Markdown 导出包含“反馈沉淀”“下一步行动”“复盘问题”和反馈记录，且章节名一致。
- 完整验证继续通过：`npm run build`、相关 `vitest`、Rust 全量测试、Playwright 全量 e2e。
