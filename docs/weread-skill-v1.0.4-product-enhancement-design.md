# 微信读书 Skill 1.0.4 产品增强设计规划

## 目标

在已完成 Skill 1.0.4 契约适配的基础上，规划下一阶段产品增强。重点不是“把新版接口全部搬进前端”，而是围绕现有 Tauri 应用的核心定位：本地优先的个人阅读资产工作台。

本规划先定义用户价值、入口设计、接口边界、调用预算和分期顺序。后续实现应按阶段拆小，避免一次性改动过大。

## 设计原则

### 1. 个人数据和公开数据分区

个人书架、个人笔记、阅读统计、AI 复盘属于“我的阅读资产”。公开点评、热门划线、共读想法属于“外部阅读信号”。

设计要求：

- 公开内容必须明确标注“公开点评”“热门划线”“共读想法”。
- 公开内容不混入个人笔记统计。
- 公开内容不默认作为 AI 判断个人偏好的证据。
- 若未来允许 AI 使用公开内容，必须在输入范围里显式展示并让用户确认。

### 2. 按需加载，限制调用量

新版 `profile.md` 已明确阅读概况只查最近 5 本电子书进度。这个原则应扩展到所有增强模块：

- 总览页不做全量逐书进度同步。
- 详情页只加载当前书的详情、进度、目录。
- 公开点评、热门划线默认只加载首屏。
- 共读想法只在用户点击某条热门划线后加载。

### 3. 不把发现页做成微信读书复刻

发现页的职责是辅助选书和候选管理，不是完整复刻微信读书搜索结果页。

设计要求：

- 默认保持“搜索、推荐、相似推荐、本地候选”的工作台式布局。
- 垂直搜索入口后置，不一次性堆满 tab。
- 搜索兜底结果不能标记为真实相似推荐。

### 4. 渐进增强，先读后写

优先做只读增强和本地缓存展示，再考虑写入或 AI 组合。

推荐顺序：

1. 当前章节定位。
2. 公开点评基础模块。
3. 微信读书概况卡片。
4. 公开点评筛选分页。
5. 热门划线。
6. 共读想法。
7. 垂直搜索和选书决策外部信号。

## 用户工作流

### 工作流 A：打开一本书，判断下一步

目标用户：已经同步书架，并从书架、总览、发现或候选页打开书籍详情的用户。

核心问题：

- 我读到哪里了？
- 这本书现在适合继续读、整理笔记，还是作为候选保留？
- 其他读者怎么看这本书？

设计入口：

- `BookDetailPage` 顶部保留当前书基本信息。
- 进度区增加“当前章节”文案。
- 目录继续高亮当前章节。
- 详情页下方新增“公开点评”模块。

首版不做：

- 不自动生成逐章任务。
- 不默认拉取热门划线和共读想法。
- 不把公开点评发送给 AI。

### 工作流 B：从相似推荐发现下一本书

目标用户：从一本已读或候选书出发找相近书。

核心问题：

- 有哪些真实相似推荐？
- 推荐结果是否可以保存到本地候选？
- 如果相似推荐失败，当前结果是否只是搜索兜底？

设计入口：

- `DiscoveryPage` 继续以真实 `/book/similar` 为主。
- 成功时标题明确为“相似推荐”。
- 失败时显示“已改用书名搜索兜底”，不混称为相似推荐。

首版不做：

- 不做复杂推荐解释。
- 不做多源推荐混排。

### 工作流 C：看总览，快速了解近期状态

目标用户：打开应用首页，希望快速知道自己最近阅读和笔记状态。

核心问题：

- 最近读了哪些电子书？
- 哪些书有进度？
- 哪些书有笔记可整理？

设计入口：

- `DashboardPage` 新增轻量“微信读书概况”卡片。
- 概况只使用书架缓存、笔记概览和最近 5 本电子书进度。
- 有进度缓存时展示进度，没有时展示最近阅读时间。

调用预算：

- 最多从 `books[]` 中按 `readUpdateTime` 取 5 本电子书。
- 不对 `albums[]`、`mp` 调 `/book/getprogress`。
- 不为概况补齐全书架历史进度。

### 工作流 D：用公开内容辅助判断，不污染个人资产

目标用户：选书时希望看一些外部信号，但仍以个人阅读资产为主。

核心问题：

- 这本书公开口碑如何？
- 大家讨论哪些点？
- 这些外部信号是否影响我的候选决策？

设计入口：

- 书籍详情页先展示公开点评基础模块。
- 候选书架或选书决策页后续可以展示“公开信号摘要”。
- AI 选书决策初期不自动使用公开信号，只展示给用户看。

## 分期设计

## P3：书籍详情增强

### P3.1 当前章节定位

用户价值：

- 打开详情页即可知道当前读到哪一章。
- 目录高亮更有意义，进度不只是百分比。

数据来源：

- `/book/getprogress` 返回 `chapterUid`、`chapterOffset`、`progress`。
- `/book/chapterinfo` 返回章节列表。

前端设计：

- 在详情页进度区域展示：
  - 当前进度百分比。
  - 当前章节标题。
  - 章节内偏移可作为弱提示，不作为主文案。
- 目录中继续高亮 `chapterUid` 对应章节。
- “在微信读书中打开”保持使用当前本地命令，不用拼展示链接。

后端设计：

- 不新增接口。
- 复用现有详情响应里的 progress 和 chapters。
- 可增加一个纯函数 helper，将 `chapterUid` 映射到章节标题。

验收标准：

- 有 `chapterUid` 且目录匹配时展示章节名。
- 无章节匹配时只展示百分比，不报错。
- 不新增任何批量进度请求。

### P3.2 公开点评基础模块

用户价值：

- 详情页能看到其他读者对当前书的公开点评。
- 辅助判断是否继续读、加入候选或生成阅读指南。

数据来源：

- `/review/list`
- 当前后端已有 `get_public_reviews` 相关能力，但前端未形成完整模块。

前端设计：

- 位置：`BookDetailPage` 详情主体下方，章节目录之后或旁侧辅助区。
- 模块标题：`公开点评`
- 副文案：`来自微信读书公开内容，不计入个人笔记。`
- 首屏展示 3 到 5 条点评。
- 每条点评展示：
  - 用户昵称或匿名占位。
  - 评分或推荐态，如果接口有稳定字段。
  - 点评正文纯文本。
  - 章节名，如果是章节点评。
  - 时间。
- 空状态：`暂无可展示公开点评。`

交互：

- 初版只加载首屏。
- 提供“刷新”按钮。
- 不做筛选、不做无限滚动。

后端设计：

- 若已有命令可用，优先复用。
- 若缺少前端 API 映射，新增：
  - `getPublicReviews(bookId, options?)`
  - `PublicReviewResponse`
  - `PublicReviewItem`
- 缓存命名建议：`public_reviews:{bookId}:default`

安全与展示：

- 优先展示纯文本 `content`。
- 不直接渲染 `htmlContent`。
- 不把公开点评合并进 `BookNotes`。

验收标准：

- 详情页能按需加载当前书公开点评。
- 公开点评失败不影响详情、目录、进度展示。
- UI 明确说明公开点评不属于个人笔记。

## P3.3 微信读书概况卡片

用户价值：

- 首页能更快看到近期阅读和笔记概况。
- 给用户一个轻量的“下一步整理入口”。

数据来源：

- `/shelf/sync` 本地缓存。
- `/user/notebooks` 本地或同步结果。
- 最近 5 本电子书的 `/book/getprogress`。

前端设计：

- 位置：`DashboardPage` 的核心指标和今日动作之间，或作为现有“本地进展”卡片的增强。
- 卡片内容：
  - 书架总数：电子书、有声书、文章收藏。
  - 笔记书籍数和总笔记数。
  - 最近 5 本电子书进度列表。
  - 一个主动作：`整理最近有笔记的书` 或 `查看阅读复盘`。

调用策略：

- 页面打开时优先使用现有缓存。
- 用户点击“刷新概况”时才同步。
- 进度请求最多 5 本电子书。

验收标准：

- 大书架不会触发 N 本进度请求。
- 有声书和文章收藏不会调用电子书进度接口。
- 无凭据时展示设置引导，不报错。

## P4：共读和发现增强

### P4.1 公开点评筛选分页

触发条件：

- P3 公开点评基础模块稳定。
- 用户确实需要查看更多点评。

设计内容：

- 增加筛选：
  - 全部点评。
  - 书评。
  - 章节点评。
  - 高赞或热门，如果接口字段稳定。
- 增加分页：
  - 使用 `maxIdx` 或 `synckey`。
  - 每次加载 10 条以内。

不做：

- 不做复杂排序面板。
- 不把点评做成全屏社区页。

### P4.2 热门划线

用户价值：

- 快速了解全书或当前章节被最多读者关注的句子。

数据来源：

- `/book/bestbookmarks`
- 可选 `chapterUid`

前端设计：

- 位置：书籍详情页的“共读信号”区域。
- 默认展示全书热门划线 5 条。
- 如果当前章节可识别，可提供“当前章节”切换。
- 每条展示：
  - 划线文本。
  - 人数或热度字段。
  - 所属章节。

调用策略：

- 默认只加载全书热门划线。
- 当前章节划线在用户切换后加载。
- 不默认加载每条划线下的想法。

验收标准：

- 热门划线与个人划线视觉分区明确。
- 没有章节 UID 时仍可展示全书热门划线。

### P4.3 共读想法

用户价值：

- 用户点开某条热门划线后，查看围绕这段文本的读者观点。

数据来源：

- `/book/readreviews`
- 参数来自 `/book/bestbookmarks` 的 `range` 和 `chapterUid`

前端设计：

- 点击热门划线后打开抽屉或内联展开。
- 展示 3 到 5 条共读想法。
- 明确标注“共读想法，不属于你的个人笔记”。

调用策略：

- 严格按点击加载。
- 不在书籍详情首屏批量请求。

验收标准：

- 打开书籍详情不会触发 N 条共读想法请求。
- 共读想法失败只影响当前展开项。

### P4.4 垂直搜索入口

用户价值：

- 用户能按意图搜索电子书、作者、有声书、书单、全文等。

设计入口：

- `DiscoveryPage` 现有 scope 控制保留。
- 不新增一排过长 tab。
- 使用“搜索类型”菜单或紧凑分段：
  - 综合。
  - 电子书。
  - 作者。
  - 有声书。
  - 书单。
  - 全文。

产品取舍：

- 默认仍是综合搜索。
- 找书意图明确时可以自动切到电子书。
- 不把 `results[].scope` 当请求 scope 使用。

验收标准：

- scope 显式传入。
- 回包分组按接口返回展示。
- `scope=10` 搜书时不会因 `results[].scope !== 10` 丢结果。

## P5：结构整理

触发条件：

- P3/P4 功能稳定后再做。
- `reading-api.ts` 继续膨胀，领域边界影响维护。

拆分建议：

- `weread-book-api.ts`：详情、进度、章节、公开点评。
- `weread-notes-api.ts`：个人笔记、热门划线、共读想法。
- `weread-discovery-api.ts`：搜索、推荐、相似推荐。
- `weread-stats-api.ts`：阅读统计。
- `command-errors.ts`：结构化错误归一化。

不建议：

- 不为了拆文件而拆文件。
- 不在 P3 开始前先做大规模 API 重排。

## 前端信息架构

### Dashboard

保留工作台定位，新增或增强概况卡：

- 阅读概况。
- 最近进度。
- 有笔记待整理。
- 跳转到笔记、复盘、书架。

### BookDetail

成为 P3/P4 的主要承载页：

- 基本信息。
- 本地整理状态。
- 当前阅读进度与章节。
- 章节目录。
- 公开点评。
- 热门划线。
- 共读想法。

### Discovery

保持选书工作流：

- 搜索。
- 推荐。
- 相似推荐。
- 保存候选。
- 垂直搜索作为增强，不做复杂社区页。

### Reading Hub

继续承载 AI 阅读成果：

- 不直接混入公开点评。
- 可以在未来提供“外部信号已可用”的入口，但不默认作为 AI 输入。

## 后端与缓存边界

### 新增接口建议

P3：

- `get_public_reviews(bookId, count?, maxIdx?)`

P4：

- `get_best_bookmarks(bookId, chapterUid?, count?)`
- `get_read_reviews(bookId, chapterUid, range, count?, maxIdx?)`

概况：

- `get_reading_overview()` 或前端组合现有接口。
- 若新增后端聚合，必须在服务层限制最近 5 本电子书进度。

### 缓存策略

- 公开点评和热门划线可以短期缓存。
- 缓存失败不阻断主详情页。
- 共读想法按划线 range 单独缓存。
- 公开内容缓存必须与个人笔记缓存分表或分 namespace。

### 错误策略

- `upgrade_required` 使用现有结构化升级提示。
- 公开模块失败只显示模块内错误。
- 详情主信息失败才进入页面级错误。

## AI 边界

默认不把以下内容送入 AI：

- 公开点评。
- 热门划线。
- 共读想法。

允许未来作为“外部信号”进入 AI 的条件：

1. 页面明确列出将使用的公开内容。
2. 用户主动确认。
3. 输出中标注“外部公开信号”。
4. 不把外部信号写入个人笔记统计。

## 验收测试规划

### 单元测试

- 当前章节 helper：`chapterUid` 匹配、缺失、目录为空。
- 公开点评 mapper：嵌套 `review.review`、章节名、时间、纯文本。
- 热门划线 mapper：章节 UID、range、热度字段。
- 共读想法 mapper：空列表、分页字段、作者信息。
- 概况聚合：最近 5 本电子书限制。

### 前端渲染测试

- 详情页展示当前章节名。
- 公开点评模块空态、错误态、正常态。
- 热门划线不混入个人笔记。
- 发现页垂直搜索不会因回包 `scope` 丢结果。

### 后端测试

- 新增 API payload 参数平铺。
- 分页参数显式传入。
- 最近 5 本进度调用限制。
- `upgrade_required` 在新增接口上继续透传。

### 构建验证

- `npx tsc --noEmit`
- `npm test`
- `cargo test --lib`
- `npm run build`

## 过度设计防线

明确不做：

- 不做微信读书社区页复刻。
- 不做全量逐书进度后台同步。
- 不做自动拉取所有热门划线和全部共读想法。
- 不做公开内容和个人笔记混合统计。
- 不在 P3 前拆分整个 `reading-api.ts`。
- 不在用户未确认时把公开内容送入 AI。

## 推荐实施顺序

1. P3.1 当前章节定位。
2. P3.2 公开点评基础模块。
3. P3.3 微信读书概况卡片。
4. P4.1 公开点评筛选分页。
5. P4.2 热门划线。
6. P4.3 共读想法。
7. P4.4 垂直搜索入口。
8. P5 结构整理。

首个实施切片建议选择 P3.1。它复用现有详情数据，不需要新增接口，风险最低，也能直接改善书籍详情页体验。
