# AI 阅读助手聊天控制增强设计

## 背景

P4 已完成 AI 阅读助手在真实 Tauri 桌面壳中的流式链路、失败路径、长数据和真实 Provider 小样本验收。当前阅读助手已经具备：

- 聊天主视图。
- 历史子视图。
- 设置子视图。
- 普通问答流式输出。
- 新书推荐和候选书决策的结构化非流式输出。
- 历史对话落库、回放、删除和清空。

接下来如果继续增强聊天体验，需要避免把阅读助手扩展成通用聊天产品。增强目标应服务于“阅读场景下更可控、更容易修正、更容易回看”，而不是新增复杂的聊天工作台。

## 问题定义

当前主要缺口有四类：

1. 用户不知道本次聊天实际使用哪个 Provider / Model，也不知道当前兼容策略是否会影响流式输出。
2. 用户提问写错后，只能重新输入一条新消息，历史里会留下错误问题和无效回答。
3. 历史对话数量增加后，当前历史子视图缺少搜索和场景筛选。
4. 用户提出“分组”诉求，但当前阅读助手天然已经有 `scope` 和 `entityId`，不应过早引入手动分组模型。

## 阶段目标

本阶段只解决三个实际问题：

- 让用户知道当前聊天使用的模型与兼容模式。
- 允许用户修正最后一条用户消息并重新生成回答。
- 让历史对话更容易查找和清理。

本阶段不做新的 AI 能力，不改变 P4 的流式/非流式边界，不新增后台自动请求。

## 非目标

本阶段明确不做：

- 不做每条消息独立选择模型。
- 不做聊天框内完整 Provider 配置表单。
- 不允许编辑助手回答。
- 不做任意历史消息编辑。
- 不做多分支对话树。
- 不做手动对话分组、文件夹、拖拽排序。
- 不做全局 AI 历史中心。
- 不把 AI 阅读助手升级为通用 ChatGPT 类产品。

## 设计原则

### KISS

优先复用现有 `ReadingAssistantPanel` 的 `chat/history/settings` 子视图，不新增独立对话管理中心。

### YAGNI

首版只支持最后一条用户消息编辑。任意消息编辑、多分支、手动分组都后置。

### DRY

模型信息读取复用现有 AI 设置状态，不在聊天模块另存一套 Provider 配置。

### SOLID

- 聊天面板只负责展示和交互编排。
- AI 设置仍由设置页负责。
- 历史一致性由后端命令保证，前端不直接拼接数据库语义。

## MVP 范围

### 1. 当前模型信息展示

在聊天主视图标题区或输入框上方展示一条紧凑模型状态：

- Provider 名称或 preset。
- Model 名称。
- 响应格式策略：
  - `jsonSchemaFirst`：严格结构化。
  - `jsonObjectFirst`：JSON 模式。
  - `noResponseFormatFirst`：宽松模式。
  - `auto`：自动。
- 可选展示当前普通问答是否可尝试流式。

交互：

- 提供“切换模型”按钮。
- 点击后进入现有设置子视图或打开全局设置中的 AI 配置区域。
- 生成中禁用切换按钮，只允许查看当前模型。

不在聊天框中直接编辑：

- Base URL。
- API Key。
- Provider preset。
- response format policy。

原因：

- Provider 配置涉及凭据、探测、模型列表和兼容策略，职责属于设置页。
- 聊天框内配置会增加状态同步和错误处理复杂度。

### 2. 最后一条用户消息编辑并重新生成

首版只允许编辑当前对话中的最后一条用户消息。

可编辑条件：

- 当前不在生成中。
- 当前消息角色为 `user`。
- 该消息是对话里的最后一条用户消息。
- 该用户消息之后最多只有一条对应助手消息。
- 对应助手消息状态为 `answered` 或 `failed`。

用户操作：

1. 用户点击最后一条用户消息旁的编辑按钮。
2. 用户消息气泡变为内联编辑态。
3. 用户可以取消，恢复原文。
4. 用户点击“保存并重新生成”。
5. 前端移除该用户消息之后的本地消息展示。
6. 后端删除该用户消息及其后续消息，并使用编辑后的文本重新生成。
7. 新回答沿用当前 `threadId`，但使用新的 input hash、当前 Provider / Model 和当前上下文设置。

语义约束：

- 这不是“修改历史文本”，而是“从最后一个问题重新生成”。
- 旧回答不再作为上下文参与新回答。
- 如果重新生成失败，历史中应保留编辑后的用户消息和失败助手消息，不能恢复到旧错误问题。

不允许：

- 编辑助手消息。
- 编辑中间用户消息。
- 编辑后保留旧回答继续显示。
- 生成中编辑。

### 3. 历史子视图搜索和场景筛选

复用现有历史子视图，增加轻量筛选能力。

搜索范围首版只包括：

- 对话标题。
- 场景标签。
- 可选实体 ID 展示名，如果当前已有可读名称。

筛选维度：

- 全部。
- 全局。
- 书籍详情。
- 笔记。
- 统计。
- 候选书。
- AI 资产。
- 本地阅读器选区。

列表展示：

- 标题。
- 场景标签。
- 更新时间。
- 消息数。
- 最近使用模型，如果已有字段可读；没有则不展示，不为首版新增复杂聚合。

已有能力继续保留：

- 打开历史对话。
- 删除单条对话。
- 清空全部历史。

清空全部历史继续需要明确二次确认，因为这是不可逆删除。

### 4. 自动分组而非手动分组

首版不新增手动分组。

替代设计：

- 历史列表按 `scope` 筛选。
- 可选按 `entityId` 在同一书籍上下文中聚合展示。
- 对当前打开的书籍详情、笔记或 AI 资产，历史页优先提供“只看当前对象相关对话”的筛选入口。

暂不新增：

- `conversation_groups` 表。
- 手动创建分组。
- 重命名分组。
- 移动对话到分组。
- 删除空分组。

原因：

- 当前 `scope + entityId` 已经能表达阅读场景归属。
- 手动分组会引入新状态、新删除边界和新空态，收益不足。

## 信息架构

### 聊天主视图

建议结构：

```text

  返回 / 标题 / 历史 / 设置 / 关闭

模型状态条
  当前 Provider · Model · 兼容策略
  [切换模型]

消息列表
  用户消息
    文本
    [编辑] 仅最后一条用户消息显示
  助手消息

输入区
  prompt 建议
  textarea
  发送 / 取消生成
```

### 历史子视图

建议结构：

```text

  返回聊天 / 清空

  全部 / 全局 / 书籍 / 笔记 / 统计 / 候选 / AI 资产 / 本地阅读器

  对话项
    标题
    场景
    更新时间
    消息数
    删除
```

## 数据设计

### 复用现有字段

继续复用：

- `ai_assistant_threads`
- `ai_assistant_messages`
- `ReadingAssistantThreadSummary`
- `ReadingAssistantThreadDetail`
- `ReadingAssistantMessage`
- `providerModel`
- `promptVersion`
- `inputHash`
- `scope`
- `entityId`

### 新增请求字段

建议扩展 `ReadingAssistantRequest`：

```ts
type ReadingAssistantRequest = {
  threadId?: string;
  scope: AssistantContextScope;
  entityId?: string;
  message: string;
  enabledContext: ReadingAssistantContextOption[];
  replaceFromMessageId?: string;
};
```

语义：

- `replaceFromMessageId` 只用于消息编辑重新生成。
- 后端必须验证该消息属于当前 `threadId`。
- 后端必须验证该消息 role 为 `user`。
- MVP 阶段后端必须验证它是当前线程最后一条用户消息。
- 验证通过后，删除该消息及其之后的消息，再插入编辑后的用户消息和新助手消息。

不建议前端单独调用“删除消息后再 ask”的两步流程。原因是两步流程中间失败会导致历史对话被删但新回答没生成，历史一致性差。

### 后端处理建议

在 `ask_reading_assistant_inner` 中处理 `replaceFromMessageId`：

1. 规范化 `replaceFromMessageId`。
2. 打开数据库事务。
3. 读取目标消息。
4. 校验：
   - `thread_id` 一致。
   - `role = user`。
   - 是最后一条用户消息。
5. 删除目标消息及其后续消息。
6. 提交事务。
7. 按现有 ask 流程重新构造上下文和生成。

注意：

- 删除旧尾部必须发生在构造 conversation history 之前，否则旧问题和旧回答会进入新 prompt。
- 如果重新生成失败，应按现有失败路径插入编辑后的用户消息和失败助手消息。
- 如果用户关闭了保存历史，编辑只影响前端本地消息数组，不需要后端删除尾部。

## 前端状态设计

新增本地状态：

```ts
type EditingUserMessageState = {
  messageId: string;
  originalContent: string;
  draftContent: string;
};
```

提交流程：

```text
点击编辑
  -> 进入 editing 状态
取消
  -> 清空 editing 状态
保存并重新生成
  -> 校验 draft 非空
  -> 清空错误
  -> 删除本地旧尾部展示
  -> 调用 askReadingAssistantStream(..., replaceFromMessageId)
  -> 收到流式事件后更新新助手 pending 消息
  -> 完成后刷新 threadId / threads
```

生成中状态：

- `isSubmitting = true` 时禁用编辑、删除、切换模型。
- 取消生成沿用现有 `cancelReadingAssistantStream`。

## 流式与结构化边界

继续保持 P4 边界：

- 普通问答尝试流式。
- 新书推荐仍非流式。
- 候选书决策仍非流式。

编辑重新生成不改变 intent 推断：

- 编辑后的消息重新推断 intent。
- 如果编辑后变成新书推荐，仍走结构化非流式。
- 如果编辑后变成普通问答，走普通问答流式路径。

## 错误处理

### 模型状态读取失败

- 不阻断聊天。
- 模型状态条显示“模型信息暂不可用”。
- “切换模型”仍可进入设置页。

### 编辑消息校验失败

前端显示明确错误：

- “只能编辑最后一条用户消息。”
- “这条消息已不在当前对话中。”
- “生成中不能编辑消息。”

### 重新生成失败

- 使用现有失败消息样式。
- 输入区恢复可用。
- 不恢复旧助手回答，避免历史语义混乱。

## 隐私与安全边界

- 模型状态条不展示 API Key。
- 不展示数据库路径、本地文件路径或原始 WeRead 响应。
- 编辑重新生成不扩大上下文范围，只使用当前 enabled context 和偏好。
- 历史搜索只搜索本地已保存的对话摘要，不触发远端请求。
- 清空历史必须二次确认。

## 验收标准

### 模型状态

- 用户能在聊天主视图看到当前 Provider / Model。
- 用户能看到当前兼容策略。
- 点击“切换模型”进入现有 AI 设置，不在聊天框内编辑 API Key。
- 生成中模型切换按钮禁用。

### 消息编辑

- 最后一条用户消息显示编辑入口。
- 非最后一条用户消息不显示编辑入口。
- 助手消息不显示编辑入口。
- 编辑后点击取消，消息内容不变。
- 编辑后点击保存并重新生成，旧助手回答从 UI 移除。
- 保存历史开启时，旧用户消息和旧助手回答不会进入新一轮 prompt。
- 重新生成失败时，保留编辑后的用户消息和失败助手消息。
- 生成中不能编辑。

### 历史管理

- 历史页可以按标题搜索。
- 历史页可以按场景筛选。
- 删除单条对话后列表刷新。
- 清空全部历史需要二次确认。
- 搜索和筛选不触发 AI 调用、不触发微信读书同步。

### 分组边界

- 不新增手动分组入口。
- 用户能通过场景筛选找到不同上下文的对话。
- 当前书籍上下文中可以只查看当前对象相关对话。

## 测试规划

### 前端测试

- `ReadingAssistantPanel` 模型状态展示。
- 编辑最后一条用户消息并重新生成。
- 非最后用户消息不显示编辑。
- 助手消息不显示编辑。
- 生成中禁用编辑和模型切换。
- 历史搜索过滤。
- 历史场景筛选。

### Rust 单测

- `replaceFromMessageId` 只能匹配当前线程用户消息。
- 非最后用户消息被拒绝。
- 删除尾部发生在上下文构造前。
- 编辑重新生成后 input hash 变化。
- 删除尾部不影响其他线程。

### E2E

- 打开助手后看到当前模型。
- 普通问答生成后，编辑最后一条用户消息并重新生成。
- 历史页搜索并打开目标对话。
- 历史页按场景筛选。

## 实施顺序

1. 模型状态条。
   - 只读展示 Provider / Model / response format policy。
   - 切换入口跳转现有设置。
2. 历史搜索和场景筛选。
   - 前端本地过滤，不改后端。
3. 最后一条用户消息编辑 UI。
   - 先支持历史关闭或本地未落库场景。
4. 后端支持 `replaceFromMessageId`。
   - 保证落库历史一致性。
5. 编辑并重新生成接入流式请求。
6. 补测试和 P4 普通问答回归。

## 后续扩展判断点

只有满足下面条件，才考虑更复杂能力：

- 历史对话数量明显增长，搜索和场景筛选仍不足。
- 用户明确需要保留多个版本回答。
- 编辑中间消息的需求高频出现。

届时可以评估：

- 任意用户消息“从此处重新生成”。
- 对话置顶或收藏。
- 当前书籍下的自动聚合分组。

仍不建议优先做：

- 手动分组文件夹。
- 多分支对话树。
- 助手回答编辑。
- 每条消息独立模型。

## 结论

本阶段推荐做“聊天控制增强”而不是“聊天系统重构”。MVP 是：

1. 当前模型状态展示。
2. 切换模型入口跳转设置。
3. 最后一条用户消息编辑并重新生成。
4. 历史搜索和场景筛选。

这能解决实际使用中的可见痛点，同时保持阅读助手的产品边界和实现复杂度可控。
