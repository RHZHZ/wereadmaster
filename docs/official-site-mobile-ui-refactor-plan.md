# 官网移动端 UI 改造文档

## 背景

官网首页当前已经把运行时视觉资源切回真实应用截图：

- 首屏主视觉：`src/assets/shelf.png`
- 下载区视觉：`src/assets/generated/set.png`

但移动端页面仍保留了较强的 AI 营销页结构：超大口号、淡化截图背景、强 CTA、抽象价值文案和较长的开源说明。用户在手机上首先看到的是“宣传感”，而不是“这就是我的实际应用界面”。这会削弱产品可信度，也会让官网和桌面应用真实体验不一致。

本次文档聚焦官网移动端，不处理桌面端整体重构。

## 补充审计：当前首屏截图问题

基于近期截图，移动端 Hero 已经从“抽象宣传页”往“真实产品页”方向修正，但还存在几个明显的首屏问题：

### 1. 标题被顶栏遮挡

当前首屏里，H1 上半部分被 sticky header 压住，用户第一眼看到的是半截标题，而不是完整主信息。

这说明移动端 Hero 的顶部安全距离仍然不足。对于手机页面，Hero 顶部必须把导航高度和安全区一起算进去，不能只按普通段落起始处理。

### 2. H1 仍然过大，换行过重

“微信读书本地管理工具”在移动端被放大后占据了过多垂直空间，并且换行后很容易形成压迫感。

这一版的问题已经不是“文案不对”，而是“字号、行高和列宽没有为手机首屏重新分配”。标题应服务于首屏理解，而不是吞掉首屏高度。

### 3. 真实截图出现得太晚

虽然首屏已经有真实产品截图，但在当前移动端排版里，它仍然被 CTA、说明文案和信任点压到较靠下的位置，用户要先扫过很多文字，才能看到实际界面。

对手机用户来说，截图应该更早出现在首屏中部，形成“标题 -> 一句话说明 -> 截图 -> 下载”的顺序。

### 4. CTA 和说明占比偏重

下载按钮、GitHub 按钮、发布说明和信任点堆在一起，会让首屏显得很满，但没有显著增强理解。

手机端首屏应该尽量只保留：

- 一个主下载动作。
- 一个次级信任动作。
- 三个极短信任点。

### 5. 截图卡片仍偏扁

当前截图卡片虽然是真实界面，但高度偏低，视觉上更像缩略图而不是产品预览。

建议手机端给截图卡片明确的最小高度，保证它能成为首屏中最有说服力的视觉对象之一。

## 当前移动端修复优先级

### P0

1. 增加 Hero 顶部安全间距，保证标题不被 sticky header 遮挡。
2. 降低移动端 H1 字号和行高，避免标题占用过高。
3. 提前真实截图的位置，让它在首屏中部就可见。

### P1

1. 压缩 CTA 区域，只保留主下载动作和一个次级动作。
2. 将长说明改成更短的双端发布说明。
3. 将信任点改成更短的 chip，降低换行破碎感。

### P2

1. 增加截图卡片高度，提升产品可读性。
2. 继续压缩“开源区”高度，避免下载入口被挤太晚。
3. 用真实界面建立信任，而不是继续依赖宣传语。

## 核心问题

### 1. 首屏对象错误

当前首屏的第一视觉对象是：

```text
大标题 + 下载按钮
```

但移动端官网更应该优先展示：

```text
真实应用界面 + 具体能做什么
```

当前真实截图被处理成背景氛围图，用户无法看清界面，也无法快速建立“这个页面和应用真实一致”的信任。

### 2. 文案抽象导致 AI 味明显

当前高频表达包括：

- 阅读资产
- 知识库
- 一站式管理
- 长期可复用
- AI 复盘
- 闭环

这些词在移动端首屏显得泛化，更接近 AI 生成的 SaaS 营销语言。实际产品更像本地桌面工具，移动端文案应优先使用事实型描述。

### 3. 信息顺序不适合手机

当前顺序：

```text
Hero
价值区
四步流程
产品截图
开源区
下载区
```

问题：

- 真实截图被推迟到第三屏以后才作为可读对象出现。
- 价值区和流程区重复解释“为什么”，但没有先解释“现在能做什么”。
- 开源区占比过大，下载入口过晚。

### 4. 导航展开状态干扰阅读

移动端菜单使用半透明浮层，展开后底层 hero 文案和截图仍明显穿透。视觉上像一层模板化玻璃浮层，不够确定，也不利于阅读菜单项。

### 5. 产品截图区缺少导读

截图区有真实截图，但说明仍是泛化句子：

```text
把书架、筛选、分类与阅读状态收进同一个桌面入口
```

缺少对截图中关键区域的指引：

- 左侧导航代表哪些工作流？
- 书架统计有什么作用？
- 分类筛选如何帮助整理？
- 同步入口在哪里？
- AI 复盘入口在哪里？

## 改造目标

1. 移动端首屏必须体现真实应用界面，而不是抽象概念图或淡背景图。
2. 降低“AI 产品宣传页”观感，回到本地工具、真实工作流、明确边界。
3. 首屏让用户在 5 秒内理解：
   - 这是微信读书本地管理工具。
   - 能同步书架、笔记、统计。
   - 能做复盘和导出。
   - AI 只在手动触发时使用。
4. 缩短移动端页面主路径，减少泛化价值区和开源区占用。
5. 所有移动端可交互控件满足 44px 触控目标。
6. 保持桌面端现有页面不被大幅牵连。

## 非目标

本次不做：

- 不重新生成图片。
- 不调用 `gpt-image2`。
- 不新增官网路由。
- 不重做桌面端完整视觉系统。
- 不修改应用主界面。
- 不改变下载链接、GitHub 链接和发布说明路径。

## 改造边界

本次改造的核心不是“去掉所有营销表达”，而是让移动端优先回答用户的实际问题：

```text
这是什么工具？
它和真实应用是否一致？
我能用它完成哪些阅读管理动作？
我的数据和 AI 调用边界在哪里？
我下一步去哪里下载或查看源码？
```

因此：

- `AI`、`知识库`、`资产` 等词不是禁止使用，而是不能成为移动端首屏主叙事。
- 四步流程不是必须删除，而是优先与能力清单合并；只有在内容重复时才删除独立区块。
- 页面总高度不是硬性指标，优先级低于首屏清晰度、截图提前出现和下载入口可达性。
- 真实截图在手机上允许局部可读，不要求完整桌面截图全部可读。

## 移动端信息架构

建议移动端改为：

```text
1. 首屏：产品名 + 真实界面截图 + 核心动作
2. 能做什么：4 个具体能力
3. 真实截图导读：书架 / 笔记 / 设置
4. 本地与 AI 边界：短说明
5. 下载与开源：压缩合并
```

### 新顺序

```text
Hero
  - 产品名
  - 一句话说明
  - 真实界面截图卡片
  - 下载 / GitHub
  - 本地优先、手动 AI、Markdown 导出

能力清单
  - 同步书架
  - 整理笔记
  - 生成复盘
  - 导出 Markdown

截图导读
  - 书架
  - 笔记
  - 设置

数据边界
  - 本地缓存
  - 手动触发 AI
  - Key 本机保存

下载
  - Windows 版
  - GitHub Releases
```

## 首屏改造方案

### 当前结构

```text
背景：真实截图低透明度铺满
前景：大 H1 + 段落 + 下载按钮 + GitHub 按钮 + bullet
```

### 建议结构

```text
产品名：微信读书个人阅读管理
说明：同步书架、笔记和统计，在本机整理复盘并导出 Markdown。

[真实应用截图卡片]
  显示 shelf.png 的可读截图，不作为背景

[下载 Windows 版] [GitHub]

本地优先 · 手动 AI · Markdown 导出
```

### 文案替换

当前 H1：

```text
把微信读书记录变成本地的阅读资产
```

建议 H1：

```text
微信读书本地管理工具
```

或：

```text
微信读书个人阅读管理
```

当前副文案：

```text
书架、笔记、统计、AI 复盘，一站式管理。本地优先，数据只在你手边，帮助你把阅读沉淀成长期可复用的个人知识库。
```

建议副文案：

```text
同步书架、笔记和统计，在本机整理复盘，并导出 Markdown。AI 只在你手动触发时使用。
```

### 样式策略

移动端：

- 不再把截图作为整屏背景。
- 增加 `.hero-product-preview` 前景截图容器。
- H1 字号从约 `54px` 降到 `34-40px`。
- Hero 高度目标控制在 `640-760px`，但优先保证真实截图和核心动作同时可见。
- CTA 保留双按钮，但下载按钮不应遮挡截图。
- bullet 改为紧凑标签，最多 3 个。

### 截图裁切策略

`shelf.png` 是桌面宽图，手机上不能简单完整缩放，否则会看不清。首版建议使用 CSS 裁切：

```css
.hero-product-preview img {
  width: 100%;
  height: 100%;
  object-fit: cover;
  object-position: 18% 0%;
}
```

裁切优先级：

1. 保留左侧导航和品牌区域。
2. 保留书架统计卡片。
3. 保留筛选控件。
4. 允许书籍卡片只露出部分。

如果 CSS 裁切仍不能表达真实界面，再单独截取一张移动端友好的真实桌面截图局部图，但不使用生成图替代。

## 移动导航改造方案

### 当前问题

- 菜单高约 `470px`。
- 半透明背景导致底层内容穿透。
- 展开后覆盖 hero，大标题和菜单互相干扰。

移动菜单改为实色面板：


交互：

- 关闭按钮保持 44px 以上。
- 点击菜单项后必须关闭菜单。

视觉：

- 移除大面积透明和毛玻璃。
- 减少阴影强度。
- 导航项使用普通列表式结构，不使用过重胶囊背景。

## 能力清单区改造

当前价值区文案偏抽象：


## 文案规范

### 推荐词

- 本地
- 手动同步
- 书架
- 笔记
- 统计
- 复盘
- 导出
- Markdown
- GitHub Releases
- 本机保存

### 限制使用

- AI
- 知识库
- 资产
- 闭环
- 赋能
- 智能
- 一站式
- 长期可复用
- 值得信赖

限制使用不等于禁止使用。使用规则：

- 首屏尽量不用或只用一次。
- 用到 `AI` 时必须同时出现“手动触发”或“本地笔记依据”。
- `知识库`、`资产` 可以放在二级说明中，不作为 H1 主语。
- 不使用“智能代理”“自动复盘”“自动沉淀”等会扩大产品承诺的说法。

### AI 表达规范

不要说：

```text
AI 智能复盘，沉淀长期可复用知识资产
```

建议说：

```text
你手动触发时，基于本地笔记生成书籍复盘。
```

## 技术改造点

- Playwright `390 x 844` 首屏验证通过：主按钮仅显示移动端文案，次按钮不显示重复副标题，无横向滚动、无 Vite/Webpack/Next 错误覆盖层、控制台无 error/warn。

## 建议实施顺序

### P0：降低 AI 味并建立真实感

1. 调整移动端 hero 文案和结构，让真实截图成为前景。
2. 移动端菜单改为实色面板，减少背景穿透。
3. 首屏保留下载和 GitHub，但弱化抽象 bullet。
4. 跑 `390 x 844` 首屏和菜单验证。

### P1：补足信息层级

1. 压缩价值区，将抽象文案改成具体能力清单。
2. 合并或压缩四步流程区。
3. 改造截图区轮播控制和导读。
4. 跑 `390 x 844`、`430 x 932`、`768 x 1024` 视口验证。

### P2：收口可信与下载

1. 压缩开源区和下载区。
2. 增加手机端 Windows 下载语境说明。
3. 检查页面总高度、下载入口可达性和性能。

## 风险与取舍

### 风险 1：移动端首屏放真实截图后显得拥挤

控制方式：

- H1 降级。
- 副文案缩短。
- 截图卡片只露出上半部分也可以，但必须可识别。

### 风险 2：桌面端结构被牵连

控制方式：

- 大部分改动放在移动端媒体查询内。
- 数据结构可以共用，但布局样式移动端单独控制。

### 风险 3：开源信息压缩后说服力下降

控制方式：

- 保留 GitHub 链接、技术栈、数据边界三项。
- 删除重复 repo card，不删除核心可信信息。

## 结论

手机版不应继续沿用桌面营销页缩放方案。它需要从“宣传式首屏”改为“真实产品说明页”：

```text
先看到真实界面
再理解具体能力
再确认数据边界
最后下载或查看 GitHub
```

这样才能降低 AI 味，提升和实际应用界面的一致性。
