# Android 底部导航与“我的”页改造方案

## 背景

当前 Android 版已经完成第一阶段移动端壳层适配：隐藏桌面窗口控制，使用移动端顶部栏、遮罩和侧边抽屉导航。第一阶段方案见 `docs/android-mobile-shell-navigation-refactor-plan.md`。

随着 Android 端从“可用”进入“更像原生移动应用”的阶段，继续保留所有一级入口在侧边抽屉中，会暴露两个体验问题：

- 高频页面需要先点菜单再选择入口，单手操作路径偏长。
- 侧边抽屉能承载全部导航，但不利于形成稳定的移动端主路径。

第二阶段建议引入底部导航，但不把所有现有一级导航硬塞到底栏。底栏只承载高频主任务，低频能力收敛到新的“我的”页。

## 目标

1. Android / 窄屏端使用底部导航承载核心入口。
2. 新增“我的”作为本机工作台和低频入口集合。
3. 保留桌面端侧边栏，不影响 Windows 桌面布局。
4. 保留现有业务路由和页面组件，优先在 `App.tsx` 壳层层面组合。
5. 明确安全区、系统手势栏和 Toast 的底部避让规则。

## 非目标

- 不重写业务页面。
- 不引入新的路由库或 UI 框架。
- 不把书架、复盘的二级入口全部提升到底部导航。
- 不把设置页拆成多个独立一级页面。
- 不变更 Android 打包、签名、发布链路。

## 现状导航

当前一级导航共有 7 个入口：

| 入口 | 频率判断 | 当前职责 | Android 处理建议 |
| --- | --- | --- | --- |
| 总览 | 高频 | 阅读状态、同步、工作台 | 放入底栏 |
| 书架 | 高频 | 微信书架、候选书架、本地书库 | 放入底栏 |
| 笔记 | 高频 | 划线、想法、批量导出 | 放入底栏 |
| 统计 | 中频 | 阅读时间、偏好、趋势 | 收敛到“我的” |
| 复盘 | 高频 | 单本复盘、阅读指南、周期报告 | 放入底栏 |
| 发现 | 中低频 | 搜索、推荐 | 收敛到“我的” |
| 设置 | 低频 | 凭据、AI、外观、导出、维护 | 收敛到“我的” |

底部导航不适合超过 5 项。超过 5 项会降低可读性，并让每个入口的语义变弱。因此第二阶段不追求“所有入口同级可见”，而是区分核心任务和低频管理。

## 推荐信息架构

Android 端底栏固定为 5 项：

```text
总览 | 书架 | 笔记 | 复盘 | 我的
```

低频入口进入“我的”：

```text
我的
├─ 顶部本机状态卡
│  ├─ 应用名称 / 本机工作台定位
│  ├─ 凭据状态 / 最近同步 / 本地保护提示
│  └─ 主操作：立即同步
├─ 快捷入口宫格
│  ├─ 统计
│  ├─ 发现
│  ├─ 设置
│  └─ 本地数据
├─ 阅读与数据
│  ├─ 阅读统计
│  ├─ 搜索与推荐
│  ├─ 本地书库管理
│  └─ 笔记导出记录
├─ 系统设置
│  ├─ AI 服务配置
│  ├─ 外观与阅读偏好
│  ├─ 代理与网络
│  └─ 更新与关于
└─ 本地安全
   ├─ API Key 仅保存在本机
   ├─ 数据库大小
   └─ 缓存 / 诊断入口
```

## 底部导航规则

### 1. 只承载顶级任务

底栏只放用户反复进入的主任务：

- `总览`：回到今日状态和同步入口。
- `书架`：进入所有书籍资产的总入口。
- `笔记`：进入划线、想法和导出。
- `复盘`：进入 AI 阅读成果。
- `我的`：进入低频入口和本机状态。

`统计` 和 `发现` 不放底栏，是为了避免底栏膨胀。它们仍然是重要功能，但不是每次打开应用都要进入的主路径。

### 2. 二级入口留在页面内

书架的二级入口继续由书架页内部承载：

- 微信书架
- 候选书架
- 本地书库

复盘的二级入口继续由复盘页内部承载：

- 书籍复盘
- 阅读指南
- 阅读报告

这样可以避免底栏承担二级导航职责，也避免底栏标签不断增加。

### 3. 保留抽屉作为兜底

第二阶段可以保留移动端抽屉，但弱化为“完整导航 / 更多菜单”兜底：

- 底栏负责高频路径。
- 抽屉可从“我的”或顶部菜单进入。
- 抽屉中仍展示完整导航树，便于快速到达低频页面。

如果实现复杂度需要进一步降低，第一版也可以只新增底栏和“我的”，暂不暴露抽屉入口；但不建议删除既有抽屉代码。

### 4. 新增入口准入规则

后续新增功能时，不默认进入底栏。先按以下准入矩阵判断入口位置：

| 问题 | 是 | 否 |
| --- | --- | --- |
| 用户是否每天或每次打开应用都会访问？ | 可考虑底栏 | 进入“我的”或页面内 |
| 是否代表一个完整主任务，而不是某个页面的筛选 / 子功能？ | 可考虑底栏 | 留在页面内 |
| 是否能用 2 个汉字清晰表达？ | 可考虑底栏 | 进入“我的”或抽屉 |
| 是否需要长期显示选中态？ | 可考虑底栏 | 使用按钮、卡片或二级 Tab |
| 是否会挤掉现有 5 个主入口之一？ | 需要产品级取舍 | 不进底栏 |

底栏变更必须满足：

- 仍然最多 5 项。
- 替换已有项时必须说明被替换入口的新归属。
- 新入口必须有对应 E2E 或手工视口验收。
- 不能因为某功能“重要”就放进底栏；底栏判断依据是高频主路径，不是功能价值。

推荐归属：

| 入口类型 | 推荐位置 |
| --- | --- |
| 主阅读流程 | 底栏 |
| 二级筛选、分段 Tab | 页面内部 |
| 设置、诊断、导出、关于 | “我的” |
| 低频管理页 | “我的”或抽屉 |
| 临时活动、实验功能 | 页面卡片或“我的”列表，不进底栏 |

## “我的”页定位

“我的”不是社交个人主页，也不是设置页改名。它应该是 Android 端的本机工作台：

- 管理本机状态。
- 收敛低频入口。
- 展示安全和数据归属。
- 提供设置、诊断、更新等系统操作入口。

产品表达建议：

```text
WxReadMaster 本机工作台
API Key 和阅读数据只保存在本机
```

这比“个人中心”更贴合当前产品，也能强化本地优先的信任感。

## “我的”页首屏布局

### 顶部状态卡

职责：让用户一眼知道当前设备和数据状态。

建议字段：

- 应用名：`WxReadMaster`
- 副标题：`本机阅读工作台`
- 同步状态：最近同步时间 / 尚未同步
- 凭据状态：已配置 / 待配置
- 本地保护提示：`API Key 和阅读数据只保存在本机`
- 主按钮：`立即同步`

不建议在首屏展示数据库路径、表记录数、完整诊断信息。这些内容应该进入折叠区或二级入口。

### 四宫格快捷入口

建议四项：

| 入口 | 目标 | 说明 |
| --- | --- | --- |
| 统计 | `stats` | 阅读时间和偏好 |
| 发现 | `discovery` | 搜索和推荐 |
| 设置 | 打开设置弹窗 | 账户、AI、外观、导出 |
| 本地数据 | 设置内本地诊断分组 | 缓存、数据库、诊断 |

四宫格每项保持短标题和一行说明，不放长文案。

### 列表分组

首屏下方可使用列表分组承载完整入口：

```text
阅读与数据
- 阅读统计
- 搜索与推荐
- 本地书库管理
- 笔记导出记录

系统设置
- AI 服务配置
- 外观与阅读偏好
- 代理与网络
- 更新与关于

本地安全
- 本机数据说明
- 数据库与缓存
- 导出诊断
```

列表项应使用 `button`，不要使用不可访问的 `div` 点击区域。

## 页面跳转策略

建议在 `App.tsx` 中新增一个 `mine` 视图标识。

```ts
type AppView =
  | "dashboard"
  | "shelf"
  | "notes"
  | "stats"
  | "readingReview"
  | "discovery"
  | "settings"
  | "mine";
```

如果当前类型不是集中定义，而是分散在状态和回调中，优先沿用现有结构，避免为新增页面引入大规模类型重构。

“我的”页内部入口只调用现有导航函数：

- `统计` 调用 `handleNavigate("stats")`
- `发现` 调用 `handleNavigate("discovery")`
- `设置` 调用 `handleOpenSettings()`
- `本地数据` 调用 `handleOpenSettings()` 并打开本地诊断分组

第一版如果设置弹窗暂不支持指定分组，可以先打开设置弹窗默认分类；不要为了这个入口重构整个设置状态。

## 组件建议

新增组件优先保持单一职责：

| 组件 | 职责 |
| --- | --- |
| `BottomNavigation` | 渲染 Android / 移动端底栏 |
| `MinePage` | 渲染“我的”页内容 |
| `MineStatusCard` | 渲染本机状态卡 |
| `MineShortcutGrid` | 渲染四宫格快捷入口 |
| `MineLinkSection` | 渲染分组入口列表 |

如果实现阶段希望更小改动，可以先把 `MinePage` 放在 `App.tsx` 同文件末尾，后续再抽到 `src/pages/MinePage.tsx`。但建议最终落到独立文件，避免 `App.tsx` 继续膨胀。

## 样式建议

### 底部安全区

在全局壳层补充底部安全区变量：

```css
.app-frame {
  --mobile-safe-bottom: env(safe-area-inset-bottom, 0px);
  --mobile-bottom-nav-height: 64px;
}
```

移动端工作区需要为底栏留出空间：

```css
.app-frame.mobile-shell .workspace {
  padding-bottom: calc(var(--mobile-bottom-nav-height) + var(--mobile-safe-bottom));
}
```

底栏固定在底部：

```css
.bottom-nav {
  position: fixed;
  right: 0;
  bottom: 0;
  left: 0;
  min-height: calc(var(--mobile-bottom-nav-height) + var(--mobile-safe-bottom));
  padding-bottom: var(--mobile-safe-bottom);
}
```

### 动态视口高度

Android WebView 和浏览器容器可能受地址栏、系统栏影响。建议在移动壳层使用 `100dvh` 兜底：

```css
.app-frame.mobile-shell {
  min-height: 100dvh;
}
```

不建议全局直接替换桌面端 `100vh`，避免破坏桌面端窗口布局。

### Toast 避让

移动端 Toast 需要避开底栏：

```css
.app-frame.mobile-shell .toast-viewport {
  bottom: calc(var(--mobile-bottom-nav-height) + var(--mobile-safe-bottom) + 12px);
}
```

如果 `localReader` 进入沉浸阅读模式，可以按页面类名单独决定是否隐藏底栏，避免底栏遮挡正文。

## 视觉规格

本节用于约束 Android 第二阶段的移动端视觉落地，避免不同页面各自发明尺寸。第一版优先使用 CSS 变量承载，不引入完整设计系统。

### 壳层尺寸

| Token | 建议值 | 用途 |
| --- | --- | --- |
| `--mobile-appbar-height` | `52px` | 顶部应用栏可点击区域 |
| `--mobile-bottom-nav-height` | `64px` | 不含安全区的底部导航高度 |
| `--mobile-safe-top` | `env(safe-area-inset-top, 0px)` | 状态栏 / 刘海避让 |
| `--mobile-safe-bottom` | `env(safe-area-inset-bottom, 0px)` | 手势条避让 |
| `--mobile-page-padding-x` | `16px` | 页面左右主内边距 |
| `--mobile-page-padding-y` | `16px` | 页面上下主内边距 |

页面可用空间必须同时考虑顶部栏和底栏：

```css
.app-frame.mobile-shell .workspace {
  min-height: 100dvh;
  padding-bottom: calc(var(--mobile-bottom-nav-height) + var(--mobile-safe-bottom));
}
```

### 底部导航

| 项目 | 规格 |
| --- | --- |
| 项数 | 固定 5 项 |
| 单项最小宽度 | `64px` |
| 单项最小高度 | `56px` |
| 图标尺寸 | `22px` |
| 标签字号 | `11px` 到 `12px` |
| 标签行高 | `1.1` 到 `1.2` |
| 顶部内边距 | `6px` 到 `8px` |
| 底部内边距 | `6px + safe-area-inset-bottom` |
| 选中态 | 主色图标 + 主色文字 + 轻背景或顶部短线 |
| 未选中态 | `var(--muted)` |

底栏标签必须使用短词：

- 推荐：`总览`、`书架`、`笔记`、`复盘`、`我的`
- 不推荐：`阅读工作台`、`微信书架`、`AI 复盘报告`

底栏选中态不应使用大面积高饱和背景，避免遮挡内容注意力。建议使用：

- 轻背景：`rgba(47, 111, 94, 0.10)`
- 或顶部短线：宽 `20px`，高 `3px`
- 或图标容器胶囊：高 `28px`，圆角 `999px`

### 阅读器沉浸模式

`localReader` 不复用普通页面的底栏安全区规则。它进入后应视为独立沉浸场景，优先保障正文高度、滚动稳定性和工具区可退让。

| 项目 | 规格 |
| --- | --- |
| 全局底栏 | 隐藏，不占用正文空间 |
| 全局顶部栏 | 隐藏，由阅读器自身返回区承载 |
| 工作区 padding | `0`，避免继承普通页面底栏避让 |
| 窄屏布局 | 单列：正文在上，检查区下沉 |
| 检查区高度 | `minmax(200px, 28vh)` 起步，正文优先 |
| 正文外边距 | `8px` |
| 正文内边距 | `clamp(24px, 8vw, 42px)` / `clamp(18px, 6vw, 34px)` |
| 工具栏 | 横向滚动，按钮不压缩正文 |
| 状态栏 | 窄屏隐藏次要进度文案和 `meter` |

第一版不引入阅读器 bottom sheet，只确保底栏不遮挡、正文不横向溢出、侧栏信息能下沉。

### 顶部应用栏

| 项目 | 规格 |
| --- | --- |
| 高度 | `calc(var(--mobile-safe-top) + 52px)` |
| 标题字号 | `18px` 到 `20px` |
| 标题字重 | `700` |
| 图标按钮 | `40px × 40px` |
| 左右边距 | `16px` 到 `18px` |
| 底部分割线 | `1px solid var(--line)` |

移动端顶部栏只展示当前页面标题，不展示两行说明。页面说明应放到内容区或导航抽屉中。

### “我的”页

| 区域 | 规格 |
| --- | --- |
| 页面左右边距 | `16px` |
| 区块间距 | `16px` 到 `20px` |
| 状态卡圆角 | `20px` 到 `24px` |
| 状态卡内边距 | `18px` 到 `20px` |
| 四宫格间距 | `10px` 到 `12px` |
| 四宫格单项高度 | `76px` 到 `88px` |
| 列表项高度 | 最小 `52px` |
| 列表项图标 | `20px` |
| 列表项右箭头 | `16px` |

“我的”页首屏应控制信息量：

- 状态卡最多 1 个主按钮。
- 四宫格最多 4 项。
- 每个快捷入口最多 1 行标题 + 1 行说明。
- 本地安全说明不超过 2 行。

### 卡片与列表

| 项目 | 移动端建议 |
| --- | --- |
| 卡片圆角 | `18px` 到 `22px` |
| 卡片内边距 | `14px` 到 `18px` |
| 卡片阴影 | 轻阴影或无阴影 |
| 列表项间距 | `8px` 到 `10px` |
| 主标题字号 | `17px` 到 `20px` |
| 二级标题字号 | `15px` 到 `16px` |
| 说明文字字号 | `12px` 到 `13px` |
| 正文行高 | `1.5` 到 `1.65` |

移动端不应在首屏出现三列以上栅格。除底栏外，常规内容优先使用单列；四宫格仅用于短入口。

### 触控与可访问性

| 项目 | 最低要求 |
| --- | --- |
| 触控目标 | `44px × 44px` |
| 主要按钮高度 | `44px` 到 `48px` |
| 次要按钮高度 | 不低于 `40px` |
| 表单输入高度 | 不低于 `44px` |
| 相邻点击区间距 | 不低于 `8px` |
| 可聚焦元素 | 必须有可见 focus 样式 |

所有列表入口和快捷入口必须使用 `button` 或可访问链接，不使用裸 `div` 承载点击。

### 深色模式

深色模式下重点检查：

- 底栏背景不应透明到露出复杂页面纹理。
- 选中态主色需要保持对比度。
- 分割线不应过亮。
- 卡片和页面背景要能分出层级。

建议底栏背景：

```css
.app-frame[data-effective-theme="dark"] .bottom-nav {
  background: rgba(22, 33, 31, 0.96);
  border-top-color: rgba(232, 238, 233, 0.10);
}
```

### 动效

第一版动效保持克制：

| 动效 | 建议 |
| --- | --- |
| 底栏切换 | 不做页面转场，只切换选中态 |
| 抽屉打开 | `180ms` 到 `220ms` |
| Bottom Sheet | `200ms` 到 `240ms` |
| 选中态变化 | `120ms` 到 `160ms` |
| Toast | 沿用现有过渡，补底部避让 |

遵循 `prefers-reduced-motion`：

```css
@media (prefers-reduced-motion: reduce) {
  .bottom-nav *,
  .sidebar,
  .sidebar-scrim {
    transition-duration: 1ms;
  }
}
```

## 页面级改造建议

底部导航只是壳层改造，后续仍需要让高频业务页符合移动端使用习惯。页面级改造应按收益和风险分阶段推进，不建议和底栏改造一次性混合提交。

推荐顺序：

```text
底部导航与我的页
→ 总览页移动化
→ 设置页移动化
→ 书架页移动化
→ 阅读器沉浸化
→ 首次 Android 引导
```

### 1. 设置页移动化

当前设置以弹窗和内部分类导航承载大量配置项，适合桌面端，但在 Android 上容易出现三类问题：

- 弹窗和系统手势区冲突。
- 弹窗内部导航、表单、说明文案共同挤压首屏。
- 高风险维护操作和日常设置在小屏内层级不够清楚。

第二阶段不必立刻重写设置页，但应确定方向：

- 桌面端继续使用设置弹窗。
- Android / 窄屏端逐步改成全屏设置页。
- 设置内部分类保留，但使用移动端列表或顶部横向分类。
- 高风险维护操作继续放在高级维护或本地诊断中，不进入首屏。

建议移动端设置结构：

```text
设置
├─ 账户与同步
│  ├─ 微信读书凭据
│  ├─ 同步状态
│  └─ 微信读书网络代理
├─ AI 服务
│  ├─ Provider
│  ├─ 模型
│  └─ Prompt / 资产偏好
├─ 外观与阅读
│  ├─ 主题
│  ├─ 字号
│  └─ 密度
├─ 导出
│  ├─ 导出位置
│  └─ 导出格式偏好
└─ 高级维护
   ├─ 本地诊断
   ├─ 缓存状态
   └─ 诊断导出
```

实现建议：

- 第一版可以让 `我的 > 设置` 继续打开现有设置弹窗。
- 当前过渡方案可把窄屏下的现有设置弹窗升级为全屏设置层，覆盖底栏并使用顶部横向分类。
- 移动端设置层需要使用 `100dvh`、顶部安全区和内部滚动区，避免内容被系统栏或底栏挤压。
- 后续新增 `SettingsMobilePage`，复用设置状态和保存逻辑。
- 不要复制两套设置业务逻辑；桌面弹窗和移动页面应共享数据模型、校验函数和保存函数。

### 2. 总览页移动化

总览页是 Android 打开后的首屏，不应直接复刻桌面 Dashboard 的信息密度。移动端总览只负责回答三个问题：

1. 当前同步和凭据是否正常？
2. 今天或最近有什么值得继续读？
3. 下一步最主要操作是什么？

建议首屏只保留：

- 顶部状态卡：同步状态、最近同步、错误提示。
- 今日阅读卡：今日阅读时间、最近读过的书。
- 主行动：立即同步 / 继续阅读 / 查看笔记。
- 轻量资产提醒：有待复盘、待导出或可更新资产时用一行提示。

建议下沉：

- 多图表趋势。
- 长文案洞察。
- 多列卡片矩阵。
- 低频维护入口。

实现原则：

- 不删除桌面端总览模块。
- 在移动端用 CSS 和轻量条件渲染调整顺序与显隐。
- 不在第一版新增复杂个性化排序。

首轮落地建议：

- 移动端将总览容器改为单列流式布局，避免桌面双列栅格压缩。
- 先展示 hero、同步状态、今日阅读工作台，再展示指标和辅助内容。
- 对所有总览直系卡片设置默认排序，避免未显式排序的卡片以 `order: 0` 抢占首屏。
- 隐藏移动端 hero 图片和 hero 快捷入口，避免首屏被装饰图和重复入口占满。
- 核心指标在小屏改为单列紧凑卡片，指标说明可下沉或隐藏。
- 今日阅读工作台保留主行动和最多两个备选动作，避免把“今日可做”列表前置到首屏。

首轮不做：

- 不拆 `DashboardPage` 的业务数据流。
- 不新增移动端专用 Dashboard 组件树。
- 不按用户行为做动态排序。
- 不移除桌面端图表、洞察和本地队列。

### 3. 书架页移动化

书架是移动端高频页，但当前桌面式工具栏和筛选项在小屏上容易占据过多首屏空间。移动端书架建议围绕“找书”和“继续处理”组织。

建议结构：

```text
书架
├─ Sticky 搜索框
├─ 横向 Tab：微信书架 / 候选书架 / 本地书库
├─ 当前筛选摘要
├─ 书籍列表
└─ 编辑模式 / 更多操作
```

关键规则：

- 搜索框固定在页面顶部或页面内容顶部。
- `微信书架 / 候选书架 / 本地书库` 使用横向分段控件。
- 分类筛选可以用横向 chips，不要在首屏铺成多行。
- 批量导出、批量维护、重新关联等操作进入“编辑模式”或更多菜单。
- 单本书卡优先展示封面、标题、作者、最近阅读/同步状态，长说明折叠。

不建议：

- 将书架二级入口放到底栏。
- 默认展示所有批量操作按钮。
- 在移动端保留桌面端多列卡片密度。

首轮落地建议：

- 搜索入口前置到筛选区之前，让“找书”成为首要动作。
- 类型、分类、书单筛选使用横向 chips，允许局部横向滚动，不撑开页面宽度。
- 统计摘要改为横向轻量卡片，不占据多行首屏。
- 书籍卡片在窄屏下保持单列，封面、标题、作者和状态不横向溢出。
- 同步按钮保持 `44px` 以上触控高度，并在窄屏下占满可用宽度。

首轮不做：

- 不改 `filterEntries` 和书架筛选业务逻辑。
- 不新增筛选 bottom sheet。
- 不把候选书架、本地书库放到底栏。
- 不重做书籍卡片内容结构。

### 4. 阅读器沉浸化

`localReader` 已经在壳层中跳过普通顶部栏，说明它天然需要独立的沉浸模式。底部导航改造不应顺手重做阅读器，而是先明确边界：

- 进入 `localReader` 时隐藏底部导航。
- 阅读器内部后续使用自己的浮动工具条。
- 目录、字号、主题、笔记、AI 操作使用 bottom sheet 或 popover。
- 阅读正文区域优先保证稳定行宽、上下留白和滚动性能。

建议后续阅读器移动端结构：

```text
阅读器
├─ 正文
├─ 轻触显示顶部返回区
├─ 底部浮动工具条
│  ├─ 目录
│  ├─ 字号
│  ├─ 主题
│  ├─ 笔记
│  └─ AI
└─ Bottom Sheet
   ├─ 目录
   ├─ 设置
   └─ 笔记/AI 面板
```

第一版已按最小闭环落地：

- `localReader` 下不渲染移动端全局底栏。
- 阅读器工作区不继承普通页面的底栏 padding。
- 窄屏改为正文在上、检查区下沉的单列结构。
- 阅读器工具栏横向滚动，避免按钮压缩正文。
- 状态栏在窄屏隐藏次要进度文案和进度条。

仍不做完整阅读器移动化：暂不新增 bottom sheet、浮动工具条或目录重构。

### 5. 首次 Android 引导

Android 用户的首次失败点通常不是导航，而是凭据、网络和代理。建议后续增加轻量引导，不做复杂 onboarding。

建议触发条件：

- 首次启动 Android 版。
- 未配置微信读书凭据。
- 同步连续失败且错误指向网络或代理。

引导内容控制在 3 步以内：

```text
1. 配置微信读书凭据
2. 如使用代理，填写 Android 代理工具提供的 HTTP/SOCKS 地址
3. 同步书架并开始使用
```

引导中必须明确：

- API Key 和阅读数据只保存在本机。
- 微信读书代理只影响同步接口，不影响 AI Provider 或应用更新。
- 如果失败，可以在设置中导出诊断。

不建议第一版做：

- 多页动画 onboarding。
- 账号体系。
- 云同步承诺。
- 自动检测第三方代理工具。

## 与本地阅读器的关系

当前 `localReader` 已经在壳层中跳过普通顶部栏。Android 第二阶段不建议立即把本地阅读器做成完整移动阅读器，但应避免底栏遮挡内容。

建议策略：

1. `activeView === "localReader"` 时默认隐藏底部导航。
2. `workspace--local-reader` 不继承普通移动页面底栏避让 padding。
3. 本地阅读器内部后续再单独设计底部浮动工具条。
4. 不在本次底部导航改造中重做目录、笔记、AI 面板。

这符合 YAGNI：先解决主壳层导航，不把阅读器沉浸体验混进同一轮改造。

## 实施阶段

范围：

范围：

- 保留现有抽屉入口，作为完整导航兜底。

范围：

- 总览页首屏降噪。
- 设置页移动端全屏化评估与拆分。

- 书架筛选不长期占据三行以上首屏空间。
- 阅读器正文不被底栏或全局导航遮挡。

### 阶段六：真机系统交互门禁

范围：

- 安全区、状态栏、手势区和三键导航验证。

- 系统返回对设置层、抽屉、阅读器和普通页面的行为验证。
- 真实网络与代理失败路径验证。

验收：

- 底栏在手势导航和三键导航下均不遮挡内容。
- 输入法弹出后关键输入框和保存按钮仍可触达。
- 系统返回优先关闭当前浮层，不破坏页面状态。
- 阅读器返回路径稳定，不直接丢失阅读上下文。
- 同步失败时用户能找到设置、代理说明和诊断导出。

- [ ] 桌面端侧边栏、标题栏、设置弹窗行为不回退。

## 夜神模拟器实时预览

如果当前没有物理 Android 设备，可以使用夜神模拟器作为 Android 端实时预览环境。夜神适合日常 UI、路由、同步错误提示和设置流程调试，但安全区、手势栏、输入法、系统代理等行为仍需在发版前尽量补真机验证。

### 前置要求

- 已安装 Android Studio 或 Android SDK。
- `adb` 在命令行可用。
- 夜神模拟器已启动。
- 优先使用 Android SDK 自带 `adb`，避免夜神自带 `adb.exe` 与系统 `adb` 版本冲突。

检查 `adb` 来源：

```powershell
where.exe adb
adb version
```

### 连接夜神

夜神常见调试端口是 `62001`：

```powershell
adb connect 127.0.0.1:62001
adb devices
```

期望看到：

```text
127.0.0.1:62001    device
```

如果使用夜神多开器，不同实例可能使用不同端口。优先在夜神多开器或安装目录配置中确认端口，再执行 `adb connect`。

### 启动实时预览

项目已提供 Android dev 脚本：

```powershell
cd "C:/Users/RHZ/CodeBuddy/wxreadmaster"
npm run android:dev
```

该脚本等价于：

```powershell
node "./scripts/run-tauri.mjs" android dev --force-ip-prompt
```

`--force-ip-prompt` 用于强制选择本机可访问 IP，避免移动端仍指向 `127.0.0.1`。不要在脚本里固定传 `--host 0.0.0.0`：`0.0.0.0` 适合 Vite 监听，但不能作为 Tauri 等待和 Android WebView 访问的真实地址。应由 Tauri CLI 提示选择类似 `192.168.x.x` 或夜神可访问的网卡地址。

### Vite host 策略

桌面开发继续默认监听 `127.0.0.1`，避免普通 `npm run dev` 暴露到局域网。

Android dev 时，Tauri CLI 会设置 `TAURI_DEV_HOST`。项目的 `vite.config.ts` 应根据该环境变量让 Vite 监听所有网卡，但 Tauri CLI 自身仍应使用真实 IP 作为 dev URL：

```ts
server: {
  host: process.env.TAURI_DEV_HOST ? "0.0.0.0" : "127.0.0.1",
  port: 5173,
  strictPort: true
}
```

这样可以兼顾桌面本地开发和 Android 模拟器实时预览。

### 常见问题

#### `adb devices` 看不到夜神

处理顺序：

1. 确认夜神已启动并完成系统加载。
2. 执行 `adb connect 127.0.0.1:62001`。
3. 如果仍失败，检查夜神多开实例端口。
4. 确认命令行使用的是 Android SDK 的 `adb`。
5. 必要时重启 adb：

```powershell
adb kill-server
adb start-server
adb connect 127.0.0.1:62001
```

#### 应用启动但页面白屏

优先检查：

- Vite dev server 是否启动在 `5173`。
- `npm run android:dev` 是否提示选择了正确 IP。
- Windows 防火墙是否阻止 Node.js。
- `vite.config.ts` 是否在 `TAURI_DEV_HOST` 存在时使用 `0.0.0.0`。
- 终端是否反复出现 `Waiting for your frontend dev server to start on http://0.0.0.0:5173/`；如果出现，说明脚本错误地把 `0.0.0.0` 传给了 Tauri CLI，应去掉 `--host 0.0.0.0`，只保留 `--force-ip-prompt`。

#### 页面加载但热更新不生效

处理方式：

- 保持 `npm run android:dev` 进程运行。
- 修改前端文件后等待 Vite HMR。
- 如果 Rust/Tauri 层修改未生效，重启 `npm run android:dev`。

#### 夜神网络代理和微信读书同步失败

夜神里的网络环境和真机不完全一致。同步失败时优先区分：

- 凭据是否配置。
- 微信读书网关是否可达。
- 是否需要在设置中填写 HTTP / SOCKS 代理地址。
- 诊断导出是否包含代理状态和同步错误详情。

### 夜神验收边界

夜神可以验收：

- 底部导航是否展示和切换。
- “我的”页布局和入口。

## 风险与取舍

### 风险 1：底栏和抽屉重复

底栏和抽屉同时存在时，导航会有两套入口。

处理方式：

- 底栏定义为高频主入口。
- 抽屉定义为完整导航兜底。
- 不在底栏中放所有入口。

### 风险 2：“我的”变成垃圾桶

如果所有放不下的功能都丢进“我的”，页面会失去结构。

处理方式：

- “我的”先定义为本机工作台。
- 入口按“阅读与数据 / 系统设置 / 本地安全”分组。
- 默认只展示高价值状态和快捷入口。

### 风险 3：底栏遮挡内容

移动端 fixed 底栏容易遮挡列表尾部、Toast 或浮层。

处理方式：

- 工作区统一增加底部 padding。
- Toast 使用底栏高度避让。
- `localReader` 默认隐藏底栏。

### 风险 4：改动范围扩大

新增底栏可能诱发页面级重构。

处理方式：

- 第一版只改壳层和新增 `MinePage`。
- 业务页面只接收更小的可用高度，不做结构重写。
- 书架和复盘二级入口继续留在页面内部。

### 风险 5：设置逻辑复制

如果为 Android 新做全屏设置页，容易复制一套桌面设置逻辑，后续导致配置保存、校验和文案不一致。

处理方式：

- 设置业务状态和保存函数保持共享。
- 桌面弹窗和移动页面只分离布局。
- 优先抽公共纯函数，不提前抽象大型设置框架。

### 风险 6：总览首屏继续桌面化

底栏完成后，如果总览仍是桌面 Dashboard 缩小版，Android 首屏会显得拥挤。

处理方式：

- 移动端总览只保留状态、今日阅读和主行动。
- 图表和洞察下沉。
- 不用移动端承载所有桌面端信息。

### 风险 7：模拟器通过但真机失败

夜神和浏览器 viewport 能快速发现布局问题，但不能完整模拟系统安全区、输入法、返回键、系统代理和厂商 WebView 差异。

处理方式：

- 浏览器和夜神用于日常快速回归。
- Android 发布前必须补真机系统交互门禁。
- PR 描述分开记录模拟器结果和真机结果。
- 真机失败项优先按“遮挡、不可保存、返回异常、同步不可恢复”排序处理。

### 拆分原则

- KISS：每个 PR 只解决一个可验收的移动端问题。
- YAGNI：不在壳层 PR 中提前实现阅读器 bottom sheet。
- DRY：设置页只拆布局，不复制保存、校验和 provider 逻辑。
- SOLID：壳层导航、页面布局、业务状态各自独立演进。

## 设计原则落地

- KISS：底栏只保留 5 个稳定主入口，不强行容纳所有导航项。
- YAGNI：不重写业务页面，不提前设计复杂的移动阅读器。
- DRY：桌面侧栏、移动抽屉、底部导航共享同一组导航动作和页面状态。
- SOLID：底栏、我的页、状态卡、快捷入口分离职责，避免继续扩大 `App.tsx`。

## 推荐结论

第二阶段 Android 改造建议采用：

```text
底栏：总览 / 书架 / 笔记 / 复盘 / 我的
我的：统计 / 发现 / 设置 / 本地数据 / 本地安全说明
抽屉：保留为完整导航兜底
```

这套方案不追求把桌面导航一比一搬到手机上，而是重新定义 Android 端的主路径：核心阅读任务一拇指可达，低频管理能力有清晰归宿。

后续页面级优化建议按“导航 → 总览 → 设置 → 书架 → 阅读器 → 首次引导”的顺序推进。这样可以先稳定应用壳层，再逐步降低核心页面的信息密度，避免一次性重写带来过高回归风险。
