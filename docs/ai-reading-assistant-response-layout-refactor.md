# AI 阅读助手回复排版与推荐卡片改造设计

## 背景

AI 阅读助手已支持普通问答流式输出、新书推荐、候选书架操作、消息编辑和历史恢复。当前主要问题不在能力缺失，而在窄聊天面板中的信息呈现方式：

- 推荐类回答经常以一整段自然语言输出，书名、推荐理由、风险、候选状态和可用性提醒混在一起。
- 下方推荐书卡片已经承载了结构化信息，但顶部 `answer` 又重复展开每本书详情。
- “搜索确认 + 加入本地候选”等操作按钮靠近标题区，容易抢占用户对推荐理由的注意力。
- 普通 Markdown-lite 段落和列表的视觉节奏偏弱，长文本看起来像日志输出。

这会导致用户在 320-420px 宽度的聊天面板中看到“文字墙”，扫读成本明显高于实际信息价值。

## 目标

将 AI 阅读助手回复从“长段落解释”收口为“摘要化回答 + 结构化卡片 + 清晰操作区”。

目标体验：

- 顶部回答先给结论，用户 3 秒内知道推荐方向。
- 推荐书详情主要由卡片承载，避免顶部和卡片重复解释。
- 操作按钮下沉到卡片底部，先理解内容，再执行动作。
- 普通问答保留 Markdown-lite，但增强段落、列表和提示信息的可读性。
- 不改变现有聊天消息、历史记录、流式输出和推荐书数据协议的主体结构。

## 非目标

本次不做：

- 不引入完整富文本渲染系统。
- 不做 Notion 式块编辑或消息块协议。
- 不引入向量数据库。
- 不做每条消息独立模型选择。
- 不做推荐卡片折叠、长文展开、复杂引用溯源。
- 不改变 `ReadingAssistantAnswer` / `ReadingAssistantMessageOutput` 的核心字段语义。
- 不把所有普通回答都强制卡片化。

## 当前问题分析

### 1. 回答职责过重

当前推荐回答常见结构：

```text
基于你已读 A 且对 B 有偏好，我推荐三本：X 是...，Y 详述...，Z 记录...。
这些书并非你本地候选书架已有项，且未在微信读书中搜索确认，请先确认...
```

这段文字同时承担：

- 推荐结论
- 用户画像解释
- 三本书详情
- 候选书架状态
- 微信读书确认提醒
- 下一步操作建议

职责过多，导致段落过长。

### 2. 顶部回答和卡片重复

`answer` 中已经解释每本书的推荐理由，`recommendedBooks` 卡片又展示 `reason`、`fit`、`risk`。这违反 DRY，也让用户重复阅读。

更合理的职责划分：

- `answer`：说明“为什么是这一组推荐”和“需要注意什么”。
- `recommendedBooks`：说明“每本书为什么适合你”。
- 卡片操作区：处理“搜索确认”“加入候选”等下一步动作。

### 3. 操作入口位置偏早

当前按钮与书名同一视觉层级，容易形成：

```text
创业维艰    搜索确认 + 加入本地候选
作者
推荐理由...
```

用户还没看完推荐理由就先看到操作。更合理的是：

```text
创业维艰
作者

为什么推荐
...

适合你
...

[搜索确认] [加入候选]
```

### 4. Markdown-lite 阅读节奏不足

普通问答虽然支持简化 Markdown，但段落、列表、强调和提示信息之间的层级较弱。长回答在窄面板里容易形成连续文本块。

## 设计原则

### KISS

优先调整 Prompt 输出约束、后端确定性回答模板和现有组件样式。不引入新消息协议。

### YAGNI

不提前建设复杂富文本、折叠详情、引用管理或向量检索能力。先解决当前截图暴露的拥挤问题。

### DRY

推荐详情只在卡片字段中展开，顶部回答不重复逐本长篇解释。

### SOLID

- 后端负责生成更稳定的回答结构和字段长度。
- 前端负责把已有结构以更清晰的层级展示出来。
- 推荐操作仍由推荐卡片负责，不把行为散落到 Markdown 文本中。

## 内容结构设计

### 推荐类回答

推荐类回答使用固定职责分层。

#### 顶部 `answer`

只保留摘要和提醒：

```md
我建议优先看这 3 本，它们都围绕科技商业人物、创业决策和失败教训展开。

选择依据：
- 你已读《奥尔特曼传》，对科技商业人物有明显兴趣。
- 当前候选书架偏人文与自我成长，缺少商业案例补充。

注意：
- 这些书需要先确认微信读书可用性，再决定是否加入候选。
```

约束：

- 总长度建议 120-220 个中文字符。
- 最多 1 个短段落 + 1 组列表 + 1 条注意事项。
- 不逐本展开 `recommendedBooks` 的 `reason`、`fit`、`risk`。
- 不在 `answer` 中内嵌操作口令，例如“点击按钮加入候选”。

#### `recommendedBooks`

每本书承担详情：

```text
创业维艰
本·霍洛维茨

为什么推荐
硅谷顶级创业者的实战回忆录，聚焦公司生死存亡时刻的真实决策。

适合你
能补足《奥尔特曼传》的组织管理视角，适合在技术阅读之间切换语境。

风险
管理案例密度较高，不适合想轻松阅读时打开。
```

字段约束：

- `reason`：1-2 句，说明书本价值。
- `fit`：1-2 句，说明与用户当前阅读画像的关系。
- `risk`：可选，说明可能不适合的点。
- 标题和作者不在 `answer` 中重复长篇描述。

### 普通问答

普通问答仍用 Markdown-lite，不强制卡片化。

建议输出形态：

```md
结论：你当前更适合先补一轮商业案例阅读，而不是继续扩展抽象方法论。

原因：
- 已读内容显示你对科技人物和组织决策有兴趣。
- 候选书架里自我成长类偏多，商业失败案例偏少。
- 下一本书最好能提供可复盘的真实场景。

建议下一步：
- 先读《创业维艰》。
- 如果想看反面案例，再读《坏血》。
```

约束：

- 优先短段落和列表。
- 避免 8 行以上连续段落。
- 重要结论放在第一屏。

### 确定性本地回答

本地确定性回答包括：

- 分类书目查询
- 统计聚合
- 微信读书搜索确认
- 单本 AI 复盘路由提示

这些回答不依赖模型自由生成，应使用稳定模板：

```md
当前本地明细可验证到 3 本经济理财类书籍。

范围说明：
- 统计聚合显示共 34 本。
- 这里只列出本地书架或本地详情可确认的书。
- 不会把统计总数展开成伪书名。
```

## UI 改造设计

### 1. 推荐卡片结构

现有卡片保留，但层级调整为：

```text
┌──────────────────────────────┐
│ 书名                          │
│ 作者 · 状态                    │
│                              │
│ 为什么推荐                    │
│ 推荐理由正文                  │
│                              │
│ 适合你                        │
│ 匹配用户偏好的说明            │
│                              │
│ 风险                          │
│ 注意事项                      │
│                              │
│ [搜索确认] [加入候选]          │
└──────────────────────────────┘
```

调整点：

- 操作按钮移动到卡片底部。
- 书名区只展示标题、作者、当前候选状态。
- `reason`、`fit`、`risk` 使用小标题分区。
- 字段为空时不渲染对应分区。
- 按钮允许换行，但不挤压标题。

### 2. Markdown-lite 样式

增强以下样式：

- 段落间距：连续段落之间增加明显间隔。
- 列表间距：列表项之间保持 4-6px 间距。
- 强调文字：`strong` 只用于重点，不扩大字号。
- 提示文本：可用轻量边框或浅色背景表示，但不新增复杂组件。
- 长词/书名：继续 `overflow-wrap: anywhere`，防止窄宽溢出。

### 3. 消息气泡宽度

保持当前用户/助手消息的左右对齐，不扩大到全宽。推荐卡片可占助手气泡内部宽度，但不单独变成外层大卡片，避免卡片套卡片。

## 后端改造点

### 1. Prompt 输出约束

修改 `reading_assistant_system_prompt()` 中推荐类输出要求：

- `answer` 必须是摘要，不逐本详述。
- 每本书详情写入 `recommendedBooks`。
- `answer` 使用 Markdown-lite：短段落 + 列表。
- 避免在 `answer` 中重复按钮动作。

### 2. 输出归一化

在 `normalize_reading_assistant_output()` 或推荐输出归一化路径中增加轻量防护：

- `suggestions` 继续限制数量。
- `recommendedBooks` 字段继续过滤空值。
- 必要时对 `reason`、`fit`、`risk` 做 trim。
- 不强行截断到破坏语义，首版以 Prompt 约束为主。

### 3. 本地确定性回答模板

调整以下本地回答生成函数：

- `answer_reading_assistant_category_books_query`
- `answer_reading_assistant_stats_aggregate`
- `answer_reading_assistant_weread_search`
- 单本 AI 复盘路由相关输出

让 `answer` 返回短段落/列表，而不是单句拼接长文本。

## 前端改造点

### 1. `ReadingAssistantMarkdownLite`

增强 Markdown-lite 渲染样式即可，暂不扩展复杂语法。

保留：

- 段落
- 有序/无序列表
- 加粗
- 行内代码

不新增：

- 表格
- 图片
- HTML
- 代码块执行

### 2. `renderRecommendedBooks`

调整推荐卡片 DOM 顺序：

- header：标题、作者、状态
- body：reason / fit / risk 分区
- footer：搜索确认、加入候选、已加入状态

按钮不再和标题同一行。

### 3. CSS

新增或调整类名建议：

```css
.reading-assistant-recommendation-card
.reading-assistant-recommendation-header
.reading-assistant-recommendation-body
.reading-assistant-recommendation-section
.reading-assistant-recommendation-section-title
.reading-assistant-recommendation-actions
```

注意：

- 不把推荐卡片做成过重的营销卡。
- 半径继续控制在 8px 内。
- 避免大面积单色背景。
- 移动和窄面板下按钮自然换行。

## 兼容性

历史消息兼容：

- 旧 `answer` 仍按 Markdown-lite 渲染。
- 旧 `recommendedBooks` 仍可展示。
- 旧长文本不会丢失，只是新消息会更结构化。

流式输出兼容：

- 流式过程中仍渲染 `answer` 文本。
- `recommendedBooks` 仍在最终结构化结果返回后展示。
- 不改变取消生成逻辑。

缓存兼容：

- 不修改缓存主键。
- 不改变 `input_hash` 语义。
- 不迁移历史记录。

## 测试计划

### Rust 单元测试

覆盖：

- 推荐类 Prompt 要求包含“answer 摘要化、详情写入 recommendedBooks”的约束。
- 分类书目查询返回 Markdown-lite 结构化短文。
- 统计聚合返回短段落/列表。
- 微信读书搜索确认不输出长段落。

### TypeScript 单元测试

覆盖：

- 推荐卡片操作按钮在 footer 区域。
- `reason`、`fit`、`risk` 分区按字段存在性渲染。
- 空字段不渲染空标题。
- Markdown-lite 段落和列表能正常静态渲染。

### 手工 QA

场景：

- “推荐 3 本可加入候选书架的新书”
- “我读过哪些经济理财类书籍”
- “帮我确认《创业维艰》微信读书是否可用”
- “基于我的笔记总结重点”
- 普通开放问答

验收：

- 320px 宽度下没有长按钮挤压标题。
- 顶部回答不再重复逐本长篇解释。
- 推荐卡片可扫读。
- 流式普通问答仍可用。
- 历史切换恢复后排版稳定。

## 分阶段实施

### P1：最小闭环

- 调整推荐类 Prompt 输出约束。
- 调整推荐卡片按钮到底部。
- 增强 Markdown-lite 段落和列表样式。
- 补推荐卡片渲染测试。

### P2：确定性回答模板

- 分类书目查询改为短段落/列表。
- 统计聚合改为短段落/列表。
- 微信读书搜索确认改为短段落/列表。
- 补 Rust 输出测试。

### P3：体验微调

- 根据真实截图调整间距、字号和按钮换行。
- 评估是否需要轻量提示块样式。
- 评估是否需要对过长 `reason` 做视觉约束。

## 风险与取舍

### 风险：Prompt 约束不稳定

模型仍可能输出长段落。

处理：

- 首版依赖 Prompt 约束。
- 如果仍不稳定，再在归一化层做轻量长度保护。

### 风险：卡片信息过少

如果顶部摘要变短，而卡片字段也过短，用户可能觉得解释不足。

处理：

- `reason` 和 `fit` 各保留 1-2 句。
- 不做硬性单句截断。

### 风险：样式改动影响其它 Markdown 输出

Markdown-lite 被普通问答复用。

处理：

- 只调间距和换行，不改语义。
- 用现有测试和手工 QA 覆盖普通问答。

## 验收标准

- 推荐新书时，顶部回答不再是一整段长文本。
- 顶部回答不逐本重复推荐卡片详情。
- 推荐卡片中“搜索确认”“加入候选”位于卡片底部。
- 卡片标题区在窄宽度下不被按钮挤压。
- 普通问答的段落和列表间距更清晰。
- 分类书目、统计聚合等本地回答不再输出拼接式长句。
- 不影响流式输出、取消生成、消息编辑、历史恢复。

## 结论

该改造是必要的体验收口，不是过度设计。

最小正确路径是：

1. 收紧 AI 输出格式。
2. 让顶部回答只负责摘要。
3. 让推荐卡片负责详情和动作。
4. 用轻量 CSS 改善 Markdown-lite 可读性。

不需要引入新数据库、新协议或复杂富文本系统。
