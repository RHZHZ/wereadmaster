# AI 阅读助手空状态与快捷问题收口设计文档

## 背景

AI 阅读助手面板已经完成 P0 主流程：右侧抽屉、上下文 chips、历史视图、设置视图、快捷问题和输入框。

最新截图中，全局助手初始态已经移除了 `最近对话` chip，但仍存在三个体验问题：

- 空状态大卡片和底部快捷问题重复展示同一个问题。
- `AI 资产` 是偏工程化的内部命名，用户理解成本较高。
- 初始态中部留白过大，视觉重心被推到页面底部。

这些问题不影响功能可用性，但会让 AI 阅读助手显得“已经有能力，但还没有被打磨过”。

## 目标

本次设计目标是让 P0 面板初始态更清晰、更少重复、更像一个随时可用的阅读助手。

核心目标：

- 空状态只承担“启动对话”的引导，不重复快捷问题。
- 快捷问题只出现一次，避免同屏重复。
- 上下文 chip 使用用户能理解的词，而不是内部工程概念。
- 初始态减少无意义留白，让视觉重心靠近输入区和快捷问题。
- 不改变后端能力、上下文构建逻辑和历史/设置视图结构。

## 非目标

本次不做：

- 不新增独立 AI 聊天页。
- 不新增一键加入候选书架。
- 不改造历史列表交互。
- 不引入复杂欢迎页或营销式介绍。
- 不新增向量知识库。
- 不改变 AI 推荐新书的后端策略。

## 当前问题分析

### 1. 空状态与快捷问题重复

当前初始态结构大致为：

```text
上下文 chips

[空状态卡片]
  推荐 3 本可加入候选书架的新书

快捷问题：
  推荐 3 本可加入候选书架的新书
  总结我的阅读偏好
  帮我制定本周阅读计划

输入框
```

问题在于第一个快捷问题被展示了两次。用户会感到这是两个不同入口，但实际点击意图相同。

### 2. `AI 资产` 命名不够用户化

`AI 资产` 对开发者表示已生成的复盘、路线、总结等缓存内容，但普通用户更自然的理解是：

- 阅读记忆
- 已生成总结
- AI 总结

如果直接显示 `AI 资产`，用户可能不知道它代表什么，也不知道为什么会参与回答。

### 3. 空状态面积过大

当前空状态使用大面积虚线卡片，占据中部空间，但信息只有一句快捷问题。它没有提供额外价值，反而让底部快捷问题和输入框显得拥挤。

## 设计原则

### 1. 一屏只出现一个主行动入口

同一个意图不要在空状态和快捷问题区重复出现。

推荐做法：

- 空状态展示轻量欢迎或当前能力提示。
- 快捷问题区承载具体可点击动作。

### 2. 上下文 chip 只展示“依据”，不展示“入口”

上下文 chip 表示本次回答可能使用的数据来源。

保留：

- 当前书
- 阅读统计
- 阅读画像
- 阅读记忆
- 复盘摘要
- 候选书

不作为顶部 chip 展示：

- 最近对话

原因：最近对话更像历史入口和连续追问能力，不应与阅读数据来源混在一起。

### 3. 面向用户命名，隐藏内部抽象

顶部 chip 应使用用户能直接理解的名词。

建议映射：

```ts
aiAssetSummary: "阅读记忆"
conversationHistory: 不在顶部 chip 展示
bookNotesSummary: "复盘摘要"
readingPersona: "阅读画像"
readingStats: "阅读统计"
```

`AI 资产` 可继续作为代码层概念存在，但 UI 不直接暴露。

### 4. 空状态要轻，不做说明书

空状态不应写太多解释性文案，也不应承担功能说明。

推荐文案：

```text
想问点什么？
```

或：

```text
围绕你的阅读记录继续提问
```

如果担心文案过于说明化，也可以只保留图标和较小标题。

## 推荐方案

### 方案 A：保留底部快捷问题，弱化空状态

这是 P0 推荐方案。

结构：

```text
AI 阅读助手                         [历史] [设置] [关闭]

阅读统计  阅读画像  阅读记忆

[轻量空状态]
  想问点什么？

快捷问题：
  推荐 3 本可加入候选书架的新书
  总结我的阅读偏好
  帮我制定本周阅读计划

输入框
```

优点：

- 改动最小。
- 不改变现有快捷问题位置。
- 去掉重复后，用户能明确看到三个可选问题。
- 输入框仍然稳定在底部。

约束：

- 空状态高度应明显收紧。
- 空状态不要再展示快捷问题文本。

### 方案 B：快捷问题进入空状态，底部只保留输入框

结构：

```text
AI 阅读助手                         [历史] [设置] [关闭]

阅读统计  阅读画像  阅读记忆

[空状态]
  推荐 3 本可加入候选书架的新书
  总结我的阅读偏好
  帮我制定本周阅读计划

输入框
```

优点：

- 初始态视觉重心更集中。
- 中部空间利用率更高。

缺点：

- 有消息后需要决定快捷问题是否仍保留。
- 会比方案 A 多一点布局分支。

### P0 选择

P0 采用方案 A。

理由：

- 当前代码已有底部快捷问题区。
- 只需要改空状态文案和高度，不需要重排对话视图。
- 风险低，符合 KISS 和 YAGNI。

## 详细设计

### 1. 顶部上下文 chip

全局助手建议展示：

```text
阅读统计  阅读画像  阅读记忆
```

书籍详情或笔记场景建议展示：

```text
当前书  复盘摘要  阅读记忆
```

候选书架场景建议展示：

```text
候选书  阅读统计  阅读画像  阅读记忆
```

`conversationHistory` 不展示在顶部 chip 中，但仍可作为后端上下文参与连续追问。

### 2. 空状态

空状态只在 `messages.length === 0` 时展示。

推荐展示：

```text
[MessageSquare icon]
想问点什么？
```

不再展示：

```text
推荐 3 本可加入候选书架的新书
```

原因：该文案已经在快捷问题区出现。

### 3. 快捷问题

快捷问题只在 `messages.length === 0` 的初始态保留在输入框上方。

全局场景：

```text
推荐 3 本可加入候选书架的新书
总结我的阅读偏好
帮我制定本周阅读计划
```

书籍/笔记场景：

```text
帮我整理 3 个问题
这本书最值得继续追问什么？
帮我提炼一个行动清单
```

候选书架场景：

```text
从候选书架里先读哪本？
这些候选书怎么取舍？
帮我缩小到 3 本
```

### 4. 空状态高度

当前空状态不应占据大面积中部区域。

建议样式：

```css
.reading-assistant-empty {
  min-height: 112px;
  padding: 24px;
}
```

如果消息区高度较大，空状态可垂直居中，但不要使用接近半屏的大卡片。

### 5. 消息出现后的布局

当 `messages.length > 0`：

- 不展示空状态。
- 消息列表占据主体区域。
- 不再展示底部通用快捷问题，避免和回答内追问建议重复。
- 如果模型返回 `suggestions`，在对应助手回答气泡内展示最多 3 条轻量追问按钮。
- 输入框保持底部稳定。

## 实施计划

### P0 改动

1. 调整 `CONTEXT_LABELS`：
   - `aiAssetSummary: "阅读记忆"`
   - 保持 `conversationHistory: "最近对话"` 作为历史依据标签，但不在顶部 chip 展示。

2. 调整空状态文案：
   - 从 `promptSuggestions[0]` 改为固定轻量文案，例如 `想问点什么？`。

3. 调整空状态样式：
   - 降低高度和 padding。
   - 避免大面积虚线卡片造成空旷感。

4. 保持快捷问题区不变：
   - 不迁移位置。
   - 不新增交互。

### P1 可选

- 根据 scope 提供更贴近场景的空状态文案。
- 快捷问题支持换行密度优化。
- 当面板高度较小时，快捷问题折叠为横向滚动或只展示前 2 个。

## 验收标准

### 视觉验收

- 初始态同屏不再重复展示同一个快捷问题。
- 顶部 chip 不出现 `最近对话`。
- 顶部 chip 不出现 `AI 资产`，改为用户可理解的 `阅读记忆`。
- 空状态高度明显小于当前截图，不形成大片空白卡片。
- 一体化输入区不被快捷问题挤压。

### 行为验收

- 点击快捷问题仍然只填充输入框，不自动发送。
- 保存对话历史开关仍然控制后端 `conversationHistory` 是否进入上下文。
- 历史按钮仍然进入最近对话视图。
- 消息出现后空状态消失。

### 回归验收

- `npx tsc --noEmit --pretty false` 通过。
- 阅读助手相关 Rust 测试通过。
- `npm run build` 通过。

## 风险与边界

### 风险：隐藏 `最近对话` 后用户不知道支持历史

缓解：

- 右上角历史图标继续保留。
- 历史视图标题继续使用 `最近对话`。

### 风险：`阅读记忆` 与 `阅读画像` 概念接近

缓解：

- `阅读画像` 表示用户偏好和统计沉淀。
- `阅读记忆` 表示已生成的 AI 总结、复盘、路线等内容。
- P0 先用更自然的词，后续可通过 tooltip 或设置页说明细化。

### 风险：空状态过轻导致初次用户不知道能做什么

缓解：

- 快捷问题仍然在输入框上方展示。
- 输入区 placeholder 保持 `问一个阅读问题`。

## 结论

当前 AI 阅读助手初始态已经符合 P0 功能结构，但需要做一次轻量 UI 收口。

P0 推荐采用“保留底部快捷问题，弱化空状态”的方案：去掉重复文案，把 `AI 资产` 改为 `阅读记忆`，降低空状态面积。这样能在不改变功能链路的前提下，让面板更清爽、更符合用户对阅读助手的直觉。
