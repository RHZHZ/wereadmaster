# 反馈成果回顾设计

## 背景

当前产品已经具备两类与反馈相关的能力：

- 书籍复盘的 `actionItems` / `reflectionQuestions` 已支持本地四态反馈，并持久化到 `ai_feedback_records`。
- 阅读指南 / 跨书路线的 `nextActions` 已具备轻量反馈与跨版本继承能力。

但现阶段“反馈”仍主要被当作状态闭环使用，缺少一个更完整的产品表达：用户在行动项、复盘问题和阅读指南上的记录，本质上也是阅读成果的一部分。下一次生成新版资产时，不应只继承状态，还应把这些成果整理为“上次沉淀”，帮助用户理解自己已经完成了什么、哪些推进方式有效、为什么新版建议发生变化。

这份文档只解决三个问题：

1. 反馈是否算阅读成果。
2. AI 是否应该在生成新版前总结这些成果。
3. 历史行动和 AI 整理后的成果回顾，最终放在哪一层展示。

## 核心判断

### 1. 反馈不仅是状态，也是阅读成果

用户对 AI 资产的反馈不只是“勾选完成”：

- `已完成` 代表已经发生的阅读整理动作。
- `暂不做` / `不适合` 代表用户对方法的筛选。
- 极短备注代表用户把阅读内容连接到现实问题、输出目标或当前节奏。
- 复盘问题的轻量回答代表新的思考结果。

因此，反馈不应只服务内部状态同步，也应被视为“用户已经沉淀出的结果”。

### 2. AI 应该做成果回顾，但不做表现评判

新版资产生成前，AI 应该对上一版反馈做一次短的整理，用来回答：

- 上一轮已经完成了哪些关键动作。
- 哪些做法对当前用户有效。
- 哪些做法已被明确放弃或判定不适合。
- 新版建议保留了什么、避开了什么、压缩了什么。

但 AI 不应做以下事情：

- 不评价用户“执行力”“完成度”“好不好”。
- 不生成长期用户画像或行为评分。
- 不把反馈上升为心理分析或教育式批注。

产品语义应是“成果回顾”或“上次沉淀”，而不是“AI 批阅”。

### 3. 展示必须按书聚合，不新增全局任务历史页

历史行动和 AI 成果回顾应该放在书籍上下文中展示，而不是做成一个全局任务页。

推荐层级保持为：

`阅读指南资产库 -> 书籍资产详情 -> 单资产版本详情`

其中：

- 资产库负责告诉用户“哪本书有资产、有没有反馈、值不值得进去看”。
- 书籍资产详情负责按书聚合“阅读指南 / 跨书路线 / 书籍复盘”。
- 单资产版本详情负责展示某一版的完整内容、行动反馈、更新依据和成果回顾。

如果后续需要补“历史行动与成果回顾”，优先作为书籍资产详情中的一个折叠区或标签区块，不新增全局历史中心。

## 用户视角

### 用户能感知到的体验

理想体验不是“系统在检查我有没有完成任务”，而是：

> 它记得我上次已经做了什么，这次建议更贴着我的真实进度。

用户路径建议如下：

1. 用户在当前版本里记录进展：
   - 标记 `待处理 / 已完成 / 暂不做 / 不适合`
   - 可选写一句极短备注
   - 复盘问题可记录轻量回答状态或简短回答
2. 用户点击重新生成时，看到说明：
   - 本次会参考上次记录的阅读成果，避免重复给出已完成或不适合的建议
3. 新版生成后，在版本详情看到一段短的“上次沉淀”：
   - 说明哪些动作已经完成
   - 哪些方式被放弃
   - 新版因此如何调整

### 推荐文案

- 反馈入口：`记录进展`
- 刷新前说明：`将参考你上次记录的阅读成果生成新版`
- 新版说明标题：`上次沉淀`
- 版本依据标题：`本次更新参考`
- 汇总区标题：`行动与反馈`

### 不推荐文案

- `执行力`
- `完成率`
- `AI 批阅`
- `用户表现`
- `反馈评价`

## 设计边界

### 要记录的内容

#### 书籍复盘

- `actionItems` 的四态反馈与极短备注
- `reflectionQuestions` 的轻量状态与极短回答

#### 阅读指南 / 跨书路线

- `nextActions` 的四态反馈与极短备注
- `reviewCheckpoints` 的轻量处理状态

### 不记录为反馈的内容

- `routeOverview`
- `books`
- `dependencies`
- 阅读统计复盘中的普通说明性段落

这些内容是 AI 资产正文的一部分，不是用户动作本身，不应强行转成反馈项。

### 不做的事情

- 不做全局任务页
- 不做提醒、排期、日历或任务系统
- 不做长篇反馈表单
- 不默认把用户完整备注发送给 AI
- 不做长期评分、用户画像或心理判断

## AI 成果回顾方案

### 目标

在新版资产生成前，利用上一版反馈生成一段短的“成果回顾”，作为新版上下文的一部分，并在新版中保留可见说明。

### 输入范围

只读取同一书籍上下文内、与当前资产相关的本地反馈：

- 同一 `feature + scope_id` 下上一有效版本的反馈
- 已完成动作
- 暂不做 / 不适合动作
- 极短备注
- 复盘问题的轻量回答

不默认扩大到整站历史，也不跨书做全局汇总。

### 输出定位

成果回顾的目的不是替代新版正文，而是为新版提供一个短的上下文总结。它应回答：

- 已经完成了什么
- 哪些方法被保留或被放弃
- 本次建议为何更轻/更重/更聚焦

### 命名建议

- `feedbackOutcomeSummary`
- `previousOutcomeReview`
- `updateFeedbackSummary`

推荐统一对外表达为：

- 页面标题：`上次沉淀`
- 内部字段：`feedbackOutcomeSummary`

### 示例

```json
{
  "feedbackOutcomeSummary": {
    "summary": "上一版中，用户已完成核心观点整理，并把其中 1 条观点连接到现实工作场景；章节卡片方式被标记为不适合，因此本次建议改为更轻量的一页复盘。",
    "appliedChanges": [
      "不再重复生成已完成的观点整理动作",
      "避开章节卡片式整理方式",
      "保留与现实应用相关的复盘输出建议"
    ]
  }
}
```

## 展示层级

### 1. 单资产版本详情

这是成果回顾的主要展示层。

建议顺序：

1. 当前版本摘要
2. 本次更新参考
3. 上次沉淀
4. 完整正文
5. 行动与反馈
6. 历史版本入口

原因：

- 成果回顾和本次更新依据都属于“为什么生成成这样”的解释信息。
- 用户在查看单个版本时最需要这类信息。
- 放在这里不会污染资产库和书籍资产详情首屏。

### 2. 书籍资产详情

这里不展示完整成果回顾正文，只做聚合摘要：

- 最近是否有反馈
- 最近一次有无成果沉淀
- 哪个资产类型最近有更新

如果需要补一个区块，建议叫 `成果回顾` 或 `行动与反馈`，默认折叠，按书聚合列出最近几次记录，不展开长文本。

### 3. 阅读指南资产库

这里只展示极轻量信号，不展示历史行动明细或 AI 回顾正文。

例如：

- 有无反馈
- 最近更新时间
- 是否建议刷新

避免把资产库变成全站任务历史堆积区。

## 数据与存储建议

### 现有反馈记录继续复用

继续使用 `ai_feedback_records` 记录用户反馈，不新增新的反馈表结构。

原因：

- 现有结构已经满足“版本绑定 + item 级状态 + 备注”的最小闭环。
- 反馈仍然是事实记录，不需要为“成果回顾”再单独拆状态表。

### 成果回顾作为资产输出的一部分

AI 生成的“成果回顾”建议写入新版资产的 `ai_outputs.output_json`，作为当前版本输出的一部分，而不是新增独立表。

原因：

- 它属于“这一版为什么这样生成”的版本内容。
- 和当前版本天然绑定，生命周期与版本一致。
- 符合 KISS / YAGNI，避免过早引入新的聚合表。

### 结构建议

对于阅读指南和书籍复盘，都允许新增可选字段：

```ts
type FeedbackOutcomeSummary = {
  summary: string;
  appliedChanges?: string[];
};
```

然后挂在对应资产结构里：

```ts
type ReadingRoute = {
  // existing fields
  feedbackOutcomeSummary?: FeedbackOutcomeSummary;
};

type BookAiSummary = {
  // existing fields
  feedbackOutcomeSummary?: FeedbackOutcomeSummary;
};
```

## 与现有文档的关系

- 本文档补充“反馈作为阅读成果”的产品口径。
- `docs/ai-feature-plan.md` 继续作为 AI 功能总方案。
- `docs/third-stage-plan.md` 继续作为“连续更新、更新依据与版本链”的阶段计划。

本方案不推翻现有“轻量反馈闭环”判断，只把它从“状态闭环”推进为“成果闭环”。

## 实施建议

### 第一阶段

- 明确文案：把“反馈”统一表达为“记录进展”。
- 在更新前说明里增加“会参考你上次记录的阅读成果”。
- 在版本详情中增加 `上次沉淀` 折叠区占位。

### 第二阶段

- 新版生成输入纳入上一版结构化摘要、当前阶段和本地反馈。
- AI 输出新增 `feedbackOutcomeSummary` 可选字段。
- 页面展示“本次更新参考”和“上次沉淀”。

### 第三阶段

- 在书籍资产详情按书聚合最近成果回顾摘要。
- 评估 `reviewCheckpoints` 是否也需要进入统一成果回顾。

## 最终结论

结论收口如下：

- 反馈需要记录，因为它本身就是阅读成果。
- AI 应该在生成新版前对这些成果做短整理，但不做用户表现评判。
- 历史行动和 AI 成果回顾应按书聚合展示，主展示层放在单资产版本详情，不单独做全局任务历史页。
