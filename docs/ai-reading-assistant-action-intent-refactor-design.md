# AI 阅读助手动作型 Intent 改造设计

## 背景

P4 真实 Provider 验收后，AI 阅读助手已经具备基础问答、结构化推荐、历史回放和隐私边界能力。但在真实使用中暴露出一个产品层问题：用户提出可执行请求时，助手仍按普通问答处理。

典型现象：

- 用户问“帮我确认《显微镜下的大明》在微信读书是否可以找到”，助手只基于通用知识回答“可能有，但无法实时确认”，而不是调用已有微信读书搜索能力。
- 用户问“总计历史记录呢”，助手只基于当前注入的局部统计回答，未明确统计口径，也未触发本地全量聚合。
- 快捷追问可能生成“帮我搜索...”这类动作型问题，但点击后仍进入普通 LLM 问答，形成体验落差。

根因不是模型质量，而是当前 AI 面板缺少“动作型 intent -> 确定性工具/本地查询 -> 结构化 UI 结果”的编排层。

## 目标

本次改造目标：

- 把可执行请求从普通问答中拆出来，优先走本地/微信读书确定性能力。
- 让助手回答和快捷追问都只承诺当前系统能完成的动作。
- 对统计类问题给出清晰口径，避免模型基于片段上下文猜测“历史记录”。
- 保持现有聊天、推荐卡片、候选书架、隐私边界不被大改。

## 阶段边界

本改造分两期：

- P1：微信读书可用性搜索 intent。解决“帮我确认某书是否能找到”这类已明确动作请求。
- P2：阅读历史/累计统计 intent。解决“总计历史记录呢”这类统计口径问题。

P1 只做搜索确认和结果展示，不做“第 N 条结果”快捷写入。加入候选仍通过结果卡片按钮完成，保留用户确认。

P2 先做轻量聚合和口径澄清，不一次性扩展完整年度/月度历史分析。

## 非目标

本阶段不做：

- 不做完整通用 tool calling 框架。
- 不让模型直接调用任意 Tauri 命令。
- 不新增远端同步、账号体系或微信读书写回。
- 不把 AI 面板改成全屏工作台。
- 不做全量自然语言 SQL 或自由数据库查询。
- P1 不做“帮我把第一条加入候选”这类序号引用写入动作。
- P2 不补齐数据库中不存在的历史明细。

## 问题拆解

### 1. 微信读书可用性确认

当前能力：

- 前端 `ReadingAssistantPanel` 已有 `searchBooks` 调用。
- 该能力只挂在推荐卡片的“搜索确认”按钮上。
- 普通聊天 intent 不会触发搜索。

用户期望：

- “帮我确认/搜索/微信读书是否有/能不能找到《X》”应直接返回搜索结果。
- 如果找到，应展示可选结果、封面、作者、类别，并允许加入本地候选。
- 如果未找到，应明确说未找到，而不是让模型凭常识猜。

当前问题：

- 模型说“系统未接入实时查询”在实现层面不完全准确，因为应用已有搜索命令。
- 模型推荐“去 App 内搜索”把用户已经发出的动作请求退回给用户。

### 2. 阅读历史/累计统计

当前能力：

- 阅读助手可注入 `readingStats`、`readingPersona`、阅读记忆和最近对话。
- 注入上下文有预算和范围限制，通常不是全量历史明细。

用户期望：

- “总计历史记录呢/全部历史呢/累计读了多少”应返回明确口径的聚合结果。
- 如果只有部分缓存，应展示当前可用范围，而不是模糊地说“更多历史未缓存”。

当前问题：

- “历史记录”语义有歧义：可能指阅读统计、AI 对话历史、书架历史、月度历史。
- 当前 intent 没有统计聚合分支，模型只能基于已有片段上下文回答。

## 设计原则

- KISS：先覆盖两类明确高频 intent，不引入通用工具系统。
- YAGNI：不做任意命令调用，只做白名单动作。
- DRY：复用现有搜索、候选确认、统计读取和展示组件。
- SOLID：LLM 负责语言解释，确定性命令负责真实数据和动作结果。

## Intent 设计

新增动作型 intent：

```rust
enum ReadingAssistantIntent {
    General,
    NewBookRecommendation,
    CandidateShelfDecision,
    WereadAvailabilitySearch,
    ReadingStatsAggregate,
}
```

### WereadAvailabilitySearch

命中条件：

- 包含“搜索”“确认”“找到”“有没有”“是否可以找到”“微信读书是否有”等词。
- 同时包含书名样式，优先识别 `《...》`。
- 或消息明确提到“微信读书”并询问某本书是否可用。

指代规则：

- 有 `《...》` 时直接使用书名。
- 在 `bookDetail` scope 下，用户说“这本书/当前书/它”时，可使用当前书标题。
- 全局 scope 下只有“这本书/它”且没有明确当前实体时，不搜索，返回口径澄清：“你指哪本书？”
- 不从最近对话里猜书名，除非最近一轮 action/result 明确绑定同一本书。

示例：

- `帮我确认《显微镜下的大明》在微信读书是否可以找到`
- `搜索一下《黑客与画家》微信读书有没有`
- `这本书微信读书有吗`

### ReadingStatsAggregate

命中条件：

- scope 为 `readingStats` 或消息包含“总计”“累计”“全部历史”“历史记录”“总阅读时长”“读了多少”等。
- 排除“AI 对话历史”“最近对话”等明确对话历史场景。

示例：

- `总计历史记录呢`
- `全部历史累计读了多久`
- `我总共读了多少本`
- `历史阅读分类怎么分布`

## 后端改造

### 1. 新增动作结果结构

动作输出使用同一份结构，同时出现在命令即时返回和历史消息 output 中：

```rust
pub struct ReadingAssistantAnswer {
    ...
    pub action: Option<ReadingAssistantActionOutput>,
}

pub enum ReadingAssistantActionOutput {
    WereadSearch(ReadingAssistantSearchOutput),
    StatsAggregate(ReadingAssistantStatsAggregateOutput),
}

pub struct ReadingAssistantMessageOutput {
    pub suggestions: Vec<String>,
    pub recommended_books: Vec<ReadingAssistantRecommendedBook>,
    pub basis_notice: String,
    pub action: Option<ReadingAssistantActionOutput>,
}
```

原因：

- `ReadingAssistantAnswer.action` 用于当前轮即时渲染。
- `ReadingAssistantMessageOutput.action` 用于历史落库和回放。
- 两者复用同一 action payload 类型，避免业务逻辑重复。

兼容策略：

- 旧历史没有 `action` 字段，按旧消息渲染。
- 新历史有 `action` 字段，按动作卡片渲染。
- 不做历史迁移。

### 2. 搜索动作处理

新增处理流程：

1. `infer_reading_assistant_intent` 命中 `WereadAvailabilitySearch`。
2. 从消息中提取搜索关键词。
3. 调用现有微信读书搜索服务，限制返回前 5 条。
4. 返回结构化搜索结果，不调用真实 Provider。
5. 历史中落库用户消息和动作结果，便于回放。

搜索结果防重：

- 渲染搜索结果前，必须检查 `shelf_entries` 和 `reading_item_states`。
- 命中 `shelf_entries` 的结果标记为“已在书架”或“已读/在读”，禁止加入本地候选。
- 命中 `reading_item_states` 且 `item_type='candidate'`、`status='toRead'` 的结果标记为“已在候选”。
- 未命中的结果才允许“加入本地候选”。

返回文案建议：

- 找到结果：`我在微信读书搜索到以下可能匹配项，请选择确认后加入本地候选。`
- 未找到结果：`当前微信读书搜索没有找到明确匹配项，可以换关键词或先保存为未确认候选。`
- 凭据/网络错误：使用现有错误文案，不让模型兜底编造。

### 3. 统计聚合动作处理

新增 `ReadingStatsAggregate` 本地轻量聚合。首版只返回当前数据库可验证的摘要，不做完整历史明细分析。

数据源建议：

- `shelf_entries`：书架条目、读完状态、最近阅读时间、分类、类型。
- 已缓存阅读统计表/服务：年度累计、最近月度、阅读时长、活跃天数。
- `reading_item_states`：本地候选、在读、待复盘/已整理状态。
- AI 资产摘要：只作为解释性上下文，不作为统计数字来源。

首版字段：

- 总阅读时长。
- 活跃天数。
- 读完/在读/候选数量。
- 分类分布。
- 当前可用范围。
- 数据更新时间。

返回时必须带口径：

```json
{
  "rangeLabel": "全部本地缓存",
  "lastSyncedAt": "...",
  "totalReadingTimeText": "...",
  "activeDays": 0,
  "finishedBooks": 0,
  "availableBreakdowns": ["年度", "月度", "分类"]
}
```

如果当前数据库缺少全量历史，只返回可验证字段，并说明缺失口径：

`当前本机缓存可验证的是 2026 年累计和最近月度；未缓存更早年份的逐月明细。`

歧义澄清：

- `历史呢`、`记录呢` 这类短句如果无法判定阅读历史还是 AI 对话历史，优先返回澄清选项，不调用 Provider。
- 明确包含“AI 对话/最近对话/聊天历史”时，进入现有 AI 历史视图或返回对话历史说明，不走阅读统计。

### 4. Prompt 收敛

动作型 intent 不应走 LLM。普通问答 prompt 需要补充：

- 不得承诺执行搜索、同步、写入或统计聚合。
- 当用户请求系统动作时，应引导或返回动作结果，不要编造远端查询结果。
- suggestions 不得生成当前系统无法直接处理的动作。

## 前端改造

### 1. 展示搜索结果卡片

复用推荐卡片里的搜索结果 UI：

- 封面。
- 书名。
- 作者。
- 分类。
- `加入本地候选`。
- `已在书架/已在候选` 状态。

区别：

- 这是用户主动搜索，不是 AI 推荐。
- 文案应使用“搜索结果”，不要写“推荐理由/适合你/取舍”。

### 2. 展示统计聚合卡片

新增轻量统计结果块：

- 总阅读时长。
- 活跃天数。
- 读完/在读/候选。
- 分类 Top 3。
- 数据范围和更新时间。

不要让模型用长段落解释所有数字；先给结构化指标，再给一句可读总结。

### 3. 快捷追问约束

搜索动作完成后建议：

- `帮我换关键词继续搜索`
- `帮我推荐同主题但未读过的书`
- `帮我对比搜索结果里的作者和版本差异`

统计动作完成后建议：

- `帮我按分类解释累计阅读偏好`
- `帮我比较今年和最近一个月的差异`
- `帮我找出下一步最值得复盘的书`

## 数据与隐私边界

- 微信读书搜索只在用户明确触发时调用。
- 不自动写回微信读书。
- 加入候选仍只写本地 `reading_item_states`。
- 统计聚合只读取本机缓存。
- 不输出 API Key、Cookie、数据库路径、原始 WeRead 响应。

## 权限与命令

优先复用已有命令：

- P1 复用 `search_books`。
- P1 加入候选复用 `upsert_reading_item_state`，但必须先做书架/候选防重。

如新增命令：

- 新增 `get_reading_stats_aggregate` 时，需要同步 Tauri permissions/capabilities。
- 新增 action 输出类型不应暴露额外敏感字段。
- 桌面 QA 必须验证真实 Tauri capability 未漏配。

## 验收用例

### 搜索确认

用例：

```text
帮我确认《显微镜下的大明》在微信读书是否可以找到
```

预期：

- 不调用真实 Provider。
- 调用微信读书搜索。
- 展示搜索结果卡片或明确未找到。
- 不再回答“系统未接入实时查询”。
- 搜索结果可加入本地候选。
- 已在书架/已读的搜索结果不可加入候选。

### 指代不明确

用例：

```text
这本书微信读书有吗
```

前置：全局 scope，无当前书实体。

预期：

- 不调用搜索。
- 返回“你指哪本书？”或展示输入书名的澄清提示。
- 不让模型猜测最近对话里的书名。

### 搜索失败

用例：

```text
搜索《不存在的测试书名XYZ》
```

预期：

- 展示未找到。
- 输入区恢复。
- 不生成虚构书籍。

### 全部历史统计

用例：

```text
总计历史记录呢
```

预期：

- 命中统计聚合 intent。
- 返回结构化统计卡片。
- 明确数据范围，例如“全部本地缓存”或“当前仅有 2026 年累计和最近月度”。
- 不把“历史记录”误解成 AI 对话历史。

### 口径澄清

用例：

```text
历史呢
```

预期：

- 如果无法判断，应给出 2-3 个可点选口径：
  - `查看全部阅读累计`
  - `查看最近月度历史`
  - `查看 AI 对话历史`
- 不直接编造统计。

## 测试计划

Rust 单测：

- `infer_reading_assistant_intent` 命中搜索 intent。
- 搜索 intent 能从 `《...》` 提取关键词。
- 全局 scope 下“这本书”不会误触发搜索。
- 统计 intent 能区分阅读历史和 AI 对话历史。
- 动作型 intent 不走 Provider。
- 动作结果能序列化/反序列化并历史回放。
- 搜索结果能标记 `shelf_entries` 已有书，避免重复候选。

前端测试：

- 搜索结果卡片渲染。
- 未找到状态渲染。
- 统计聚合卡片渲染。
- 动作型消息历史回放。

桌面 QA：

- 真实 Tauri 中搜索一本存在书。
- 搜索一本不存在书。
- 断网/凭据失败时错误可读。
- 统计聚合不横向溢出。
- 历史回放动作卡片稳定。

## 实施顺序

P1：

1. 扩展 `ReadingAssistantIntent` 和搜索 intent 推断。
2. 新增 `ReadingAssistantMessageOutput.action` 并接入历史序列化。
3. 实现 `WereadAvailabilitySearch` 后端处理，复用 `search_books`。
4. 增加搜索结果防重：检查 `shelf_entries` 和 `reading_item_states`。
5. 前端渲染搜索结果动作卡片。
6. 收紧搜索后的快捷追问，不生成“第 N 条写入”动作。
7. 补单测、前端测试和桌面 QA。

P2：

1. 扩展统计 intent 推断。
2. 明确统计聚合数据源和字段。
3. 实现轻量 `ReadingStatsAggregate`。
4. 前端渲染统计聚合卡片和口径说明。
5. 补统计口径澄清测试和桌面 QA。

## 风险与取舍

- 搜索结果依赖微信读书接口可用性，失败时必须展示明确错误。
- 书名抽取先只支持 `《...》` 和简单关键词，不做复杂 NLP。
- 统计聚合必须坚持“有什么数据说什么”，不要为了完整感补齐不存在的历史。
- 动作型 intent 增多后可能需要统一 action registry；本阶段先保持白名单分支，避免过度设计。
- 首版不做序号引用写入动作，避免 action result 引用、二次确认和历史回放复杂化。

## 当前结论

这次改造的关键不是继续调 prompt，而是给 AI 阅读助手补一层小而确定的动作编排。搜索和统计这两类请求应该由本地能力回答，LLM 只负责解释结果和生成后续追问。
