# 书架页视觉密度改造方案

## 结论

当前书架页的信息架构是合理的，但视觉密度偏松。

页面已经按用户任务组织成：同步书架、查看统计、按类型/分类/书单筛选、搜索书籍、进入书籍详情。这条路径是对的。问题在于顶部 Hero、统计卡和类型分段控件的视觉权重偏高，导致首屏更像展示型仪表盘，而不是高效的书架管理工具。

这次改造不改变筛选逻辑，不重写书架列表，只收紧页面顶部和筛选区的尺寸体系，让用户更快进入“找书、筛书、打开书”的任务。

## 与已有文档的关系

已有文档：

- `docs/bookshelf-filter-ui-refactor-plan.md`

该文档主要解决筛选语义：类型、分类、书单之间的层级关系。

本方案补充页面级视觉密度：

- Hero 是否过大。
- 统计卡是否过重。
- 类型分段控件是否过厚。
- 分类和书单 chip 是否占用过多首屏。
- 搜索框是否应该更靠近结果区。

两份文档不冲突。实施时应先保留已有筛选语义，再压缩视觉尺寸。

## 当前问题

1. 顶部 Hero 占比过高

- `bookshelf-toolbar` 中只有页面标题、说明和同步按钮。
- 但当前区域高度接近一个展示型 Hero。
- 对书架页来说，用户进入后更关心筛选和书籍列表，而不是再次阅读页面说明。

2. 统计卡过大

- `shelf-summary-row` 展示了 6 张统计卡：全部、电子书、有声书、文章收藏、私密、微信书单。
- 统计值有参考意义，但不应和筛选、搜索争夺首屏。
- 统计卡高度和间距偏大，让页面第一屏被摘要占据过多。

3. 类型分段控件视觉过重

- `bookshelf-filter-tabs--type` 的按钮高度和选中态都很强。
- 它像主导航，而不是当前页面内的筛选控件。
- 在书架页里，类型筛选重要，但不应压过搜索和结果。

4. 分类 chip 行距偏松

- 分类数量多，换行后会形成较高的 chip 区。
- `展开更多 4` 的权重偏高，容易被误认为主分类。
- 分类行应该是辅助筛选，不应成为首屏视觉主体。

5. 搜索框视觉稍厚

- 搜索是核心能力，宽度应保留。
- 但搜索框高度、留白和上方筛选区间距可以收紧。
- 搜索应更贴近结果列表，强化“筛选后找书”的任务路径。

## 改造目标

1. 首屏更像书架工具

- 保留页面识别和同步入口。
- 更快露出筛选、搜索和结果区域。
- 减少展示型大块留白。

2. 控件主次清晰

- 类型是主筛选。
- 分类和书单是从属筛选。
- 搜索是结果过滤工具。
- 统计是摘要，不是主任务。

3. 不改变数据逻辑

- 不修改书架同步逻辑。
- 不修改类型、分类、书单筛选口径。
- 不修改搜索匹配字段。
- 不改变书籍卡片点击和菜单行为。

4. 优先 CSS 级改造

- 第一阶段以 `src/styles.css` 为主。
- 只有在结构无法表达层级时，才调整 `BookshelfPage.tsx`。
- 不新增复杂状态，不拆大组件。

## 具体改造建议

### 1. Hero 收成标题工具条

当前：

- 标题区较高，说明文案和同步按钮分散在大卡内。

建议：

- `bookshelf-toolbar` 改成更紧凑的标题工具条。
- 标题、说明、同步按钮保持同一区域。
- 同步按钮保留右上角，但高度降低。

目标尺寸：

- Hero 高度降低约 `25%` 到 `35%`。
- `bookshelf-toolbar` padding 控制在 `20px` 到 `24px`。
- 标题字号控制在 `28px` 到 `34px`。
- 说明文案最多一行或两行，不做大段展示。

### 2. 统计卡改成紧凑摘要

当前：

- 6 张卡片横排，每张高度较大。

建议：

- 第一阶段仍保留 6 张卡，但降低高度。
- 第二阶段可评估改成一行摘要条。

第一阶段目标：

- `.shelf-summary-row` gap 降低。
- 统计卡 padding 降低。
- 卡片高度控制在 `72px` 到 `84px`。
- 数字字号控制在 `28px` 到 `32px`。

第二阶段可选：

```text
全部 498 · 电子书 472 · 有声书 25 · 文章收藏 1 · 私密 7 · 书单 4
```

不建议第一阶段直接改成摘要条，避免影响现有测试和视觉节奏。

### 3. 类型筛选降权但保持主筛选身份

当前：

- `bookshelf-filter-tabs--type` 较厚，选中态很强。

建议：

- 保留分段控件。
- 降低高度和按钮 padding。
- 选中态仍明确，但不需要像页面主导航。

目标：

- 高度控制在 `44px` 到 `48px`。
- 字号控制在 `15px` 到 `16px`。
- 外层 padding 降低。
- 不再形成过大的白色胶囊区域。

### 4. 分类和书单 chip 收紧

当前：

- 分类和书单 chip 数量多，换行后首屏占用明显。

建议：

- chip 高度控制在 `34px` 到 `38px`。
- 行间距减少。
- `展开更多` 改为低权重文本 chip。
- 分类和书单标签保留，但降低视觉重量。

注意：

- 不改变已有 `展开更多` 逻辑。
- 不把分类和书单混成同一行。
- 不让书单筛选默认折叠，除非移动端空间不足。

### 5. 搜索框靠近结果区

当前：

- 搜索框宽度合理，但视觉厚度略高。

建议：

- 搜索框高度控制在 `44px` 左右。
- 与书单筛选的间距收紧。
- 与结果列表保持明确关联。

目标：

- 搜索不抢过类型筛选。
- 搜索又足够明显，能作为进入结果列表前的最后一步。

## 实施顺序

1. 收紧 `bookshelf-toolbar` 高度、padding、标题字号和说明行距。
2. 收紧 `shelf-summary-row` 卡片高度、gap 和数字字号。
3. 降低 `bookshelf-filter-tabs--type` 的高度和按钮 padding。
4. 收紧分类、书单 chip 的 padding、间距和字号。
5. 收紧搜索框高度和上下间距。
6. 检查移动端是否仍能横向滚动或自然换行。
7. 跑书架相关单测和一次桌面/移动视觉截图。

## 验收标准

- 首屏明显减少展示型留白。
- 同步按钮仍清晰可见。
- 统计区不再压过筛选和搜索。
- 类型筛选仍是第一层筛选，但视觉不再像主导航。
- 分类和书单 chip 可扫读，不形成厚重标签墙。
- 搜索框清晰但不臃肿。
- 桌面端和移动端不出现文字溢出、按钮重叠、横向页面溢出。
- 书架筛选、搜索、展开更多、同步书架行为不回退。

## 不做范围

- 不修改书架同步接口。
- 不修改书架缓存数据结构。
- 不新增新的筛选维度。
- 不重做书籍卡片。
- 不引入虚拟列表。
- 不重新设计候选书架。
- 不改变暗色主题整体配色。

## 风险

- Hero 压缩过度可能让“本地缓存”边界说明不明显。
- 统计卡缩小后，大数字的视觉冲击会下降，但这符合工具页目标。
- chip 压缩过度可能影响移动端触控，需要保留至少 `32px` 到 `36px` 的点击高度。
- 如果同时改筛选数据逻辑和视觉，问题定位会变难；建议本轮只改视觉密度。

## 推荐落地方式

第一阶段建议采用 CSS-first：

- 只改 `src/styles.css`。
- 不改筛选派生数据。
- 不改书架卡片。
- 不改同步流程。

如果 CSS-first 后仍显得臃肿，再进入第二阶段：

- 将统计卡合并为摘要条。
- 将 Hero 改为真正的标题工具条。
- 移动端将分类和书单改为横向滚动容器。

这样符合 KISS 和 YAGNI：先解决当前明确的尺寸和层级问题，不提前重做书架信息架构。
