# 单本笔记章节目录与正文排版改造规划

## 背景

当前单本笔记页的列表视图存在两个体验问题：

- 章节快速目录常驻铺在顶部，章节数较多时会形成一大片 chip，挤压正文。
- 章节项主文案优先显示 `章节 550` 这类技术编号，用户难以判断它对应书中的哪一段。

从截图看，页面功能是可用的，但视觉更像数据列表或调试视图，不像阅读笔记页。笔记页的核心应是划线、想法和章节上下文，目录只是跳转工具，不应抢正文注意力。

## 目标

1. 降低顶部章节 chip 的视觉占用。
2. 明确章节目录与正文的主次关系。
3. 避免新增常驻左栏与全局侧边栏打架。
4. 章节项优先展示可读标题，技术编号降级为辅助或兜底。
5. 保持当前笔记展示、筛选、展开/收起能力不回退。

## 非目标

- 不重做单本笔记页整体导航。
- 不新增全局侧边栏或第二级应用导航。
- 不改微信读书笔记同步接口。
- 不引入新的 UI 组件库。
- 不改变划线、想法、导出和 AI 复盘的数据口径。
- 不在本轮实现复杂虚拟列表。

## 前置数据规则

章节展示顺序必须先稳定，否则 UI 再调整也会让用户不信任。

排序原则：

- 有章节目录时，按真实 `chapterIdx` 排序。
- 缺少章节目录时，按 `chapterUid` 升序兜底。
- 同一章节内，划线和想法按 `range` 起点排序；没有位置时再按创建时间兜底。

这条属于数据展示基础规则，不应只在某一个 UI 组件里临时处理。排序应尽量在映射层完成，让列表视图、导出、AI 输入都拿到稳定顺序。

## 推荐信息架构

采用保守方案：顶部轻工具栏 + 可折叠章节目录 + 正文区域。

```text
单本笔记页
├─ 顶部书籍信息与统计
├─ 视图切换 / 筛选 / 导出
└─ 笔记列表视图
   ├─ 章节管理工具栏
   │  ├─ 111 个章节可浏览
   │  ├─ 全部章节 / 只看有想法
   │  ├─ 章节目录
   │  └─ 展开全部
   ├─ 章节目录折叠面板
   └─ 章节正文流
```

不推荐第一版做常驻局部左栏。它虽然适合桌面端，但当前应用已经有全局侧边栏，再增加左侧目录容易造成层级竞争。后续如果目录使用频率很高，再评估桌面端局部目录栏。

## 推荐 UI

### 顶部工具栏

保留轻量控制：

```text
章节管理
111 个章节可浏览

[全部章节] [只看有想法]        [章节目录] [展开全部]
```

说明：

- `章节目录` 是开关按钮，控制下面的目录面板显示/隐藏。
- 默认不展开目录，避免一进入页面就看到大量 chip。
- `展开全部` 只作用于当前筛选后的可见章节。

### 章节目录面板

目录面板作为内容区内的折叠区域，不贴屏幕边缘，不使用全局导航样式。

建议布局：

```text
章节目录
[后记             6划线 · 1想法]
[第一章 xxx       2划线]
[第二章 xxx       1想法]
...
```

桌面端：

- 面板展开后使用两列或三列紧凑列表。
- 每个目录项高度稳定，避免标题长短造成大跳动。
- 当前展开章节可以轻微高亮。

窄屏端：

- 第一版可先复用折叠面板，不做抽屉。
- 如果面板过长，再升级为底部抽屉或全屏轻面板。

### 章节项文案

优先级：

1. 真实章节标题，例如 `后记`、`第一章 重新认识阅读`。
2. 如果标题缺失，才显示 `章节 550`。
3. `chapterUid` 不作为主标题；需要展示时放到弱辅助信息里。

示例：

```text
后记
6 划线 · 1 想法
```

而不是：

```text
章节 550
6 划线 · 1 想法
```

### 正文区域

章节正文继续作为主内容。

建议：

- 章节块之间保留更明显的纵向间距。
- 章节标题区展示真实标题，计数作为轻量 badge。
- 划线卡片减少强边框数量，更像摘录块。
- 想法/点评与划线保持明确分区，但不让每条内容都显得像独立重卡片。

示例：

```text
后记                                      6 划线 · 1 想法

划线
│ 我坚信人们对于我们的脊骨那无数次的探索...
  2023-12-29 · 位置 4852-5213

想法/点评
│ ...
```

## 交互规则

### 章节目录开关

- 默认收起。
- 点击 `章节目录` 后展开目录面板。
- 再次点击收起。
- 切换书籍时回到默认收起状态。

### 快速跳转

- 点击目录项后滚动到对应章节。
- 被跳转章节自动展开。
- 目录面板可以保持展开，第一版不自动关闭，避免用户需要连续跳转时重复打开。

### 筛选

- `全部章节` 展示有划线或想法的全部章节。
- `只看有想法` 只展示 `thoughts.length > 0` 的章节。
- 切换筛选后，目录和正文使用同一份 `visibleGroups`。
- 如果当前筛选无结果，保留现有空态。

### 展开全部

- 只展开当前筛选下的可见章节。
- 如果已全部展开，则收起全部。
- 不改变目录面板开关状态。

## 数据与组件边界

当前入口：

```tsx
<NoteList groups={notes.chapterGroups} />
```

推荐拆分：

```text
NoteList
├─ NoteListToolbar
├─ ChapterDirectoryPanel
├─ NoteGroupSection
├─ HighlightCard
└─ ThoughtCard
```

职责：

- `NoteList`：持有筛选、目录开关、展开章节状态。
- `NoteListToolbar`：只负责工具栏展示和回调。
- `ChapterDirectoryPanel`：只负责章节目录和跳转入口。
- `NoteGroupSection`：负责单个章节块。
- `HighlightCard / ThoughtCard`：继续负责具体内容卡片。

不建议把排序逻辑放进这些 UI 组件。排序应来自数据映射层或独立 helper，UI 只消费稳定顺序。

## 实施步骤

### 1. 稳定章节排序

- 修正 `chapterGroups` 顺序。
- 增加 Rust mapper 测试，覆盖章节目录存在和缺失两种情况。

验收：

- 截图中的章节不再出现 `550 -> 549 -> 424 -> 548` 这种跳序。
- 组内划线按位置从前到后展示。

### 2. 收起顶部常驻章节 chip

- 移除常驻横向 `note-chapter-index`。
- 新增 `isDirectoryOpen` 状态。
- 增加 `章节目录` 开关按钮。

验收：

- 页面初始状态顶部不再被大量章节 chip 占满。
- `全部章节 / 只看有想法 / 展开全部` 保持可用。

### 3. 新增章节目录折叠面板

- 展开后展示当前筛选下的章节目录。
- 目录项主标题使用章节标题。
- 点击目录项跳转并展开目标章节。

验收：

- 目录打开后可快速跳转。
- 目录与正文顺序一致。
- 技术编号不再作为主视觉信息。

### 4. 优化正文排版

- 章节标题区更聚焦真实标题。
- 弱化计数 badge 和卡片边框。
- 调整划线/想法块的间距和层级。

验收：

- 第一屏更像阅读笔记，而不是数据管理表。
- 划线正文阅读阻力降低。
- 长文本不溢出、不重叠。

### 5. 响应式验证

- 桌面端检查目录面板多列布局。
- 窄屏端检查目录面板高度和可滚动性。
- 如果窄屏目录过长，再评估底部抽屉。

验收：

- 小屏不出现横向溢出。
- 目录面板不会遮挡或挤坏正文。

## 验证清单

需要覆盖：

- 章节顺序是否稳定。
- `章节目录` 展开/收起。
- 点击目录项跳转并展开章节。
- `全部章节` 与 `只看有想法` 的目录和正文一致。
- `展开全部 / 收起全部` 对当前筛选生效。
- 空态仍然正确。
- 桌面端和窄屏端无重叠、溢出、跳动。

建议命令：

```bash
cargo test --manifest-path src-tauri/Cargo.toml mappers::notes::tests
npm test -- --run
npm run build
```

如启动前端，还需要用浏览器截图验证：

- 初始目录收起状态。
- 目录展开状态。
- 只看有想法状态。
- 窄屏状态。

## 工程原则

- KISS：第一版只做折叠目录，不引入桌面常驻局部侧栏。
- YAGNI：不提前做虚拟列表、复杂抽屉、章节搜索。
- DRY：目录和正文共用 `visibleGroups`，不复制筛选逻辑。
- SOLID：排序、目录、正文展示分层处理，避免 UI 组件承担数据清洗职责。

## 风险

- 章节标题可能缺失，需要保留 `章节 ${chapterUid}` 兜底。
- 书籍章节很多时，折叠面板仍可能很长，需要限制最大高度并允许内部滚动。
- 如果用户高频在章节间跳转，未来可能需要桌面端局部目录栏，但不应第一版就常驻。
- 目录展开后如果自动关闭，连续跳转效率会差；第一版建议保持展开。

## 推荐落地顺序

1. 确认章节排序修复已覆盖真实数据。
2. 改 `NoteList` 状态和工具栏，移除常驻 chip。
3. 新增 `ChapterDirectoryPanel`。
4. 调整正文视觉层级。
5. 浏览器验证桌面和窄屏。
