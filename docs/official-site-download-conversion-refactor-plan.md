# 官网下载转化与引流改造文档

## 文档状态

- 状态：草案，待确认后进入实施。
- 范围：现有官网引导页，不新增复杂产品站或内容系统。
- 目标页面：`website/App.tsx`、`website/index.html`、`website/styles.css`。
- 核心目标：让访问官网的人更快理解产品价值，并愿意下载桌面端或 Android 版。

## 背景

当前产品必须下载安装后才能真正使用，因此官网不能把主要目标放在“在线体验”。官网应承担四个职责：

```text
被搜索到
被快速看懂
建立下载信任
引导安装和首次使用
```

当前官网已经具备基本结构：

- 首屏：产品名、真实截图、下载按钮、GitHub 按钮。
- 功能区：同步书架、整理笔记、生成复盘、导出 Markdown。
- 截图区：书架、笔记、设置三类真实截图。
- 开源区：GitHub、技术栈、数据边界。
- 下载区：Windows 与 Android 下载说明。
- SEO：`title`、`description`、Open Graph、`SoftwareApplication` 结构化数据已存在。

这说明页面已经有基础，不需要推倒重做。下一步应把它从“产品介绍页”进一步收敛成“下载转化页 + 搜索入口页”。

## 核心判断

对于只能下载后使用的桌面应用，获客链路应是：

```text
搜索 / 社区 / 分享
  -> 官网引导页
  -> GitHub Releases / 商店页
  -> 下载安装
  -> 首次同步成功
```

因此，官网首页不需要塞进所有教程，也不需要做成文章页。它应该优先让用户完成两个动作：

1. 判断“这是不是我需要的工具”。
2. 点击下载或查看 GitHub。

专题页、教程页和案例页可以后续扩展，但 P0 阶段先改好现有引导页。

## 改造目标

### 业务目标

- 提高下载按钮点击率。
- 提高 GitHub Releases 访问率。
- 降低用户对“桌面端才能用”的疑虑。
- 让搜索用户能确认这是“微信读书笔记整理 / AI 读书复盘 / 阅读统计 / Markdown 导出”相关工具。

### 用户目标

用户进入官网后，应在 5 秒内知道：

- 这是微信读书个人阅读管理工具。
- 需要下载安装，主要是 Windows 桌面端，也提供 Android 版。
- 能同步书架、整理笔记、做 AI 复盘、导出 Markdown。
- 数据优先保存在本机，AI 只在手动触发时使用。
- 下载来自 GitHub Releases，项目开源可查。

### 技术目标

- 不破坏现有 Vite 官网构建。
- 不引入复杂 CMS。
- 不新增远端依赖。
- 不新增需要后端支持的下载统计闭环。
- 保留现有真实产品截图，不改用概念图。

## 非目标

本阶段不做：

- 不做完整内容矩阵。
- 不新增多路由专题页。
- 不实现账号系统、在线 Demo 或网页版功能。
- 不改应用主界面。
- 不接入付费广告平台。
- 不承诺下载量统计完全准确。
- 不为了 SEO 堆关键词。

## 当前问题

### 1. 首屏下载意图还可以更直接

当前首屏已有下载按钮，但“为什么需要下载桌面端”解释不足。用户可能会想：

```text
为什么不能在线用？
下载后数据安全吗？
是不是必须配置很多东西？
Windows 和 Android 有什么区别？
```

首屏或紧随首屏的位置需要明确回答这些问题。

### 2. 搜索关键词覆盖仍偏自然，不够主动

当前 `title` 和 `description` 已覆盖：

- 微信读书个人阅读管理
- 阅读复盘
- 笔记导出
- Windows
- Android

但页面正文还可以更明确覆盖用户真实搜索意图：

- 微信读书笔记整理
- 微信读书导出 Markdown
- AI 读书复盘
- 阅读统计管理
- 微信读书书架同步
- 本地阅读工作台

这些词应自然出现在可见正文、FAQ 和截图导读里，而不是塞在 meta 中。

### 3. 下载前信任信息不够集中

当前信任信息分散在首屏、开源区和下载区。更适合的结构是：

```text
下载按钮附近直接放 3 个低成本信任点：
- GitHub Releases 分发
- 开源项目可查看源码
- API Key 和阅读数据优先留在本机
```

这样用户不必滚到底部才确认下载来源。

### 4. 缺少下载 FAQ

桌面端产品的下载转化经常卡在安装疑虑，而不是功能疑虑。当前页面需要增加短 FAQ：

- 支持哪些系统？
- 为什么需要桌面端？
- 是否需要微信读书 API Key？
- AI Key 是否必须配置？
- 数据会不会上传？
- 安装包从哪里下载？

FAQ 既能服务转化，也能补搜索长尾。

## 推荐信息架构

保留单页结构，但调整叙事顺序：

```text
1. Hero：产品名 + 真实截图 + 下载 CTA
2. 下载前确认：平台、数据边界、开源来源
3. 核心场景：同步书架 / 整理笔记 / AI 复盘 / Markdown 导出
4. 截图导读：书架 / 笔记 / 设置
5. 为什么是桌面端：本地缓存、安全存储、长期整理
6. FAQ：安装、Key、AI、数据、安全
7. 下载区：再次转化
8. 开源区：源码和反馈入口
```

### P0 建议顺序

为了少改动，P0 不必真的大幅重排所有区块。优先改：

```text
Hero
功能区
新增 FAQ
下载区
SEO 文案
```

开源区可以暂时保留现状，只减少重复文案。

### 为什么需要桌面端

新增短区块，标题：

```text
为什么是桌面端？
```

正文：

```text
阅读笔记、AI Key、导出目录和本地数据库都更适合放在你自己的设备上。桌面端可以把同步、整理、复盘和导出放进同一个长期工作台里，而不是把阅读材料散落在多个网页服务中。
```

三点说明：

```text
本地缓存：书架、笔记和统计优先写入本机。
安全存储：微信读书 API Key 和 AI API Key 不在页面明文展示。
长期整理：复盘、候选书和导出文件可以持续积累。
```

### FAQ 文案

建议 P0 增加 6 个 FAQ。

#### 支持哪些平台？

```text
当前提供 Windows 桌面端和 Android 签名 APK，安装包通过 GitHub Releases 分发。
```

#### 为什么不能直接在网页里使用？

```text
应用需要管理本地数据库、导出目录和安全存储中的 Key。桌面端更适合长期保存阅读数据，也能减少把阅读内容交给网页服务托管。
```

#### 是否必须配置微信读书 API Key？

```text
需要。同步书架、笔记和统计前，需要在设置里填入微信读书 Skill 页面提供的 API Key。
```

#### 是否必须配置 AI Provider？

```text
不必须。书架、笔记、统计和导出可以先使用；AI 复盘、阅读助手等功能需要你在本机配置自己的 AI Provider 和 Key。
```

#### 我的阅读内容会自动上传吗？

```text
不会。应用只在你主动同步或主动生成 AI 内容时执行对应动作。AI 调用也只在你点击生成时发送确认范围内的内容。
```

#### 下载是否免费？

```text
项目当前开源发布，安装包在 GitHub Releases 获取。后续发布策略以项目说明为准。
```

## 引流内容边界

现有引导页只承接高意图访问，不承接全部内容营销。

### 适合放在首页

- 产品是什么。
- 核心场景。
- 真实截图。
- 下载入口。
- 数据和 AI 边界。
- FAQ。
- GitHub / 开源信息。

### 不适合放在首页

- 长篇教程。
- 多篇博客。
- 详细竞品对比。
- 大量关键词段落。
- 更新日志全文。
- 用户故事长文。

这些内容适合后续做专题页或文档页。

## 事件埋点建议

P0 可以先不接第三方分析工具，但代码结构上应预留事件函数。

建议事件：

| 事件 | 触发时机 | 目的 |
| --- | --- | --- |
| `download_click` | 点击主下载按钮 | 衡量下载转化 |
| `github_click` | 点击 GitHub 项目 | 衡量信任验证 |
| `release_click` | 点击 GitHub Releases | 衡量安装意图 |
| `faq_expand` | 展开 FAQ | 判断用户疑虑 |
| `screenshot_tab_click` | 切换截图导读 | 判断关注场景 |

首版可以只在控制台或内部函数占位，不一定上报。

如果后续接入统计，必须避免采集：

- API Key。
- 用户本地文件路径。
- 阅读笔记内容。
- 书名明细。
- AI 请求内容。

## 验收标准

### 内容验收

- 首屏明确出现“微信读书”。
- 首屏明确出现下载入口。
- 页面明确说明 Windows 与 Android 发布方式。
- 页面明确说明为什么需要桌面端。
- 页面明确说明数据和 AI 调用边界。
- FAQ 至少覆盖平台、Key、AI、数据、下载来源。

### SEO 验收

- `title` 覆盖“微信读书”“笔记整理”“AI 读书复盘”或“Markdown 导出”中的核心词。
- `description` 不超过常规摘要长度，表达完整。
- 页面正文自然出现核心搜索词，不堆砌。
- `SoftwareApplication` 结构化数据保持有效。
- 如添加 FAQPage，页面必须有同内容的可见 FAQ。

### 视觉与交互验收

- 390px 宽度下没有横向滚动。
- 下载按钮高度不低于 44px。
- FAQ 可点击或可读，不遮挡内容。
- 真实截图仍可见，不被 FAQ 和信任文案挤掉。
- 桌面端首屏能看到主 CTA。
- 移动端首屏能看到主 CTA 或明确的下载提示。

## 风险与取舍

### 风险 1：首页过长

处理：

- FAQ 控制在 6 个以内。
- “为什么是桌面端”只保留 3 点。
- 不把教程内容塞进首页。

### 风险 2：下载导向太强，削弱信任

处理：

- CTA 附近放下载来源和开源说明。
- 保留真实截图和 GitHub 入口。
- 不使用夸张承诺。

### 风险 3：SEO 文案变成关键词堆砌

处理：

- 关键词只放在真实语境里。
- 每段只表达一个用户问题。
- 不新增隐藏文本。

### 风险 4：移动端 Windows 下载语境不自然

处理：

- 移动端主 CTA 优先显示 Android 版。
- 同时说明 Windows 安装包也在 GitHub Releases。
- 下载区标题使用“下载 Windows 与 Android 版”。

## 推荐实施顺序


## 结论

现阶段最划算的路线是：

```text
先改现有引导页，让它成为下载转化页。
再根据 Search Console 和社区反馈，决定是否拆专题页。
```

这个策略符合 KISS 和 YAGNI：不提前做复杂内容系统，只先解决“用户看懂并愿意下载”的关键问题。
