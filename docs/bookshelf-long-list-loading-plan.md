# 书架长列表加载改造计划

## 文档状态

- 状态：已实现，后续仅保留真实大书架数据下的性能观察。
- 目标范围：微信读书书架页面的长列表展示、筛选后的可见数量控制、加载更多交互。
- 关联文件：`src/pages/BookshelfPage.tsx`、`src/pages/bookshelf-filter.ts`、`src/pages/BookshelfPage.test.tsx`。
- 当前触发问题：书架页目前会把筛选后的 `filteredEntries` 全量渲染为卡片；当用户书架条目达到几百到上千时，Tauri WebView 中的首次渲染、筛选重绘和滚动性能可能明显下降。

## 背景

当前书架页的数据流是：

```text
bookshelf.snapshot.entries
  -> filterEntries(entries, type, category, query)
  -> filteredEntries.map(ShelfEntryCard)
```

这个实现简单、直观，也适合几十到一两百条的书架。但随着后续能力扩展，条目数量会增长：

- 微信书架电子书。
- 微信有声书。
- 微信文章收藏。
- 用户后续可能同步更多书架索引数据。
- 本地书架与微信书架如果合并展示，也会进一步扩大列表规模。

因此需要先做一层轻量的长列表承载，避免页面一次性渲染所有卡片。

## 改造目标

一句话目标：

```text
保留现有书架卡片式浏览、搜索和筛选体验，同时限制首批渲染数量，并通过“加载更多”逐步展示后续条目。
```

具体目标：

- 初始只渲染一批可见条目，降低首屏渲染成本。
- 搜索、类型筛选、分类筛选仍然作用于全量数据。
- 筛选条件变化后，自动回到首批展示数量。
- 用户可以显式点击“加载更多”继续浏览。
- 列表区域展示当前可见数量与筛选后总数。
- 少量结果不显示加载更多控件。
- 保持现有空态、同步、打开详情、复制标题、保存候选等能力不变。

## 非目标

- 不做后端分页。
- 不做数据库分页查询。
- 不把微信读书接口改成分页同步。
- 不做传统页码分页。
- 不做 URL 页码状态。
- 不做每页条数设置。
- 不引入第三方虚拟列表库。
- 不重构书架卡片视觉设计。

这些能力在当前阶段都不是必要复杂度。第一阶段只解决“不要一次性渲染过多卡片”的问题。

## 方案选择

### 推荐方案：加载更多

默认参数：

```text
初始可见数量：96
每次追加数量：96
展示控件阈值：120
```

行为规则：

- 当 `filteredEntries.length <= 120` 时，直接展示全部结果，不显示加载更多。
- 当 `filteredEntries.length > 120` 时，展示前 `visibleCount` 条。
- 用户点击“加载更多”后，`visibleCount += 96`。
- 当 `visibleCount >= filteredEntries.length` 时，不再显示加载更多按钮。
- 搜索关键词、类型筛选或分类筛选变化后，`visibleCount` 重置为 `96`。

### 不推荐传统分页

书架页面是浏览型界面，不是数据表。页码分页会带来几个问题：

- 搜索结果容易被用户误解为“当前页搜索”还是“全部搜索”。
- 分类切换后页码重置会产生额外认知负担。
- 卡片浏览天然适合连续下滑和追加加载。
- 页码控件会把界面推向管理后台风格，不符合当前书架体验。

### 暂不做虚拟列表

虚拟列表性能更强，但也会引入额外复杂度：

- 网格卡片高度不一定完全稳定。
- 搜索、焦点、键盘导航、滚动定位要额外处理。
- 当前阶段没有证据表明必须支撑上万条书架。

如果后续书架条目超过数千，或“加载更多”仍有明显卡顿，再单独升级虚拟网格。

## 交互设计

### 列表计数

在书架条目区域上方或底部展示轻量文本：

```text
已显示 96 / 共 438 条
```

当全部展示时：

```text
共 86 条
```

计数使用筛选后的总数，不使用原始书架总数。

### 加载更多按钮

按钮文案：

```text
加载更多
```

辅助文案：

```text
继续显示 96 条
```

最后一批不足 96 条时，辅助文案显示真实剩余数量：

```text
继续显示 18 条
```

### 筛选变化

以下任一变化都重置可见数量：

- 类型筛选变化。
- 分类筛选变化。
- 搜索关键词变化。
- 清空搜索。
- 重置筛选。
- 书架同步完成后 entries 变化。

重置后首个结果仍可通过回车聚焦，保持当前键盘体验。

### 空态

空态沿用现有逻辑：

- 未同步书架：显示同步引导。
- 有书架但筛选无结果：显示清空搜索或查看全部书架。
- 加载中：显示骨架屏。

加载更多不参与空态判断。

## 数据与状态设计

新增状态：

```ts
const BOOKSHELF_INITIAL_VISIBLE_COUNT = 96;
const BOOKSHELF_VISIBLE_COUNT_STEP = 96;
const BOOKSHELF_LOAD_MORE_THRESHOLD = 120;

const [visibleCount, setVisibleCount] = useState(BOOKSHELF_INITIAL_VISIBLE_COUNT);
```

派生数据：

```ts
const shouldLimitVisibleEntries =
  filteredEntries.length > BOOKSHELF_LOAD_MORE_THRESHOLD;

const visibleEntries = shouldLimitVisibleEntries
  ? filteredEntries.slice(0, visibleCount)
  : filteredEntries;

const remainingEntryCount = Math.max(0, filteredEntries.length - visibleEntries.length);
```

渲染逻辑只使用 `visibleEntries.map(...)`。

`filteredEntries` 保持作为全量筛选结果，用于：

- 搜索结果总数。
- 空态判断。
- 是否显示加载更多。
- 回车聚焦首个结果的判断。

## 实现步骤

1. 在 `BookshelfPage.tsx` 增加长列表常量与 `visibleCount` 状态。
2. 基于 `filteredEntries` 计算 `visibleEntries`、`remainingEntryCount`、`nextLoadCount`。
3. 将卡片渲染从 `filteredEntries.map` 改为 `visibleEntries.map`。
4. 增加 `useEffect`，在筛选条件、搜索关键词或 entries 引用变化后重置 `visibleCount`。
5. 在列表底部增加计数和“加载更多”控件。
6. 保持 `firstResultRef` 指向当前可见列表第一项。
7. 为长列表行为补充测试。

## 测试计划

单元/结构测试：

- 长列表默认只渲染首批条目。
- 长列表显示“已显示 X / 共 Y 条”。
- 长列表显示“加载更多”按钮。
- 少量条目不显示加载更多。
- 搜索无结果时仍走现有空态。
- 筛选 helper 不需要知道分页状态。

交互测试如后续引入 Testing Library DOM 事件，可补：

- 点击加载更多后可见条目增加。
- 切换筛选后可见数量重置。
- 清空搜索后可见数量重置。

手工验证：

- 构造 200、500、1000 条书架数据预览页面。
- 检查首屏渲染是否明显快于全量渲染。
- 检查滚动到底部后加载更多位置是否自然。
- 检查打开详情、复制标题、保存候选菜单仍可用。

## 风险与边界

### 搜索仍是全量前端搜索

本方案只减少渲染数量，不减少 `filterEntries` 对全量数组的遍历。对几百到一两千条足够；如果后续数据量非常大，需要再考虑搜索索引或后端查询。

### 卡片数量增加后仍会累计渲染

加载更多不是虚拟列表，用户多次加载后页面里仍可能有很多卡片。这个行为符合“显式继续浏览”的预期，也比首屏全量渲染更可控。

### 同步后列表重置

同步完成导致 entries 更新时，列表会回到首批展示。这比保留旧滚动位置更可预测，也避免同步后卡片数量突然撑大页面。

## 验收标准

- 书架结果超过 `120` 条时，默认只展示 `96` 条。
- 点击“加载更多”后追加展示下一批条目。
- 剩余条目不足一批时，按钮辅助文案显示真实剩余数量。
- 搜索、类型筛选、分类筛选变化后，展示数量重置。
- 少量结果不显示加载更多区域。
- 现有书架空态、同步按钮、详情入口和卡片菜单不回归。
