# AI 阅读对话助手需求分析

## 结论

建议新增 AI 对话助手，但产品定位必须收口为：

```text
基于本地阅读上下文的 AI 阅读对话助手。
```

它可以采用“随时打开、自然提问”的对话形态，但不应成为泛用聊天机器人，也不应绕过现有 AI 功能的手动触发、输入确认、缓存和隐私边界。

本需求适合进入新阶段预研与 MVP 设计，原因：

1. 用户在阅读过程中确实存在即时解惑需求，对话入口比固定模板更自然。
2. 应用已经沉淀书架、候选、笔记、统计、阅读画像、复盘和阅读指南等本地资产，具备上下文基础。
3. 现有 AI 能力偏“生成资产”，缺少一个低摩擦的解释、追问和推荐入口。

第一版不应追求“什么都能问”，而是优先解决三个问题：

- 我现在读的内容或书籍，哪里不理解。
- 我最近适合读什么，为什么。
- 我的阅读记录说明了什么偏好，下一步怎么读。

## 背景

当前 AI 功能已经覆盖单本书笔记总结、阅读统计复盘、阅读指南、跨书路线图和选书决策。这些能力的共同特点是：

- 用户主动进入明确页面。
- 生成结构化结果。
- 输出可保存、可导出或可作为阅读资产继续使用。
- 输入范围相对固定，便于控制隐私和成本。

但用户在实际阅读和整理时，还会出现更碎片化的问题：

- 看到一个概念，想立即解释。
- 看完一段笔记，想知道它和过去读过的书有什么关系。
- 不想进入完整选书决策流程，只想问“最近我该读什么”。
- 想追问某个 AI 复盘结论，但当前页面不是对话式。

因此，对话助手的必要性来自交互效率，而不是新增一个无限能力入口。

## 产品定位

### 应该是什么

AI 阅读对话助手应该是一个“上下文受限的阅读协作入口”：

- 可以全局唤起。
- 能识别当前页面上下文。
- 能基于用户授权的本地阅读数据回答。
- 能解释推荐依据和数据边界。
- 能把复杂任务引导到现有结构化功能，例如阅读指南、选书决策和单本复盘。

### 不应该是什么

- 不做泛 AI 聊天。
- 不做无限自动推荐流。
- 不做后台自动生成。
- 不做全量书架和全量笔记默认上传。
- 不做整本书全文问答，除非用户明确在本地阅读器中选择文本。
- 不替代现有的阅读指南、选书决策和统计复盘结果页。

## 目标用户

### 普通阅读用户

核心诉求：

- 快速问清楚一本书适不适合自己。
- 不知道下一本读什么时获得建议。
- 对推荐理由和阅读路径有解释。

典型问题：

- “我最近适合读什么？”
- “这本书适合我现在读吗？”
- “为什么推荐这本？”

### 深度阅读用户

核心诉求：

- 围绕笔记、划线、想法做追问。
- 把散乱观点整理成可执行复盘。
- 关联以前读过的书和已有复盘。

典型问题：

- “这条笔记能怎么展开？”
- “这本书和我之前读的那本有什么差异？”
- “把我最近的笔记整理成 3 个主题。”

### 目标导向用户

核心诉求：

- 围绕目标生成阅读路线。
- 根据过往阅读记录调整难度和顺序。
- 获得短周期可执行计划。

典型问题：

- “我想提升表达能力，按我的历史推荐阅读路径。”
- “本周只读 3 小时，先读哪本？”
- “我最近阅读偏科了吗？”

## 核心场景

### 场景 1：全局即时解惑

用户在任意页面打开 AI 助手，提出自然语言问题。

需求：

- 助手能识别用户当前所在页面。
- 没有足够上下文时，必须明确说明需要用户补充。
- 对超出阅读上下文的问题，可以回答有限通用解释，但不应伪装成基于本地数据。

示例：

```text
用户：我最近适合读什么？
助手：基于你最近的阅读记录、候选书和已生成复盘，我建议优先考虑……
```

### 场景 2：当前书籍问答

用户在书籍详情、笔记页、阅读指南页打开助手。

需求：

- 默认注入当前书籍元数据、阅读状态、已缓存复盘摘要和本地整理状态。
- 不默认发送原始笔记正文，除非用户确认“使用本书笔记”。
- 回答必须区分“基于本地笔记”和“基于书籍基础信息”的依据。

示例问题：

- “这本书我现在该继续读还是先搁置？”
- “基于我的笔记，这本书最值得复盘的点是什么？”
- “我读这本书时应该重点关注什么？”

### 场景 3：本地阅读画像问答

用户在总览、统计、阅读报告或复盘中心打开助手。

需求：

- 默认只使用结构化统计、阅读画像和已生成缓存。
- 不发送笔记正文。
- 对阅读偏好使用“倾向”“本周期更接近”等表达，避免人格定论。

示例问题：

- “总结我的阅读偏好。”
- “我最近阅读有什么盲区？”
- “给我一个下周阅读建议。”

### 场景 4：个性化书籍推荐

用户询问下一本书或某类主题书籍。

需求：

- 首版优先基于本地候选书、书架、最近阅读和结构化统计推荐。
- 推荐结果必须包含依据、取舍和下一步动作。
- 没有候选书时，可以引导用户去发现页搜索或保存候选，不生成虚假的长书单。

推荐输出必须结构化：

```ts
type AssistantBookRecommendation = {
  title: string;
  author?: string;
  source: "local_candidate" | "bookshelf" | "generated_suggestion";
  reason: string;
  basis: string[];
  tradeoff?: string;
  nextAction: string;
  confidence: "low" | "medium" | "high";
};
```

### 场景 5：对已有 AI 资产追问

用户在单本复盘、阅读指南、选书决策、统计复盘结果页打开助手。

需求：

- 默认注入当前 AI 资产的结构化摘要，而不是重新发送原始输入。
- 用户追问时，回答需要引用当前资产字段，例如行动项、复盘问题、推荐候选。
- 如果用户要求重新生成正式资产，应跳转到对应功能页，而不是在聊天里静默覆盖缓存。

## 功能范围

### P0：MVP 必须具备

1. 全局 AI 助手入口
   - 可从主要页面打开。
   - 支持文本输入和多轮对话。
   - 对话窗口不遮断当前页面主要上下文。

2. 页面上下文注入
   - 书籍详情：当前书、阅读状态、本地状态、已缓存复盘摘要。
   - 笔记页：当前书和笔记概览，原始笔记正文需要用户确认。
   - 统计页：当前周期结构化统计和阅读画像。
   - 候选书架：候选书列表摘要和已有选书决策。
   - AI 资产页：当前资产结构化结果摘要。

3. 快捷问题
   - 推荐 3 本可加入候选书架的新书。
   - 从候选书架里选择下一本。
   - 总结我的阅读偏好。
   - 解释这本书我该怎么读。
   - 帮我制定本周阅读计划。
   - 解释当前 AI 结果的依据。

4. 依据说明
   - 回答必须说明使用了哪些本地上下文。
   - 数据不足时明确提示，而不是编造结论。
   - 推荐新书必须给出推荐理由，并提示用户确认后再加入候选书架。
   - 候选书架决策必须基于已有候选书给出取舍。

5. 隐私开关
   - 用户可以关闭个性化上下文。
   - 用户可以清空对话历史。
   - 用户可以在发送前看到本次将使用的上下文类型。

6. 本地阅读上下文库
   - 首版只沉淀结构化阅读上下文，不建立完整向量知识库。
   - 上下文来源包括最近阅读、候选书、本地状态、结构化统计、阅读画像和已生成 AI 资产摘要。
   - 助手每次回答只按问题选择最小必要上下文，不能默认带上全部数据。

### P1：体验增强

- 对话历史按本地保存，支持按书籍或页面上下文回看。
- 推荐反馈：喜欢、不感兴趣、已读过、推荐不准。
- 对话中支持把建议转为阅读行动项。
- 支持引用已有复盘、指南、选书决策中的具体条目。
- 支持将一次对话总结导出为 Markdown。
- 建立阅读资产摘要库，用于沉淀可被助手复用的书籍摘要、笔记摘要、主题摘要和 AI 资产摘要。

### P2：后续评估

- 基于本地索引的跨笔记语义搜索。
- 本地向量知识库，用于跨书召回、观点对比和高相关笔记检索。
- 多书观点对比。
- 长周期阅读目标追踪。
- 更细粒度的引用定位和来源跳转。
- 本地阅读器内选区问答与全局助手的历史打通。

## 非目标

第一版明确不做：

- 不做通用知识库聊天。
- 不做全书全文问答。
- 不做首版完整向量知识库。
- 不做自动向量化全量笔记。
- 不做后台自动推荐。
- 不做联网搜索外部书单。
- 不做自动购买、自动加入书架或写回微信读书。
- 不做复杂 Agent 编排。
- 不做跨设备同步对话历史。

## 信息架构建议

推荐采用“全局助手 + 场景快捷入口”的结构。

### 全局入口

入口位置：

- 桌面端：全局侧边栏或右下角悬浮按钮。
- 移动端：底部导航中的次级入口或页面右上角按钮。

职责：

- 承接随时提问。
- 显示当前页面上下文状态。
- 提供快捷问题。

### 场景入口

在已有页面中放置轻量入口：

- 书籍详情：问问这本书。
- 笔记页：基于笔记提问。
- 统计页：问问我的阅读偏好。
- 候选书架：问问下一本读什么。
- AI 资产页：追问这份结果。

场景入口只负责带上下文打开助手，不在页面内铺开长聊天流。

## 数据边界

### 默认允许使用

- 当前页面标识和上下文类型。
- 当前书籍基础信息。
- 本地阅读状态。
- 最近阅读摘要。
- 候选书摘要。
- 结构化统计。
- 已生成 AI 资产的结构化摘要。
- 用户显式输入的问题。

### 需要用户确认后才能使用

- 当前书原始划线。
- 当前书原始想法。
- 多本书笔记摘要。
- 对话历史作为后续上下文。
- 推荐反馈历史。

### 禁止发送

- AI API Key。
- 微信读书凭据。
- 本地数据库路径。
- 本地文件路径。
- 原始 WeRead API 响应。
- 全量书架原始数据。
- 全量笔记正文。
- 未经用户选择的本地书全文。

## 本地阅读记忆库

### 定位

需要建立本地阅读记忆库，但第一版不应直接做完整向量知识库。

推荐定义：

```text
本地阅读记忆库 = 结构化阅读上下文 + 阅读资产摘要 + 可追溯来源引用。
```

它的目标不是让 AI “记住一切”，而是让 AI 在用户提问时能稳定调用与问题相关的本地阅读资产。

### 分层设计

#### 第一层：结构化阅读上下文

P0 先做这一层。

内容包括：

- 最近阅读。
- 在读、已读、候选、待复盘、本地整理状态。
- 阅读统计和分类偏好。
- 阅读画像。
- 已生成单本复盘摘要。
- 已生成阅读指南摘要。
- 已生成选书决策摘要。

这一层不需要向量检索，主要用于回答：

- “我最近适合读什么？”
- “我的阅读偏好是什么？”
- “这本书现在该继续读吗？”
- “本周怎么安排阅读？”

#### 第二层：阅读资产摘要库

P1 再做这一层。

它不存整本书全文，而是沉淀可被 AI 调用的摘要资产：

```ts
type ReadingMemoryItem = {
  id: string;
  type:
    | "book_summary"
    | "note_summary"
    | "theme_summary"
    | "reading_route"
    | "book_decision"
    | "stats_review";
  bookId?: string;
  title: string;
  summary: string;
  tags: string[];
  sourceRefs: string[];
  updatedAt: string;
};
```

这一层用于回答：

- “我之前读过哪些和时间管理有关的内容？”
- “我的笔记里反复出现什么主题？”
- “这本书和我以前读过的哪些书有关？”

#### 第三层：本地向量知识库

P2 才评估这一层。

适用条件：

- 对话助手 P0/P1 已验证高频使用。
- 用户确实需要跨书笔记召回。
- 已有稳定的来源引用和隐私开关。
- 本地索引更新、清理和重建策略清楚。

首版向量化范围应限制为：

- 用户笔记摘要。
- 用户划线摘要。
- 已生成 AI 复盘摘要。
- 阅读指南摘要。
- 选书决策摘要。

不建议首版向量化：

- 全量书架原始数据。
- 全量原始笔记正文。
- 整本书全文。
- 微信读书公开点评和共读内容。

### 召回原则

- 优先结构化过滤，再做语义召回。
- 优先使用用户本地资产，不把公开内容混入个人记忆。
- 每个召回结果必须带 `sourceRefs`。
- 回答中必须能说明“基于哪些阅读记忆”。
- 数据不足时返回不足，不补写不存在的记忆。

### 更新策略

P0 不做后台自动索引。

推荐策略：

- 用户同步、生成复盘、生成阅读指南、生成选书决策后，更新对应摘要。
- 用户删除 AI 输出或清空对话记录时，不误删正式阅读资产摘要。
- 用户清空本地缓存或关闭阅读记忆时，停止调用并可清理摘要索引。
- 向量索引如果进入 P2，必须支持重建和清除。

### 隐私要求

- 阅读记忆默认保存在本机。
- 用户可以关闭“允许 AI 使用阅读记忆”。
- 用户可以查看本次回答使用了哪些记忆类型。
- 用户可以清空对话历史，但不应默认删除书籍复盘、阅读指南等正式资产。
- 如果用户选择清空阅读记忆，需要明确说明影响范围。

## 技术边界

### 调用边界

- AI 调用继续放在 Tauri/Rust trusted layer。
- 前端只提交用户问题、上下文选择和页面上下文引用。
- Rust 侧负责组装最小必要上下文、调用 Provider、缓存结果。

### 建议新增服务语义

```ts
type AssistantContextScope =
  | "global"
  | "book-detail"
  | "book-notes"
  | "reading-stats"
  | "candidate-shelf"
  | "ai-asset"
  | "local-reader-selection";

type ReadingAssistantRequest = {
  scope: AssistantContextScope;
  userMessage: string;
  includeSensitiveContext: boolean;
  contextRefs: string[];
};
```

### 缓存建议

首版可以只缓存对话记录，不把每轮问答拆成新的 AI 资产。

建议使用独立 feature：

- `feature`: `reading-assistant-chat`
- `scope_id`: `scope:{scope}:{optional_entity_id}`
- `prompt_version`: `reading-assistant-chat-v1`

对话历史和正式 AI 资产要分离：

- 对话是低摩擦解释层。
- 阅读指南、选书决策、单本复盘仍是正式资产层。
- 对话不能静默覆盖正式资产缓存。

## Prompt 要求

首版提示词必须固定版本，并包含以下规则：

- 只基于提供的上下文回答。
- 不编造未提供的阅读记录、笔记、评分、书籍内容。
- 推荐必须说明依据和不确定性。
- 数据不足时主动说明。
- 使用简体中文。
- 不输出心理诊断、人格定论或能力评价。
- 不承诺已经读取整本书，除非输入中明确包含全文。
- 当用户请求正式生成阅读指南、复盘或选书决策时，引导使用对应功能。

## 输出要求

普通回答可以是自然语言，但推荐、计划和复盘建议必须尽量结构化。

### 推荐回答

必须包含：

- 推荐书籍。
- 推荐理由。
- 数据依据。
- 取舍说明。
- 下一步动作。
- 不确定性说明。

### 阅读计划

必须包含：

- 时间范围。
- 阅读对象。
- 每次阅读动作。
- 验收标准。
- 复盘产出。

### 解释类回答

必须包含：

- 简明解释。
- 与当前书或用户阅读记录的关联。
- 可继续追问的问题。
- 数据边界说明。

## 隐私与安全

这是 P0 要求，不是后置优化。

必须满足：

- 默认不发送原始笔记正文。
- 默认不发送全量书架。
- 每次使用敏感上下文前要有显式确认。
- 对话记录只保存在本机。
- 用户可以清空对话记录。
- 用户可以关闭个性化上下文。
- 日志不得打印完整 prompt、API Key、原始 Provider 响应和用户原始笔记正文。

对话窗口需要展示轻量上下文提示，例如：

```text
本次回答将使用：当前书籍、最近阅读摘要、候选书摘要。
```

如果用户开启原始笔记上下文，则提示：

```text
本次将额外使用当前书的笔记正文，用于回答你的问题。
```

## 风险分析

### 风险 1：退化为泛聊天

表现：

- 全局入口变成无边界聊天框。
- 回答大量通用知识，与本地阅读资产无关。

应对：

- Prompt 明确阅读助手定位。
- UI 显示当前使用的阅读上下文。
- 对不相关问题给出简短回答，并引导回阅读任务。

### 风险 2：破坏现有 AI 资产体系

表现：

- 用户在聊天中生成了阅读指南，但不会保存到正式资产库。
- 聊天结果和阅读指南、选书决策产生冲突。

应对：

- 聊天只做解释和草案。
- 正式复盘、阅读指南、选书决策仍跳转现有页面生成。
- 聊天中不能静默写入正式资产缓存。

### 风险 3：隐私边界扩大

表现：

- 为了回答更聪明，默认发送全量笔记和书架。

应对：

- 上下文分级。
- 敏感上下文显式确认。
- 默认发送摘要，不发送原文。

### 风险 4：推荐幻觉

表现：

- AI 编造用户没读过的书、没保存的候选、没出现过的偏好。

应对：

- 推荐优先从本地候选和书架中选择。
- 生成型推荐必须标记为 `generated_suggestion`。
- 输出必须附带依据和置信度。

### 风险 5：成本失控

表现：

- 多轮对话持续携带大量上下文。

应对：

- 上下文摘要化。
- 历史对话只传最近轮次和摘要。
- 原始笔记按需传输。

### 风险 6：知识库过早复杂化

表现：

- 第一版就引入全量向量化、后台索引和跨书全文问答。
- 用户难以理解 AI 到底使用了哪些阅读数据。

应对：

- P0 只做结构化阅读上下文。
- P1 只做阅读资产摘要库。
- P2 才评估本地向量知识库。
- 所有阅读记忆调用都必须展示来源类型和可追溯引用。

## 优先级

| 优先级 | 能力 | 判断 |
|--------|------|------|
| P0 | 全局助手入口 | 必须有，否则不能满足随时解惑 |
| P0 | 页面上下文注入 | 必须有，否则只是普通聊天 |
| P0 | 本地阅读上下文库 | 必须有，否则 AI 无法稳定理解用户阅读情况 |
| P0 | 本地画像推荐 | 必须有，是核心差异化 |
| P0 | 隐私开关和上下文提示 | 必须有，否则风险不可控 |
| P0 | 数据不足提示 | 必须有，防止伪结论 |
| P1 | 对话历史 | 有价值，但首版可本地轻量实现 |
| P1 | 推荐反馈 | 有价值，用于后续画像优化 |
| P1 | 转行动项 | 有价值，但不应变成复杂任务系统 |
| P1 | 阅读资产摘要库 | 有价值，用于沉淀长期阅读记忆 |
| P2 | 语义搜索 | 价值高，但需要索引和引用策略 |
| P2 | 本地向量知识库 | 后置，必须先验证使用价值和隐私边界 |
| P2 | 多书深度对比 | 后置，需要更稳定的来源引用 |

## 验收标准

P0 完成后应满足：

- 用户可以从主要页面打开 AI 助手。
- 助手能显示当前使用的上下文类型。
- 在书籍详情页提问时，回答能引用当前书和本地状态。
- 在统计页提问时，回答只使用结构化统计和画像，不发送笔记正文。
- 用户询问下一本读什么时，优先基于本地候选和近期阅读给出推荐。
- 推荐回答包含推荐理由、依据、取舍和下一步动作。
- 助手能使用结构化阅读上下文，但不会默认读取全量笔记正文。
- 助手回答时能说明使用了哪些阅读记忆类型。
- 数据不足时给出明确提示和下一步建议。
- 用户可以关闭个性化上下文。
- 用户可以关闭 AI 使用阅读记忆。
- 用户可以清空本地对话历史。
- 未配置 AI Provider 时，引导去设置，不展示假回答。
- 对话不会自动生成或覆盖阅读指南、选书决策、单本复盘等正式资产。
- P0 不包含完整向量知识库、后台自动索引或整书全文问答。

## 实施建议

建议分三步推进：

### 阶段 1：需求原型

- 只做交互原型和上下文选择模型。
- 不接入复杂语义搜索。
- 验证全局入口、场景入口和上下文提示是否清晰。

### 阶段 2：P0 MVP

- 接入 Tauri/Rust trusted layer 调用。
- 支持全局对话、页面上下文、快捷问题和个性化推荐。
- 接入结构化阅读上下文库。
- 支持本地对话历史和清空。
- 支持关闭个性化上下文。

### 阶段 3：资产联动

- 对接已有复盘、阅读指南、选书决策资产。
- 建立阅读资产摘要库，优先复用已有 AI 资产摘要。
- 支持追问已有 AI 结果。
- 支持把对话建议转为轻量阅读行动。

### 阶段 4：本地检索增强

- 评估本地向量知识库。
- 只索引摘要资产和用户授权的笔记摘要。
- 补齐索引重建、清除、来源引用和隐私开关。

## 工程原则

- KISS：先做上下文受限的文本对话，不做复杂 Agent。
- YAGNI：不提前做全文问答、向量库和联网推荐。
- DRY：复用现有 AI Provider、缓存、设置和错误处理链路。
- SOLID：上下文收集、Prompt 构建、Provider 调用、对话存储和 UI 展示分层，避免前端直接拼完整 Provider payload。

## 最终建议

可以做，而且建议做成对话助手形态。

但第一版的成功标准不是“能聊得多广”，而是：

```text
用户随时提问时，助手能清楚说明它基于哪些本地阅读数据，给出可执行的阅读解释、推荐或下一步动作。
```

只要坚持本地优先、手动触发、上下文最小化和正式资产分层，这个需求可以自然补齐当前 AI 能力中“随时解惑”的缺口，而不会把产品带偏成泛聊天工具。
