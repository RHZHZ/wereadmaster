# 手机端触屏交互可用性改造方案

## 结论

当前手机端主流程没有确认的未修复阻断性问题。移动底栏、书架、设置、阅读报告、批量导出、书籍复盘导出、本地阅读器详情弹窗、选区相关记录浮层、输入范围弹窗、行动反馈弹窗、资产更新确认弹窗、通用确认弹窗、统计时间跳转弹窗、更新说明弹窗和横屏短视口均已有端到端交互验证。

本轮已确认并修复的手机端交互问题集中在三类：触屏不可发现、短视口裁切、定位上下文导致的浮层越界。其中“发现”页推荐横栏卡片的更多操作入口是首个 P1：按钮默认透明，只在 `hover` 或 `focus-within` 时显示。触屏设备没有稳定的悬停状态，用户看不到入口；同时透明按钮仍覆盖卡片右上角，会截获该区域的触摸，造成“点卡片却打开菜单”的非预期行为。

改造应以输入设备能力为边界，不重写推荐列表，不改变候选书保存逻辑。桌面端继续保留悬停显现，粗指针设备上常显更多操作并扩大触控热区。

## 与已有文档的关系

已有文档：

- `docs/android-mobile-shell-navigation-refactor-plan.md`

该文档负责 Android 壳层、顶部栏、抽屉和安全区。本方案只处理业务内容中的触屏交互可发现性，不调整移动导航结构，也不重复定义平台壳层。

## 当前问题

### P1：推荐卡片更多操作在触屏上不可发现

代码位置：

- `src/components/RecommendationList.tsx`
- `src/styles.css` 中 `.recommendation-rail-menu`

当前行为：

```css
.recommendation-rail-menu {
  opacity: 0;
}

.recommendation-rail-card:hover .recommendation-rail-menu,
.recommendation-rail-card:focus-within .recommendation-rail-menu {
  opacity: 1;
}
```

问题表现：

- 手机用户无法通过悬停发现更多操作。
- 菜单触发按钮使用通用 `.shelf-card-menu-trigger`，当前尺寸为 30 × 30，触控容错偏低。
- `opacity: 0` 不会移除命中区域，透明按钮仍可能截获触摸。
- 用户点击卡片右上角时，行为可能与点击卡片其他位置不同，但界面没有任何提示。

影响范围：

- 发现页推荐横栏中的“保存候选”等操作。
- 使用触屏的 Android 设备、触屏平板和部分触屏 Windows 设备。
- 桌面鼠标和键盘操作当前基本正常。

### P2：推荐卡片文案依赖悬停显示

推荐卡片的书名、作者和推荐理由默认透明，只在卡片 `hover` 或 `focus-within` 时显示。触屏用户首次点击前无法稳定看到推荐对象和推荐理由。

粗指针设备上应让 `.recommendation-rail-copy` 默认可见；桌面端继续保留悬停和键盘聚焦显现。

### P3：共享卡片菜单触控热区和弹层偏移不足

书架、候选书架和推荐卡片复用 `.shelf-card-menu-trigger`。桌面基础尺寸为 30 × 30；推荐卡片单独扩大按钮后，如果仍使用 `top: 36px` 的弹层偏移，菜单会覆盖触发按钮。

粗指针设备应统一使用 44 × 44 热区，并将菜单弹层起点调整到触发按钮下方。卡片右侧 padding 应同步增加，避免按钮压住标题。

### P4：悬浮 AI 按钮需要显式避让底部导航

窄屏下右下角 AI 阅读助手按钮与底部导航距离较近，截图上容易被按钮阴影判断为重合。移动端定位应显式基于 `--mobile-bottom-nav-height` 和 `--mobile-safe-bottom` 计算，确保按钮底部不进入 `.bottom-nav` 的矩形范围。

### P5：本地阅读器触屏选区缺少原生事件入口

本地阅读器正文只通过 `mouseup` 和 `keyup` 读取选区。手机长按选中文字通常依赖浏览器原生 `selectionchange`，不一定触发鼠标事件，因此会出现“文字已选中但本地划线/写想法工具条不出现”的问题。

阅读器应监听 `selectionchange`，并在正文 `touchend` 后复查选区。该修复只补齐事件入口，不改变选区解析、划线保存或 AI 提问逻辑。

### P6：本地阅读器侧栏图标操作触控热区过小

本地阅读器侧栏中的划线删除、想法删除、AI 提问复制/定位/删除按钮基础尺寸为 24 × 24。手机端需要较高精确点按，容易误点到卡片主体。粗指针设备上应扩大到 44 × 44，并同步增加卡片右侧留白，避免按钮覆盖文本。

### P7：AI 阅读助手在软键盘短视口下输入区被裁切

390 × 360 这类被软键盘压缩后的手机视口中，AI 阅读助手仍显示三条可选建议，导致底部输入区向面板外溢出。面板使用 `overflow: hidden`，发送按钮下半截被裁切，实际点按区域不完整。

修复策略：

- 在 `max-width: 720px` 且 `max-height: 430px` 时隐藏 `.reading-assistant-suggestions`，把空间让给输入区。
- 压缩输入区内部间距和 textarea 最小高度，但保留消息区滚动。
- 将 `.reading-assistant-model-chip` 和模型菜单操作项提升到 44 × 44 触控热区。

### P8：本地阅读器书内搜索浮层在短视口左侧越界

书内搜索浮层原本使用绝对定位并 `right: 0` 贴合工具按钮。手机端工具栏可横向滚动，当搜索按钮不在视口最右侧时，宽度接近视口的浮层会被按钮定位上下文带出屏幕，出现 `left < 0`，输入框和上一条按钮被裁切。

修复策略：

- 在手机断点下将 `.local-reader-search-popover` 改为 `position: fixed`。
- 使用 `left: 12px; right: 12px; width: auto` 约束到视口内。
- 增加 `max-height` 和 `overflow-y: auto`，避免短视口内容再次溢出。

### P9：本地阅读器目录/字号/行距/主题/导出浮层在短视口越界

目录、字号、行距、主题和导出浮层与书内搜索共享同一条移动端风险：它们原本依赖工具按钮的绝对定位。390 × 360 下，字号、行距、主题和导出浮层会向右或向下超出视口，导出主按钮还会被更具体的 `min-height: 36px` 规则压回低于 44px。

修复策略：

- 在手机断点下统一将 `.local-reader-outline-popover` 和 `.local-reader-tool-popover` 改为视口内 fixed 定位。
- 使用同一套 `left/right/max-height/overflow-y` 约束，避免每个工具按钮单独计算偏移。
- 用更具体的触屏规则把 `.local-reader-tool-primary-action` 提升到 44px 热区。

### P10：发现页搜索主操作按钮触控高度不足

390 × 360 下发现页搜索面板可以通过内容区滚动避让底部导航，没有横向溢出；但“智能范围”和“搜索”两个按钮实际高度为 43px，低于触屏 44px 基线。

修复策略：

- 在窄屏搜索表单规则中为 `.discovery-search-form .text-button` 和 `.discovery-search-form .secondary-action` 显式设置 `min-height: 44px`。
- 回归测试按两个阶段验证：搜索表单可触达；范围 tabs 可继续滚动到底栏上方。

### P11：设置 AI 表单输入和操作按钮触控高度不足

390 × 360 下设置弹窗本身会全屏覆盖底部导航，关闭按钮可达，内容区可滚动；但 AI 设置中的文本输入、下拉选择和卡片操作按钮实际高度约 43px，低于触屏 44px 基线。

修复策略：

- 在触屏/窄屏热区规则中只提升 `.credential-input input/select`、`.settings-card-actions > button`、AI 模型刷新按钮和模型候选按钮。
- 不修改 checkbox 本体尺寸，避免破坏现有偏好项布局。

### P12：选书决策输入弹窗按钮触控高度不足

390 × 360 触屏短视口下，“调整选书决策输入范围”三步弹窗整体未越界，且层级高于底部导航；但关闭按钮实测约 34 × 28，“下一步”和“取消”高度约 43px，低于 44px 触控基线。

修复策略：

- 补齐通用 `.icon-button` 基础样式，避免弹窗关闭按钮退回浏览器默认小按钮。
- 在移动端 `.reading-route-dialog` 范围内提升 `.icon-button`、`.secondary-action`、`.sync-button`、`.text-button`、搜索输入和下拉选择的 `min-height: 44px`。
- 不改选书决策三步状态和提交逻辑，只收敛按钮/输入控件触控面积。

### P13：阅读指南输入弹窗底部操作在短视口下不常驻可见

390 × 360 触屏短视口下，“调整阅读指南输入范围”弹窗本身有滚动能力，底部导航也被弹窗遮罩覆盖；但内容较长时底部“完成”按钮初始位置会落到视口下方，需要滚动后才能触达。

修复策略：

- 在移动端只针对 `.reading-route-input-panel` 内的弹窗使用“头部 / 当前书 / 可滚动候选区 / footer”四行布局，让“完成”保持在弹窗可视区底部。
- 继续让候选区内部滚动，避免 footer 覆盖候选说明或把候选书列表压缩到不可读。
- 保持遮罩 `z-index` 高于 `.bottom-nav`，因此不会出现按钮被底部 bar 抢点击的问题。

### P14：行动反馈编辑弹窗底部保存操作短视口不可直接触达

390 × 360 触屏短视口下，“编辑状态与记录”弹窗整体可滚动，但底部“取消 / 保存反馈”初始位置会落到视口外，保存前必须滚动；状态按钮和保存按钮也需要明确 44px 触控高度。

修复策略：

- 为 `.dialog-close` 补齐 44 × 44 关闭热区，避免所有复用关闭按钮的弹窗退回浏览器默认小命中区。
- 在短视口下把 `.ai-action-feedback-dialog` 改为固定行布局，隐藏次要“图片附件”说明，把备注 textarea 作为可伸缩区域。
- 为反馈状态按钮和底部保存/取消按钮设置 `min-height: 44px`，不改变反馈数据结构和保存逻辑。

### P15：资产更新确认弹窗按钮触控热区不足

390 × 360 触屏短视口下，“更新前确认”弹窗边界、滚动区和底部操作区可见，但关闭按钮约 30 × 24，“取消”按钮宽约 32px，主操作高度约 43px，低于手机触控基线。

修复策略：

- 复用 P14 的 `.dialog-close` 44px 关闭热区。
- 为 `.ai-asset-update-dialog-actions` 内的文本按钮和主按钮设置最小 44px 宽高。
- 保持 `.ai-asset-update-dialog-body` 局部滚动，不改变“进入生成页确认更新”的导航行为。

### P16：通用确认弹窗操作按钮触控高度不足

390 × 360 触屏短视口下，设置高级维护中的“确认清除 AI 输出缓存？”弹窗整体不越界，遮罩层级也高于底部导航；但“取消”和“确认清除”按钮实测高度约 43px，低于 44px 触控基线。

修复策略：

- 为 `.dialog-actions button` 设置 `min-width: 44px` 和 `min-height: 44px`，覆盖所有复用 `ConfirmDialog` 的操作按钮。
- 不调整确认弹窗的标题、文案和确认回调，避免影响数据维护流程。

### P17：统计时间跳转弹窗按钮触控高度不足

390 × 360 触屏短视口下，“跳到月份”弹窗边界在视口内，遮罩层级高于底部导航；但年份、月份和内联操作按钮依赖 padding 撑高，部分按钮实际高度会落到 43px。

修复策略：

- 为 `.stats-period-jump-grid button` 设置 `min-height: 44px`。
- 为 `.stats-period-jump-inline-actions .sync-button` 设置最小 44px 宽高。
- E2E 使用真实交互链路：打开统计页时间锚点，选择年份，再选择月份完成跳转。

### P18：更新说明弹窗短视口操作区被内部内容挤出视口

390 × 360 触屏短视口下，“发现新版本”更新说明弹窗外框被 `max-height` 限制在视口内，但内部 grid 没有确定高度，版本信息和摘要会把 `.update-dialog-actions` 推到视口下方，导致“稍后再说 / 查看详情 / 前往下载”无法直接点击，并可能与底部导航区域重合。

修复策略：

- 在移动断点下为 `.update-dialog` 设置确定高度：不超过当前视口减 24px，且最多 640px。
- 将 `.update-dialog-notes` 保持为可滚动内容区，操作区固定留在 grid 底部。
- 在 `max-height: 430px` 的短视口里隐藏次要描述和版本 meta，保留“标题 / 更新摘要 / 操作区”三段主结构。
- 为 `.update-dialog-actions` 内按钮设置最小 44px 宽高，并在移动端均分操作按钮宽度。

### P19：批量导出弹窗短视口操作区被设置区挤出视口

390 × 360 触屏短视口下，“批量导出向导”在启用“先同步缺失笔记再导出”后，三条策略、并发选择和排除开关会把 `.bulk-export-actions` 推到视口下方。实测操作区底部可到 753px，用户无法直接点击“重新预检 / 开始导出”，底部导航区域也会抢占视觉空间。

修复策略：

- 在短视口下将 `.bulk-export-dialog` 改为纵向 flex 布局，操作区固定为最后一段。
- 隐藏次要摘要、边界说明和策略描述，保留策略标题与控件。
- 将 `.bulk-export-setup` 改为局部滚动，确保策略、并发和排除开关仍可触达。
- 为 `.bulk-export-actions` 内文本按钮和主按钮补齐 44px 高度。

### P20：书籍复盘导出弹窗短视口操作区被列表内容挤出视口

390 × 360 触屏短视口下，“导出书籍复盘”选择页会显示步骤、摘要、搜索工具栏和大列表，`.bulk-export-actions` 初始会被挤到 459px 附近，导致“取消 / 下一步”不在视口内。

修复策略：

- 在短视口下将 `.book-review-export-dialog` 改为纵向 flex 布局。
- 隐藏次要步骤条、摘要和标题说明，把空间让给搜索工具栏、列表和底部操作。
- 让 `.book-review-export-body` 成为可伸缩区域，列表和设置确认区内部滚动。
- 为工具栏文本按钮和底部操作按钮补齐 44px 高度。

### P21：阅读报告生成弹窗短视口 footer 超出视口

390 × 360 触屏短视口下，“阅读报告生成”第一步仍保留说明文案、步骤条和四张大类型卡，`.monthly-report-poster-dialog-actions` 会被内容挤到视口外。实测类型步骤 footer 底部可到 382px，导致“关闭 / 下一步：选择月报时间”初始不可完整触达，并可能落入底部导航视觉区域。

同一路径中，统计页“生成阅读报告”入口按钮在短视口下实测约 43px 高，略低于 44px 触控基线，虽然可滚动点击，但需要纳入同一触屏路径约束。

修复策略：

- 在移动短高断点下为 `.monthly-report-poster-dialog` 设置确定高度，并按步骤拆分为“标题 / 可滚动主体 / footer”或“标题 / tabs / 可滚动预览 / footer”。
- 隐藏说明文案、步骤条和次要 hint，把空间让给当前步骤的核心选择控件。
- 让 `.monthly-report-kind-selector`、`.monthly-report-dialog-toolbar` 和 `.monthly-report-poster-preview-shell` 在弹窗内部局部滚动。
- 为入口按钮、类型卡、周期按钮、预览 tabs 和 footer 操作补齐 44px 触控高度。

### P22：本地阅读器详情弹窗触控热区不足

390 × 360 触屏短视口下，本地阅读器的划线详情、想法详情和 AI 提问详情弹窗本身可以保持在视口内，但弹窗内的复制、关闭、划线类型切换和 footer 操作仍沿用桌面紧凑尺寸。实测划线详情中“复制”为 28px 高，“划线 / 设为标记 / 设为疑问”为 30px 高，低于 44px 触控基线。

修复策略：

- 只在 `(hover: none), (pointer: coarse)` 下扩大本地详情弹窗控件热区，避免增加桌面端视觉密度。
- 覆盖 `.local-reader-thought-modal-close`、`.local-reader-thought-modal-copy`、`.local-reader-highlight-tone-actions button`、`.local-reader-ai-followups button` 和 `.local-reader-thought-modal-actions button`。
- 保持弹窗现有 grid 结构，正文继续在 `.local-reader-thought-modal-body` 内部滚动，footer 不覆盖正文。

### P23：本地 AI 提问浮层关闭按钮触控热区不足

390 × 360 触屏短视口下，本地阅读器“写想法”和“AI 提问”浮层可以保持在视口内，底部取消/保存按钮已被移动规则提升到 44px；但 AI 提问浮层右上关闭按钮被更具体的 `.local-reader-ai-composer header button` 规则压回 28px 高，触屏关闭需要较精确点按。

- 不改变 AI 提问保存、提交和追问逻辑，只补齐关闭按钮命中面积。
- 对写想法浮层和 AI 提问浮层共同增加 390 × 360 布局断言，确保 footer、textarea 和按钮均留在视口内。

### P24：带相关记录的选区浮层短视口压住状态栏

390 × 360 触屏短视口下，本地阅读器选区浮层如果同时出现“相关想法”和“相关 AI 提问”，仍按桌面相关分组高度估算，实际 bottom 可到 465px；阅读状态栏 top 约 173px，浮层会越过状态栏并落入底部区域，相关卡片需要在被覆盖区域中滚动点按。

修复策略：

- 在短高触屏断点下将 `.local-reader-selection-popover.has-thoughts` 改为自身滚动容器，并把最大高度收敛到 96-148px 区间。
- 让相关记录分组取消内层高度限制，统一交给外层浮层滚动，避免嵌套滚动抢手势。
- 让 `LocalReaderPage` 的选区浮层高度估算与短视口 CSS 上限一致，并在定位时用 `.local-reader-statusbar` 作为下边界。
- 增加 390 × 360 触屏用例，验证浮层不压状态栏/底部区域、按钮和相关卡片达到 44px、相关想法和相关 AI 提问详情都能点开。

### P25：窄屏主页面高频控件触控热区不足

390 × 844 触屏窄屏下，书架筛选 chips、书架搜索框、笔记同步/批量导出按钮、笔记筛选按钮和我的页同步按钮等高频控件存在 35-43px 的实际高度，略低于 44px 触控基线。复盘页搜索输入本体高度更小，但用户实际命中区域应以 `.search-field` 包裹热区为准。

同一路径里还发现主页面滚动容器会延伸到固定底部导航下方：例如“我的”页第二行快捷卡片首屏只露出 14px，可滚动后能完整点击，不属于阻断性不可交互；但完整露出的可操作目标需要确保不被底部 bar 抢点击。

修复策略：

- 只在移动 shell 范围内统一提升 `.sync-button`、`.search-field`、`.filter-tabs button` 和 `.bookshelf-filter-tabs button` 的 `min-height: 44px`。
- 校准 `--mobile-bottom-nav-height` 为底栏实际高度，并让移动 shell 的非阅读器 `.workspace` 可视高度截止到底部导航上方。
- 不修改各页面业务组件结构，不改变筛选、搜索、同步、导出和导航逻辑。
- 增加 390 × 844 触屏窄屏巡检用例，覆盖书架、笔记、复盘和我的页完整露出的高频控件，断言不存在小于 44px 或被底部导航抢点击的可操作目标。

### P26：主页面切页滚动位置继承与通用按钮热区不足

390 × 844 触屏窄屏下继续扫描所有可见交互目标时发现，总览页部分 `.text-button` 高度只有 21-40px，复盘/空状态里的 `.secondary-action` 高度约 43px；书架底部卡片的 `.shelf-card-menu-trigger` 在 mobile shell 下仍可能保持 30 × 30，因为原 44px 规则只依赖 `(hover: none), (pointer: coarse)`，在窄屏自动化或部分触屏桌面环境里不一定命中。

同一路径还发现，用户在一个主页面滚到底后点击底部导航切到复盘等页面，新页面会继承旧的 `.workspace.scrollTop`。结果顶部卡片会半露在 sticky topbar 下方，中心点被 topbar 命中，表现为“切页后第一屏不是页面顶部，顶部卡片不好点”。

修复策略：

- 在移动 shell 范围内统一提升 `.text-button` 和 `.secondary-action` 的最小宽高到 44px，保持桌面端紧凑按钮样式不变。
- 在移动 shell 范围内显式提升 `.shelf-card-menu-trigger` 到 44 × 44，并同步卡片右侧留白和菜单偏移，避免只依赖粗指针媒体查询。
- 在 `App` 中给 `.workspace` 增加 ref，主视图或复盘子视图切换后将非本地阅读器工作区滚动位置归零。
- 扩展 390 × 844 主页面巡检用例，覆盖总览底部按钮、书架卡片菜单、笔记/复盘底部次级操作，并验证切页后没有被 sticky topbar 或底部导航抢点击。

## 推荐方案

### 1. 明确隐藏控件的命中规则

桌面默认状态下，菜单不可见时同时禁用指针命中：

```css
.recommendation-rail-menu {
  opacity: 0;
  pointer-events: none;
}

.recommendation-rail-card:hover .recommendation-rail-menu,
.recommendation-rail-card:focus-within .recommendation-rail-menu {
  opacity: 1;
  pointer-events: auto;
}
```

这样可以避免透明按钮截获卡片点击。`pointer-events` 不影响键盘 Tab 聚焦；按钮获得焦点后，父卡片的 `focus-within` 会恢复菜单显示和指针命中。

### 2. 使用输入能力媒体查询处理触屏

优先根据输入设备能力适配，不使用 JavaScript user agent 判断：

```css
@media (hover: none), (pointer: coarse) {
  .recommendation-rail-menu {
    opacity: 1;
    pointer-events: auto;
  }

  .recommendation-rail-menu .shelf-card-menu-trigger {
    width: 44px;
    height: 44px;
  }
}
```

选择该方案的原因：

- CSS 直接表达“不能悬停”或“粗指针”，职责清晰。
- 能覆盖手机、平板和触屏桌面设备。
- 不需要新增 React 状态或全局平台服务。
- 保留鼠标设备现有视觉密度。

如果实际 Android WebView 对组合媒体查询支持不一致，可退化为在现有 `@media (max-width: 980px)` 中常显入口，但应先以真机结果为依据。

### 3. 保持卡片操作边界清晰

推荐卡片继续使用两个独立按钮：

- 卡片主体按钮负责打开书籍详情。
- 右上角更多按钮负责打开操作菜单。

不要把长按卡片作为唯一菜单入口，也不要让整个卡片同时承担打开详情和打开菜单两个动作。

触屏常显后需要检查：

- 更多按钮不会遮挡封面中的关键文字。
- 菜单弹层不会被横向滚动容器裁切。
- 点击菜单项后菜单关闭，状态更新可见。
- 点击菜单外或切换页面后不会遗留浮层。

### 4. 保留现有无障碍语义

现有按钮已具备书名相关的 `aria-label` 和 `aria-expanded`，应继续保留：

- 关闭时 `aria-expanded="false"`。
- 打开时 `aria-expanded="true"`。
- 菜单项使用真实 `button`，不改成可点击 `div`。
- 聚焦轮廓不能只依赖颜色变化。

## 自动化测试方案

### 1. 粗指针上下文

新增独立 Playwright context，至少包含：

```ts
const context = await browser.newContext({
  baseURL: "http://127.0.0.1:5173",

  isMobile: true
```
