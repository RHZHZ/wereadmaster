# AI 阅读助手 P3.1.1 数据范围选择与总计阅读检索设计

## 背景

用户在 AI 阅读助手中询问“我读过的理财类书籍”时，AI 回答只基于 2026 年度阅读统计判断“经济理财分类仅 1 本”。从体验看，用户预期是查询总计阅读或全部历史阅读，而不是只看最新年度统计。

当前问题不是单纯的模型回答问题，而是上下文数据范围不明确：

- `read_route_latest_stats` 只读取 `reading_stats` 中 `updated_at` 最新的一条统计。
- `read_route_latest_stats_review` 只读取最新统计复盘摘要，不区分 `overall`、`annually`、`monthly`。
- 全局 AI 助手默认不注入最近阅读书籍。
- 即使注入最近阅读，也只取最近 5 本，不是全部已读书籍。
- 上下文 chips 只显示“阅读统计”，没有说明是“年度统计”还是“总计统计”。

因此 AI 容易把“最新周期统计”误当成“总计阅读事实”。

## 目标

P3.1.1 目标是让 AI 阅读助手在回答前明确“本次使用的数据范围”，并允许用户手动选择范围。

目标：

- 用户可以选择本次提问的数据范围。
- 默认支持自动判断，不增加每次提问的操作负担。
- “总计阅读”问题优先使用 `overall` 统计，而不是最新年度统计。
- 回答和上下文 chips 明确展示数据范围。
- 针对“我读过哪些理财书”这类问题，补充轻量的本地书籍匹配，不直接做向量知识库。

## 非目标

本阶段不做：

- 不做完整向量知识库。
- 不做整本书全文语义检索。
- 不把范围选择做成复杂全局设置中心。
- 不读取原始 WeRead 响应。
- 不保证能找出所有语义相关书籍，只做结构化字段和 AI 资产摘要的轻量匹配。
- 不让“最近阅读 5 本”冒充“全部已读书籍”。

## 推荐方案

采用“自动判断 + 本次手动覆盖”的轻量交互。

### 前端交互

在 AI 输入区附近增加一个数据范围选择入口。

默认值：

- `自动`

可选项：

- `自动`
- `总计阅读`
- `本年阅读`
- `最近阅读`
- `当前页面`
- `候选书架`

交互建议：

- 桌面端可以是轻量下拉菜单或 segmented menu。
- 不放到复杂设置页作为主入口。
- 用户选择后，只影响本次提问。
- 发送后在消息上方或上下文 chips 中展示实际使用范围。

文案建议：

```text
范围：自动
范围：总计阅读
范围：2026 年度阅读
范围：最近阅读
```

### 请求结构

建议在 `ReadingAssistantRequest` 中增加可选字段：

```ts
type ReadingAssistantContextRange =
  | "auto"
  | "overall"
  | "annual"
  | "recent"
  | "current"
  | "candidate";

type ReadingAssistantRequest = {
  threadId?: string;
  scope: AssistantContextScope;
  entityId?: string;
  message: string;
  enabledContext: ReadingAssistantContextOption[];
  contextRange?: ReadingAssistantContextRange;
};
```

说明：

- `auto` 是默认值。
- 前端不传时按 `auto` 处理，兼容旧调用。
- 不建议直接暴露底层 `mode/baseTime` 给 UI，避免用户理解成本过高。

## 自动判断规则

当 `contextRange = "auto"` 时，后端根据用户消息做轻量规则判断。

建议规则：

| 用户表达 | 推断范围 |
| --- | --- |
| 总计、全部、历史上、我读过的、所有、曾经读过 | `overall` |
| 今年、本年、年度、2026 年 | `annual` |
| 最近、近期、这段时间、接下来 | `recent` |
| 当前这本、这本书、这篇笔记 | `current` |
| 候选、候选书架、待读、加入候选 | `candidate` |

规则边界：

- 自动判断只决定数据范围，不改写用户问题。
- 用户手动选择优先级高于自动判断。
- 如果自动判断不确定，保持当前 scope 的默认行为。

## 后端上下文策略

### 1. 总计阅读

当范围为 `overall`：

- 优先读取 `reading_stats` 中 `mode = 'overall'`、`base_time = 0` 的统计。
- 如果存在对应统计复盘，读取 `reading-stats-review` 的 `overall:0` 资产。
- 上下文 key 建议使用：

```json
{
  "overallStatsSignals": {},
  "overallStatsReview": {}
}
```

used context label：

```text
总计阅读统计
总计统计复盘
```

如果没有 overall 缓存：

- 不回退成年度统计并声称是总计。
- 可以补充 latest 统计，但必须标注为“当前只找到年度/最新统计”。

### 2. 本年阅读

当范围为 `annual`：

- 读取最新 annual 统计，或根据用户文本中的年份读取对应年份。
- used context label 必须带年份，例如：

```text
2026 年度阅读统计
2026 年度统计复盘
```

### 3. 最近阅读

当范围为 `recent`：

- 可继续使用最近阅读、最近 AI 资产、最近对话。
- 明确最多 N 条，不当作全集。

used context label：

```text
最近阅读 · 5
最近 AI 资产 · 5
```

### 4. 当前页面

当范围为 `current`：

- 维持当前书、当前笔记、当前 AI 资产逻辑。
- 不额外注入总计统计，除非用户问题同时要求总计。

### 5. 候选书架

当范围为 `candidate`：

- 注入候选书架和最新选书决策。
- 用于候选比较、加入候选、待读决策。

## 理财类书籍检索策略

针对“我读过的理财类书籍应该不止这一本”这类问题，仅靠统计 category 不够。

建议新增轻量匹配：

### 输入

从用户问题中提取主题关键词：

```text
理财、投资、基金、指数基金、财务、财富、经济、金融
```

### 数据源

优先使用结构化本地数据：

- `shelf_entries.title`
- `shelf_entries.author`
- `shelf_entries.category`
- `book_details.title`
- `book_details.category`
- `notebook_books.title`
- `ai_outputs` 中已生成的摘要类资产标题和概览摘要

### 输出上下文

建议 key：

```json
{
  "matchedReadBooks": [
    {
      "bookId": "...",
      "title": "...",
      "author": "...",
      "category": "...",
      "matchReason": "分类包含经济理财",
      "source": "shelf_entries"
    }
  ]
}
```

限制：

- 最多 12 本。
- 不读取原始笔记。
- 不读取整本书全文。
- 不把匹配结果说成绝对全集，应表述为“本地结构化记录中匹配到”。

## 回答约束

AI 系统提示词需要补充：

- 必须说明本次依据的数据范围。
- 不得把年度统计说成总计阅读。
- 不得在没有 `overallStatsSignals` 或 `matchedReadBooks` 时断言“总计没有其他书”。
- 如果范围不足，应主动说明限制并建议切换范围。

示例回答：

```text
我刚才只看到了 2026 年度统计，所以不能代表你的全部阅读历史。
切到总计阅读后，本地结构化记录里匹配到这些理财/投资相关书籍：……
```

## 验收标准

### 用例 1：总计阅读查询

问题：

```text
我读过的理财类书籍有哪些？
```

期望：

- 自动选择 `overall`。
- chips 显示 `总计阅读统计` 或 `本地匹配书籍`。
- 不再回答“根据 2026 年年度统计”。

### 用例 2：年度查询

问题：

```text
我今年读过哪些理财类书？
```

期望：

- 自动选择 `annual`。
- chips 显示 `2026 年度阅读统计`。
- 回答明确限定为今年。

### 用例 3：无总计缓存

问题：

```text
我历史上读过哪些投资书？
```

期望：

- 如果没有 overall 缓存，回答说明“当前没有总计统计缓存”。
- 不拿年度统计冒充总计。
- 可提示用户先同步总计统计或切换为本年范围。

### 用例 4：用户手动覆盖

用户选择：

```text
范围：本年阅读
```

问题：

```text
我读过哪些理财类书？
```

期望：

- 手动选择优先。
- 回答限定为本年。

## 风险与取舍

### 风险 1：范围选项过多

处理：

- 默认显示 `自动`。
- 其余选项放在菜单中，不常驻铺开。

### 风险 2：用户以为匹配是完整语义检索

处理：

- 回答使用“本地结构化记录中匹配到”。
- 不使用“全部”“完整”等绝对措辞，除非数据源确实覆盖。

### 风险 3：自动判断错误

处理：

- 用户可以手动覆盖。
- chips 展示实际范围，让错误可见。

### 风险 4：实现过重

处理：

- 先做 SQL/结构化字段匹配。
- 不做 embedding。
- 不做后台索引任务。

## 推荐实施顺序

1. 增加 `contextRange` 类型和请求字段。
2. 前端 AI 面板增加范围选择入口。
3. 后端实现自动范围推断。
4. 后端按范围读取 `overall` / `annual` / `recent` 上下文。
5. 新增 `matchedReadBooks` 轻量匹配。
6. 更新 used context label。
7. 补充前后端测试。

## 结论

增加数据范围选择是必要的，但应做成“本次提问范围控制”，不要先做复杂全局设置。

这能直接解决“用户想查总计阅读，AI 却只看年度统计”的问题，也为后续 P3.2 流式输出前的数据准确性打底。
