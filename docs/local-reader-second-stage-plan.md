# 本地书库第二阶段开发计划

## 阶段定位

第二阶段的目标不是把本地阅读器做成完整 Kindle、微信读书或 PDF 标注工具，而是把第一阶段的概念稿和预览闭环推进为可交付的桌面端本地图书能力。

一句话目标：

```text
用户能导入自己本机合法持有的 TXT/EPUB 文件，在应用内稳定阅读、划线、写想法、保存进度、导出本地阅读资产，并且不会和微信读书数据发生隐性冲突。
```

第二阶段继续坚持本地优先：

- 本地书库只处理用户主动选择的本机文件。
- 本地阅读进度、划线、想法、AI 提问记录和导出结果都归属 `source: "local"`。
- 微信读书书架、微信读书导入书和微信读书笔记继续归属 `source: "weread"`。
- 同名书和同作者候选不自动合并；当前只在标题/作者唯一且不冲突时提示，ISBN 等额外信号留到后续版本。

## 当前基础

第一阶段已经完成：

- 本地书库入口、卡片、格式筛选、搜索和导入入口。
- TXT/EPUB 轻量正文阅读器。
- 本地阅读进度、划线、想法、标记、疑问、复制。
- 正文附近选区浮层和想法编辑浮层。
- 长划线、长想法使用摘要卡和详情弹窗，避免撑大侧栏卡片。
- 书内搜索、轻量目录、字号、行距、主题和 Markdown 导出。
- AI 提问记录保存、复制、定位、详情查看和清除；未配置 AI 时只保存草稿态记录。
- AI 选区提问请求边界已有纯函数约束，只允许发送选中文本、选区前后文窗口、用户问题、书名作者和本地来源 key。
- 本地来源和微信来源隔离提示。
- Tauri 本地图书命令、权限、TXT/EPUB 导入测试链路、EPUB OPF 标题/作者元数据读取、阅读进度驱动的最近阅读排序，以及正文读取失败时的重试/返回恢复路径。

第二阶段应在这些能力上做真实数据闭环、稳定性和验收，不重新设计信息架构。

Markdown 作为第三种本地文本格式的后续扩展，独立规划见 [本地书库 Markdown 导入扩展计划](./local-reader-markdown-import-plan.md)。该扩展只接入 `.md / .markdown` 的文本阅读闭环，不把本地书库升级为 PDF/DOCX/HTML 通用文档阅读器。

## 范围

### 必须做

- 真实导入 TXT/EPUB 文件，并保存稳定的本地书籍记录。
- 解析文件元数据、正文文本、文件 hash、格式、大小、导入时间和最近阅读时间。
- 已导入文件可从本地书库打开到阅读器。
- 阅读器能保存和恢复滚动进度、字号、行距、主题、划线、想法和 AI 提问记录。
- 导出 Markdown 时保留本地来源 front matter，不混入微信读书笔记。
- 导入失败、解析失败、文件丢失、重复导入和权限失败都有明确提示。
- 真实 EPUB 仍按“保守文本抽取”交付，不承诺精排版。

### 可以做

- 允许用户手动把本地书和某个微信读书条目建立“同一本书的不同版本”关联。
- 在本地书库卡片上提示“疑似已存在微信读书版本”。
- 在统一资产页后续展示来源版本，但只做只读并排，不合并笔记和进度。
- 在 AI Provider 已配置时，支持对用户选中文本发起一次性提问。

### 不做

- 不做移动端阅读器。
- 不做微信读书网页壳阅读。
- 不导入微信读书正文。
- 不破解 DRM，不处理受保护文件。
- 不做 PDF 阅读器、PDF 坐标标注和图片批注。
- 不做完整 EPUB CFI、分页引擎、复杂脚注/图文精排。
- 不把本地划线写回微信读书。
- 不把微信读书笔记转换为本地划线。
- 不做后台自动 AI、全文问答、整本书摘要或自动向量化。

## 数据边界

### 来源命名

本地书：

```ts
type LocalBookSource = {
  source: "local";
  sourceId: string;
  fileHash: string;
};
```

微信读书书架书，包括用户通过微信网页接口导入到微信书架的书：

```ts
type WereadBookSource = {
  source: "weread";
  sourceId: string; // 微信 bookId
};
```

这两类记录不能共用裸 `bookId`。跨来源引用必须使用：

```ts
type SourceItemKey = {
  source: "local" | "weread";
  sourceId: string;
};
```

当前前端已有 `source-item-keys` 工具负责创建、序列化、解析和比较 `SourceItemKey`。解析时会拒绝裸 `bookId`、未知来源和空 `sourceId`，后续手动关联必须复用该工具。

### 同书处理

如果本地书库和微信读书里出现同名书：

- 默认显示为两个版本。
- 本地版本显示“本地版本”。
- 微信版本显示“微信读书版本”。
- 疑似同书只提示，不自动合并。
- 用户手动关联后，只产生 `assetLink`，不移动、不覆盖、不合并任何原始数据。

当前关联底座使用“双来源版本对”模型，而不是把两个来源拆成松散的单条记录：

```ts
type ReadingAssetLinkPair = {
  id: string;
  assetId: string;
  local: SourceItemKey;
  weread: SourceItemKey;
  linkedBy: "user";
  createdAt: string;
};
```

前端已有 `reading-asset-links` 工具负责创建、读写、去重和取消这类关联。关联记录只保存来源版本关系，不保存划线、想法、进度、AI 提问记录或微信笔记内容。

### 笔记和进度

本地笔记继续按字符 offset 保存：

```ts
type LocalReaderHighlight = {
  id: string;
  bookId: string;
  source: "local";
  startOffset: number;
  endOffset: number;
  text: string;
  tone: "yellow" | "green" | "blue";
  createdAt: string;
};
```

微信读书笔记继续按微信返回的章节、位置和 note id 保存。两套 anchor 不互转。

## 实施顺序

### 2.1 真实导入闭环

目标：文件从用户选择到本地书库记录稳定落库。

任务：

- 完善 `choose_local_book_file` 和 `import_local_book` 的错误处理。
- TXT 使用明确编码策略，优先 UTF-8，失败时给出可理解提示。
- EPUB 提取标题、作者和纯文本正文；正文抽取失败时不得生成空书。
- 超大源文件或超大正文在导入阶段返回明确错误，避免一次性读取拖垮阅读器。
- 计算文件 hash，用于识别同一文件重复导入。
- 保存原始文件名、格式、大小、hash、导入时间、最近打开时间和应用内存储路径；不保存用户原始绝对路径。
- 重复导入同一 hash 时由后端返回已有记录和重复导入状态，前端提示“已在本地书库”，允许打开已有记录，不创建重复记录。
- 重复导入命中已有记录但应用内源文件丢失时，使用用户重新选择的同一文件修复本地源文件，不创建重复记录。
- 重复导入命中已有记录但应用内源文件内容被污染时，校验 hash 并使用用户重新选择的同一文件覆盖修复，不创建重复记录。
- 重复导入命中已有记录但应用内规范源文件位置被污染成目录时，使用用户重新选择的同一文件覆盖修复，不留下临时备份目录。
- 重复导入命中已有记录但应用内存储路径被坏数据污染时，恢复到规范应用内路径并同步修复文件记录，不创建重复记录。
- 导入和重复导入修复写入应用内源文件时，先写同目录临时文件；覆盖已有目标时先移到同目录备份，替换失败会尝试恢复备份并清理临时/备份文件。
- 重复导入修复旧记录时，如果源文件复制后 DB 修复失败，清理本次复制的规范源文件，不留下孤儿文件。
- 前端重复导入提示基于后端结果，不因页面列表过期把“已存在”误报为新导入。
- 导入命令成功后，阅读进度读取失败只作为附带提醒，不把已完成导入误报为失败。
- 书库加载时单本阅读进度读取失败不阻塞整页展示，进度作为可缺省的附属数据处理。
- 阅读器打开时阅读进度读取失败不阻塞正文加载，进度作为可缺省的附属数据处理，并给出附带提醒。
- 阅读进度保存失败时保留阅读状态并显示“暂未保存”，同一错误不反复刷屏；系统会自动重试一次，保存成功后恢复为“已保存”。
- 切换书籍或重新打开时，旧阅读进度保存请求的返回结果不回写当前阅读器状态。
- 阅读器打开或重新打开时先清空旧书头部、正文、选区浮层、详情弹窗和待确认删除状态，避免正文读取失败时混入上一本文案或旧操作面板。

验收：

- 导入一个 TXT 后能在本地书库看到并打开。
- 导入一个 EPUB 后能在阅读器看到正文文本和“保守文本抽取”边界。
- 重复导入同一文件不会生成两张卡。
- 文件不存在、格式不支持、解析失败都有明确错误态。
- 数据库持久化失败时清理已复制的本地图书源文件，不留下孤儿文件。
- 应用内源文件丢失后，重新导入同一文件能修复阅读器打开路径。
- 应用内源文件内容损坏后，重新导入同一文件能修复阅读器打开内容。
- 应用内源文件位置被目录污染后，重新导入同一文件能恢复为规范源文件，且不残留临时备份目录。
- 应用内源文件写入失败或修复失败时，不留下临时复制文件或替换备份文件。
- 应用内存储路径异常后，重新导入同一文件能修复规范路径和阅读器打开路径。
- 应用内存储路径修复失败时，清理本次复制的源文件，不留下孤儿文件。
- 导入成功但阅读进度读取失败时，书籍仍进入本地书库列表，并显示附带提醒。
- 单本阅读进度读取失败时，书库仍显示其余本地图书。
- 阅读器打开时阅读进度读取失败，正文仍可阅读，并提示“阅读正文已打开，但阅读进度暂时无法读取”。
- 阅读进度保存失败时只对同一错误提示一次，不阻塞继续阅读；系统会自动重试一次，下一次成功保存后状态恢复。
- 旧书或旧加载轮次的保存进度结果返回后，不覆盖当前书的保存状态或进度。
- 正文读取失败时，阅读器保留错误提示，并提供“重新打开”和“返回本地书库”操作。
- 正文读取失败或重新打开过程中，不展示上一本文档的标题、正文浮层、详情弹窗或待删除确认态。

### 2.2 阅读器稳定化

目标：真实导入文件的阅读体验不依赖预览数据。

任务：

- 打开真实本地图书时读取本地正文缓存或解析结果。
- 保存并恢复滚动进度。
- 保存并恢复主题、字号、行距偏好。
- 保存本地划线、想法、标记、疑问和 AI 提问记录。
- 长选区、长想法、长 URL 和连续英文 token 不撑开浮层、卡片、侧栏或弹窗。
- 书内搜索只定位当前本地正文，不跨书搜索。
- 目录识别只做轻量标题识别，识别失败时隐藏目录列表或给出空态。

验收：

- 关闭再打开同一本本地书，进度和标记仍在。
- 点击划线能在正文附近显示小浮层。
- 写想法在正文附近完成，不跑到侧边栏工作台。
- 划线卡片只展示摘要，点击后通过详情弹窗查看完整内容，并可继续复制、定位、写想法、问 AI 或切换划线类型。
- 想法卡片只展示摘要，点击后通过详情弹窗查看完整内容。
- 长文本不会造成横向溢出或卡片高度异常。

### 2.3 本地导出和资产边界

目标：导出物可以作为本地阅读资产交付，同时不污染微信读书数据。

任务：

- Markdown front matter 固定包含 `source_kind: local`、`book_id`、`file_hash`、`format` 和导出时间。
- 导出内容包含本地划线、想法、标记、疑问和 AI 提问记录。
- 只有 AI 提问记录、没有划线或想法时，也能导出，并保留空划线和空想法占位。
- 不读取微信读书笔记。
- 不调用 AI。
- 导出文件名使用书名和“本地标记”后缀。

验收：

- 导出 Markdown 可打开且内容结构稳定。
- 导出中看不到微信读书 note id、微信章节位置或远端同步字段。
- 本地书和微信同名书同时存在时，导出仍只包含本地版本内容。

### 2.4 可选 AI 接入

目标：只在用户主动选择片段时提供 AI 帮助，不把 AI 做成阅读器依赖。

前置条件：

- AI Provider 已配置。
- 用户主动选中文本。
- 用户主动点击“问 AI”并提交问题。

任务：

- 请求体只包含选中文本、选区前后文窗口、用户问题、书名、作者和本地来源标识。
- 不发送整本书正文。
- 不发送本地文件路径、数据库路径、API Key 或微信读书凭据。
- 回答保存为本地 AI 提问记录，归属 `source: "local"`。
- Provider 未配置时只保留草稿和复制能力，不展示假回答。

验收：

- 未配置 AI 时没有“AI 回答”“赞同回答”“不赞同回答”等假控件。
- 配置 AI 后，只有用户手动提交才发起请求。
- AI 记录不会进入微信读书笔记列表。

当前状态：

- 已有 `local-reader-ai-requests` 请求边界工具，确保请求体不包含整本书正文、本地文件路径、文件 hash、API Key 或微信读书凭据。
- 已接入桌面端一次性 Provider 调用：配置 AI 后，用户提交问题才会请求模型；未配置或 Web 预览仍只保存本地草稿态记录。
- Provider payload 只包含书名、作者、用户选中文本、选区前后文窗口、选区 offset 和问题；本地来源 ID 仅用于本机缓存归属，不进入模型输入。
- AI 回答缓存归属 `local-reader-selection-qa` / `source: "local"`，不会进入微信读书笔记列表。
- AI 提问展示已收纳为“正文选区触发、侧栏摘要列表、详情弹窗查看完整内容”；不在章节、分区或正文段落下默认渲染 AI 卡片。
- 继续追问已归入原 AI 提问记录的追问线程，侧栏不新增顶层卡片；摘要卡只显示轮次摘要，完整追问在线程区查看。

### 2.5 来源关联提示

目标：解决“同一本书本地和微信都存在”的认知问题，但不做自动合并。

当前状态：

- 已有本地书库到微信书架的疑似版本提示。
- 已有微信书籍详情到本地书库的疑似版本提示。
- 已有 `SourceItemKey` 和 `ReadingAssetLinkPair` 数据底座。
- 已在本地书库卡片和微信书籍详情页开放手动关联 / 取消关联 UI。
- 关联后的 UI 只展示“已关联微信版本 / 已关联本地版本”，不进入统一资产页，也不合并任意来源数据。
- 关联、取消和去重由 `reading-asset-links` 的纯函数统一处理，避免两个页面出现行为分叉。

任务：

- 本地书库卡片可提示“可能存在微信读书版本”。
- 疑似提示只读取已同步微信书架中的电子书条目，作者冲突、同名多候选或同名同作者多候选不提示，有声书和文章收藏不参与匹配。
- 微信书籍详情可提示“可能存在本地版本”。
- 微信书籍详情的本地版本提示只读取本地书库列表，不进入笔记、进度或 AI 缓存合并。
- 疑似版本对内部使用 `SourceItemKey` 表示本地版本和微信版本，不使用裸 `bookId` 作为跨来源 key。
- 提供手动关联入口。
- 关联后在当前 UI 中展示“同一本书的两个来源版本”状态。
- 关联只影响展示，不影响导出、笔记、进度和 AI 缓存归属。
- 取消关联只删除 `ReadingAssetLinkPair` 记录，不修改两边原始数据。

验收：

- 同名同作者且唯一候选时出现提示，但不自动合并。
- 手动关联不会复制、迁移或合并任何本地/微信笔记。
- 取消关联后两边恢复独立显示。
- 本地笔记和微信笔记仍按来源筛选。

## UI 约束

- 本地书库继续是书架子菜单下的独立入口。
- 不把本地书库塞进微信书架的 `book / album / mp` 类型筛选。
- 阅读器首屏以正文为主，侧栏只承载摘要、列表和状态，不承载主要输入。
- 选区动作必须出现在正文附近小浮层。
- 长划线和长想法通过“摘要卡 + 详情弹窗”处理，不扩大列表卡片。
- AI 提问不作为每个章节或分区的默认模块渲染；只在选区浮层、AI tab 摘要列表和详情弹窗中出现。
- AI tab 无记录时只显示一次空态，不在正文或每个分区重复显示。
- 长 AI 问题、长选区和长回答通过“摘要卡 + 详情弹窗”处理，不扩大侧栏卡片。
- 继续追问属于同一 AI 提问记录线程，不归入想法分区，不新增右侧顶层 AI 卡片。
- 多轮追问长列表在 AI 详情内局部滚动，不拉长侧栏和详情面板。
- 不展示未接入的按钮、伪筛选、假排序或假 AI 反馈。
- 桌面端优先；手机版本阶段不投入。

## 测试计划

### 单元测试

- 本地书 hash 去重。
- TXT/EPUB 元数据解析 fallback。
- 本地阅读进度保存和恢复。

## 风险和应对

### EPUB 兼容性

风险：不同 EPUB 的目录、章节、编码和 HTML 结构差异很大。

应对：第二阶段只承诺纯文本抽取；失败时明确提示，不展示精排版承诺。

### 来源混淆

风险：微信读书导入书和本地导入书标题一致，用户误以为笔记会自动同步。

应对：所有 UI、导出和数据模型保留来源字段；同书只做手动关联。

### AI 过度读取

风险：用户以为 AI 会理解整本书，或者应用静默发送整书内容。

应对：只允许选区提问；请求前展示范围说明；未配置时只保留草稿能力。

### 阅读器复杂度膨胀

风险：EPUB 精排、PDF、分页、跨设备同步和全文问答同时进入，导致主线失控。

应对：第二阶段只做本地文本阅读闭环；重型阅读器单独立项。

## 开发原则

- KISS：优先稳定导入和阅读闭环，不做重型阅读器。
- YAGNI：不提前实现 PDF、EPUB CFI、向量库和跨设备同步。
- DRY：复用已有本地阅读器状态、导出和测试工具。
- SOLID：来源、书籍、笔记、进度、AI 记录分层，不让 UI 直接依赖裸 `bookId`。
