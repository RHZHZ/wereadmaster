# 微信读书 Skill 1.0.4 产品增强需求分析

## 结论

这些增强项不应等量设计。当前应用的核心定位是“本地优先的个人阅读资产工作台”，不是微信读书客户端复刻。因此需求判断应以三个问题为准：

1. 是否直接帮助用户处理“继续读、整理笔记、选择下一本书”。
2. 是否能复用现有数据链路，且不会引入大规模批量调用。
3. 是否能保持个人资产与公开内容的边界。

按这个标准，推荐分为四类：

| 类别 | 增强项 | 结论 |
|------|--------|------|
| 必须设计，优先实现 | 当前章节定位、真实相似推荐来源标识 | 价值清晰，成本低，直接改善已有工作流 |
| 需要设计，但轻量实现 | 公开点评基础模块、微信读书概况卡片 | 有明确用户价值，但必须控制调用量和信息边界 |
| 需要预留设计，后置实现 | 热门划线、共读想法、公开点评筛选分页 | 接口已可用，但交互和数据类型成本更高 |
| 暂不作为产品增强设计 | 完整垂直搜索页、社区页复刻、P5 API 拆分 | 容易过度设计，或者属于技术整理而非用户需求 |

推荐实施顺序：

1. P3.1 当前章节定位。
2. P3.2 公开点评基础模块。
3. P3.3 轻量微信读书概况卡片。
4. P4.2 热门划线。
5. P4.3 共读想法。
6. P4.1 公开点评筛选分页。
7. P4.4 搜索 scope 交互收敛。
8. P5 API 拆分。

## 需求边界

### 核心用户任务

当前应用已经覆盖书架、详情、笔记、统计、发现、AI 复盘和本地候选。新增能力应服务三个核心任务：

- 单本书详情：我读到哪里了，是否继续读，是否整理笔记。
- 选书发现：这本书是否值得加入候选，有哪些真实相似书。
- 总览回顾：最近读了哪些书，哪些书有笔记待整理。

不建议把增强项扩展到以下目标：

- 复刻微信读书的完整社区、搜索、点评、互动体验。
- 后台维护全量逐书进度。
- 将公开内容默认并入个人笔记、统计或 AI 偏好判断。
- 为未来可能扩展而提前拆大文件、重做状态管理。

## 单项需求判断

### 1. 当前章节定位

结论：需要设计，优先级最高。

理由：

- 用户打开书籍详情时，最自然的问题是“我读到哪里了”。
- 现有详情链路已经有 `progress.chapterUid`、`progress.chapterOffset`、`progressPercent` 和章节列表。
- 当前 `ChapterList` 已能高亮当前章节，但详情页主信息区还没有把“当前章节名”作为明确阅读状态展示出来。
- 不需要新增接口，不增加批量请求。

设计范围：

- 在详情页进度区域展示当前章节标题。
- 目录继续高亮当前章节。
- 找不到章节名时回退到百分比。
- 章节内偏移只做弱提示，不作为主文案。

不做：

- 不生成逐章任务。
- 不做后台章节追踪。
- 不把章节作为 AI 阅读路线的强制任务清单。

必要性判断：这是“已有数据未充分展示”的补齐，不属于过度设计。

### 2. 真实相似推荐

结论：需要保留并继续产品化，但不需要扩大设计。

理由：

- 用户从一本书出发找下一本书，是发现页的核心场景。
- 新版 Skill 明确了 `/book/similar` 的分页参数，旧版本容易失败后降级搜索的问题可以真正修复。
- 当前前端已有“先相似推荐，失败后书名搜索兜底”的流程，设计重点是结果来源清晰。

设计范围：

- 成功时明确显示“相似推荐”。
- 失败兜底时明确显示“已改用书名搜索兜底”。
- 搜索兜底结果不能混称为相似推荐。

不做：

- 不做复杂推荐解释。
- 不把多源推荐混排。
- 不为了相似推荐新增独立页面。

必要性判断：需要，但已经接近完成，后续只做来源标识和异常兜底的收敛。

### 3. 公开点评基础模块

结论：需要设计，但必须轻量实现。

理由：

- 公开点评能帮助用户判断是否继续读、是否保存为候选。
- 后端已经有 `get_public_reviews` 命令和 mapper，接口契约也明确了双层 `review` 结构。
- 目前缺少前端详情页入口，导致这类外部信号没有被产品化。

主要风险：

- 公开点评容易和个人笔记混淆。
- 长点评、HTML 内容、分页筛选会把详情页拖成社区页。
- 如果默认送入 AI，会污染“个人阅读资产”的边界。

设计范围：

- 放在 `BookDetailPage` 的详情主体下方，作为“公开点评”模块。
- 首屏只展示 3 到 5 条。
- 使用纯文本 `content`，不直接渲染 `htmlContent`。
- 显式标注“来自微信读书公开内容，不计入个人笔记”。
- 模块错误只影响该模块，不影响书籍详情、目录、进度。

不做：

- 初版不做筛选、不做无限滚动。
- 不将公开点评并入个人笔记统计。
- 不默认作为 AI 复盘或选书决策输入。

必要性判断：需要。它是详情页的决策辅助信息，但第一版必须控制成一个小模块。

### 4. 微信读书概况卡片

结论：谨慎设计，可做轻量版本，不应变成新总览体系。

理由：

- 用户打开应用首页时，确实需要快速知道最近阅读和笔记概况。
- 新版 `profile.md` 明确要求阅读进度只查最近 5 本电子书，这能防止大书架过量调用。
- 当前 Dashboard 已有较多信息，新增概况卡片可能造成重复。

设计范围：

- 只作为 Dashboard 的轻量卡片或现有卡片增强。
- 展示书架总数、电子书/有声书/文章收藏拆分、笔记概况、最近 5 本电子书进度。
- 优先使用已有缓存。
- 用户点击刷新时才补最近 5 本电子书进度。

调用约束：

- 只从 `books[]` 按 `readUpdateTime` 取最近 5 本。
- 不对 `albums[]`、`mp` 调 `/book/getprogress`。
- 不为概况页补齐全量书架历史进度。

不做：

- 不新增完整 Profile 页面。
- 不替代现有 Dashboard。
- 不把最近 5 本限制强加到单本详情页。

必要性判断：可做，但排在当前章节和公开点评之后。它是体验增强，不是契约适配必需项。

### 5. 热门划线

结论：需要预留设计，后置实现。

理由：

- `/book/bestbookmarks` 现在返回全书热门划线文本和人数，能帮助用户快速了解一本书的公共关注点。
- 它比“共读想法”更轻，只需要展示划线原文、热度和章节。
- 但它会新增数据类型、缓存策略、详情页分区和公开内容边界。

设计范围：

- 放在书籍详情页的“共读信号”区域。
- 初版展示全书热门划线 5 条。
- 如果当前章节可识别，可提供“当前章节”切换，但点击后再加载。
- 明确标注不属于个人划线。

不做：

- 不首屏加载每个章节的热门划线。
- 不默认加载每条划线下的共读想法。
- 不和个人划线混排。

必要性判断：有价值，但不应和公开点评同一阶段实施。先做公开点评，验证详情页承载能力后再做热门划线。

### 6. 共读想法

结论：后置，只做按需查看，不做首屏模块。

理由：

- 共读想法依赖热门划线的 `range` 和 `chapterUid`，数据链路比热门划线更长。
- 它属于更深的公共讨论层，用户价值存在，但不是详情页首屏必需信息。
- 如果批量请求，会形成 N 条热门划线乘以 N 次评论请求的调用风险。

设计范围：

- 用户点击某条热门划线后，再请求 `/book/readreviews`。
- 每次只展示 3 到 5 条。
- 使用抽屉或内联展开。
- 单条想法详情 `/review/single` 继续后置。

不做：

- 不在打开书籍详情时批量加载。
- 不做评论、点赞完整互动页。
- 不进入个人笔记统计或 AI 默认输入。

必要性判断：不是首轮需求。只有热门划线模块稳定后才需要设计。

### 7. 公开点评筛选分页

结论：后置增强，不进入公开点评第一版。

理由：

- 筛选和分页只有在用户愿意持续阅读公开点评时才有价值。
- 第一版的目标是让用户知道“其他读者怎么看”，不是浏览完整点评页。
- 如果一开始加入推荐、最新、差评、一般、分页、好友摘要，会让详情页复杂度过高。

触发条件：

- 公开点评基础模块稳定。
- 用户确实需要查看更多点评。
- 数据字段和分页游标在真实回包中表现稳定。

设计范围：

- 增加推荐、最新、差评、一般筛选。
- 加载更多每次控制在 10 条以内。
- 使用 `maxIdx` 和 `synckey`。

必要性判断：不是当前必要项。它是基础公开点评模块的二阶段能力。

### 8. 垂直搜索入口

结论：低优先级，当前更适合做交互收敛，而不是新增完整设计。

理由：

- 当前发现页已经有搜索 scope 入口，并且 scope 项已经较多。
- 新版 Skill 明确了 scope 语义，但应用目标是候选管理和阅读扩展，不是完整搜索客户端。
- 如果继续增加 tab、筛选和结果解释，会削弱发现页主任务。

设计范围：

- 保留综合和电子书作为高频入口。
- 作者、有声书、书单可放入“更多搜索类型”。
- 全文、公众号、文章、网文继续作为低频能力，不做主入口。
- 添加候选书场景默认 `scope=10`，主题探索默认 `scope=0`。

不做：

- 不新增完整搜索结果页。
- 不把所有 scope 做成平级 tab。
- 不把全文搜索包装成可读正文能力。

必要性判断：需要对齐，但不是新增产品增强重点。现阶段主要做减法和文案校准。

### 9. P5 API 拆分

结论：暂不作为产品需求设计。

理由：

- API 拆分是维护性工作，不直接产生用户价值。
- 如果在 P3/P4 前拆分，容易扩大改动范围并引入回归。
- 只有当公开点评、热门划线、共读想法等能力稳定后，领域边界才更清楚。

触发条件：

- `reading-api.ts` 的命令调用、映射、错误归一化继续膨胀。
- P3/P4 功能已经稳定，测试覆盖足够。
- 拆分能减少真实维护成本，而不是为了文件大小而拆。

必要性判断：技术上合理，但当前设计它属于过早优化。

## 前端设计取舍

### BookDetailPage

应该承载：

- 当前章节定位。
- 公开点评基础模块。
- 后续热门划线和按需共读想法。

不应该承载：

- 完整社区页。
- 大量点评筛选条件。
- 公开内容与个人笔记混排。

### DashboardPage

应该承载：

- 轻量阅读概况。
- 最近 5 本电子书进度。
- 笔记待整理入口。

不应该承载：

- 全量逐书进度。
- 新的复杂 Profile 页面。
- 大量与现有统计卡重复的信息。

### DiscoveryPage

应该承载：

- 综合搜索。
- 电子书搜索。
- 推荐和真实相似推荐。
- 本地候选保存。

不应该承载：

- 微信读书完整搜索客户端。
- 所有 scope 平铺成主入口。
- 搜索兜底结果伪装成相似推荐。

### Reading Hub 和 AI

应该承载：

- 个人笔记、阅读统计、本地状态、已生成复盘。

不应该默认承载：

- 公开点评。
- 热门划线。
- 共读想法。

如果未来允许 AI 使用公开内容，必须作为“外部公开信号”显式选择，并在输出中标注来源。

## 遗漏检查

当前增强规划总体完整，但需求分析层面还需要补三条约束：

### 1. 公开内容的失败隔离

公开点评、热门划线、共读想法都必须是模块级失败。书籍详情主信息、目录和个人进度不应因为公开内容失败而不可用。

### 2. 公开内容的纯文本策略

接口可能返回 `htmlContent`，第一版只展示纯文本字段。后续若需要富文本，必须先设计白名单清洗和测试，不能直接渲染服务端 HTML。

### 3. 概况卡片的缓存优先策略

最近 5 本电子书进度限制不只是接口调用限制，也应体现在交互上：页面打开优先展示缓存，用户主动刷新才补进度，避免 Dashboard 成为隐性同步入口。

## 二次遗漏补充

以下内容不改变主实施顺序，但需要提前写清楚，否则实现时容易把边界做宽。

### 1. `/book/underlines` 不应并入热门划线首版

新版 `notes.md` 同时提供 `/book/underlines` 和 `/book/bestbookmarks`，但二者不是同一个产品能力：

- `/book/bestbookmarks` 返回热门划线原文和人数，适合做“热门划线”模块。
- `/book/underlines` 只返回章节内划线热度统计，不返回原文，更适合阅读器内展示“X 人划线”热度标签。

当前 Tauri 应用没有微信读书正文阅读器，首版热门划线不需要接 `/book/underlines`。如果未来本地阅读器要展示公共热度标签，也必须先解决原文位置映射，不能把 `range` 直接当成可在本地文本中稳定定位的坐标。

### 2. 公开内容缓存需要刷新边界

公开点评、热门划线、共读想法是外部公开信号，和个人书架/笔记缓存不同。后续实现缓存时应满足：

- 缓存 namespace 与个人笔记分离。
- UI 显示缓存更新时间或提供手动刷新。
- 默认短期缓存，避免长期展示过旧的公开口碑。
- 缓存失败不阻断详情页主流程。
- 不为了刷新公开内容触发多个模块并发请求。

首版可以先只做内存状态或沿用 raw cache，但必须保留“公开内容可过期”的产品认知。

### 3. 公开用户信息最小展示

公开点评和共读想法可能包含昵称、头像、userVid 等公开用户字段。首版只需要展示昵称和内容，不应扩大处理范围：

- 不持久化公开用户画像。
- 不导出公开用户信息到个人笔记 Markdown。
- 不把公开用户行为用于个人偏好建模。
- 头像加载失败不影响点评内容展示。

### 4. 新增接口也必须接入升级错误闭环

后续新增 `get_public_reviews` 前端 API、热门划线、共读想法接口时，都必须复用现有 `upgrade_required` 结构化错误处理：

- 升级错误展示 `SkillUpgradeNotice` 或同等专门提示。
- 模块内保留重试入口。
- 不把 `upgrade_required` 压成普通“获取失败”。

这属于契约适配的延续，不应等到 P5 再统一修。

### 5. Web Preview 和无 Tauri 环境要降级

`reading-api.ts` 已有 Web 预览分支。新增公开点评、热门划线、概况聚合时，需要明确 Web Preview 行为：

- 没有预览数据时展示模块不可用或空态。
- 不在浏览器预览中调用 Tauri command。
- 不因为公开模块缺失导致书籍详情页整体不可渲染。

### 6. 适用对象需要加门禁

不是所有书架条目都适合调用电子书详情增强接口：

- 当前章节定位只适用于 `books[]` 里的电子书。
- 最近 5 本进度只适用于 `books[]`，不包含 `albums[]` 和 `mp`。
- 公开点评、热门划线、共读想法只对有有效 `bookId` 的微信读书电子书开放。
- 本地导入书、文章收藏、有声书/专辑不应复用这些接口。

### 7. 评分口径不能复用个人想法评分

公开点评评分是百分制阶梯：`20=一星`、`40=二星`、`60=三星`、`80=四星`、`100=五星`。个人想法/点评里的 `star` 可能是 `0-5` 或 `-1`。

公开点评模块应单独做评分格式化，不要复用个人笔记想法的评分展示逻辑。

### 8. 书架 `deepLink` 透传不等于新增主入口

新版 `shelf.md` 说明书架条目也可能返回 `deepLink`。这需要遵守“接口返回才展示，不手动拼接”的契约，但不意味着书架列表要增加新的主操作。

建议：

- 书架卡片继续以打开详情为主。
- 如要显示微信读书直达入口，放在详情页或次级图标按钮。
- 不把 `deepLink` 持久化到候选书状态。
- 没有 `deepLink` 时继续使用本地 `open_book_in_weread` 命令作为用户主动打开能力，但不要把构造链接当接口字段展示。

### 9. 首屏外部信号预算

书籍详情页已经需要加载基本信息、进度、目录。新增公开点评、热门划线后，必须控制首屏外部信号请求：

- P3 首屏最多自动加载一个公开模块，建议只加载公开点评基础模块。
- 热门划线进入 P4 后，可以选择默认不加载，或仅在用户展开“共读信号”时加载。
- 共读想法永远点击后加载。
- 不在首屏同时请求公开点评、热门划线、多个章节热度和多条共读想法。

## 过度设计判断

以下设计属于当前阶段过度：

- 为公开点评做完整筛选面板和无限滚动。
- 为热门划线一次性加载所有章节。
- 将 `/book/underlines` 做成热门划线首版依赖。
- 为每条热门划线预加载共读想法。
- 把作者、有声书、书单、全文、公众号、文章、网文全部做成主 tab。
- 为候选书持久化 `deepLink`。
- 在 P3 前拆分 `reading-api.ts`。
- 把公开内容默认传入 AI。

以下设计不算过度：

- 详情页展示当前章节名。
- 公开点评首屏展示 3 到 5 条。
- 相似推荐成功与搜索兜底的来源文案区分。
- 概况卡片限制最近 5 本电子书进度。
- 热门划线只做全书前 5 条。

## 推荐的阶段门槛

### P3.1 完成门槛

- 详情页能展示当前章节名。
- 目录当前章节高亮不回退。
- 没有章节匹配时不报错。
- 不新增接口调用。

### P3.2 完成门槛

- 详情页按需加载公开点评。
- 首屏不超过 5 条。
- 模块有空态、错误态、刷新态。
- 文案明确“不计入个人笔记”。
- 不使用 `dangerouslySetInnerHTML`。

### P3.3 完成门槛

- Dashboard 不触发全量逐书进度。
- 最近进度最多请求 5 本电子书。
- 有声书和文章收藏不调 `/book/getprogress`。
- 无凭据时展示设置引导。

### P4 完成门槛

- 热门划线和个人划线视觉分区明确。
- 共读想法严格点击后加载。
- 点评筛选分页只在基础模块稳定后加入。
- 所有公开内容失败都不影响个人资产页主流程。

## 最终建议

可以开始设计和实施，但应从最小闭环开始：

1. 先做当前章节定位，这是最稳的用户价值补齐。
2. 再做公开点评基础模块，验证详情页能否承载外部信号。
3. 概况卡片作为轻量首页增强，但必须坚持最近 5 本电子书限制。
4. 热门划线和共读想法不要提前并入首轮。
5. 搜索 scope 以收敛现有入口为主，不扩展成完整搜索客户端。
6. API 拆分等 P3/P4 稳定后再做。
