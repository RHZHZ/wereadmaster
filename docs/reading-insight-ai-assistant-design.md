# 阅读洞察与 AI 助手设计文档

## 背景

当前产品已经形成本地优先的阅读资产闭环：

- `笔记` 承载单本划线、想法和 Markdown 导出。
- `单本 AI 复盘` 把一本书的笔记整理为结构化资产。
- `复盘中心` 归档书籍复盘、阅读指南和周期报告。
- `AI 阅读助手` 提供带上下文的临时问答、追问和推荐。
- `统计` 提供周期阅读数据、本地解读和 AI 阅读体检报告。

flomo 式“洞察”的价值不在于增加一个聊天入口，而在于把用户长期记录里的关注主题、反复问题和思维张力显性化。放到本产品里，它应当服务“从阅读素材到个人判断”的环节，而不是替代现有 AI 助手。

## 问题定义

现有 `AI 复盘` 更像“把这本书整理清楚”，但用户还需要回答另一类问题：

- 我为什么会标记这些内容？
- 我的笔记里反复出现什么主题？
- 哪些想法之间存在张力、矛盾或未完成的问题？
- 这些洞察下一步如何转成写作、复盘或阅读计划？

如果直接把这些问题放进 AI 助手，入口会过于泛化，用户需要自己组织上下文和问题。更合适的方式是：

```text
笔记 / 复盘页生成洞察
AI 助手负责追问、串联和召回
复盘中心负责沉淀
总览负责提醒下一步动作
```

## 目标

1. 在单本笔记和单本 AI 复盘中引入“阅读洞察”能力。
2. 明确 AI 助手在洞察场景中的职责：追问、解释、跨书召回，而不是主生成入口。
3. 保持本地优先和用户手动触发，不新增后台自动分析。
4. 让洞察结果可保存、可导出、可回看，并尽量能回到原始笔记依据。
5. 用最小改造复用现有 AI 资产、缓存、导出和阅读记忆机制。

## 非目标

本阶段不做：

- 不新增一级导航“洞察”。
- 不做通用知识图谱。
- 不做向量数据库或全文语义检索。
- 不自动扫描全量笔记生成洞察。
- 不把 AI 助手改成全屏工作台。
- 不把聊天结果默认写入正式 AI 资产。
- 不跨书发送原始笔记正文。
- 不基于洞察输出心理诊断、人格定论或能力评价。

## 核心设计判断

### 1. 洞察是阅读资产，不是聊天消息

洞察需要稳定保存、导出和回看。因此它应优先挂在 `BookAiSummary` / AI 资产体系上，而不是只存在于助手线程里。

### 2. AI 助手是洞察的对话层

助手负责：

- 对某条洞察继续追问。
- 把洞察转成写作提纲、复盘问题或行动项。
- 在全局范围召回已生成的 AI 资产摘要。
- 解释统计变化和阅读节奏。

助手不负责：

- 默认读取全量笔记。
- 自动生成正式洞察资产。
- 替代单本复盘入口。

### 3. 首版应优先复用现有结构

现有 `BookAiSummary` 已包含：

- `keyIdeas`
- `myFocus`
- `actionItems`
- `representativeQuotes`
- `reflectionQuestions`

这些字段已经能支撑一个基础“洞察”展示层。首版可以先通过 UI 重组和 prompt 约束强化洞察表达；只有当展示和引用关系不足时，再扩展新字段。

## 信息架构

### 推荐入口优先级

| 位置 | 角色 | 建议 |
| --- | --- | --- |
| 单本笔记页 `BookNotesPage` | 生成入口 | 保留 `AI 复盘`，增加“洞察”语义提示，不新增独立复杂入口 |
| 单本 AI 复盘页 `BookAiSummaryPage` | 主展示 | 增加“阅读洞察”区块，承接关注点、张力、追问和来源 |
| 复盘中心 `ReadingHubPage` | 资产归档 | 展示已生成洞察/复盘，不负责首次生成 |
| 总览页 `DashboardPage` | 下一步提醒 | 只展示最近或最需要处理的一条洞察动作 |
| AI 阅读助手 `ReadingAssistantPanel` | 追问层 | 提供快捷追问、跨书召回和洞察解释 |
| 统计页 `StatisticsPage` | 周期解释 | 保持“阅读统计复盘”，不混入基于笔记正文的洞察 |

### 单本笔记页

入口仍放在操作区：

```text
导出 Markdown   刷新   AI 复盘
```

首版不建议新增并列按钮 `生成洞察`，原因：

- 会让用户误以为“复盘”和“洞察”是两套不同资产。
- 后端会出现相似 prompt、相似缓存和相似导出。
- 违反 KISS / DRY。

建议将按钮旁的状态说明或复盘状态卡文案调整为：

```text
AI 复盘会整理关键观点、阅读洞察、行动项和复盘问题。
```

### 单本 AI 复盘页

在 `overview` 下方新增“阅读洞察”区块：

```text
阅读洞察

1. 你反复关注的是……
   依据：代表性摘录 / 章节 / 想法
   追问：……

2. 这些笔记里存在的张力是……
   依据：……
   追问：……
```

首版可以用现有字段组合生成：

- 洞察标题：来自 `myFocus`
- 洞察说明：来自 `keyIdeas`
- 依据：来自 `representativeQuotes`
- 追问：来自 `reflectionQuestions`

如果组合效果不足，再进入 P1 字段扩展。

### 复盘中心

`ReadingHubPage` 继续作为归档层。

建议：

- 资产卡片仍使用“书籍复盘 / 阅读指南 / 跨书路线”分类。
- 不新增“洞察库”。
- 在版本详情页展示洞察内容。
- 后续如果 `BookAiSummary` 新增 `insightCards`，版本详情页也复用同一展示组件。

### 总览页

总览只承担提醒，不承担生成。

可展示：

- 最近生成但未处理的洞察。
- 有复盘问题但没有反馈的书。
- 洞察对应的下一步行动。

不展示：

- 长篇洞察正文。
- 多条洞察列表。
- 复杂筛选。

### AI 阅读助手

助手承接三类洞察使用场景：

1. 当前书追问：
   - “我为什么会关注这些内容？”
   - “这些洞察能整理成写作提纲吗？”
   - “这条洞察对应哪些行动？”

2. AI 资产追问：
   - “这份复盘最值得继续推进的一个问题是什么？”
   - “帮我把这条洞察压缩成一段输出。”
   - “把洞察转成 3 个复盘问题。”

3. 全局召回：
   - “我最近几本书反复关注什么？”
   - “哪些主题在不同书里重复出现？”
   - “哪些复盘问题还没有被解决？”

助手当前已经具备 `bookNotes`、`aiAsset`、`global`、`readingStats` 等 scope，首版优先调整快捷问题和上下文说明，不新增新 scope。

## 用户流程

### 流程 1：从单本笔记生成洞察

```text
用户进入单本笔记页
  -> 确认划线 / 想法数量
  -> 点击 AI 复盘
  -> 进入单本 AI 复盘页
  -> 点击生成复盘
  -> 展示概览、阅读洞察、行动项、复盘问题、代表性摘录
  -> 用户复制 / 导出 / 标记已整理
```

### 流程 2：对某条洞察追问

```text
用户在 AI 复盘页看到洞察
  -> 点击“追问”
  -> 打开 AI 阅读助手
  -> 助手 scope = aiAsset 或 bookNotes
  -> 输入框预填追问模板
  -> 用户确认发送
```

首版如果不做预填输入框，也可以只新增快捷问题，降低改造成本。

### 流程 3：跨书召回洞察

```text
用户打开全局 AI 助手
  -> 使用“我最近反复关注什么？”快捷问题
  -> 助手只读取已生成 AI 资产摘要、统计和阅读记忆
  -> 输出跨书主题和来源类型
  -> 不读取跨书原始笔记正文
```

### 流程 4：总览提醒下一步

```text
用户打开总览
  -> 总览发现最近有已生成复盘但行动反馈为空
  -> 展示“继续处理《X》的阅读洞察”
  -> 点击进入单本 AI 复盘页或复盘中心版本详情
```

## 数据结构设计

### P0：不新增后端字段

P0 复用 `BookAiSummary`：

```ts
type BookAiSummary = {
  overview: string;
  keyIdeas: string[];
  myFocus: string[];
  actionItems: string[];
  themeTags: string[];
  representativeQuotes: BookAiRepresentativeQuote[];
  reflectionQuestions: string[];
  // existing fields
};
```

前端通过一个纯函数把现有字段映射成展示模型：

```ts
type BookInsightViewModel = {
  title: string;
  description: string;
  sourceQuotes: BookAiRepresentativeQuote[];
  followUpQuestions: string[];
};
```

建议位置：

- `src/lib/book-insights.ts`
- `src/lib/book-insights.test.ts`

职责：

- 只做字段组合。
- 不调用 AI。
- 不读写本地存储。
- 不修改原始 `BookAiSummary`。

### P1：可选新增结构化洞察字段

如果 P0 的组合质量不足，再在 `BookAiSummary` 中新增可选字段：

```ts
type BookInsightCard = {
  title: string;
  observation: string;
  tension?: string;
  sourceQuotes: BookAiRepresentativeQuote[];
  followUpQuestions: string[];
};

type BookAiSummary = {
  // existing fields
  insightCards?: BookInsightCard[];
};
```

约束：

- `sourceQuotes.quote` 必须来自输入划线或想法原文，可截短，不可改写。
- `sourceQuotes.noteType` 只能是 `划线` 或 `想法`。
- `followUpQuestions` 只生成问题，不生成任务。
- 字段缺失时前端回退到 P0 组合逻辑。

### 为什么不新增独立表

洞察是当前 AI 复盘资产的一部分，应继续存入 `ai_outputs.output_json`。

不新增表的原因：

- 当前不需要跨洞察检索和聚合统计。
- 复用已有缓存失效规则：笔记变化导致 `input_hash` 变化。
- 导出和版本详情已经围绕 `BookAiSummary` 工作。
- 降低迁移和兼容成本。

只有后续出现以下需求时，再评估独立表：

- 跨书洞察搜索。
- 单条洞察反馈状态。
- 洞察级引用跳转。
- 用户手动编辑洞察。

## Prompt 策略

### P0

不强制新增 prompt version。先通过前端展示重组，让已有 `myFocus`、`keyIdeas`、`representativeQuotes` 和 `reflectionQuestions` 显示为洞察。

如果发现现有输出不够稳定，再进入 P1。

### P1

新增字段时必须升级 prompt version：

```text
book-notes-summary-v4
```

系统 prompt 新增要求：

- 额外返回可选 `insightCards`。
- 每条洞察必须说明“用户为什么可能会关注它”，而不只是复述书中观点。
- 每条洞察必须附带 1-3 条来源摘录。
- 可以指出笔记中的张力或未完成问题，但不得做心理诊断。
- 不得编造未提供的书籍内容。
- 不得声称读完整本书。
- 输出必须为简体中文 JSON。

JSON schema 需要补：

- `insightCards`
- `insightCards[].title`
- `insightCards[].observation`
- `insightCards[].tension`
- `insightCards[].sourceQuotes`
- `insightCards[].followUpQuestions`

解析逻辑要求：

- 老缓存没有 `insightCards` 时不报错。
- AI 返回超长洞察时截断到展示上限。
- `sourceQuotes` 不可为空；为空的卡片应被丢弃或回退。

## AI 助手设计

### 快捷问题调整

在 `ReadingAssistantPanel.tsx` 的 `suggestionsForScope` 中调整：

#### `bookNotes`

建议：

- “我为什么会关注这些笔记？”
- “这些笔记背后有什么反复问题？”
- “把这些笔记整理成 3 个复盘问题。”

#### `aiAsset`

建议：

- “这份复盘最值得继续追问什么？”
- “把当前洞察整理成写作提纲。”
- “哪些行动项应该先做？”

#### `global`

建议：

- “我最近反复关注什么主题？”
- “哪些复盘问题还没有处理？”
- “基于最近阅读资产，下一步适合整理哪本书？”

#### `readingStats`

建议：

- “这个周期最明显的阅读变化是什么？”
- “哪些数据说明我需要调整节奏？”
- “这个周期适合复盘哪几本？”

### 上下文边界

助手使用现有上下文开关：

- `bookNotesSummary`
- `aiAssetSummary`
- `readingMemory`
- `readingStats`
- `conversationHistory`

默认不打开 `rawBookNotes`。

跨书问题只使用已生成资产摘要和统计，不发送多本原始笔记正文。用户显式开启原始笔记时，也应只作用于当前书上下文。

### 洞察追问入口

P0 可先不新增入口，只增加快捷问题。

P1 可在洞察卡片上增加轻量按钮：

```text
追问
```

点击后：

- 打开 `ReadingAssistantPanel`。
- `scope = "aiAsset"`。
- `entityId` 指向当前书或当前 AI 资产。
- 输入框预填问题，例如：

```text
围绕这条洞察继续追问：{洞察标题}
请结合当前复盘和来源摘录，给出 3 个后续问题。
```

实现上建议新增可选 prop：

```ts
type ReadingAssistantPanelProps = {
  // existing props
  initialDraft?: string;
};
```

约束：

- `initialDraft` 只填入输入框，不自动发送。
- 切换 `scope/entityId` 后清空过期草稿。
- 不把洞察卡片全文塞入前端状态；上下文由后端根据 `scope/entityId` 读取。

## 页面改造建议

### `BookAiSummaryPage`

新增组件：

```text
BookInsightSection
```

输入：

```ts
type BookInsightSectionProps = {
  summary: BookAiSummary;
  onAskInsight?: (draft: string) => void;
```

职责：

- 不写缓存。

- 只有 `detail.feature === "book-review"` 且存在 `bookSummary` 时显示。
- 不在资产列表卡片中展示完整洞察。

### `DashboardPage`

只增加轻量提醒，不新增长模块。

候选触发条件：

- 最近有 `book-review` 资产。
- 该资产有 `reflectionQuestions` 或 `actionItems`。
- 对应反馈为空或未完成。

展示文案示例：

```text
继续处理《X》的阅读洞察
把 1 个复盘问题转成输出或行动。
```

### `StatisticsPage`

不引入笔记洞察。

保留当前职责：

- 本地统计解读。
- AI 阅读体检报告。
- 周期报告导出。

统计页助手只解释数据，不声称读取笔记正文。

## 技术实施计划

### P0：无 schema 改造版本

目标：低风险验证“洞察”信息架构是否成立。

改造点：

1. 新增 `book-insights` 纯函数和测试。
2. 在 `BookAiSummaryPage` 增加“阅读洞察”区块。
3. 在 `ReadingHubPage` 版本详情复用该区块。
4. 调整 AI 助手快捷问题。
5. 调整单本笔记页文案，说明 AI 复盘包含洞察。

验收：

- 旧缓存可正常展示洞察区块。
- 没有代表性摘录时不显示空引用容器。
- 没有 AI Key 时不影响查看旧缓存。
- 助手快捷问题随 scope 正确变化。
- 不新增数据库迁移。

### P1：结构化洞察字段

触发条件：

- P0 组合结果不够稳定。
- 用户需要每条洞察绑定明确来源。
- 导出需要独立“阅读洞察”章节。

改造点：

1. `BookAiSummary` 新增 `insightCards?`。
2. Rust `BookAiSummary` struct 增加可选字段。
3. `book_notes_summary_json_schema()` 增加可选字段。
4. prompt version 升级到 `book-notes-summary-v4`。
5. normalize / sanitize 逻辑限制数量和长度。
6. Markdown 导出增加“阅读洞察”章节。
7. 单元测试覆盖旧缓存兼容和新字段解析。

验收：

- 老 `book-notes-summary-v3` 缓存继续可读。
- 新生成结果包含 1-4 条洞察卡片。
- 每条洞察至少有 1 条来源摘录。
- 来源摘录不能由 AI 改写。
- 导出 Markdown 不重复输出同一内容过多。

### P2：洞察追问入口

改造点：

1. `ReadingAssistantPanel` 支持 `initialDraft`。
2. 洞察卡片提供“追问”按钮。
3. `App.tsx` 维护助手草稿状态。
4. 助手打开后只预填，不自动发送。

验收：

- 点击追问能打开助手并预填问题。
- 关闭助手后草稿不污染其他 scope。
- 切换书籍后旧草稿清空。
- 不自动调用 AI。

### P3：总览提醒

改造点：

1. 从 AI 资产摘要和反馈状态中找出最近未处理复盘。
2. 在总览行动区增加一条轻量建议。
3. 点击跳转到单本 AI 复盘或复盘中心版本详情。

验收：

- 没有 AI 资产时不显示。
- 已整理或反馈已完成时降低优先级。
- 不额外触发 AI 调用。

## 隐私与安全边界

- 洞察生成必须由用户手动触发。
- AI 调用仍在 Rust trusted layer。
- 前端不直接读取或发送 AI Key。
- 不跨书发送原始笔记正文。
- `rawBookNotes` 默认关闭。
- 洞察输出必须说明依据来自当前书本地笔记。
- 助手跨书召回只使用已生成 AI 资产摘要和统计。
- Markdown 导出不包含 API Key、本地数据库路径和原始内部 ID。

## 兼容性要求

- 老缓存没有洞察字段时，使用 P0 组合逻辑。
- Web 预览只读环境不应触发真实生成。
- AI Provider 不支持严格 JSON schema 时，沿用现有 response format 兼容策略。
- Android / 移动端不新增复杂双栏布局。

## 风险与取舍

### 风险 1：概念和 AI 复盘重叠

处理方式：

- 不新增一级入口。
- 不新增独立“洞察生成”命令。
- 首版把洞察作为 AI 复盘内部区块。

### 风险 2：AI 输出变成空泛鸡汤

处理方式：

- 洞察必须绑定来源摘录。
- prompt 禁止心理诊断和泛化评价。
- UI 展示 `basisNotice`。

### 风险 3：助手变成泛聊天

处理方式：

- 助手只做追问和召回。
- 正式洞察仍由 AI 复盘页生成。
- 聊天结果默认不写入正式资产。

### 风险 4：结构扩展过早

处理方式：

- P0 不新增后端字段。
- P1 以可选字段扩展，兼容旧缓存。
- 不新增数据库表。

## 工程原则应用

- KISS：首版复用 `BookAiSummary`，不新增独立洞察系统。
- YAGNI：不做知识图谱、向量库、洞察表和全局洞察导航。
- DRY：复盘页和复盘中心复用同一洞察展示组件。
- SOLID：
  - 展示模型构建放到纯函数，保持页面组件职责单一。
  - AI 助手只负责对话，不负责正式资产生成。
  - Rust AI 服务继续负责 prompt、schema、缓存和可信调用边界。

## 建议优先级

推荐先做 P0。

理由：

- 不改数据库。
- 不破坏旧缓存。
- 不增加 AI Provider 兼容风险。
- 能快速验证用户是否真的需要“洞察”这个表达层。
- 后续 P1 可以在已有 UI 验证成立后再扩展结构化字段。

## 下一阶段灰度计划

当前实现已经完成 P0 验证，可以进入 P1 灰度验证。P1 的目标不是扩展更多能力，而是判断“阅读洞察 + AI 助手追问”是否在真实阅读复盘中降低用户继续思考的成本。

### 灰度验证问题

P1 重点回答：

1. 用户是否能理解“阅读洞察”是复盘内部的表达层，而不是新的资产类型。
2. 用户是否会点击洞察追问，并在助手草稿基础上继续编辑。
3. 洞察追问是否比用户手动复制复盘内容再提问更省步骤。
4. 洞察卡片是否有足够依据，避免变成没有来源的泛化结论。
5. AI 助手是否能直接回答洞察、反馈沉淀、下一步行动和来源摘录追问，而不是误导用户生成正式复盘。

### 灰度样本

建议选择 5-8 本书做人工检查：

- 笔记和摘录都较多的书，用于验证洞察来源展示。
- 只有想法、缺少代表摘录的书，用于验证空来源降级。
- 已经生成多版复盘的书，用于验证历史版本详情。
- 有行动反馈的书，用于验证反馈沉淀追问。
- 行动项较弱或较泛的书，用于观察用户是否仍需要 P1 质量校验。

### 人工检查清单

每本书检查：

1. 阅读洞察是否能从关注点、关键观点、代表摘录和复盘问题中组合出稳定主题。
2. 洞察卡片长标题、长摘录和多条追问在桌面与窄屏下不溢出。
3. 点击洞察“追问”后，AI 阅读助手打开并预填草稿，不自动发送。
4. 点击反馈沉淀“追问”后，助手草稿包含真实反馈摘要，不编造执行结果。
5. 点击下一步行动“拆解”后，助手草稿要求拆解步骤和最小完成版本。
6. 三类追问都不出现“生成 AI 复盘”动作卡。

### P1 非目标

- 不新增独立洞察数据库表。
- 不新增全局洞察列表或成果库。
- 不自动把助手回答写回复盘。
- 不把洞察追问结果作为新版本复盘依据，除非用户明确记录为反馈。
- 不引入向量库、知识图谱或跨书原文召回。

### 进入 P2 的条件

只有同时满足以下条件，才考虑进入 P2：

- 用户在真实样本中稳定使用洞察追问，而不是只阅读静态洞察。
- 洞察追问产生的回答能帮助用户形成更具体的行动或复盘问题。
- 旧缓存、空摘录、低笔记量场景没有明显误导。
- 全量 e2e 和相关单测保持通过。
- 仍然没有明确需求要求把洞察升级成独立成果库。
