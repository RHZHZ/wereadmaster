# 手机端书本详情页目录折叠/滚动改造方案

## 结论

需要改造。

手机端书本详情页的目录不应继续完整长展开。推荐方案是：桌面端保持当前展开目录和内部滚动体验；手机端默认折叠目录，只展示标题、章节数和当前进度，用户点开后再进入一个有最大高度限制的目录滚动区。

只加滚动条不是最佳方案。当前截图中的目录有 55 个章节，如果手机端仍默认展开，即使列表内部可以滚动，目录仍会在首屏形成过强视觉占比，用户进入详情页后很难快速看到本书状态、热门划线、公开点评和后续动作。折叠负责降低首屏占用，内部滚动负责展开后的可控浏览，两者应该一起做。

## 当前实现

相关文件：

- `src/pages/BookDetailPage.tsx`
- `src/components/ChapterList.tsx`
- `src/styles.css`
- `tests/e2e/app-smoke.spec.ts`

当前结构中，`BookDetailPage` 在书籍元信息和阅读进度之后渲染 `ChapterList`：

```tsx
<ChapterList
  chapters={detailResponse.chapters}
  progress={detailResponse.progress}
  isOpening={isOpening}
  onOpenChapter={onOpenChapter}
/>
```

`ChapterList` 目前无折叠状态。只要 `chapters.length > 0`，就直接渲染完整的 `ol.chapter-list`。

桌面基础样式已经给目录列表设置了滚动：

```css
.chapter-panel {
  max-height: min(72dvh, 760px);
  overflow: hidden;
}

.chapter-list {
  max-height: calc(min(72dvh, 760px) - 92px);
  overflow-y: auto;
}
```

但移动端断点又把目录高度限制和滚动取消了：

```css
.book-info-panel,
.chapter-panel {
  max-height: none;
  overflow: visible;
}

.chapter-panel .chapter-list {
  max-height: none;
  overflow: visible;
}

.book-grid,
.local-book-grid,
.chapter-list,
... {
  max-height: none;
  overflow-y: visible;
}
```

因此在手机端，长目录会完整展开到页面流里。55 个章节这种场景会占用多个屏高，用户需要长距离滚动才能离开目录。

## 当前问题

### P1：目录在手机端抢占详情页主体

书本详情页是工具型页面，用户打开后首先需要快速确认：

- 当前书是谁。
- 当前进度到哪里。
- 有没有可复盘材料。
- 能不能继续打开、看笔记、生成复盘或查看公共内容。

完整长目录默认展开后，这些信息被目录列表分隔开，页面从“书本详情”变成“目录长列表”。这对短屏手机尤其明显。

### P2：已有滚动能力在移动端被覆盖

桌面端已经有目录内部滚动，但移动端为了避免多个列表嵌套滚动，把 `.chapter-list` 统一改回 `overflow-y: visible`。这个全局规则对普通卡片列表合理，但对目录这种可折叠、可定位的长列表不够精确。

需要在移动端单独恢复 `.chapter-panel .chapter-list` 的受控滚动，而不是撤销整段移动端列表策略。

### P3：当前章节高亮不等于当前章节可达

目录会高亮 `progress.chapterUid` 对应章节。但长目录展开后，如果当前进度在中后段，用户仍需要从列表顶部手动滑很久才能看到当前章节。

展开目录后可以考虑自动把当前章节滚到列表中部。这个能力不改变数据模型，也不会影响打开章节逻辑，但能明显提升长目录场景的实用性。

### P4：只加滚动条仍然不是首屏最优

如果只恢复移动端 `.chapter-list` 的 `max-height` 和 `overflow-y: auto`，目录面板仍会默认占据半屏以上。用户仍必须先处理目录，再继续看详情页其他模块。

折叠和滚动的职责不同：

- 折叠：降低目录在详情页首屏的占用。
- 滚动：展开后控制目录浏览范围。

首版建议同时做，范围仍然很小。

## 改造目标

1. 手机端默认收起目录

- 目录面板只展示标题、章节数、当前进度和展开按钮。
- 章节为空时保持现有空态，不显示无意义的折叠按钮。
- 桌面端保持当前默认展开体验。

2. 展开后目录内部滚动

- 展开后列表最大高度受视口约束。
- 不把 55 个章节完整摊进页面流。
- 保留章节层级缩进、当前章节高亮、付费/免费图标和打开入口。

3. 触控操作稳定

- 展开/收起按钮使用真实 `button`。
- 触控热区不低于 44px。
- 使用 `aria-expanded` 和 `aria-controls` 表达状态。

4. 不扩大改造范围

- 不修改微信读书详情 API。
- 不改变 `Chapter` / `ReadingProgress` 数据结构。
- 不重排整张书籍详情页。
- 不改热门划线、公开点评、本地版本提示等模块。

## 推荐方案

### 方案 A：移动端默认折叠，展开后滚动

推荐采用。

行为：

- 桌面端：目录默认展开，沿用当前内部滚动。
- 手机端：目录默认折叠。
- 用户点击“展开目录”后，目录列表显示在面板内部。
- 展开状态下，列表高度约束为 `min(52dvh, 420px)` 一类的安全范围。
- 用户点击“收起目录”后，面板恢复为摘要状态。

优点：

- 首屏干净。
- 长目录不会吞掉详情页。
- 用户仍可完整浏览目录。
- 改动集中在 `ChapterList` 和 CSS，符合 KISS。

### 方案 B：只给手机端目录加滚动条

不推荐作为最终方案。

优点是实现更少，但缺点明显：

- 目录仍默认占据较大首屏空间。
- “55 个章节”这种书看起来依旧像一个长列表模块。
- 用户想看后续内容时仍需要先越过目录面板。

如果担心新增折叠状态有风险，可以把它作为过渡方案，但不应作为最终交互。

### 方案 C：拆成底部弹层目录

暂不建议。

底部弹层适合阅读器目录，但书本详情页不是沉浸阅读场景。为了一个列表新增弹层、遮罩、焦点管理和关闭手势，范围会明显扩大，也会和当前页面已有卡片结构不一致。

## 实施步骤

### 第一步：给 `ChapterList` 增加折叠状态

文件：

- `src/components/ChapterList.tsx`

建议改造：

- 引入 `ChevronDown` / `ChevronUp` 或同类 lucide 图标。
- 增加本地状态，例如 `isExpanded`。
- 初始状态按视口决定：
  - `window.matchMedia("(max-width: 720px)")` 命中时默认 `false`。
  - 非手机端默认 `true`。
  - `typeof window === "undefined"` 时默认 `true`，避免现有静态渲染测试被无关折叠状态影响。
- 面板 header 中增加展开/收起按钮。
- 按钮写入：
  - `aria-expanded`
  - `aria-controls`
  - `aria-label="展开目录"` / `aria-label="收起目录"`

建议文案：

- 折叠时：`展开目录`
- 展开时：`收起目录`

不建议把整个 `.chapter-panel` 都做成点击区。手机端误触成本高，折叠控制应是明确按钮。

### 第二步：移动端恢复目录内部滚动

文件：

- `src/styles.css`

建议样式方向：

```css
@media (max-width: 720px) {
  .chapter-panel {
    overflow: hidden;
  }

  .chapter-panel.is-collapsed .chapter-list {
    display: none;
  }

  .chapter-panel.is-expanded .chapter-list {
    max-height: min(52dvh, 420px);
    overflow-y: auto;
    padding-right: 8px;
    scrollbar-gutter: stable;
    -webkit-overflow-scrolling: touch;
  }
}
```

注意点：

- 现有移动端有一段宽选择器会把 `.chapter-list` 统一改成 `overflow-y: visible`，新规则必须放在其后，或使用更具体的 `.chapter-panel.is-expanded .chapter-list` 覆盖。
- 不要取消其他列表的移动端策略，只处理书籍详情目录。
- 不建议第一版加 `overscroll-behavior: contain`，避免产生滚动困住的副作用。只有验证发现滚动穿透，再单独处理。

### 第三步：展开后定位当前章节

文件：

- `src/components/ChapterList.tsx`

建议作为 P2 增强，但可以和第一版一起做。

行为：

- 用户在手机端展开目录时，如果存在当前章节，把对应 `li.is-current` 滚到目录容器中部。
- 桌面端不主动滚动，避免干扰现有目录浏览。
- 没有 `progress.chapterUid` 或章节未匹配时，不做额外行为。

实现边界：

- 使用 `ref` 定位列表容器和当前章节。
- 只在用户展开时触发，不在每次渲染时触发。
- 不持久化目录展开状态到 localStorage，首版只保留页面生命周期内状态。

### 第四步：补充移动端回归测试

文件：

- `tests/e2e/app-smoke.spec.ts`

建议新增用例：

`手机端书籍详情长目录默认折叠且展开后内部滚动`

覆盖流程：

1. 设置 `390 x 844` 手机视口。
2. 注入包含 50+ 章节的书籍详情 mock。
3. 从书架进入书籍详情页。
4. 断言目录 header 可见，显示章节数和当前进度。
5. 断言折叠状态下 `.chapter-list` 不占据页面高度。
6. 点击 `展开目录`。
7. 断言 `.chapter-list` 可见，`overflow-y` 为 `auto` 或 `scroll`。
8. 断言目录列表高度小于视口高度，例如小于 `0.6 * window.innerHeight`。
9. 点击某个章节，确认仍调用打开章节逻辑或按钮保持可点击。
10. 点击目录底部区域，确认最后一条章节不会被右下角悬浮助手按钮遮住。
11. 点击 `收起目录`，确认列表收起且后续内容可更容易到达。

建议补充断言：

- 页面没有横向溢出。
- 展开/收起按钮触控高度不小于 44px。
- 当前章节高亮仍存在。
- 底部导航不覆盖展开/收起按钮。
- 右下角悬浮助手按钮不覆盖目录最后一条章节。

## 验收标准

### 手机端

- 390px 宽视口下，目录默认折叠。
- 折叠状态下只显示目录摘要，不显示完整章节卡片列表。
- 展开按钮可见、可点，触控区域不小于 44px。
- 展开后章节列表在目录面板内滚动，不把整页撑长多个屏高。
- 55 个章节场景下，列表高度受控，不与底部导航重合。
- 当前章节仍高亮；有当前章节时，展开后可快速定位。
- 点击章节仍能打开对应微信读书章节。
- 后续模块可以通过正常页面滚动访问，不被目录长期阻断。

### 桌面端

- 目录默认展开。
- 原有内部滚动体验保留。
- 当前章节高亮、章节缩进、锁图标和打开入口不回退。
- 页面整体布局不因为移动端折叠能力发生变化。

### 空态

- 章节为空时继续展示“暂未获取到章节”。
- 不显示“展开目录”按钮。
- 空态文案和暗色主题样式不回退。

## 风险与控制

### 风险 1：新增折叠状态影响静态渲染测试

控制：

- `typeof window === "undefined"` 时默认展开。
- 保持空态和桌面端默认结构接近现状。
- 必要时补 `ChapterList` 单元测试，但不为了这个小改造引入新的测试库。

### 风险 2：移动端嵌套滚动造成手势不顺

控制：

- 只有展开目录时启用内部滚动。
- 列表最大高度不超过视口一半左右。
- 不先使用强制滚动捕获规则。
- 用真实手机视口 E2E 检查展开、滚动、收起和继续滚动页面。

### 风险 3：宽选择器覆盖新规则

控制：

- 新规则放在现有移动端 `.chapter-list` 重置规则之后。
- 使用 `.chapter-panel.is-expanded .chapter-list` 提升针对性。
- 不修改 `.book-grid`、`.local-book-grid`、`.recommendation-stack` 等无关列表策略。

### 风险 4：当前章节自动定位打断用户浏览

控制：

- 只在“从折叠到展开”的用户操作后触发一次。
- 不在用户手动滚动后反复定位。
- 未匹配当前章节时不做任何自动滚动。

## 不做内容

- 不把目录移到全屏页面或底部弹层。
- 不改书籍详情的数据加载顺序。
- 不新增章节搜索。
- 不新增章节收藏、批量打开或目录管理。
- 不修改微信读书打开章节接口。
- 不调整热门划线、公开点评、本地版本提示和 AI 入口。
- 不新增目录搜索或悬浮目录面板。

## 推荐落地范围

第一版只改：

- `src/components/ChapterList.tsx`
- `src/styles.css`
- `tests/e2e/app-smoke.spec.ts`

可选补充：

- 如果现有 E2E mock 不方便构造 50+ 章节详情，可先在 `BookDetailPage.test.tsx` 里补静态渲染断言，再追加一条 smoke 用例覆盖真实移动端布局。

## 优先级

- P0：手机端默认折叠目录。
- P0：展开后目录内部滚动，并覆盖现有移动端宽选择器。
- P1：展开后自动定位当前章节。
- P1：补移动端 E2E 回归。
- P2：后续再评估是否需要目录内搜索。

## 文档复核结论

该方案不过度。

它只处理长目录在手机端占屏和浏览不受控的问题，不借机重构书籍详情页，也不改变任何数据能力。折叠解决首屏信息密度，内部滚动解决 50+ 章节浏览，当前章节定位作为轻量增强，整体符合 KISS 和 YAGNI。
